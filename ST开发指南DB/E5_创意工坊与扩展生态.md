## 1. 运行层与总体架构

### 1.1 四种容易混叫“插件”的东西

| 运行层 | 载体 | 负责 | 本册 |
|---|---|---|---:|
| ST UI Extension | 浏览器宿主扩展 | manifest、设置、事件、prompt、宿主 UI | 否，D5 |
| ST Server Plugin | Node.js 服务端 | 路由、文件系统、服务端能力 | 否，D5 |
| TH 卡内脚本 | `about:srcdoc` iframe / ScriptTree | 随卡脚本、Wand、世界书桥、卡内运行时 | 是 |
| 创意工坊 | 卡内客户端 + Gateway + 存储 | 内容浏览、预览、安装、发布、审核、撤回 | 是 |

四层 API 不可互换：

- `SillyTavern.getContext()` 是 UI Extension 宿主入口；
- `eventOn/getButtonEvent/getScriptTrees/updateWorldbookWith` 是当前 TH iframe 全局函数；
- Gateway HTTP API 只提供包和发布产品链，不可获得本地世界书写权限；
- 卡内 bridge 把用户已确认的有限动作从工坊 iframe交给 TH 脚本执行。

### 1.2 工坊三方云端 + 卡内客户端

```text
GitHub Pages
  schemas / examples / read-only index / fixed runtime
             ↓
卡内 TH 启动器 → 宿主模态框 → 工坊 iframe
             ↕             ↕
Worldbook Bridge       Workshop Gateway
             ↓             ↓
聊天世界书          发布/审核/owner/revision
                           ↓
                     Cloudreve 包/媒体
```

- GitHub Pages：只读机器契约和静态基线；
- Gateway：动态包索引、审核状态、publisher ownership、revision 和撤回；
- Cloudreve：公开 JSON 包与可选媒体；
- TH 卡内脚本：唯一拥有本地世界书写入能力的一侧；
- 工坊包：内容数据，不是任意脚本、prompt patch 或 MVU patch。

### 1.3 核心铁律

工坊导入只写受控聊天世界书条目，例如 `[世界因子]当前设定`、`[创意工坊][类型]标题`；**绝不直接修改 MVU 存档变量**。Zod/MVU 核心、开局主流程、prompt patch 和宿主主题留在卡内受信版本，不从通用远程包热替换。

## 2. Tavern Helper 当前脚本树契约

### 2.1 ScriptTree 类型

本地 2026-05-08 类型快照与生产卡使用数组树结构：

```typescript
type ScriptButton = {
  name: string;
  visible: boolean;
};

type Script = {
  type: 'script';
  enabled: boolean;
  name: string;
  id: string;
  content: string;
  info: string;
  button: {
    enabled: boolean;
    buttons: ScriptButton[];
  };
  data: Record<string, unknown>;
  export_with: {
    data: boolean;
    button: boolean;
  };
};

type ScriptFolder = {
  type: 'folder';
  enabled: boolean;
  name: string;
  id: string;
  icon: string;
  color: string;
  scripts: Script[];
};

type ScriptTree = Script | ScriptFolder;
```

规则：

- `id` 是机器稳定标识；`name` 是可变显示名；
- 一个脚本可有多个 `button.buttons[]`；
- 数组顺序就是显示/执行树顺序；
- `export_with.data` 与 `export_with.button` 分别控制导出；
- 文件夹内仍要递归处理脚本，不能只扫顶层；
- 写回保留未知字段，避免旧/新版本扩展字段被删。

旧式 scripts 键值对象、`displayName`、`button:true`、`export_with:true` 只作为历史迁移输入，不再是当前范式。

### 2.2 管理 API 必须带作用域

```javascript
const scope = { type: 'character' };

const trees = getScriptTrees(scope);

replaceScriptTrees(nextTrees, scope);

const updated = await updateScriptTreesWith(async (currentTrees) => {
  return migrateOrPatchTrees(currentTrees);
}, scope);
```

当前 `type` 可为 `global | preset | character`。示例操作角色卡脚本，必须显式 `{type:'character'}`；无参调用或把 scope 写成 `chat` 都不符合当前类型契约。

### 2.3 历史格式迁移

```text
read raw scripts
→ detect object vs ScriptTree[]
→ backup original
→ assign stable ids
→ displayName → name
→ button:true → button.enabled + buttons[]
→ export_with:true → {data:true,button:true}
→ preserve order/content/data/unknown fields
→ validate tree
→ replace once with {type:'character'}
→ reread and compare
```

重名显示名不能当 ID。迁移产生 ID 时需要稳定映射和冲突报告，不能每次导入重新随机生成导致升级找不到原脚本。

## 3. 卡内脚本初始化、API 与卸载

### 3.1 初始化与单例

卡内 iframe 可能在宿主页面 ready 后才创建，不只依赖一次性 `DOMContentLoaded`。按当前 TH 指南使用 `$(() => init())` 或目标版本推荐的就绪方式。

```javascript
const RUNTIME_KEY = '__MY_WORKSHOP_RUNTIME__';
window[RUNTIME_KEY]?.destroy?.();
window[RUNTIME_KEY] = createWorkshopRuntime();

window.addEventListener('pagehide', () => {
  window[RUNTIME_KEY]?.destroy?.();
}, { once: true });
```

新实例必须先销毁旧实例，防热重载后重复按钮、重复 message listener 和重复世界书写入。

### 3.2 宿主窗口定位

```javascript
function hostWindow() {
  try {
    if (window.parent && window.parent !== window && window.parent.document) {
      return window.parent;
    }
  } catch (_error) {}
  return window;
}

function hostDocument() {
  return hostWindow().document || document;
}
```

多层嵌套可按能力评分，但必须明确来源：有 ST DOM、目标 TH API、URL 不是 `about:srcdoc`。不要在循环中吞掉所有异常后静默选错窗口。

### 3.3 事件与提示词

```javascript
const subscription = eventOn(tavern_events.CHAT_CHANGED, () => {
  resetChatContext();
});
disposers.push(() => subscription.stop());

injectPrompts([{
  id: 'my-workshop-note',
  position: 'in_chat',
  depth: 0,
  role: 'system',
  content: '...',
  should_scan: false,
}], { once: true });
```

当前类型中 `eventOn` 返回 `{stop()}`；`injectPrompts` 第二参数精确为 `{once?:boolean}`。TH `injectPrompts` 与 ST UI Extension 的 `setExtensionPrompt` 参数结构不同，不能互换。

### 3.4 Worldbook 是当前推荐 API

当前类型方向：Worldbook API 推荐；`getLorebookEntries/...` 是 deprecated 兼容层。

| API | 当前签名要点 | 返回 |
|---|---|---|
| `getCharWorldbookNames('current')` | 参数必填，返回 primary/additional | `CharWorldbooks` |
| `getChatWorldbookName('current')` | 参数必填 | `string|null` |
| `getOrCreateChatWorldbook('current', name?)` | 获取/创建聊天世界书 | `Promise<string>` |
| `getWorldbook(name)` | 条目是数组，不是键值对象 | `Promise<WorldbookEntry[]>` |
| `updateWorldbookWith(name, updater, options?)` | updater 收/返数组 | 更新后的数组 |
| `createWorldbookEntries(name, entries, options?)` | 批量创建 | `{worldbook,new_entries}` |
| `deleteWorldbookEntries(name, predicate, options?)` | 第二参数是 predicate，不是 UID 数组 | `{worldbook,deleted_entries}` |

`updateWorldbookWith/replaceWorldbook/createWorldbookEntries/deleteWorldbookEntries` 的 `render` 为 `'debounced' | 'immediate'`；默认用性能更好的 debounced，用户正在查看且需要同步反馈时再 immediate。

按 UID 更新：

```javascript
const updated = await updateWorldbookWith(targetWorldbook, (entries) => {
  const index = entries.findIndex((entry) => entry.uid === targetUid);
  if (index < 0) throw new Error(`entry not found: ${targetUid}`);
  const next = entries.map((entry) => ({ ...entry }));
  next[index] = { ...next[index], content: newContent };
  return next;
}, { render: 'debounced' });
```

删除：

```javascript
const result = await deleteWorldbookEntries(
  targetWorldbook,
  (entry) => ownedUids.has(entry.uid),
  { render: 'immediate' },
);
```

UID 只在单个世界书内部有效；跨世界书不能复用。按 `entries[targetKey]` 访问会把 UID/字符串误当数组下标，禁止作为更新范式。

### 3.5 跨脚本全局与完整清理

`initializeGlobal(name, api)` 会写入 window 并触发 `global_${name}_initialized`；`waitGlobalInitialized` 可等待。销毁时要撤销：

- `eventOn` subscription；
- 父/当前 window 的 `message`、click、keydown；
- timer、rAF、observer；
- popup、iframe、宿主 DOM/style；
- AbortController；
- prompt 注入；
- 全局 API 引用；
- 未完成的 bridge request。

TH 只会清理自己管理的 iframe 生命周期资源；挂到父窗口的监听仍由脚本负责。

## 4. 魔棒与功能入口

### 4.1 TH 原生按钮（优先）

元数据：

```json
{
  "button": {
    "enabled": true,
    "buttons": [
      { "name": "打开创意工坊", "visible": true }
    ]
  }
}
```

监听：

```javascript
const buttonName = '打开创意工坊';
const eventName = getButtonEvent(buttonName);
const subscription = eventOn(eventName, () => openWorkshop());
disposers.push(() => subscription.stop());
```

`getButtonEvent()` 匹配 `button.buttons[].name`，不是脚本 `name`，更不是历史 `displayName`。一个脚本可以注册多个按钮。

### 4.2 直接挂载 ST 魔棒菜单（版本敏感）

星月生产组件还会寻找 `#extensionsMenu`、`.extensionsMenu`，最后回退 `document.body`。这条路线依赖 ST 私有 DOM，不是稳定公共 API；必须：

- 用稳定容器/按钮 ID 防重复；
- 点击只路由到统一 command registry；
- CHAT_CHANGED/宿主重绘后重挂；
- destroy 删除节点和 listener；
- 目标 ST 版本复核选择器；
- TH 原生按钮或页面按钮作为可恢复入口。

不要在 E5 复制控制中心业务代码。控制中心案例见 C2，悬浮球/轮盘/命令注册表见 C11。

## 5. 创意工坊 Bridge 协议

### 5.1 能力动作

当前生产架构可抽象为：

```text
handshake
getContext
getEntries
createWorldbook
createEntries
updateEntry
deleteEntries
upsertWorkshopEntries
```

Bridge 只暴露有限动作，不把宿主 window 或任意函数交给 iframe。

### 5.2 请求/响应包络

```javascript
// request
{
  protocol: 1,
  requestId: crypto.randomUUID(),
  action: 'getEntries',
  payload: { worldbookName: '...' }
}

// response
{
  protocol: 1,
  requestId: 'same-id',
  ok: true,
  result: {},
  error: null
}
```

强制要求：

- 固定双向 origin allowlist；
- `event.source === expectedIframe.contentWindow`；
- 禁止 `postMessage('*')`；
- requestId 配对、超时、一次性 resolve；
- action/payload/result schema 校验；
- handshake 交换协议版本和能力；
- 未识别 action 明确拒绝；
- iframe 重载/销毁时 reject 旧 pending 请求；
- OAuth 登录 URL 固定 origin/path，禁止任意跳转。

### 5.3 最小 Context

工坊 iframe 通常只需：

- cardScope/作品 ID；
- 非敏感当前聊天标识；
- 目标世界书名与可用能力；
- 已安装来源摘要；
- 客户端协议/运行时版本。

不默认发送完整聊天、所有世界书、宿主对象、Cookie、API key 或 Gateway token。

## 6. 工坊包契约与内容边界

### 6.1 机器契约优先

`rolecard-diy-workshop/schemas/workshop-package.schema.json` 与 `shared/workshop-package-contract.js` 是包结构真相源。文档示例只帮助理解，不覆盖 schema。

```json
{
  "packageVersion": "1.0.0",
  "id": "publisher.package-id",
  "type": "world_factor",
  "cardScope": "xingyue",
  "title": "示例世界因子",
  "revision": 3,
  "contentHash": "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
  "reviewStatus": "approved",
  "payload": {
    "worldFactors": [
      {
        "title": "示例世界因子",
        "content": "示例内容"
      }
    ]
  }
}
```

`contentHash` 是 Gateway 管理的整包字段，格式为 **64 位小写十六进制**，不带 `sha256:` 前缀。客户端不应让发布者手填并信任它。

### 6.2 内容白名单

当前允许：

- `character`
- `user_identity`
- `world_factor`
- `shop_item`
- `blueprint`
- `recipe`
- `skill`
- `function`

拒绝：

- `opening_pack`
- `prompt_patch`
- `ui_theme`

这些拒绝类型会绕过卡内受信开局、提示词或全局主题主链，影响范围远大于普通世界书内容。禁止换类型名后塞入 payload。

### 6.3 开局内容的现有合法路线

`opening_pack` 仍是拒绝类型；当前机器契约已经为开局内容定义了更窄的合法路线：

```json
{
  "packageVersion": "1.0.0",
  "id": "publisher.opening-example",
  "type": "world_factor",
  "cardScope": "xingyue-opening-v1",
  "title": "开学日示例",
  "revision": 1,
  "contentHash": "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
  "reviewStatus": "approved",
  "payload": {
    "target": "xingyue.opening_day_body",
    "schemaVersion": 1,
    "compatibility": {
      "minRuntimeVersion": "3.4.9"
    },
    "gradeScope": ["all"],
    "worldFactors": [
      {
        "title": "开学日示例",
        "content": "受约束的开局正文"
      }
    ]
  }
}
```

硬边界：

- `type` 必须是 `world_factor`，`cardScope` 必须是 `xingyue-opening-v1`；
- `target` 固定为 `xingyue.opening_day_body`，`schemaVersion` 固定为 `1`；
- `worldFactors` 恰好一项，条目标题必须与包标题一致；
- `gradeScope` 按 schema 枚举校验；含 `all` 时不能再混入其他年级；
- `compatibility.minRuntimeVersion` 与正文宏都按当前 contract 校验。

客户端必须把四个动作拆开：

1. **详情预览**：只读校验 schema、兼容性、信任级别和 diff；零写入。
2. **本地导入草稿**：当前星月 3.4.9 会立即写入开局草稿的 `packages/worldFactors`，并默认关闭该包；它属于未审核开发输入，不等于安装了可信来源。
3. **安装为来源**：开局包必须来自带有效正整数 `revision` 的可信工坊详情；写成默认禁用的单个世界书来源，保留来源元数据，但不改当前 `openingDay` 正文。
4. **应用到当前草稿**：先确认来源已安装、未被本地修改，且详情 revision 与已安装 revision、实际 hash、记录 hash、预期 hash 全部一致；再校验聊天上下文、年级范围和正文宏并展示 diff。用户再次确认后才写当前开局草稿，仍不发送消息、不修改 MVU。真正开始游玩继续走 C10 的确认发送链。

上面的 JSON 是 **Gateway approved 详情形状**。若去掉 Gateway 管理字段，它仍可作为本地导入草稿通过 schema，但不能安装为开局来源。

因此，“支持开局内容”不等于允许工坊包直接插入聊天或替换首楼。

### 6.4 安装前校验

- packageVersion；
- id/type/cardScope；
- runtime/协议范围；
- payload schema 与大小；
- 来源身份与信任级别；
- Gateway approved 包的 revision、contentHash 与 reviewStatus；
- revision 与重复安装；
- 目标世界书能力和用户授权。

来源必须分流处理：

- **Gateway approved 包**：必须有正整数 `revision`、Gateway 生成的 64 位 `contentHash`，且 `reviewStatus === 'approved'`；三者参与远端更新身份与冲突检查。
- **本地 JSON**：schema 允许省略上述 Gateway 管理字段；即使文件内自带这些字段，也不能据此信任。客户端须标记为 detached / untrusted / development，不能宣称 owner 身份、approved 状态或远端自动更新关系。

## 7. 预览与带条件补偿的安装事务

### 7.1 预览零写入

```text
选择包
→ 拉取详情/payload
→ schema/cardScope/version/hash 校验
→ 读取当前世界书和安装清单
→ 计算 create/update/conflict/unchanged/remove
→ 展示 diff、来源和权限
→ 用户确认
```

预览不创建世界书、不改 Persona、不写 MVU、不插消息。世界书不存在时只显示“确认后将创建”。

这里的“预览”专指只读详情/diff 路径；调用星月 3.4.9 的 `importPackageToDraft()` 已经会写开局草稿，不能把该函数命名中的 import 当成零写入预览。

### 7.2 Diff 分类

| 分类 | 条件 | 默认动作 |
|---|---|---|
| create | 本地无相同 packageIdentity | 新建一条包条目 |
| unchanged | revision 与条目基线一致 | 跳过 |
| update-safe | 当前本地仍等于 installedEntryHash | 更新 |
| modified-local | 玩家已改条目 | 冲突，人工选择 |
| collision | 名称相同、来源不同 | 不覆盖，改名/选目标 |
| remove | 用户明确卸载该 packageIdentity | 先保留用户副本，再移除受管条目 |

名称不是稳定所有权；包级 hash 也不能判断某一个条目是否被玩家修改。

星月 3.4.9 的生产模型是 **一包一条世界书条目**：`worldFactors[]` 等 payload 内容先由 `packageToWorldbookText(pkg)` 合成为单条正文。稳定身份 `packageIdentity` 是三元组：

```text
packageId + packageType + packageTarget
```

其中 `packageTarget = payload.target || packageTarget || type`。当前 v1 schema 的内容项没有 `slot` 或稳定 item id；不得读取 `item.slot`，也不得按 `worldFactors[]` 下标建立多条受管世界书条目。数组插入、删除或重排只会改变这一包合成后的预期正文 hash，再按整条 entry 做安全升级判断。

### 7.3 两类 hash、稳定身份与条目元数据

必须分开：

- `packageContentHash`：整个远端包的 Gateway hash，用于确认拿到的远端详情；
- `installedEntryHash`：当前生产代码中的 `entry.meta.contentHash`，是安装后世界书正文的 hash，不等于 Gateway 整包 hash；
- `expectedEntryHash`：把新包通过同一 `packageToWorldbookText()` / canonical 规则转换后得到的预期正文 hash；
- `packageId + packageType + packageTarget + revision`：受管来源身份与版本；
- `uid`：当前世界书中的实际条目 UID。

星月 3.4.9 直接把安装清单写进单条世界书 entry 的 `meta/extra`：

```javascript
entry.meta = {
  source: 'xingyue-workshop',
  kind: 'workshop_package',
  packageId: pkg.id,
  packageType: pkg.type,
  packageTarget: pkg.payload?.target || pkg.packageTarget || pkg.type,
  programOnly: pkg.payload?.target === 'xingyue.opening_day_body',
  revision: pkg.revision,
  contentHash: hashWorldbookContent(entry.content),
  installedAt: new Date().toISOString(),
};
```

`uid` 保留在 entry 本体。条目 hash 必须使用稳定正文/canonical 规则，并排除 `installedAt` 等每次变化的诊断字段。Gateway `packageContentHash` 可另存于来源/草稿记录，但不能冒充 `entry.meta.contentHash`。

### 7.4 世界书 API 只有更新回调，不提供跨系统原子事务

本册所说“事务”是 **带条件补偿的应用层事务**：

```javascript
const previewSnapshot = await getWorldbook(target);
const previewPlan = buildPreviewPlan(previewSnapshot, pkg, installManifest);

const written = await updateWorldbookWith(target, (latestEntries) => {
  // 并发窗口收口：按 latestEntries 重新核对 packageIdentity/UID/installedEntryHash。
  const checkedPlan = revalidatePlan(previewPlan, latestEntries);
  if (checkedPlan.conflicts.length) throw new Error('worldbook changed since preview');
  return applyPlan(latestEntries, checkedPlan);
}, { render: 'debounced' });

assertWrittenState(written, previewPlan.expectedWrites);
```

原则：

1. 预览快照只用于展示，真正 updater 内必须对当前数组重验；
2. updater 返回完整数组；
3. 写后重读/返回值校验；
4. 同一 packageIdentity 的旧 revision 拒绝；同 revision 但预期 hash 不同也拒绝；
5. 双击/网络重试不能创建第二套条目；
6. 回滚只在目标仍等于“本事务刚写入的 entry hash”时执行；
7. 并发修改或不可逆步骤出现时停止，展示部分提交 diff，不用旧整本世界书覆盖。

跨 Persona、世界书、Gateway 和聊天的操作没有原生原子性。不可逆动作最后执行，每一步记录结果和补偿条件。

## 8. 来源所有权、升级、卸载与迁移

### 8.1 条目来源元数据

使用 7.3 的当前生产形状：`source/kind/packageId/packageType/packageTarget/programOnly/revision/contentHash/installedAt`。匹配受管条目时同时要求：

```text
meta.source === 'xingyue-workshop'
meta.kind === 'workshop_package'
packageId、packageType、packageTarget 三者全部相等
```

显示名、comment 和 payload 内数组位置都不能证明所有权。先生成合并后的世界书正文并求 `installedEntryHash`，再加入时间戳等诊断字段，避免自引用和每次 hash 改变。

### 8.2 安全升级是三方比较

```text
条目 meta 中记录的 installedEntryHash
vs 当前本地 canonical entry hash
vs 新包转换后的 expectedEntryHash
```

- 当前仍等于安装基线：可自动更新；
- 当前正文或行为已改：显示“覆盖 / 保留当前 / 脱离为用户副本”；
- `incomingRevision < installedRevision`：拒绝回滚；
- revision 相同但新包预期 hash 与已记录来源不一致：拒绝并要求刷新详情；
- packageContentHash 只证明包，不证明某条本地内容未改。

### 8.3 安全卸载

删除必须同时满足：

1. 明确 `packageId + packageType + packageTarget`；
2. entry meta 能对应 UID/基线；
3. 当前条目仍等于 installedEntryHash，或用户明确确认删除修改版；
4. updater 内再次核对；
5. 删除后重读验证；
6. 失败补偿不覆盖并发修改。

星月 3.4.9 卸载前会先生成 detached 用户副本，再移除受管条目；若开局草稿仍记录该来源，则仅移除来源绑定，保留当前开局正文。云端包 withdrawn 不会远程删除玩家已安装内容。

### 8.4 历史 `🧩mod ` 前缀

前缀只能作为旧条目候选，不能证明所有权。迁移时通过已知包、内容与 `packageId/type/target` 高置信匹配并让用户确认，再补 source metadata；不得批量删除所有同前缀条目。

## 9. 发布、审核、owner 与撤回链

### 9.1 产品状态

```text
本地草稿
→ 登录 + Discord 服务器成员校验
→ POST 发布（pending）
→ 管理员审核
   ├─ approved → 进入公开索引
   └─ rejected → owner 可见原因并修订
→ owner PUT 更新（revision 保护）
→ owner DELETE 撤回（withdrawn）
```

未登录仍可浏览 approved 公开索引、导入/导出本地 JSON；发布、更新、撤回才要求登录。

### 9.2 Gateway 契约摘要

| 路径 | 用途 |
|---|---|
| `GET /api/workshop/health` | 健康检查 |
| `GET /api/workshop/packages` | approved 公开索引与过滤 |
| `GET /api/workshop/packages/:id` | 包详情，保留 payload |
| `GET /api/workshop/me` | publisher 登录/成员状态 |
| `GET /api/workshop/me/packages` | owner 的全部发布和审核状态 |
| `POST /api/workshop/packages` | 新发布，默认 pending |
| `PUT /api/workshop/packages/:id` | owner 更新，带 revision 防并发 |
| `DELETE /api/workshop/packages/:id` | owner 撤回，移出公开索引 |
| `/api/admin/review/...` | 管理员审核队列/决定 |

旧 revision 更新应返回 `409 package-conflict`。客户端刷新远端详情并重新比较，不能最后写入者静默覆盖。

### 9.3 撤回语义

- 从公开索引移除；
- 保留最小 ownership/审核记录；
- owner UI 显示 withdrawn；
- 不删除玩家本地世界书；
- 恢复/再发布按 Gateway 契约，不由客户端猜。

## 10. 安全、鉴权与隐私

### 10.1 Bridge 与 CORS

- 生产 Gateway 配置固定 CORS allowlist；默认 `*` 只能视为未收口风险，不是生产安全配置；
- iframe 双向验证 origin + source；
- 禁止 `postMessage('*')`；
- payload、包大小、schema、type、cardScope、hash 全校验；
- OAuth 登录 URL 固定 origin/path；
- 工坊 iframe 不获得任意 Worldbook updater 或宿主 window。

### 10.2 Bearer token

当前 Gateway 使用无状态 HMAC Bearer token，卡内客户端保存在 localStorage 并随鉴权请求发送。必须明确：

- localStorage 不是安全密钥库，同源 UI Extension/TH 脚本可读；
- token 不进入角色卡导出、聊天、世界书、脚本 `data`、错误报告或日志；
- 前端不把 token postMessage 给非固定工坊 origin；
- 文档说明 TTL、过期、退出清除、SESSION_SECRET 轮换和服务端撤销边界；
- XSS/恶意同源扩展可能窃取 token，不能用“在 iframe/localStorage”宣称隔离；
- 认证失败要回到未登录浏览态，不无限重试。

### 10.3 隐私表述要收窄

publisher 身份注册表长期持久化只保留 salted Discord hash、生成的 publisherId，以及最小 provider/createdAt/lastLoginAt；不存邮箱、昵称、头像、原始 Discord ID、IP 或行为画像。

Gateway 业务上仍会存包、审核状态、revision、拒绝原因、撤回/最小审计等数据，因此不能笼统写“Gateway 只存 publisherId/hash”。不同表的数据保留和删除策略要分别说明。

### 10.4 生产脚本不能直接当安全范式

`参考案例/4.2_1.components/tavern_helper_scripts/05_创意工坊.js` 可证明架构与 API 可行，但当前审计发现：

- 世界书 bridge 已有 origin 检查，可保留思路；
- Discord auth listener 缺完整 origin/source 校验；
- 认证转发存在 `postMessage('*')`；
- 登录 URL 需要固定 allowlist；
- `removePanel()` 未完整注销 message/TH button listener；
- 工坊主链被无关 MVU 等待阻断。

新实现必须修正，不能因为“生产在用”就原样复制。

## 11. 高频错误与版本敏感项

| 错误 | 后果 | 正确方向 |
|---|---|---|
| 把 E5 写成第二份插件教程 | 与 D5 漂移 | ST 插件统一转 D5 |
| 旧 scripts 对象当主格式 | 当前 TH 无法按契约管理 | `ScriptTree[]` + scope |
| `getButtonEvent(scriptName)` | 按钮无响应 | 传 `button.buttons[].name` |
| 把 Worldbook 称旧 API | 新旧方向反了 | Worldbook 推荐，Lorebook deprecated |
| `entries[targetKey]` | 把 UID/字符串当数组下标 | `find/findIndex` + uid/packageIdentity |
| `deleteWorldbookEntries(name, ids)` | 参数类型错误 | predicate |
| 包级 hash 判断条目未改 | 误覆盖玩家修改 | installedEntryHash |
| 预览计划直接写旧快照 | 并发覆盖 | updater 内重验 latestEntries |
| 回滚整本世界书 | 覆盖并发/用户内容 | 仅补偿本事务刚写入且未变化的条目 |
| iframe 当安全沙盒 | 过度授权 | 最小 bridge + 主动代码审查 |
| `postMessage('*')` | token/动作泄漏 | 固定 origin + source |
| token 进脚本 data/世界书 | 随卡/存档泄露 | 仅受控 localStorage，明确同源风险 |
| 工坊硬等 MVU | 无关依赖导致入口失效 | 最低能力依赖 |
| `🧩mod ` 前缀批量删除 | 误删用户条目 | 只作迁移候选 |
| withdrawn 自动远删本地 | 破坏玩家存档 | 只移除公开索引 |

版本敏感项：

- TH 当前上游版本与 `minimum_client_version`；
- ScriptTree 和 Worldbook 类型；
- `#extensionsMenu/.extensionsMenu` 私有 DOM；
- `tavern_events` 常量与 payload；
- Gateway package schema、API 和 CORS；
- OAuth/token TTL/交接协议；
- ST/TH iframe 同源与 CSP 行为。

施工记录“目标版本/commit、能力检测、核验日期和降级路径”，不写“最新版通用”。
