## 1. 机制总览：Chat Completion vs Text Completion 双轨

**high `[文档·多源]`**

ST 的 prompt 构建管线分为完全独立的两套：

```
API 类型                    Prompt 构建控制面板
──────────────────────────────────────────────────
Chat Completion             Prompt Manager（本文档主讲）
  (OpenAI / Claude / etc.)
Text Completion             Advanced Formatting 面板
  (KoboldAI / tabbyAPI 等)   → Context Template (Story String)
                             → Instruct Mode（仅此模式需要）
──────────────────────────────────────────────────
```

两套不互通。角色卡开发基本针对 Chat Completion（OpenAI 兼容接口 + Claude API），故本文档重点覆盖 Prompt Manager 路径。

来源：docs.sillytavern.app/prompts + docs.sillytavern.app/advanced-formatting 交叉验证  
适用版本：v1.14.0–v1.18.0

### 2.1 系统预设 Prompt 完整列表

**high `[运行时源码]`**

以下 identifier 为 ST 硬编码的系统预设，顺序参照 `PromptManager.js` 中 `promptManagerDefaultPromptOrder`（注意：Default.json 预设文件的 character_id=100000 顺序与此略有差异，且 Default.json 的 character_id=100000 中无 `personaDescription`，仅 character_id=100001 的全局 group 默认顺序才含此项）：

| identifier | 显示名 | 类型 | 默认 role | 默认 enabled |
|---|---|---|---|---|
| `main` | Main Prompt | 内容型 | system | ✅ |
| `worldInfoBefore` | World Info (↑Char) | 标记型（marker） | — | ✅ |
| `personaDescription` | Persona Description | 标记型 | — | ✅（仅在 promptManagerDefaultPromptOrder 中；Default.json character_id=100000 无此项） |
| `charDescription` | Character Description | 标记型 | — | ✅ |
| `charPersonality` | Character Personality | 标记型 | — | ✅ |
| `scenario` | Character Scenario | 标记型 | — | ✅ |
| `worldInfoAfter` | World Info (↓Char) | 标记型 | — | ✅ |
| `dialogueExamples` | Chat Examples | 标记型 | — | ✅ |
| `nsfw` | Auxiliary Prompt | 内容型 | system | ✅ |
| `enhanceDefinitions` | Enhance Definitions | 内容型 | system | ❌（默认关） |
| `chatHistory` | Chat History | 标记型 | — | ✅ |
| `jailbreak` | Post-History Instructions | 内容型 | system | ✅ |

说明：
- `marker=true` 的条目是占位符，内容由 ST 运行时动态填充，用户**不能编辑正文**。
- `system_prompt=true` 的条目在 PromptManager.js 内受到特殊保护，不允许被角色卡 override 覆盖。
- Default.json 实际默认顺序（character_id=100000）为：main → worldInfoBefore → charDescription → charPersonality → scenario → enhanceDefinitions(关) → nsfw → worldInfoAfter → dialogueExamples → chatHistory → jailbreak。

来源：本地 v1.14.0 `PromptManager.js` `chatCompletionDefaultPrompts` + `promptManagerDefaultPromptOrder`（直读）+ `default/content/presets/openai/Default.json`（直读）

### 2.2 每个 Prompt 条目的完整字段

**high `[运行时源码]`**

来源：`PromptManager.js` release branch

```json
{
  "identifier": "my_custom_prompt",    // string，自定义用 UUID 或自命名
  "name": "显示名称",                   // UI 显示用，不发给模型
  "content": "实际注入内容，支持{{宏}}",
  "role": "system",                    // "system" | "user" | "assistant"
  "enabled": true,                     // bool，是否启用
  "system_prompt": false,              // bool，系统预设才为 true；禁止用户新建时设为 true
  "marker": false,                     // bool，标记型占位符；用户不能设为 true
  "injection_position": 0,             // 0=RELATIVE, 1=ABSOLUTE(In-Chat)
  "injection_depth": 4,                // injection_position=1 时生效，深度值
  "injection_order": 100,              // 同 role+depth 时的次序（数值越小越靠前）
  "forbid_overrides": false,           // bool，禁止角色卡 override
  "extension": false,                  // bool，是否由扩展插件注入
  "injection_trigger": []              // string[]，限制触发生成类型（见 2.4）
}
```

适用版本：v1.14.0–v1.18.0

### 2.3 injection_position 详解

**high `[运行时源码]`**

来源：`PromptManager.js` INJECTION_POSITION 枚举

| 值 | 常量名 | 行为 |
|---|---|---|
| `0` | `RELATIVE` | 按 UI 拖拽顺序放置，不进入聊天历史区 |
| `1` | `ABSOLUTE`（In-Chat） | 按 `injection_depth` 插入聊天历史中，忽略拖拽顺序 |

**深度语义**（high `[文档·多源]`）：

```
depth=0 → 插在最后一条消息之后（对话末尾）
depth=1 → 插在最后一条消息之前（倒数第1条的上方）
depth=2 → 插在倒数第2条消息之前
depth=N → 插在倒数第N条消息之前
```

默认值 `injection_depth=4`（与世界书 DEFAULT_DEPTH 一致）。

**同 depth 的多条 prompt 排序**（**high `[运行时源码 + 文档·多源]`**）：  
按 `injection_order` **数值越小越靠前**（在最终输出中更早出现）。官方文档原文："The lower the number, the earlier it is sent."。源码机制：`openai.js` `populationInjectionPrompts()` 中先以 order 降序(`b - a`)处理（大数值先 splice），再整体 `messages.reverse()`，净效果是小数值 order 的 prompt 在最终 prompt 序列中靠前。同 injection_order 时，roles 按 `system → user → assistant` 顺序推入（注释"most important go lower"，反转后靠后即更接近生成末端）。

来源：本地 v1.14.0 `openai.js` L760-791（直读）+ docs.sillytavern.app/prompt-manager（多源交叉）  
适用版本：v1.14.0–v1.18.0

### 3.2 system_prompt / post_history_instructions 生效条件

**medium·单源孤证 `[文档·单源 + 运行时源码（默认值）]`**

- 角色卡的 `data.system_prompt` 字段非空时，**仅在用户 ST 设置中开启「Prefer Character Prompt」**后才覆盖 preset 的 main prompt；**该选项默认为开启**（`[运行时源码]` 本地 v1.14.0 `power-user.js` L208：`prefer_character_prompt: true`）。
- 角色卡的 `data.post_history_instructions` 字段非空时，同理需要开启「Prefer Character Instructions」（`prefer_character_jailbreak`）；**该选项默认同样为开启**（`power-user.js` L209）。
- 若卡的这两个字段为空字符串，则完全不影响用户 preset 设置。
- `{{original}}` 宏可在角色卡 `system_prompt` 中引用全局 preset 的 main prompt 内容（见 7.2 节宏表）。

来源：docs.sillytavern.app/prompt-manager（单源）+ 本地 v1.14.0 `power-user.js` L208-209（默认值）  
仍需真机验证：override 的精确内容替换还是追加行为（见悬案 P2）。

## 4. Post-History Instructions（PHI）机制

**high `[运行时源码]`**（role 字段源码直读；"优先级最高"为推断，单独标注见下）

PHI 在 Prompt Manager 中对应 `identifier="jailbreak"` 的条目（尽管 UI 显示名是"Post-History Instructions"），其特殊之处：

- 作为 **system role 注入**（`[运行时源码]` 本地 v1.14.0 `chatCompletionDefaultPrompts` 直读），按 UI 拖拽顺序位于 chatHistory 之后，插在 prompt 末尾（最近的 assistant 回复请求之前）
- 优先级最高——因为紧靠生成请求，模型最后读到它（**medium·推导未验证 `[推断]`**：此处为逻辑推断，不同模型权重分布可能不同）
- 可包含宏（`{{char}}`、`{{user}}` 等）

> **[运行时源码] 校准**：本地 v1.14.0 `openai.js` L2034-2038 与 `PromptManager.js` L2034-2038 均明确 `jailbreak` 的 `role` 为 `"system"` 而非 `"user"`。原文"user role 的不可见注入"为错误描述，已更正。用户可自行在 Prompt Manager 中将 role 改为 user，但默认即 system。

标准 Chat Completion prompt 末端结构：

```
[...聊天历史...]
[Post-History Instructions（system role，默认；用户可改为 user/assistant）]
[最后一条 assistant 前缀（或空 assistant 开头消息触发回复）]
```

来源：docs.sillytavern.app/prompt-manager + PromptManager.js `chatCompletionDefaultPrompts`（本地 v1.14.0 直读）  
适用版本：v1.14.0–v1.18.0

## 5. Context Template（Text Completion 专用）

**high `[文档·多源]`**

Context Template **仅影响 Text Completion API** 的 prompt 格式化，Chat Completion 完全不用此配置。

### 5.1 Story String（Handlebars 语法）

范式（**medium·单源孤证 `[文档·单源]`**）：

```handlebars
{{#if system}}{{system}}
{{/if}}{{#if description}}{{char}}'s personality: {{description}}
{{/if}}{{#if personality}}{{char}}'s persona: {{personality}}
{{/if}}{{#if scenario}}Scenario: {{scenario}}
{{/if}}{{#if persona}}[{{user}}'s persona: {{persona}}]
{{/if}}
```

可用 Handlebars 参数：

| 参数 | 说明 |
|---|---|
| `{{description}}` | 角色描述字段 |
| `{{scenario}}` | 场景字段 |
| `{{personality}}` | 角色性格字段 |
| `{{system}}` | 系统提示词内容 |
| `{{persona}}` | 用户人设描述 |
| `{{char}}` | 角色名 |
| `{{user}}` | 用户名 |

来源：docs.sillytavern.app/context-template  
适用版本：v1.14.0–v1.18.0

## 6. Instruct Mode（Text Completion 专用）

**high `[文档·多源]`**

Instruct Mode **仅适用于 Text Completion API**。Chat Completion（OpenAI/Claude）完全不需要也不支持此设置——配置了 Instruct Mode 对这些 API 没有任何作用。

来源：docs.sillytavern.app/instructmode + 源码分析

### 6.1 核心序列字段

**high `[文档·多源]`**

```json
{
  "name": "ChatML",
  "user_alignment_message": "",          // 用户填充消息（user-first 约束时使用）
  "system_prompt": "",                   // 独立系统提示词（已从序列中分离）

  // 消息包裹序列
  "input_sequence": "<|im_start|>user\n",
  "input_suffix": "<|im_end|>\n",

  "first_output_sequence": "",           // 第一条 assistant 消息前缀（聊天历史专用）
  "last_output_sequence": "",            // 最后一条 assistant 消息前缀（实际生成用）
  "system_instruction_sequence": "",     // 后台静默生成时的系统前缀

  "stop_sequence": "<|im_end|>",         // 停止序列（生成到此截断）

### 6.2 `{{name}}` 宏（Instruct 专用）

**high `[文档·多源]`**

在 Instruct 序列中，`{{name}}` 返回当前消息的实际发言者名（而非固定的 `{{char}}` 或 `{{user}}`），适合群聊或 `/sendas` 场景。此宏**仅在 Instruct 序列字段**内有意义，在普通 prompt content 中行为未定义。

### 8.3 atDepth 四步注入流水线

源码调用链 **high `[运行时源码]`**；最终 splice+reverse 落位 **medium·推导未验证 `[推断]`**

```

2. script.js 调用 setExtensionPrompt(id, content, IN_CHAT=1, depth, false, role)

## 10. Claude API 的特殊处理（prompt-converters.js）

**high `[运行时源码]`**（原描述有两处错误，已就地校准，见下）

来源：`C:\Users\Administrator\OneDrive\ST-\SillyTavern-1.14.0\src\prompt-converters.js`（本地 v1.14.0 直读）

Claude 的 Messages API 对消息角色有严格约束，ST 通过 `convertClaudeMessages()` 处理（路径：server 侧 `src/endpoints/anthropic.js` 调用）：

- **头部 system 消息提取**：messages 数组最前端连续的 `role=system` 消息被提取为 Claude API 的独立 `system` 参数（仅在 `useSysPrompt=true` 时启用）
- **后续 system 消息转换为 user**：messages 数组中位于非头部的 `role=system` 消息一律被**转换为 `role=user`**（源码 L257 `message.role = 'user'`），而非"合并"
- **PROMPT_PROCESSING_TYPE.CLAUDE** 已标注 `@deprecated`，实际 Claude 处理不走 `STRICT mergeMessages`

> **[运行时源码] 校准**：原文"连续同角色消息通过 mergeMessages() 合并（STRICT 模式）"有误——CLAUDE 类型对应的是 `strict: false` 的 merge，且该路径已废弃。实际生效的是 `convertClaudeMessages()` 的"提取头部 system + 后续 system 转 user"双步逻辑。

**实践影响**：atDepth 注入的 system role WI 条目，在 Claude API 下如果位于 messages 数组非头部位置，会被**静默转换为 user role**（非合并），看起来像用户消息混入对话流。是否与真实用户消息合并取决于 `mergeMessages` 后处理（悬案 P3 依然成立）。如需保持 system 语义，应确保该注入位于 messages 头部。

适用版本：v1.14.0–v1.18.0（Claude API 路径）

## 12. 高频避坑

**high `[运行时源码 + 真机/生产卡 + 文档·多源]`**，个别条目见下方来源标注

1. **Preset 与角色卡同名陷阱**：Preset 若与某角色卡同名，打开该角色时会自动切换到该 preset——通常为非预期行为。Preset 命名建议加前缀区分（如 `[通用]写实风格`）。

2. **Claude API system 消息转换（非合并）**：messages 数组头部连续的 `role=system` 消息被提取为独立 system 参数；头部之后的 `role=system` 消息被**逐条转换为 `role=user`**（非合并成一条），会混入对话流（见悬案 P3）。原"合并"描述不准确——实为类型转换。`[运行时源码]` 本地 v1.14.0 `src/prompt-converters.js` L257 直读。

3. **depth=0 的歧义**：In-Chat depth=0 是"最末尾"（最新消息**之后**），而非"最顶部"。Post-History Instructions 本质上就是 depth=0 的 user role 注入。

4. **marker 标记型条目不能编辑正文**：`chatHistory`、`charDescription`、`worldInfoBefore` 等 `marker=true` 的系统条目，只能设置 enabled/disabled，不能修改其 content——内容由 ST 运行时自动填充。

5. **AN 频率=0 时 Top/Bottom of AN 的 WI 条目被静默跳过**：不报错，仅无效。使用 At Depth（position=4）代替。

6. **WI atDepth 同 depth+role 合并**：同一 depth+role 的多个 WI 条目在 prompt 中只产生一条消息（内容用换行拼接）——无法通过 WI 在同深度产生多条独立消息。

7. **`{{original}}` 宏只在角色卡 override 上下文有效**：在角色卡的 `system_prompt` 字段里使用 `{{original}}` 可引用当前 preset 的 main prompt 内容；在其他地方行为未定义。**high `[文档·单源]`**

8. **`{{outlet::Name}}` 禁止放在 WI 内容里**：文档明确不支持，仅在 Prompt Manager 的内容型条目正文中使用。**high `[文档·单源]`**

9. **Instruct Mode 仅对 Text Completion 生效**：Chat Completion（OpenAI/Claude）完全不走 Instruct 逻辑，配置了 Instruct Mode 对这些 API 没有任何作用。

10. **`injection_trigger` 为空 `[]` ≠ 任何类型都不触发**：空数组是"全部类型均触发"，相当于不过滤。若要只在特定类型触发，必须显式填写类型列表。

11. **`system_prompt=true` 的条目不允许用户新建时设为 true**：PromptManager.js 内对此有保护；新建的自定义条目务必保持 `"system_prompt": false`。

来源：PromptManager.js + prompt-converters.js + docs.sillytavern.app + A3_世界书优化.md  
适用版本：v1.14.0–v1.18.0
