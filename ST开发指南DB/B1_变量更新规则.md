## 1. MVU 系统三层架构

**核心结论**：MVU（MagVarUpdate）由三层构成，各层职责分离，缺一不可。

| 层 | 载体 | 职责 |
|---|---|---|
| **世界书条目** | `[mvu_update]`标记的WI条目 | 向LLM传递变量JSON + 更新规则/schema描述 |
| **bundle脚本** | `artifact/bundle.js` | 解析LLM输出的`<UpdateVariable>`块，执行JSONPatch操作，触发生命周期事件 |
| **zod schema脚本** | `zod_schema.js` | 定义变量结构约束，注册到MVU做变量合法性验证 |

**变量存储结构**（MvuData）：

| 变量 | 用途 | 注入给 LLM |
|---|---|---|
| `stat_data` | 下一轮 LLM 读取当前状态 | 是（通过世界书条目） |
| `display_data` | 前端状态栏渲染（纯值/纯对象，无二元组） | 否 |

**VWD [value, desc] 二元组状态 — 核心结论**：**VWD [value, desc] 格式已在实战卡中彻底废弃。** 本地 InitialVariables.json（星月2.5.0、交错宙域2.6.0）均为纯值结构，路径直接指向值本身（如 `/环境/时间`），不是 `/环境/时间/0`。描述职责已完全迁移到 zod schema 的 `.describe()` 和 `prefault` 机制。bundle.js 仍保留 VWD 检测代码仅用于兼容旧卡，新卡**不应**生产此格式。

**CDN 地址**：

```

https://cdn.jsdelivr.net/gh/MagicalAstrogy/MagVarUpdate/artifact/bundle.js

# 备用镜像（部分生产卡以此为主，如星月2.5.0 mvu_zod_cn.js）
https://testingcf.jsdelivr.net/gh/MagicalAstrogy/MagVarUpdate/artifact/bundle.js
```

## 2. JSONPatch (RFC 6902 方言) 完整规范

### 2.1 UpdateVariable 块结构

**核心结论**：LLM输出必须包裹在 `<UpdateVariable>` 块内，分 analysis 和 JSONPatch 两段。

```xml
<UpdateVariable>
<analysis>
变量预分析（详述正文中哪些状态发生了变化，每行对应至少一个path）
</analysis>
<JSONPatch>
[
  {"op":"replace","path":"/环境/时间","value":"14:25"},
  {"op":"add","path":"/任务/任务名","value":{...完整对象...}}
]
</JSONPatch>
</UpdateVariable>
```

**bundle.js 标签解析正则**：

```javascript
/<(update(?:variable)?|variableupdate)>/gi  // 外层匹配（大小写不敏感）
we(t, 'Analysis')        // 提取 analysis 内容
we(t, 'json_?patch')     // 提取 JSONPatch 内容（匹配 JSONPatch 和 json_patch 两种写法）
```

> 注：`we()` 函数实现为 `e.match(new RegExp('^\\s*<${t}>\\s*([\\s\\S]*?)\\s*</${t}>\\s*$','i'))`，第二参数传 `'json_?patch'`（正则模式），因此同时兼容 `<JSONPatch>` 和 `<json_patch>` 两种标签写法。

来源：`[运行时源码]`（bundle.js CDN 实测） | 置信度：**high**

### 2.2 op 类型完整枚举（MVU方言，非纯RFC6902）

**核心结论**：MVU支持6种op，但LLM输出层只应使用4种。

| op | RFC6902标准 | MVU特有 | LLM可用 | 说明 |
|---|---|---|---|---|
| `replace` | ✅ | — | ✅ | 替换已存在路径的值；路径不存在则报错 |
| `add` | ✅ | 扩展 | ✅ | 新增字段/record项；也用于追加数组元素 |
| `remove` | ✅ | — | ✅ | 删除指定路径 |
| `move` | ✅ | — | ✅ | 字段迁移，`from`→`path`（或`to`） |
| `delta` | ❌ | ✅ | ❌ | MVU专有：数值增量操作，`value`须为number；等同`field += value` |
| `insert` | ❌ | ✅ | ❌ | MVU专有：在数组指定位置插入；与`add`等价，在anyOf中合并处理 |

**关键区别**：

- `delta`：`{"op":"delta","path":"/user/体力","value":-10}` → 在当前值基础上加减
- `add` vs `insert`：功能几乎等价（两者在 anyOf 中合并为 `'insert'|'add'`），均可新增record/追加数组
- `replace`：**路径必须已存在**，否则禁止使用；路径不存在时必须用 `add`

**本地 mvu_update_full.txt 规则**（星月2.5.0）明确只允许：`add、replace、remove、move`，未包含 delta 和 insert。

来源：`[运行时源码]`（bundle.js 确认 delta/insert op 存在）+ `[本地实战]`（星月2.5.0 mvu_update_full.txt 规则文本确认 LLM 只用4种 op）| 置信度：op 技术存在：**high**；"delta/insert LLM不可用"结论：**high**（来自生产卡世界书强制规则文本，属规则层而非 bundle 强制拦截）

### 2.3 路径语法

**核心结论**：路径格式为 `/顶层容器/键名/子字段`，使用 JSON Pointer 风格的 `/` 分隔符。

| 规则 | 说明 |
|---|---|
| 分隔符 | `/`（JSON Pointer风格） |
| 数组追加 | `/-` 等效于数组末尾追加（bundle.js 中 `'-' === o ? c.length : o`） |
| 转义 | 使用 `lodash.toPath()` 而非标准 RFC6902 的 `~0`/`~1` 转义 |
| 中文键名 | 直接使用，不需转义 |

**嵌套路径示例**：

```
/环境/时间                          → string字段
/user/基本属性/等级                  → 嵌套对象字段
/当前互动角色/张三/衣着/上装          → record下的子字段
/当前互动角色/张三                    → 整个record项（add/replace完整对象）
/成就/-                              → 追加到数组末尾
```

来源：`[运行时源码]` + `[本地实战]` | 置信度：**high**

## 3. Zod Schema 集成

### 3.1 prefault 详解

**核心结论**：`prefault` 是 Zod v4 的**标准内置方法**（非prototype扩展、非TavernHelper注入），与 `default` 和 `catch` 有本质区别。

**定义**："pre-parse default"——在验证前应用后备值。当输入为 `undefined` 时，prefault 值被送入完整的验证管道（包括transform）。

```javascript
// prefault vs default 的关键区别
z.string().transform(val => val.length).prefault("tuna")
  .parse(undefined)  // → 4（"tuna"经过transform变为4）

z.string().transform(val => val.length).default("tuna")
  .parse(undefined)  // → "tuna"（绕过transform直接返回默认值）
```

**三者对比**：

| 方法 | 触发条件 | 后备值经过验证pipeline | 适用场景 |
|---|---|---|---|
| `prefault(v)` | 输入为 `undefined` | ✅ 是 | 后备值需经transform处理时 |
| `default(v)` | 输入为 `undefined` | ❌ 否（绕过） | 后备值已是最终类型时 |
| `catch(v)` | **任何验证失败** | ❌ 否 | 不可信输入的全兜底 |

**在角色卡中的惯用法**：

```javascript
// 字符串字段：undefined → ''
z.string().prefault('')

// 对象字段：undefined → {}（再经内部字段各自prefault）
z.object({ 上装: z.string().prefault('') }).prefault({})

// 数值字段（配合coerce）：undefined → 0
z.coerce.number().transform(v => _.clamp(v, 0, 100)).prefault(0)

// 布尔字段
z.coerce.boolean().prefault(false)
```

**`passthrough()` 的作用**：允许 schema 对象接受未声明的额外字段。LLM可能写入任意扩展字段时使用。

来源：`[文档·多源]`（Zod v4官方文档 + DeepWiki + Medium文章）+ `[本地实战]` | 置信度：**high**（prefault/default/catch 三者行为对比）；prefault 在 z.preprocess 嵌套中的触发顺序：**medium·推导未验证**（见 §9 悬案 7.3）

### 3.2 registerMvuSchema API

**核心结论**：`registerMvuSchema` 由 StageDog/tavern_resource 的 mvu_zod.js 提供，是连接 zod schema 与 MVU 变量管理器的桥接层。

**导入与注册**：

```javascript
// 1. 导入（带fallback）
let registerMvuSchema;
try {
  ({ registerMvuSchema } = await import(
    'https://cdn.jsdelivr.net/gh/StageDog/tavern_resource/dist/util/mvu_zod.js'
  ));
} catch (error) {
  ({ registerMvuSchema } = await import(
    'https://testingcf.jsdelivr.net/gh/StageDog/tavern_resource/dist/util/mvu_zod.js'
  ));
}

// 2. 定义schema（必须是z.object(...)，对应stat_data层级）
export const Schema = z.preprocess(normalizeRoot, z.object({
  环境: z.object({...}).prefault({}),
  user: z.object({...}).prefault({}),
  // ...其他顶层容器
}));

// 3. 注册（必须在$()回调中）
$(() => { registerMvuSchema(Schema); });
```

**内部原理（mvu_zod.js实现）**：

```javascript
function registerMvuSchema(r) {
  const schema = () => {
    const t = typeof r === 'function' ? r() : r;
    const e = t instanceof z.ZodObject ? z.looseObject(t.shape) : t;
    // 包装为 {stat_data: schema} 注册到MVU变量管理器
    if (typeof registerVariableSchema === 'function') {
      registerVariableSchema(z.object({ stat_data: e }), { type: 'message' });
    }
    return e;
  };
  // 监听MVU生命周期事件，在变量初始化/命令解析/更新结束时执行验证
}
```

**Zod import URL（mvu_zod.js内部）**：

```
https://testingcf.jsdelivr.net/npm/zod/v4/core/+esm
```

**`registerVariableSchema`（底层函数）**：由 JS-Slash-Runner（TavernHelper v4.3.7+）注入到全局环境，`mvu_zod.js` 只是检测并调用它。

来源：`[运行时源码]`（mvu_zod.js CDN 实测）+ `[本地实战]`（星月2.5.0 zod_schema.js）+ `[文档·单源]`（TavernHelper官方更新日志，v4.3.7 明确记录"新增 registerVariableSchema 函数"）| 置信度：**high**

### 3.3 $meta 结构详解

**核心结论**：`$meta` 是变量 JSON 中可附加的元信息对象，控制 MVU 对该变量的处理行为。**生产卡中不使用**（依赖 zod schema 约束结构）。

| 字段 | 类型 | 说明 |
|---|---|---|
| `$meta.extensible` | boolean | 对象/数组是否允许添加未在schema中声明的新字段 |
| `$meta.recursiveExtensible` | boolean | 递归地对所有子字段启用extensible |
| `$meta.template` | object/array | 模板，新增数组/对象元素时的默认结构 |
| `$meta.required` | string[] | 必需字段名数组（缺失时验证失败） |
| `$meta.strictTemplate` | boolean | 严格模板模式（新值必须符合template结构） |
| `$meta.strictSet` | boolean | 严格设置模式（禁止添加template以外的字段） |
| `$meta.concatTemplateArray` | boolean | 数组模板时是否拼接（true）还是替换（false） |

来源：`[运行时源码]`（bundle.js CDN 摘要读取） | 置信度：字段名称存在：**medium·单源孤证**；各字段具体行为：**medium·推导未验证**（生产卡不用 $meta，无真机验证）

## 4. zod 体系与跨实例风险

### 4.1 三套 schema 体系对比

**核心结论（已回源坐实）**：「变量 schema」在 ST 生态里有**三套完全独立、互不调用**的体系，重建时务必认清当前用的是哪套（混淆是 zod_schema.js bug 的根源领域）：

| 体系 | API / 形态 | 作用 | 运行时是否拦截写入 |
|---|---|---|---|
| **JS-Slash-Runner** | `registerVariableSchema(schema, {type})` 单函数，type ∈ 5 种作用域 | 仅变量管理器 UI 校验显示 | 否（纯 UI） |
| **MagVarUpdate (MVU)** | 内嵌 `MvuData.schema`（自定义 `SchemaNode` 类型，**非 zod**），写在变量结构里 | 指导路径校验 / 模板填充 / extensible 控制 | 是（运行时起效） |
| **ST-Prompt-Template** | `setVariableSchema()`（EJS 路线，已废弃） | EJS 写入前拦截校验 | 是 |

来源：`[运行时源码]`（本地 variables.d.ts JSDoc 逐字核实）| 置信度：JS-Slash-Runner 侧 **high**；MVU SchemaNode 侧 **medium·单源孤证**（MVU 私有仓，本地无佐证）

---

### 4.2 registerVariableSchema 纯 UI 不拦截写入

**核心结论（已回源坐实）**：`registerVariableSchema` **仅影响变量管理器 UI 的校验显示**，对代码层面零影响——不影响 `parseMessage` / `replaceMvuData` 的实际写入，不会在运行时拦截非法写入。

依据 `@types/function/variables.d.ts` 的 JSDoc 原文：「这只是方便使用变量管理器这一 UI 查看和管理变量，对于代码层面没有任何影响」

它是**单一函数**靠 `option.type`（global/preset/character/chat/message）切换作用域，**不存在** `registerGlobalVariableSchema` 等多个独立函数。

```typescript
// 签名
function registerVariableSchema(
  schema: z.ZodType<any>,
  option: { type: 'global' | 'preset' | 'character' | 'chat' | 'message' }
): void;
```

### 4.3 跨 zod 实例风险（三实例地雷）

**核心结论**：JS-Slash-Runner 内置 zod `^4.4.3`、MVU（mvu_zod.js）内部从 CDN 加载 zod `v4/core`，卡内若再从 CDN 引第三个 zod → `instanceof` 跨实例必失败、`._def` 取不到、`parse` 静默产出 NaN/undefined。

**铁律：卡内一律复用 `window.z`（JS-Slash-Runner 注入的同一实例），绝不自带 CDN zod。**

```javascript

const Schema = z.object({ 积分: z.number() });

// 危险：从 CDN 引入独立 Zod（可能是不同 v4 patch 或 v3）
// import { z } from 'https://cdn.jsdelivr.net/npm/zod@3/...'; // 禁止
```

**Zod v4 重大 API 变化**：`._def` 内部属性迁移到 `._zod.def`；`ZodType` 类不再互通。跨版本实例 `instanceof z.ZodType` 必然失败。

```typescript

// iframe 内的 window.z 与 parent.z 是同一对象引用
```

## 8. 高频避坑

来源：`[本地实战]` + `[运行时源码]` | 置信度：**high**

1. **replace路径不存在报错**：`replace` 要求路径已存在。首次写入 record 项必须 `add` 完整对象，不能直接 `replace` 其子字段。

2. **新增record子字段的顺序**：先 `add /顶层容器/recordKey`（完整对象），才能对其子字段使用 `replace`。

3. **VWD格式已废弃**：InitialVariables.json 中不写 `[value, desc]` 二元组，直接写纯值（字符串/数字/空对象）。路径直接是 `/环境/时间` 而非 `/环境/时间/0`。

4. **prefault vs default的选择**：对象字段用 `.prefault({})`（值先经过 transform），不要与 `.default({})` 混用；`.default` 的后备值绕过 transform。

5. **passthrough()的必要性**：LLM 可能写入 schema 中未声明的字段时，需在对象末尾加 `.passthrough()`，否则这些字段会被 stripping 掉。

6. **$meta在生产卡中不用**：生产卡通过 zod schema 约束结构，不用 `$meta` 来控制 extensible 等。

7. **registerMvuSchema 需在`$()`回调中调用**：必须等 DOM 就绪后注册，不能在模块顶层直接调用。

8. **z 是全局变量，无需 import**：TavernHelper（v3.3.2+）自动注入 `z`，直接使用 `z.object()` 等，无需 import zod。卡内绝不从 CDN 另引 zod 实例（三实例地雷，见 §4.3）。

9. **analysis每行必须有对应patch**：`<analysis>` 中每行描述至少对应一个 JSONPatch 操作，禁止分析中提到变化但 patch 中遗漏。

10. **LLM输出不使用 delta/insert**：这两个 op 仅限脚本代码调用；LLM输出的 JSONPatch 只用 `add / replace / remove / move` 四种。

11. **VARIABLE_INITIALIZED 必须用常量引用**：事件字符串值中有 typo（`initiailized`，多一个 i），硬编码极易出错，必须用 `Mvu.events.VARIABLE_INITIALIZED` 引用（见 §5.2）。

12. **iframe_events 有 `js_` 前缀**：iframe 内监听 ST 事件要用 `iframe_events.GENERATION_STARTED`（`'js_generation_started'`），不是 `tavern_events.GENERATION_STARTED`（`'generation_started'`）。

13. **registerVariableSchema 不拦截写入**：调用此函数只影响变量管理器 UI 显示，无法在运行时阻止 LLM 写非法值。真正的运行时拦截靠 zod schema 经 `registerMvuSchema` 注册（见 §4.2）。

## 9. 悬案（未决项）

### 悬案 P1 — delta op 是否为 LLM 可用 op

**状态**：已有倾向性结论，但未真机验证  
**描述**：MVU bundle.js 使用 `delta` op（数值增量）但 RFC6902 标准中无此 op。本地 mvu_update_full.txt 规则（星月2.5.0）明确只允许 `add、replace、remove、move`，**未包含 delta**。  
**倾向结论**：LLM输出格式仅用4种op（add/replace/remove/move），`delta` 是内部API或早期版本遗留，不暴露给LLM。  
**验证路径**：查看卡的世界书条目中 `[mvu_update]` 的op限制说明；或在 JSONPatch 中使用 `delta` 后观察 bundle 是否接受。
