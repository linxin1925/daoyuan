## 0 置信度分级体系（全手册适用）

> **收集日期统一声明**：本手册全部条目收集于 **2026-06-27**。回源核实基于本地 SillyTavern **v1.14.0** + JS-Slash-Runner **v4.8.9** 源码；MVU（MagVarUpdate）/ ST-Prompt-Template 为第三方仓，本地仅有类型声明、无运行时实现，相关条目已在标注处单独说明。

每条结论采用 **置信度等级 + 依据类型** 双轴标注。

### ① 置信度三级

| 级别 | 含义 | 判定标准 |
|---|---|---|
| **high** | 已坐实，可直接施工 | 有运行时证据（真机实测 / 读到运行时实现源码 / 生产卡验证）；非 beta 未合并特性；结论不含跨步推导；不在悬案清单 |
| **medium** | 待验证，施工前须实测 | 强制二选一子标签：`medium·单源孤证`（单一来源、无第二源、无实测）/ `medium·推导未验证`（源码或逻辑推导成立但未跑通，**须随附真机验证路径**）|
| **low** | 孤证无源，禁止据此施工 | 零来源猜测 / 纯历史快照（须标年份 + 窄缩适用范围）|

### ② 依据类型标签（与置信度正交，强制标注）

`[运行时源码]`、`[真机]`、`[文档·多源]`、`[文档·单源]`、`[类型声明]`、`[历史孤证:年份]`、`[无源·待考]`

### ③ 两条铁规

1. **仅由 `[类型声明]` 支撑的「运行时行为 / 可用性」结论，最高只能 `medium·推导未验证`**——除非另有 `[运行时源码]` 或 `[真机]` 佐证才升 high。（"类型文件里如此定义"这一**类型层事实**本身可标 high `[类型声明]`，但不得外推为运行时保证。）
2. **节级标注 = 节内条目下确界**——禁止整节标 high 却内含悬案条目；冲突即拆节。正文断言凡命中悬案章（本手册 §15 P 系列）的，置信度不得高于该悬案对应级别，并交叉引用。

## 1 核心架构

### 核心结论

MVU 引擎在每轮对话结束后（GENERATION_ENDED 事件）读取 LLM 输出的差分块，基于上一楼最新 stat_data 合并生成两个变量，均存储在当前楼层当前 swipe 的消息变量中：

| 变量 | 用途 | 注入给 LLM | 格式 |
|---|---|---|---|
| `stat_data` | 下一轮 LLM 读取当前状态 | 是（通过世界书条目） | `[值, 描述]` 二元组 |
| `display_data` | 前端状态栏渲染 | 否 | `"旧值->新值(原因)"` 字符串 |

`display_data` 不进 prompt，可在此做任意美化（数值→阶段别名、隐藏字段等），不增加 prompt 复杂度。

**前端读取优先级**（high，多源一致）：

```javascript
const message_data = await getChatMessages(getCurrentMessageId());
var gameData = message_data[0].data.stat_data;
if (_.has(message_data[0].data, 'display_data')) {
    gameData = message_data[0].data.display_data;
}
```

**MvuData 完整 TypeScript 类型定义**（high，来自 `@types/iframe/exported.mvu.d.ts`）：

```typescript
type MvuData = {
  initialized_lorebooks: Record<string, any[]>;
  stat_data: Record<string, any>;
  // 以下字段在 .d.ts 被 [key: string]: any 兜底，实际存在：
  // schema: ObjectSchemaNode  (来自 variable_def.ts，.d.ts 未暴露)
  // display_data?: Record<string, any>   // @deprecated
  // delta_data?: Record<string, any>     // @deprecated
  [key: string]: any;
};
```

**注意**：`display_data` 和 `delta_data` 据 MVU 私有仓 `variable_def.ts` 标记 `@deprecated`（本地 `exported.mvu.d.ts` 并未暴露此信息，属单源转述），但状态栏渲染中仍被实际使用。

> 来源：MagVarUpdate doc/stat_var.md + exported.mvu.d.ts ｜ 适用版本：beta 分支 + TH v3.4.0+ ｜ 置信度：MvuData 基础字段（`initialized_lorebooks` / `stat_data`）= `high [类型声明]`；而 `schema` / `display_data` / `delta_data` 的存在与 `@deprecated` 状态 = `medium·单源孤证 [文档·单源]`
>
> ⚠️ **来源订正（回源核实 2026-06-27）**：本地 v1.14.0 的 `exported.mvu.d.ts` 中 `MvuData` 仅为 `{ initialized_lorebooks, stat_data, [key]: any }`，**不含** `schema` / `display_data` / `delta_data` / `@deprecated` 任何字样。上述隐藏字段实际来自 MVU 私有仓 `variable_def.ts`（本地无）或社区推断，**不应挂在 `exported.mvu.d.ts` 名下当 high**。

## 2 变量初始化

### VWD（ValueWithDescription）格式

每个叶字段用 `[初始值, "描述/更新条件"]` 二元组声明。描述字段直接对 LLM 可见，指导更新时机和有效范围，不需要单独写 prompt。

世界书条目名称必须包含 `[InitVar]` 字符串（可在 MVU 配置中禁用此约定，但默认开启）。

**范式代码**（high，多源一致）：

```json
{
    "日期": ["03月15日", "今天的日期，格式为 mm月dd日"],
    "时间": ["09:00", "格式为 hh:mm，每次行动后更新"],
    "user": {
        "身份": ["新来的牧师", "随故事进展改变"],
        "当前位置": ["教堂", "user所在位置，移动后改变"]
    },
    "角色名": {
        "好感度": [0, "[-100,100]，情感变化时更新"],
        "着装": [["默认服装"], "更换着装时更新"]
    },
    "$meta": { "extensible": false }
}
```

**场景特定初始化（问候语 greeting 中的覆盖块）**（high）：

```xml
<UpdateVariable>
_.set('user.身份', '未知', '新来的牧师');//故事开始的设定
_.set("角色.好感度", 0, 15);//故事开始的设定
</UpdateVariable>
```

问候语块在开局时执行，优先级高于 `[InitVar]` 默认值。注意：此块可以没有 `<Analysis>` 节。

**EJS 路线的等价方案**（需 ST-Prompt-Template 插件）（medium）：

```html
<% setvar('stat_data', { '日期': ['03月15日', '今天的日期'] }, { dryRun: true, flags: 'nx' }) %>
```

`flags: 'nx'` = 不存在时才写，防重复初始化；`dryRun: true` = 准备阶段多次评估也安全。

> 来源：MagVarUpdate tutorial.md + supplementary-tutorial.md ｜ 适用版本：beta 分支 ｜ 置信度：VWD 格式 / `[InitVar]` 约定 = `high [文档·多源]`；问候语覆盖块"优先级高于 [InitVar]"的执行行为 = `medium·单源孤证 [文档·单源]`（tutorial.md 单源，未实测）

## 4 五种操作命令

> ⚠️ **整章废弃**：`_.set`/`_.add`/`_.assign`/`_.remove`/`_.move` 命令式方言**已彻底废弃**（被 JSONPatch 取代），VWD `[value,desc]` + `[0]` 索引同废。本章仅作"读旧卡"历史参考，新卡一律用 JSONPatch（第 5 章）。

### _.set — 设置值

（high，多源一致）

```javascript
// 两参数（直接覆写）
_.set('当前日期', '2000-01-01');
_.set('生命', 14);

// 三参数（含旧值校验，推荐）
_.set('悠纪.好感度', 33, 35);//愉快的一次讨论
_.set('经历天数[0]', 1, 5);//第二天

// [value, description] 结构必须用 [0] 精确定位值层
_.set('悠纪.好感度[0]', 10, 12);//正确
// _.set('悠纪.好感度', 10, 12);   // 若好感度是 [value,desc] 结构，这会替换整个数组！

// newValue 参数可用 math.js 表达式（beta，medium）
_.set('悠纪.好感度', 10, math.pow(2, 3) + math.sin(math.PI));
```

**[0] 索引铁律**（high，多源一致）：当变量值本身是对象或数组时（即 VWD 的 `[value, "desc"]` 中 value 是嵌套结构），路径必须显式加 `[0]` 才能进入值层：

```javascript
_.set('福建.舰载机[0].补给中.J-35', 8, 9);  // 正确
// _.set('福建.舰载机.补给中.J-35', 8, 9);   // 错误——操作描述字段而非数据字段
```

### _.add — 数值增减

（high，来自 beta 分支 supplementary-tutorial.md）

```javascript
_.add('gold', 10);         // 增加 10
_.add('health', -5);       // 减少 5
_.add('悠纪.好感度[0]', 2);  // [value,desc] 结构须加 [0]
```

**注意**：`assign` 和 `remove` 不支持 shortcut input，必须使用完整路径。

### _.assign — 插入元素

（high，来自 beta 分支）

```javascript
// 数组末尾追加（2参数）
_.assign('inventory', '新获得的钥匙');

// 数组指定索引插入（3参数，第二参为数字索引）
_.assign('inventory', 0, '古老的卷轴');

// 对象添加键值对（3参数，第二参为字符串键名）
_.assign('achievements', 'FIRST_MEETING', '与悠纪的初次相遇');

// 向根对象添加新角色命名空间
_.assign('', '新角色', {"好感度": [0, "描述"], "着装": [["..."], "描述"]});
```

**[0] 索引对称坑（high）**：与 `_.set` 完全对称——若目标本身是 VWD 结构（`[值, "描述"]`），要进值层数组追加/插入也须显式 `[0]`：

```javascript
_.assign('背包[0]', '新物品');   // 正确：进入值层数组追加
// _.assign('背包', '新物品');   // 错误：操作外层 VWD 数组，覆写描述位
```

### _.remove — 删除元素

（high，来自 beta 分支）

```javascript
// 删除整个变量（1参数）
_.remove('temp_quest_marker');

// 按值从数组删除（2参数，第二参为字符串）
_.remove('inventory', '用掉的药水');

// 按索引从数组删除（2参数，第二参为数字）
_.remove('inventory', 2);

// 从对象按键名删除（2参数，第二参为字符串键名）
_.remove('achievements', 'FIRST_MEETING');
```

### _.move — 路径间移动

（high，来自 CommandInfo TypeScript 类型定义）

```javascript
// MoveCommandInfo.args = [from: string, to: string]
// 具体语法格式需参考 MagVarUpdate 源码或示例，TypeScript 层面只有两个路径参数
```

### CommandInfo 完整 TypeScript 类型（官方）

（high，来自 `@types/iframe/exported.mvu.d.ts`）

```typescript
type CommandInfo = SetCommandInfo | InsertCommandInfo | DeleteCommandInfo | AddCommandInfo | MoveCommandInfo;

type SetCommandInfo = {
  type: 'set';
  full_match: string;
  args:
    | [path: string, new_value_literal: string]
    | [path: string, expected_old_value_literal: string, new_value_literal: string];
  reason: string;
};

type InsertCommandInfo = {  // 对应 _.assign
  type: 'insert';
  full_match: string;
  args:
    | [path: string, value_literal: string]
    | [path: string, index_or_key_literal: string, value_literal: string];
  reason: string;
};

type DeleteCommandInfo = {  // 对应 _.remove
  type: 'delete';
  full_match: string;
  args: [path: string] | [path: string, index_or_key_or_value_literal: string];
  reason: string;
};

type AddCommandInfo = {  // 对应 _.add
  type: 'add';
  full_match: string;
  args: [path: string, delta_or_toggle_literal: string];
  reason: string;
};

type MoveCommandInfo = {
  type: 'move';
  full_match: string;
  args: [from: string, to: string];
  reason: string;
};
```

## 5 JSONPatch 方言

（high，来自 beta 分支 supplementary-tutorial.md）

符合 RFC 6902 标准，支持 `replace / add / remove / move` 四种 op（`add/remove/replace` 基本操作 + `move`）。ST-Prompt-Template 的 jsonPatch 实现额外支持非标准 `set/assign` 作为别名。

```xml
<UpdateVariable>
    <Analysis>
        - [Numerical Calculation]: Old Value + Delta = New Value
        - [Array Modification]: Identify index to remove or use /- to append
        - [Object Modification]: Check if key exists. New key = 'add', Existing = 'replace'
    </Analysis>
    <JSONPatch>
[
  { "op": "replace", "path": "/当前时间", "value": "2022-10-17T16:30:00" },
  { "op": "move", "from": "/主线任务/进行中/0", "path": "/主线任务/已完成/-" },
  { "op": "replace", "path": "/玲奈/亲密度", "value": 0.507 },
  { "op": "add", "path": "/玲奈/重要成就/2022年10月18日", "value": "玲奈与<user>相遇。" },
  { "op": "remove", "path": "/玲奈/着装/0" }
]
    </JSONPatch>
</UpdateVariable>
```

**路径规则**（RFC 6901 JSON Pointer）：
- 路径以 `/` 开头
- 数组末尾追加用 `/-`
- 键名含 `/` 需转义为 `~1`，含 `~` 需转义为 `~0`

> 来源：MagVarUpdate supplementary-tutorial.md ｜ 适用版本：beta 分支 ｜ 置信度：JSONPatch 格式规范（RFC 6902/6901）= `high [文档·多源]`（公开标准）；MVU 是否支持及是否已进 @master CDN = `medium·单源孤证 [文档·单源]`（见 §15 P1）

## 6 Analysis 节规范

（medium，来自 WebFetch 摘要，未见完整原文）

**Analysis 节是否必需**：存在矛盾——问候语 greeting 块示例没有 Analysis 节，tutorial.md 常规块都有。推断 Analysis 节**可选**，框架缺少时仍能解析执行命令。

**命令式方言的 Analysis 格式**（中文，Y/N 列举）：

```xml
<Analysis>
    悠纪.好感度: Y
    暮莲.日程.周三.上午: Y
    当前位置: N
</Analysis>
```

**JSONPatch 方言的 Analysis 格式**（要求英文推导）：

```xml
<Analysis>
    - calculate time passed: current 14:00, action takes 2h = 16:00
    - [Numerical Calculation]: Old Value (33) + Delta (2) = New Value (35)
    - [Array Modification]: Identify index to remove (e.g., 0 for first item)
    - [Object Modification]: Check if key exists. New key = 'add', Existing key = 'replace'
</Analysis>
```

**避坑**：若 LLM 系统提示/思维链中提到了 `<analysis>` 标签（裸标签提及），正则必须用"不跨同名标签"写法，防止把中间正文吞入 analysis 框：

```javascript
// 错误（会从思维链裸 <analysis> 一路匹配到正文后的 </analysis>）
/<analysis>\s*([\s\S]*?)\s*<\/analysis>/

// 正确（遇中途同名标签即停）
/<analysis>\s*((?:(?!<\/?analysis>)[\s\S])*?)\s*<\/analysis>/gi
```

> 来源：MagVarUpdate tutorial.md + supplementary-tutorial.md ｜ 适用版本：beta 分支 ｜ 置信度：Analysis 格式细节 = `medium·单源孤证 [文档·单源]`（WebFetch 摘要，未见完整原文）；不跨标签正则 = `high [真机]`（已在生产角色卡真机验证）

## 7 $meta 结构保护机制

（high，来自 supplementary-tutorial.md）

`$meta` 嵌在变量初始化 JSON 中控制结构约束，**初始化完成后自动移除，不消耗后续 token**。每次结构变动后系统重新生成 schema，LLM 执行增删命令时执行模式校验。

### $meta 字段速查表

| 字段 | 类型 | 说明 | 置信度 |
|---|---|---|---|
| `extensible` | boolean | false=锁定（禁止 _.assign 新增键/_.remove 删已有键）；true=开放 | high |
| `required` | string[] | 列入的键名不允许被 _.remove 删除（即使 extensible:true） | high |
| `recursiveExtensible` | boolean | true=所有后代节点均可扩展，遇子节点显式 extensible:false 才截断 | high |
| `template` | object/array | _.assign 新增元素时自动套用的默认结构 | high |
| `strictSet` | boolean | true=禁用 VWD 语义，_.set 直接覆写标量（丢弃描述字段） | high（仅 beta） |
| `strictTemplate` | boolean | 模板合并类型约束（默认 false） | medium |
| `concatTemplateArray` | boolean | 数组模板合并策略，默认 true=concat 追加 | medium |

**注意**：文档说"$meta 接受四个属性"后又出现 strictSet/strictTemplate/concatTemplateArray，这三个字段可能是根级别专属选项，非普通子节点 $meta 选项。分类不清晰，需源码确认。

### 常用范式

**对象局部锁定 + 子对象开放**：

```json
{
  "$meta": { "extensible": false },
  "角色": {
    "核心属性": { "$meta": { "extensible": false } },
    "背包": { "$meta": { "extensible": true } },
    "任务": { "$meta": { "extensible": true } }
  }
}
```

**extensible:true + required 保护必需键**：

```json
{
  "$meta": { "extensible": true, "required": ["核心属性", "基础状态"] },
  "核心属性": { "生命": [100, "描述"] },
  "可选扩展": {}
}
```

**数组可扩展标记**（`$__META_EXTENSIBLE__$`）：

数组默认不可扩展（与对象行为不同）。需在数组值部分（索引0）的数组内放置此标记：

```json
{
  "背包": [
    ["$__META_EXTENSIBLE__$", "初始物品A", "初始物品B"],
    "背包内容变化时更新"
  ]
}
```

**初始化完成后此标记自动移除**，不出现在后续 stat_data 中。

**数组元素模板（$arrayMeta）**：

```json
{
  "记忆": [
    {
      "$meta": { "template": ["重要的记忆"] },
      "$arrayMeta": true
    }
  ]
}
```

执行 `_.assign('记忆', ["成为了猫娘"])` 后自动套用模板得到 `["成为了猫娘", "重要的记忆"]`。

**recursiveExtensible 高自由度目录树**：

```json
{
  "/": {
    "$meta": { "recursiveExtensible": true },
    "home": {
      "$meta": { "template": { "notes.txt": "document" } },
      "alice": { "notes.txt": "document" }
    },
    "etc": {},
    "var": {}
  }
}
```

**strictSet 禁用 VWD 描述保留**：

```json
{
  "$meta": { "strictSet": true },
  "生命": [20, "受到伤害时减少"]
}
```

启用前：`_.set('生命', 14)` → `[14, "受到伤害时减少"]`（保留描述）
启用后：`_.set('生命', 14)` → `14`（直接标量）

> 来源：MagVarUpdate supplementary-tutorial.md ｜ 适用版本：beta 分支 ｜ 置信度：$meta 主字段（extensible/required/recursiveExtensible/template）= `medium·单源孤证 [文档·单源]`（仅 beta supplementary-tutorial.md 单源、无第二源、无运行时实测，是否已进 @master 未证）；strictSet/strictTemplate/concatTemplateArray 层级归属 = `medium·单源孤证 [文档·单源]`（文档自相矛盾，见 §14.4）

### $meta 避坑

- `extensible:false` 只阻止 _.assign 新增键和 _.remove 删键，**不阻止** _.set 修改已有键的值
- 被屏蔽的非法操作具体表现未定义（静默忽略 vs 报错），LLM 不会收到反馈，可能持续生成被丢弃的命令
- 数组默认不可扩展（与对象不同），忘记加 `$__META_EXTENSIBLE__$` 则 _.assign 到数组会被屏蔽
- `required` 只防删除该键，不防止 _.set 修改该键的内部内容

## 14 分歧与矛盾点

以下矛盾来自不同源，**不强行统一，各保留原始说法**。

### 14.1 命令格式：旧方言 vs 新方言

- **stat_var.md（master 分支）**说法：更新格式为 `set|${variable}=${old}→${new}|(reason)`，包裹在 `<UpdateVariable>` 标签
- **tutorial.md（beta 分支）**说法：格式为 `_.set('path', oldValue, newValue);//reason`，附带 `<Analysis>` 块
- **分歧本质**：两者是不同版本的迭代，旧格式可能已被新格式取代，也可能同时受支持
- **重建时**：优先使用新格式（`_.set` 风格），若旧版 bundle.js 解析失败则降级测试旧格式

### 14.2 display_data 箭头符号

- **stat_var.md 示例**：使用 `→`（Unicode U+2192）
- **tutorial.md 示例**：使用 `->`（ASCII）
- **两种写法均在文档中出现**，具体符号由 `function.ts`（私有仓）的生成代码决定，状态栏正则/CSS 匹配时应兼容两种

### 14.3 EJS 路线 vs MVU 路线变量竞争

- **MVU 方案**：MagVarUpdate 引擎维护自己的 state store，通过 `Mvu.getMvuData/replaceMvuData` 操作
- **EJS 方案（ST-Prompt-Template）**：通过 `setvar/getvar` 操作 ST 原生 `chat_metadata.variables`
- **是否能共存**：两套系统都需要独占变量管理权，文档均未明确说明能否共存。架构推断认为混用相同变量键会出现竞争。**实测确认路径**：在同一卡片内用 `/setvar foo 123`，然后用 `Mvu.getMvuData()` 读 `stat_data.foo`，观察是否读到 ST 写入的值

### 14.4 $meta 字段层级归属

- **文档说"$meta 接受四个属性"**：extensible / required / recursiveExtensible / template
- **同一文档后文又出现**：strictSet / strictTemplate / concatTemplateArray
- **两种可能**：后三者是根级别 $meta 专属；或文档分类不清晰，实际均可用于任意层级
- **重建时**：strictSet 只在根级别 $meta 使用，观察是否有作用

function SafeGetValue(value, defaultValue = "") {
