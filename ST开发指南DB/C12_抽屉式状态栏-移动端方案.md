## 1. 使用场景与非目标

### 1.1 为什么移动端需要抽屉

桌面浮窗依赖可拖动、可缩放和持续可见的剩余屏幕空间；在手机上会同时遇到：

- 视口窄，缩放把手和内容争夺空间；
- 浏览器地址栏动态伸缩，`100vh` 不等于真正可见高度；
- 软键盘改变 `visualViewport`，底部浮窗容易覆盖输入区；
- 粗指针需要更大热区，不能复用桌面小把手；
- ST 顶栏、底部发送区与 PWA 安全区都可能变化。

抽屉把交互简化成“边缘手柄 + 单轴开合”，更适合移动端作为完整 HUD 的终极外壳方案。

### 1.2 抽屉不是什么

- 不是消息楼层里的 `<details>` 折叠块；
- 不是首楼同层前端内部的路由面板；
- 不是可以任意拖动、自由缩放的窗口；
- 不负责读写 MVU / 数据库，也不定义状态栏字段；
- 不应挂在消息 iframe 内，否则无法可靠覆盖宿主视口；
- 不等于“移动端把状态栏缩小一点”。

## 2. 共享 HUD 宿主架构

抽屉和浮窗应复用同一个 HUD Host Core，只替换 ShellAdapter：

```text
Wand / Orb / Page Button
          ↓ command registry
HUD Host Core
  phase / generation / AbortController
  contextRevision / visible / dirty / iframe
          ↓ ShellAdapter
       DrawerShell
          ↓
      iframe + bridge
          ↓
    状态栏真身 / 数据链
```

共享核心建议至少维护：

```javascript
const hudSession = {
  phase: 'idle',             // idle/loading/ready/failed/destroyed
  mode: 'drawer',
  host: null,
  iframe: null,
  blobUrl: null,
  abortController: null,
  generation: 0,
  contextRevision: 0,
  visible: false,
  dirty: true,
  pendingChatReset: false,
  disposers: [],
};
```

抽屉只实现：

- 建立/销毁外壳；
- 设置 top/bottom 几何；
- 打开、关闭、手势和可见性；
- 把 HUD body 容器交给共享核心；
- 通知共享核心 `visible` 变化。

加载远程 HTML、注入 bridge、变量刷新和上下文守卫不应在 DrawerShell 复制一份。

## 3. 抽屉 DOM 与状态机

### 3.1 DOM 结构

```html
<aside id="card-status-drawer"
       data-placement="bottom"
       data-open="false"
       data-phase="idle">
  <button class="drawer-handle"
          type="button"
          aria-controls="card-status-drawer-panel"
          aria-expanded="false">
    <span class="sr-only">打开状态栏</span>
  </button>
  <section id="card-status-drawer-panel"
           class="drawer-panel"
           aria-hidden="true">
    <div class="hud-body"></div>
  </section>
</aside>
```

外层 wrapper 可用 `pointer-events:none` 避免挡住酒馆；手柄和打开后的面板恢复 `pointer-events:auto`。

### 3.2 状态机

```text
unmounted
  → closed-idle
  → opening → open-ready
  → closing → closed-idle(dirty? false/true)
  → failed → retrying
  → switching-shell
  → destroyed
```

外壳开合与内容加载是两个正交状态：抽屉可以已经打开但内容还在 `loading`，也可以关闭但 iframe 仍为 `ready`。不要用一个 `isOpen` 同时表示两者。

推荐分开：

```javascript
drawer.open;       // 外壳是否展开
hudSession.phase; // 内容是否 ready
hudSession.visible;
hudSession.dirty;
```

### 3.3 关闭不是销毁

普通关闭应：

- 把 panel 移出可视区；
- 同步 `aria-hidden` / `aria-expanded`；
- 若焦点在 panel/iframe 内，先归还给手柄；随后设置 `panel.inert=true`，必要时把 iframe `tabIndex=-1`；
- 设置 `visible=false`；
- 保留 iframe，避免每次重拉资源；
- 隐藏期间把数据变化合并为一个 `dirty=true`。

只有切换外壳模式、切卡、热重载、永久关闭功能或运行时销毁时，才真正撤销 iframe、bridge、Blob URL 和监听。

## 4. Top / Bottom / Auto 策略

### 4.1 三种位置模式

| 模式 | 适合 | 主要风险 |
|---|---|---|
| `top` | 底部输入区长期占用；顶部留白稳定 | ST 顶栏测高错误产生空白或遮挡 |
| `bottom` | 顶栏复杂；用户更习惯拇指下缘操作 | 发送框、软键盘和 Home 条竞争 |
| `auto` | 通用发行 | 必须稳定、可解释并允许用户覆盖 |

`auto` 不应每次 resize 都来回跳。设备策略确定后，只有在明确断点、输入区状态或用户设置变化时切换。

### 4.2 宿主模式自动选择

```javascript
function hostWindow() {
  try {
    if (window.parent && window.parent !== window && window.parent.document) {
      return window.parent;
    }
  } catch (_error) {}
  return window;
}

function chooseHudShell() {
  const host = hostWindow();
  const coarse = host.matchMedia('(pointer: coarse)').matches;
  const narrow = host.innerWidth <= 768;
  return coarse || narrow ? 'drawer' : 'floating';
}
```

入口与抽屉挂在 ST 宿主时，`document/visualViewport/matchMedia/getComputedStyle/localStorage` 和全局单例都必须从 `hostWindow()`/`hostWindow().document` 取得；TH iframe 的裸 `window` 不是宿主视口。

这只是最低能力规则。可进一步加入：

- 用户强制选择；
- PWA / 移动 WebView；
- 横屏高度过低；
- 桌面触摸屏但有足够空间；
- ST 宿主 movingUI/侧栏占用。

自动模式必须允许用户手动覆盖，并提供“恢复自动”。

### 4.3 Top 位置

不能直接写 `top:0`。应测量当前可见的 ST 顶部 chrome：

```javascript
function measureTopAnchor(doc, viewport, handleHeight = 44) {
  const host = doc.defaultView || hostWindow();
  const top = viewport.y;
  const bottom = viewport.y + viewport.height;
  const maxChromeHeight = Math.min(120, Math.max(24, viewport.height / 3));
  const candidates = [
    '#top-settings', '#navbar', '#sheld_header',
    '#top-settings-holder', '#top-bar', '.top-bar',
  ].flatMap((selector) => [...doc.querySelectorAll(selector)]);

  const anchors = candidates.flatMap((node) => {
    const style = host.getComputedStyle(node);
    const rect = node.getBoundingClientRect();
    const visible = style.display !== 'none'
      && style.visibility !== 'hidden'
      && rect.width > 0 && rect.height > 0
      && rect.height <= maxChromeHeight
      && rect.bottom > top && rect.bottom <= top + maxChromeHeight;
    return visible ? [rect.bottom + 4] : [];
  });

  const candidate = Math.min(
    anchors.length ? Math.max(...anchors) : top + 4,
    top + maxChromeHeight,
  );
  const upper = Math.max(top, bottom - handleHeight);
  return Math.round(Math.min(Math.max(candidate, top), upper));
}
```

返回值是 layout viewport 坐标系中的绝对 `top` anchor，不是相对偏移。选择器是脆弱观察值，必须可配置并带回退、可视性、相交与上限保护；若测得异常，应回到 `viewport.y + gap` 并记录诊断，而不是留下大片空白。

### 4.4 Bottom 位置

底部模式应测量 `#form_sheld`、`#send_form`、`#send_textarea` 等实际可见输入区的顶部，并取可见视口底边与输入区顶边的差值作为避让量。

不要把输入区高度写死。移动端软键盘、快捷回复栏、附件面板和主题都可能改变它。

## 5. 动态视口、安全区与软键盘

### 5.1 使用 visualViewport

```javascript
function getViewportRect() {
  const host = hostWindow();
  const vv = host.visualViewport;
  if (!vv) {
    return { x: 0, y: 0, width: host.innerWidth, height: host.innerHeight };
  }
  return {
    x: vv.offsetLeft,
    y: vv.offsetTop,
    width: vv.width,
    height: vv.height,
  };
}
```

抽屉重排至少监听：

- `window.resize`；
- `orientationchange`；
- `visualViewport.resize`；
- `visualViewport.scroll`；
- ST 顶栏或输入区 `ResizeObserver`。

高频事件合并到单个 `requestAnimationFrame`，避免每个 scroll 都同步测量和写样式。

### 5.2 CSS 安全区

```css
#card-status-drawer {
  position: fixed;
  left: max(8px, env(safe-area-inset-left));
  right: max(8px, env(safe-area-inset-right));
  z-index: var(--card-hud-z, 30000);
  pointer-events: none;
}

#card-status-drawer .drawer-panel {
  max-height: min(82dvh, 720px, var(--hud-available-height));
  background: rgba(20, 22, 30, 0.94);
  overflow: hidden;
}
```

横屏时左右安全区尤其重要。底部还要处理 `env(safe-area-inset-bottom)`；顶部处理 `env(safe-area-inset-top)`，但不要和已经包含刘海高度的宿主顶栏测量重复相加。

### 5.3 `dvh` 与回退

优先使用 `dvh` 表示动态视口高度，并为不支持的 WebView 提供 `vh` 或 JS CSS 变量回退：

```css
.drawer-panel {
  height: min(76vh, 640px);
  height: min(76dvh, 640px);
}
```

不要只写 `100vh`。移动浏览器地址栏收放会使它大于实际可见区域。

### 5.4 软键盘策略

底部抽屉和软键盘同时出现通常会让输入区不可用。可选策略：

1. 发送框获得焦点时自动关闭 bottom drawer；
2. 抽屉改为 top；
3. 按 `visualViewport` 剩余高度压缩，但设最小可用高度；
4. 给用户“保持打开”高级选项。

默认推荐 1：聚焦发送框时平滑关闭底部抽屉，并保留 `dirty` 状态，键盘关闭后不自动抢焦点重开。

## 6. 点击、滑动与无障碍

### 6.1 手势规则

- 点击手柄：开/关；
- top drawer 向下滑打开、向上滑关闭；
- bottom drawer 向上滑打开、向下滑关闭；
- 单轴位移超过阈值才判为滑动；
- 横向滚动或轻微抖动不应误触；
- `pointercancel` 恢复到手势前状态。

```javascript
const SWIPE_THRESHOLD = 34;

function shouldOpenFromSwipe(placement, deltaY) {
  return placement === 'top'
    ? deltaY >= SWIPE_THRESHOLD
    : deltaY <= -SWIPE_THRESHOLD;
}
```

手势结束后只提交一次目标状态，不要一边拖一边改变业务 `open` 真值。

### 6.2 触控热区

在窄屏或 `(pointer:coarse)` 下，手柄可点击区域至少 44 CSS px。视觉线条可以只有 4–6px，但外层 button 不能同样纤细。

### 6.3 无障碍

- 手柄使用原生 `button`；
- `aria-controls` 指向 panel；
- `aria-expanded` 与实际开合一致；
- panel 的 `aria-hidden` 同步；
- `Enter` / `Space` 开关；
- `Escape` 关闭并把焦点还给手柄；
- 打开后不要强制把焦点劫持到 iframe；
- 读屏文本说明“打开/关闭状态栏”，不是只读出“按钮”。

### 6.4 Reduced Motion 与低端设备

```css
@media (prefers-reduced-motion: reduce) {
  #card-status-drawer,
  #card-status-drawer .drawer-panel {
    transition: none !important;
  }
}
```

`backdrop-filter` 必须有不支持和低性能时的纯色回退。移动端掉帧时首先禁用 blur、阴影和复杂背景，不应减少触控热区或破坏内容可读性。

## 7. 加载、隐藏刷新与失败恢复

### 7.1 打开流程

```text
open requested
  → ensure shell
  → shell visible
  → phase idle/failed ? start load : reuse ready iframe
  → bridge ready
  → dirty ? refresh once : no-op
```

可以先展示外壳和 loading 占位，再等待 iframe；不要在网络完成前完全无反馈。

### 7.2 隐藏期间的 dirty 合并

```javascript
function onHudDataChanged() {
  hudSession.dirty = true;
  void flushDirtyHud();
}

function onDrawerOpened() {
  hudSession.visible = true;
  drawerPanel.inert = false;
  hudIframe.tabIndex = 0;
  void flushDirtyHud();
}

async function flushDirtyHud() {
  if (!hudSession.visible || !hudSession.dirty || hudSession.refreshing) return;
  hudSession.refreshing = true;
  const revision = hudSession.contextRevision;
  try {
    await refreshHudOnce({ revision });
    if (revision === hudSession.contextRevision) hudSession.dirty = false;
  } catch (error) {
    hudSession.dirty = true;
    hudSession.phase = 'failed';
    reportHudRefreshFailure(error);
  } finally {
    hudSession.refreshing = false;
  }
}
```

关闭期间不要继续做每 token、每 MutationObserver 的完整渲染。只记录“需要刷新”，再次打开补一次。

### 7.3 失败状态

至少区分：

- 远程内容拉取失败；
- bridge 注入失败；
- iframe 加载超时；
- 依赖（jQuery/lodash/MVU/TH）未就绪；
- 当前聊天上下文已变化；
- 内容脚本主动报告错误。

失败面板必须提供：错误摘要、重试、关闭。不要自动无限重试；也不要以 HTTP 200 代替应用 ready。

## 8. 模式切换与异步竞态

### 8.1 Drawer ↔ Floating 单宿主原则

抽屉与浮窗必须共享同一个活动 HUD 内容实例，或在切换时明确迁移/重建；不能同时留下两个 iframe 写同一份变量。

```text
drawer active
  → request floating
  → mark switching
  → abort old load / detach old bridge
  → destroy drawer shell
  → mount floating shell
  → attach/reload one iframe
```

若内容 iframe 可安全移动 DOM，可复用；若 bridge 把宿主 window 捕获在闭包中，应销毁重建。必须由实现明确选择，不能半迁移。

### 8.2 generation 防旧响应覆盖

```javascript
async function loadHudContent() {
  const generation = ++hudSession.generation;
  hudSession.abortController?.abort();
  const controller = new AbortController();
  hudSession.abortController = controller;

  const html = await fetchHud({ signal: controller.signal });
  if (generation !== hudSession.generation) return;
  mountHudHtml(html);
}
```

三类失效必须分开：

- shell 或内容源变化：`generation++` 并 abort 旧加载；
- `CHAT_CHANGED` / Swipe / Message Updated：`contextRevision++`，设置 `dirty/pendingChatReset`，不无故中止与聊天无关的静态 HUD bundle；
- destroy：generation 与 contextRevision 都失效并清全部资源。

只有 HUD bundle 本身与聊天绑定时，切聊天才取消并立即重启加载。旧请求即使最后成功，也不能覆盖新 shell。

### 8.3 上下文守卫

变量写回前同时校验：

- `chatId`；
- `messageId`；
- `contextRevision`；
- runtime 未销毁；
- 仍是发起交互的同一 HUD generation。

否则用户在异步确认框打开期间切聊天，结果可能写进旧楼层。

## 9. 性能与生命周期

### 9.1 布局读写分离

一次重排中先读：

- visualViewport；
- 顶栏 rect；
- 输入区 rect；
- panel 内容需要的最小高度。

然后在下一阶段集中写 CSS 变量。不要在循环中交替 `getBoundingClientRect()` 和 `style.*`。

### 9.2 资源清理

销毁时必须处理：

- `ResizeObserver`；
- window / visualViewport 监听；
- 手柄 Pointer Events；
- focusin/focusout 监听；
- iframe 与 bridge subscriber；
- AbortController；
- Blob URL；
- animation frame / timer；
- 宿主 DOM 和 style。

普通关闭不销毁这些资源，但切换到浮窗模式时要按共享核心策略处理，确保全局仍只有一套订阅。

## 11. 高频错误

| 错误 | 后果 | 修复方向 |
|---|---|---|
| 抽屉挂消息 iframe | 被楼层裁切，无法覆盖宿主 | 挂宿主 `document.body` |
| 只用 `100vh` | 地址栏/键盘遮挡 | `dvh` + visualViewport |
| 只看 `innerHeight` | 软键盘弹出仍误判高度 | 监听 visualViewport |
| 顶栏选择器误命中 | 顶部出现大段空白 | 可见性、上限、回退与诊断 |
| bottom 覆盖输入框 | 无法继续发送消息 | 测发送区，聚焦时关闭或切 top |
| safe-area 重复相加 | 刘海设备产生额外偏移 | 明确宿主 chrome 是否已包含安全区 |
| 手柄只有细线可点 | 移动端难以触发 | 视觉细、热区至少 44px |
| 关闭仍高频渲染 | 掉帧耗电 | hidden 只记 dirty |
| top/bottom 同一 transform | 一边开合方向反了 | placement 明确方向函数 |
| 切模式不 abort | 旧 iframe 覆盖新 shell | generation + AbortController |
| `aria-hidden` 不同步 | 读屏状态错误 | 单一开合提交函数 |
| 无限自动重试 | 离线时持续请求 | 可见失败态 + 用户重试 |
| 移动端强用浮窗 | 小把手难用、内容过窄 | auto 路由到 drawer |

## 12. 验收矩阵

| 维度 | 必验项 |
|---|---|
| 自动模式 | `<=768px`、coarse pointer 进入 drawer；桌面回到 floating；用户覆盖有效 |
| 位置 | top/bottom 紧贴有效宿主 chrome，无额外空白或遮挡 |
| 浏览器 | 地址栏展开/收起、PWA、WebView、缩放后仍在可视区 |
| 键盘 | 输入框聚焦、软键盘弹出、bottom drawer 正确关闭/避让 |
| 安全区 | iPhone 刘海、Home 条、横屏左右安全区 |
| 手势 | 点击、正确方向滑开/滑关、轻微抖动不误触、pointercancel |
| 数据 | 关闭期间只记 dirty；重开补一次；切聊天、Swipe、编辑后刷新 |
| 模式切换 | drawer ↔ floating 始终只有一个 host/iframe/订阅链 |
| 失败恢复 | 网络失败、bridge 超时、依赖缺失、重试、切换途中取消 |
| 无障碍 | 44px、button、ARIA、键盘、焦点返回、reduced-motion |
| 生命周期 | listener、observer、Blob、iframe、timer 在销毁后全清 |

稳定完成门：至少在一台 iOS Safari/WebView 与一台 Android Chrome/WebView 真机通过 top、bottom、软键盘、旋转和模式切换；桌面模拟触摸不能替代真机。
