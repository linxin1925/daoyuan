## 1. 定义、使用场景与非目标

### 1.1 定义

浮窗式状态栏是在 **ST 宿主 document** 中创建的独立 content surface：

- 相对可视视口定位；
- 可以拖动和缩放；
- 内容通常由 iframe 或隔离容器承载；
- 关闭后可由 Wand、悬浮球或页面按钮重新打开；
- 刷新或重载后可恢复位置和尺寸；
- 切聊天时保留外壳，但内容必须切换到新上下文。

### 1.2 适用场景

- 桌面端需要边聊天边查看较完整状态；
- 状态栏有多个 Tab、人物、背包、任务等较大内容；
- 不希望把控制中心锁死在某条消息楼层；
- 同一 HUD 真身还要复用到移动端抽屉；
- 需要网络加载、bridge 握手、失败重试和内容热更新。

### 1.3 非目标

- 不在消息 iframe 内写 `position:fixed` 假装宿主浮窗；
- 不把 ST movingUI 当作角色卡稳定公开 API；
- 不在手机上强行保留细小 resize handle；
- 不通过浮窗外壳直接实现 MVU/数据库读写；
- 不把“关闭浮窗”“收起浮窗内容”“状态栏内部折叠区”混成一个状态；
- 不把项目枚举名 `orb` 解释为球形浮窗。项目中它可以只是“从悬浮球入口打开 floating shell”。

## 2. 共享 HUD session

浮窗和抽屉应复用一个核心会话：

```javascript
const hudSession = {
  phase: 'idle',
  mode: 'floating',
  host: null,
  iframe: null,
  blobUrl: null,
  abortController: null,
  generation: 0,
  contextRevision: 0,
  visible: false,
  dirty: true,
  pendingChatReset: false,
  subscribers: new Set(),
  disposers: [],
};
```

职责分配：

| 模块 | 负责 |
|---|---|
| HudHostCore | phase、加载、bridge、上下文、刷新、AbortController、Blob URL |
| FloatingShell | 浮窗 DOM、几何、拖动、缩放、可见性、无障碍 |
| DrawerShell | top/bottom、visualViewport、手柄与单轴开合 |
| HudBody | 状态栏内容、Tab、字段渲染、写回 |
| EntryRegistry | Wand/悬浮球/页面按钮到命令的路由 |

共享 session 的意义不是“所有代码放一个大函数”，而是保证任意时刻只有一个活跃内容宿主、一套上下文订阅和一份异步真值。

## 3. 浮窗 DOM 与双状态机

### 3.1 推荐 DOM

```html
<section id="card-floating-hud"
         role="region"
         aria-label="角色状态栏"
         data-shell-state="visible"
         data-content-phase="ready">
  <header class="hud-drag-handle">
    <span class="hud-title">状态栏</span>
    <button type="button" data-action="collapse" aria-expanded="true">收起</button>
    <button type="button" data-action="close">关闭</button>
  </header>
  <div class="hud-body"></div>
  <div class="hud-resize-handle" aria-hidden="true"></div>
</section>
```

iframe 放入 `.hud-body`，拖动把手和关闭按钮留在宿主外壳，确保内容加载失败时仍可操作。这里使用非模态 `region`；若产品改成真正模态 dialog，才增加完整焦点圈与 `aria-modal` 管理。

### 3.2 外壳状态

```text
closed-hidden
  → opening
  → visible
     ├─ dragging
     ├─ resizing
     └─ collapsed-content
  → load-failed
  → destroyed
```

### 3.3 内容状态

```text
idle → loading → ready
          └────→ failed → loading(retry)
ready → stale/dirty → refreshing → ready
any → destroyed
```

两套状态必须分开。浮窗可以 `visible + loading`，也可以 `collapsed-content + ready`。若只用 `open/closed`，通常会出现“收起后留下大块空壳”或“加载失败时关闭按钮也消失”。

## 4. 几何、夹取与持久化

### 4.1 绝对像素真值

浮窗稳定真值建议为：

```javascript
const rect = { x: 240, y: 80, w: 430, h: 640 };
```

宿主 transform、长页面和 CSS containing block 可能让 `right/bottom` 或 `translate(-50%)` 定位跑到不可见区域。恢复后的稳定状态使用明确 `left/top/width/height`：

```css
#card-floating-hud {
  position: fixed;
  left: var(--hud-x);
  top: var(--hud-y);
  width: var(--hud-w);
  height: var(--hud-h);
}
```

初始化居中可以临时计算，但计算完成后立即落成绝对像素，不把 transform 保留为持久化的一部分。

### 4.2 可视视口

```javascript
function hostWindow() {
  try {
    if (window.parent && window.parent !== window && window.parent.document) {
      return window.parent;
    }
  } catch (_error) {}
  return window;
}

function viewportRect() {
  const host = hostWindow();
  const vv = host.visualViewport;
  return vv
    ? { x: vv.offsetLeft, y: vv.offsetTop, w: vv.width, h: vv.height }
    : { x: 0, y: 0, w: host.innerWidth, h: host.innerHeight };
}
```

桌面通常 `offsetLeft/Top=0`，但浏览器缩放、移动模拟和嵌入 WebView 并不保证如此。浮窗挂宿主 document 时，视口、`matchMedia/getComputedStyle/localStorage`、事件和全局单例都从 `hostWindow()` 获取；TH iframe 的裸 `window` 不是宿主。

### 4.3 夹取函数

```javascript
const MIN_W = 280;
const MIN_H = 320;
const EDGE_GAP = 8;

function clampHudRect(input, viewport) {
  const availableW = Math.max(1, viewport.w - EDGE_GAP * 2);
  const availableH = Math.max(1, viewport.h - EDGE_GAP * 2);
  const minW = Math.min(MIN_W, availableW);
  const minH = Math.min(MIN_H, availableH);
  const w = Math.min(Math.max(input.w, minW), availableW);
  const h = Math.min(Math.max(input.h, minH), availableH);
  const minX = viewport.x + EDGE_GAP;
  const minY = viewport.y + EDGE_GAP;
  const maxX = viewport.x + viewport.w - w - EDGE_GAP;
  const maxY = viewport.y + viewport.h - h - EDGE_GAP;
  return {
    x: Math.min(Math.max(input.x, minX), maxX),
    y: Math.min(Math.max(input.y, minY), maxY),
    w,
    h,
  };
}
```

当可用视口小于正常最小值时，上例会临时降低最小值以保证不出屏；产品更推荐直接返回 `useDrawer=true` 切换 C12，避免浮窗内容被压到不可用。

### 4.4 保存与恢复

```javascript
function saveHudRect(rect) {
  hostWindow().localStorage.setItem('my-card:hud-rect', JSON.stringify({
    version: 1,
    ...rect,
  }));
}
```

恢复流程：

1. 解析并校验有限数字；
2. 处理版本迁移；
3. 用当前视口重新夹取；
4. 应用样式；
5. 只在拖动/缩放结束后保存，避免高频写存储。

用户从 4K 屏切到笔记本、浏览器缩小或侧栏展开后，旧矩形必须重夹取。保存成功不等于恢复可见。

## 5. 拖动、缩放与 iframe 指针仲裁

### 5.1 Pointer Events + pointer capture

拖动和缩放应使用统一 Pointer Events。开始时记录起始指针和矩形，移动时基于起始值计算，避免累计误差。

```javascript
function beginDrag(event) {
  if (event.button !== 0) return;
  event.currentTarget.setPointerCapture(event.pointerId);
  dragState = {
    pointerId: event.pointerId,
    startX: event.clientX,
    startY: event.clientY,
    startRect: { ...currentRect },
  };
  host.dataset.shellState = 'dragging';
  host.classList.add('hud-shell-busy');
}
```

`.hud-shell-busy iframe { pointer-events:none; }` 可避免指针进入 iframe 后丢失 `pointermove`。结束或取消时恢复 pointer events、夹取、保存。

### 5.2 拖动区域

- 只从明确 drag handle 开始；
- 按钮、输入框、Tab、滚动区不触发拖动；
- 双击标题栏可选执行“恢复默认尺寸”，但不要和收起冲突；
- 拖动时暂时禁用文本选择；结束后恢复；
- `pointercancel` 也要结束 busy 状态。

### 5.3 缩放

根据左上角和当前视口动态计算最大宽高：

```javascript
const maxW = viewport.x + viewport.w - currentRect.x - EDGE_GAP;
const maxH = viewport.y + viewport.h - currentRect.y - EDGE_GAP;
```

缩放把手只适合精细指针。`(pointer:coarse)` 下应隐藏或放大，并优先自动切换 drawer。不要把桌面 `mousedown` 拖动工具直接当移动端模板。

### 5.4 写入节流

pointermove 中只更新内存和 `requestAnimationFrame` 内的样式；不要每帧写 localStorage、触发 iframe 重排或保存聊天元数据。

## 6. iframe bridge 与内容加载

### 6.1 推荐加载链

```text
fetch primary source
  → failed? fetch fallback
  → validate protocol / content
  → inject trusted bridge bootstrap
  → create Blob URL
  → create iframe
  → wait iframe load
  → wait bridge + required libraries + app-ready marker
  → phase=ready
```

HTTP 200、`load` 事件和真正应用 ready 是三件不同的事。生产握手应由状态栏真身主动设置 `data-*-ready=1` 或发送带协议版本的 ready 消息。

### 6.2 Bridge 最小协议

```javascript
const BRIDGE_PROTOCOL = 1;

const bridge = {
  protocol: BRIDGE_PROTOCOL,
  getContextSnapshot,
  getVariables,
  requestVariableAction(action) {
    // action 仅允许宿主注册的有限类型；宿主校验 schema、作用域和上下文后执行。
    return applyValidatedVariableAction(action);
  },
  subscribe,
  notifyReady(details) {
    validateReady(details);
    markHudReady();
  },
};
```

不要把 `replaceVariables` 或整个 `window.parent` 交给远程内容。即使同源 iframe 能访问父窗口，也应只注入最小能力。远程内容只提交有限 action/patch；宿主限制为当前 message 变量，clone 旧值，二次检查 chat/message/contextRevision，做 schema 验证后才调用正式写回。只有完全受信本地内容才可在宿主侧使用 `updateVariablesWith` 一类 updater，而且 updater 不跨 bridge 传递。

### 6.3 Blob URL 与脚本

Blob iframe 便于把已核 HTML 与本地 bridge 组合，但要注意：

- 切换内容前 revoke 旧 URL；
- CSP、字体和相对资源路径可能变化；
- 远程 `<script>` 不应默认执行；
- 如果运行可信本地 bundle，明确其版本与完整性；
- iframe 销毁后删除所有 bridge 引用，避免闭包保留聊天数据。

### 6.4 超时与重试

为 fetch、iframe load、bridge ready 分别设置合理超时。失败状态保留外壳上的“重试/关闭”，而不是只在 iframe 内显示错误——iframe 自己可能根本没加载成功。

## 7. 异步上下文与写回守卫

### 7.1 为什么需要 contextRevision

用户可以在浮窗确认框、网络请求或 iframe 初始化期间：

- 切换聊天；
- Swipe 到另一分支；
- 删除/编辑消息；
- 切卡；
- 切换 drawer/floating 模式。

如果异步回调只记住 `messageId`，它可能把数据写回旧聊天。

### 7.2 上下文令牌

```javascript
function captureHudContext() {
  return HudContextAdapter.capture({
    revision: hudSession.contextRevision,
    generation: hudSession.generation,
  });
}

function assertHudContext(token) {
  return HudContextAdapter.matches(token)
    && token.revision === hudSession.contextRevision
    && token.generation === hudSession.generation
    && hudSession.phase !== 'destroyed';
}
```

适配层负责从实际宿主/TH 上下文获取 chat 与 message 身份；不要假定消息 iframe 便利函数在宿主浮窗脚本中也是全局函数。

所有写操作在“用户点击”和“真正提交前”各检查一次。若失败，取消并提示“聊天已切换，请重新操作”，不要偷偷改成写当前聊天。

### 7.3 隐藏 dirty

浮窗隐藏或内部收起时，数据变化只设置 `dirty=true`。再次可见时调用 C12/C13 共享的 `flushDirtyHud()`：await 刷新成功且 contextRevision 未变化后才清 dirty；失败保留 dirty，并用 `refreshing` 防重复并发。

## 8. 打开、关闭、收起与失败恢复

### 8.1 三种动作要分开

| 动作 | 外壳 | iframe | visible | 适用 |
|---|---|---|---:|---|
| 关闭浮窗 | 隐藏/移除 | 通常保留或按策略释放 | false | 暂不查看 |
| 收起内容 | 标题栏保留 | 保留 | 可视但内容 false | 节省空间 |
| 销毁运行时 | 移除 | 销毁 | false | 切卡/热重载/卸载 |

“收起”时要把外壳高度缩到标题栏，而不是只隐藏 iframe 留下整块磨砂背景。

视觉关闭前若焦点在 host/iframe 内，先归还给打开入口；随后设置 `host.inert=true`，必要时 `iframe.tabIndex=-1`。打开时恢复 inert/tabIndex。`aria-hidden` 只影响辅助技术树，不能单独阻止不可见控件进入 Tab 顺序。

### 8.2 打开流程

```javascript
async function openFloatingHud() {
  ensureFloatingShell();
  applyRect(clampHudRect(loadRect() || defaultRect(), viewportRect()));
  setShellVisible(true);
  hudSession.host.inert = false;
  if (hudSession.iframe) hudSession.iframe.tabIndex = 0;
  hudSession.visible = true;

  if (hudSession.phase === 'idle' || hudSession.phase === 'failed') {
    await loadHud();
  } else {
    await flushDirtyHud();
  }
}
```

入口可以重复调用，但 `ensureFloatingShell()` 与 `loadHud()` 必须幂等。双击入口不能创建两个 host。

### 8.3 失败恢复

失败不应自动回退为静默关闭。外壳显示：

- 失败阶段（网络/iframe/bridge/依赖）；
- 简短可理解原因；
- 重试；
- 切换到 drawer（若适用）；
- 关闭；
- 可选复制诊断信息。

重试前先 abort 上一代任务、撤销旧 Blob URL、清掉半挂载 iframe，再启动新 generation。

## 9. 桌面/移动模式切换

### 9.1 自动模式

```javascript
function preferredShell() {
  const host = hostWindow();
  return host.innerWidth <= 768 || host.matchMedia('(pointer: coarse)').matches
    ? 'drawer'
    : 'floating';
}
```

当断点跨越时：

1. 防抖确认新模式；
2. 保存当前浮窗 rect；
3. 标记 session `switching`；
4. 中止旧加载并解除旧 shell 监听；
5. 挂载新 shell；
6. 迁移或重建唯一 iframe；
7. 保持 `contextRevision` 和 dirty 语义；
8. 恢复可见状态。

### 9.2 何时重建 iframe

可以直接移动 iframe 的条件：

- bridge 不捕获旧宿主节点；
- 尺寸观察器会重新绑定；
- CSS 和相对资源不依赖旧层级；
- 移动期间不会触发错误 unload。

否则应销毁重建。两种路线都可以，但必须有明确协议；不能既保留旧订阅又创建新 iframe。

### 9.3 强制浮窗

用户可以选择在平板横屏强制浮窗，但设置页需提示：粗指针缩放和软键盘不属于主支持路线。320–430px 手机不应把强制浮窗作为默认。

## 10. 安全、性能与无障碍

### 10.1 安全边界

- 远程状态栏 HTML/JS 是主动代码供应链，不因放进 iframe 就天然安全；
- bridge 只暴露最小能力，写入须校验上下文和数据 schema；
- `postMessage` 必须固定 origin 并校验 `event.source`；
- 不把 API key、Cookie、完整聊天或宿主对象发给不可信来源；
- 加载源、协议版本和完整性快照应可诊断；
- 关闭/卸载时撤销 Blob URL 和全局 bridge。

### 10.2 性能

- 拖动/缩放样式写入合并到 rAF；
- busy 时禁用 iframe pointer，不反复重建 iframe；
- 隐藏期间暂停动画、轮询和高频渲染；
- `backdrop-filter` 提供纯色 fallback；
- 长列表由内容层虚拟化，外壳不截获内部滚动；
- 加载超时有上限，不无限轮询 ready。

### 10.3 无障碍

- 明确标题和 `role="dialog"` 或更合适的区域语义；
- 关闭、收起是原生按钮，有可读标签；
- 键盘用户可关闭和恢复默认位置；
- 不把拖动作为访问内容的前置条件；
- `Escape` 关闭时不吞掉 ST 其他模态框的 Escape；
- 收起/展开同步 `aria-expanded`；
- reduced-motion 下关闭开场、吸附和缩放过渡。

## 11. 生命周期与资源清理

完整销毁顺序建议：

```text
mark destroyed / generation++
→ abort fetch
→ stop ready polling and timers
→ unsubscribe data and bridge events
→ remove pointer/resize/keyboard listeners
→ disconnect observers
→ remove iframe and host
→ revoke Blob URL
→ remove styles/global API references
```

```javascript
function destroyHudSession() {
  if (hudSession.phase === 'destroyed') return;
  hudSession.phase = 'destroyed';
  hudSession.generation += 1;
  hudSession.contextRevision += 1;
  try { hudSession.abortController?.abort(); } catch (_error) {}
  while (hudSession.disposers.length) {
    try { hudSession.disposers.pop()?.(); } catch (error) { console.warn('hud cleanup failed', error); }
  }
  try { hudSession.subscribers?.clear?.(); } catch (_error) {}
  try { hudSession.bridge?.destroy?.(); } catch (_error) {}
  try { hudSession.iframe?.remove(); } catch (_error) {}
  try { hudSession.host?.remove(); } catch (_error) {}
  try { hudSession.style?.remove(); } catch (_error) {}
  try {
    if (hudSession.blobUrl) hostWindow().URL.revokeObjectURL(hudSession.blobUrl);
  } catch (_error) {}
  hudSession.iframe = hudSession.host = hudSession.style = null;
  hudSession.blobUrl = hudSession.bridge = null;
}
```

在 `pagehide`、热重载、切卡、脚本禁用和运行时替换前调用。普通 `CHAT_CHANGED` 通常不销毁外壳，但必须递增 `contextRevision`、取消旧交互并刷新内容。

## 12. 星月 3.4.9 生产样本

> 已核文件：`rolecard-diy-workshop/runtime/xingyue/3.4.9/control-center.js`。当前 runtime 比模块化 fragments 更新，HUD shell 取材必须以前者为准。

生产样本已经包含：

- 共享 HUD session：phase、mode、host、Blob、AbortController、generation、contextRevision、visible、dirty；
- 430×640 默认矩形、居中、恢复后视口夹取；
- `{x,y,w,h}` 持久化与最小/最大尺寸；
- 宿主 fixed 玻璃外壳、纯色 fallback 与 busy 时 iframe 禁 pointer；
- Pointer Events、pointer capture、拖动/缩放结束保存；
- bridge 注入、Blob iframe、依赖与 ready 标记握手；
- 主/备用资源、12 秒握手上限、AbortController 和 generation；
- `chatId::messageId + contextRevision` 写回守卫；
- 自动在窄屏/coarse pointer 切换为抽屉；
- 销毁时 abort、revoke、取消订阅和移除节点。

项目历史还验证了一个关键坑：在 ST 顶层包含 transform 的环境里，用 `right/bottom` 或 `translate(-50%)` 维持定位会把浮窗带到长页面底部或错误 containing block。当前通用范式因此坚持 `left/top` 绝对像素真值。

模块化 `fragments/11_sidebar.js` 仍是早期浮窗骨架，没有完整 `hudSession`、抽屉 shell、AbortController 和 contextRevision；不能用它否定当前 runtime 的生产能力。

推荐 DOM 是本册增强目标，不是对 3.4.9 的逐行复刻：当前星月把部分关闭控制注入 iframe header，加载失败时宿主外壳的关闭/重试仍应加强；已打开浮窗也应在 resize、orientationchange、visualViewport 变化时主动重新 clamp，而不只在恢复/重开时夹取。

## 13. 高频错误

| 错误 | 症状 | 修复方向 |
|---|---|---|
| 在消息 iframe 内 fixed | 被楼层边界裁切 | 挂宿主 document |
| 用 right/bottom/translate 持久化 | 浮窗跑到长页底部 | 保存 left/top 像素 |
| 拖动时 iframe 吃事件 | 指针中途丢失 | pointer capture + busy 禁 pointer |
| 不重夹旧矩形 | 换屏后窗体找不到 | 每次恢复按当前视口 clamp |
| 收起只隐藏 iframe | 留下整块空磨砂壳 | 同步缩小外壳高度 |
| 关闭/收起/销毁同一布尔值 | 重开丢内容或泄漏 | 外壳与内容双状态机 |
| 内容源/外壳旧请求覆盖新宿主 | 新 shell 显示旧 bundle | AbortController + generation |
| 切聊天后旧交互仍写回 | 数据落入旧聊天 | contextRevision + dirty + 写回二次校验 |
| 只等 iframe load | 空白但被判成功 | bridge + 依赖 + app-ready 握手 |
| Blob URL 不撤销 | 重开越多内存越高 | 每代加载/销毁 revoke |
| 写回不校验 chat | 数据落入旧楼层 | context token 二次校验 |
| 移动端强用小把手 | 难以操作 | 自动切 drawer |
| 隐藏仍持续渲染 | CPU 占用与掉帧 | visible/dirty 合并 |
| 双入口各建一窗 | 出现两个 iframe | 全局单例 + 命令注册表 |
| 把 movingUI 当公共 API | ST 更新后失效 | 自有 shell + 能力检测 |

## 14. 验收矩阵

| 维度 | 必验项 |
|---|---|
| 几何 | 默认居中、拖动、缩放、四边夹取、最小尺寸、浏览器缩放 |
| 恢复 | 保存后重开；4K→笔记本、视口缩小后仍可见 |
| iframe | 拖动时不抢事件，内容操作和滚动时正常响应 |
| 外壳语义 | 关闭、内部收起、重开、失败态不留下空壳 |
| 数据 | chat/message/contextRevision 守卫；隐藏 dirty；重开只补一次 |
| 加载 | 主源失败→备用源；fetch/load/bridge 分阶段超时；retry |
| 竞态 | 切聊天后旧交互/写回失效；shell/content 变化时该取消的加载被取消 |
| 桌面/移动 | 桌面完整；移动自动转 drawer；用户覆盖有明确提示 |
| 安全 | origin/source、schema、最小 bridge、无敏感信息外泄 |
| 无障碍 | 原生按钮、标题、键盘关闭/重置、焦点、reduced-motion |
| 生命周期 | Blob、bridge、subscriber、observer、timer、DOM/style 全清 |

稳定完成门：最低支持版和当前核验版 ST 各跑一次；Chrome/Edge 桌面必验，目标发行若承诺 Firefox/Safari 也须加入；任何“偶发消失”都应先检查几何真值与宿主 transform，不能用增加重试掩盖。
