## 0. 置信度分级体系（全手册适用）

> **收集日期统一声明**：本手册全部条目收集于 **2026-06-27**，交叉验证跨 6 个独立来源。源码级断言的回源核实基于本地 SillyTavern **v1.14.0**（`world-info.js` 等）；手册中标注的 commit/sha（如 5e68410、staging）为**采集时快照**，与本地 v1.14.0 机制一致但行号可能漂移。

每条结论采用 **置信度等级 + 依据类型** 双轴标注。

### ① 置信度三级

| 级别 | 含义 | 判定标准 |
|---|---|---|
| **high** | 已坐实，可直接施工 | 有运行时证据（真机实测 / 读到运行时实现源码 / 生产卡验证）；结论是源码/文档的**直接事实**而非下游推导；不在悬案清单 |
| **medium** | 待验证，施工前须实测 | 强制二选一子标签：`medium·单源孤证`（单一来源、无第二源、无实测）/ `medium·推导未验证`（源码或逻辑推导成立但未跑通，**须随附真机验证路径**）|
| **low** | 孤证无源，禁止据此施工 | 零来源猜测 / 纯历史快照（须标年份 + 窄缩适用范围）|

### ② 依据类型标签（与置信度正交，强制标注）

`[运行时源码]`、`[真机]`、`[文档·多源]`、`[文档·单源]`、`[历史孤证:年份]`、`[无源·待考]`

### ③ 两条铁规

1. **「从源码事实再推一步」的下游推导（如 sort→注入位置、splice→reverse→落位、getScore→激活门槛），即使源码本身已直读，该推导结论最高只能 `medium·推导未验证`**——除非已真机验证；源码内部事实仍标 `high [运行时源码]`。
2. **节级标注 = 节内条目下确界**——禁止整节标 high 却内含悬案条目；冲突即在节内局部降级（**§7.1 是正面范例**）。正文断言凡命中悬案章（本手册 §21 U 系列）的，置信度不得高于该悬案对应级别，并交叉引用。

## 1. 基本定义与核心概念

**核心结论（high）**：World Info（Lorebook / Memory Book）是动态字典，仅当关键词出现在扫描缓冲区时才将对应条目内容插入 prompt。内容对 AI 可见，但不保证 AI 一定使用。

| 属性 | 说明 |
|---|---|
| 作用域 | Global WI：全角色共用；Character Lorebook：绑定到特定角色；Chat Lorebook：绑定到单个聊天（v1.17.0）；Persona Lorebook：绑定到 Persona（v1.17.0） |
| 激活时机 | 每次生成前扫描聊天历史，关键词命中 → 内容注入 prompt |
| 效果保证 | 仅保证内容进入 context，不保证模型遵循 |

来源：docs.sillytavern.app ｜ 适用版本：v1.12.x+ ｜ 置信度：high `[文档·多源]`

## 3. 激活策略（Strategy）

**核心结论（high）**：三种策略通过 UI 图标切换，决定条目的激活触发方式。

| 图标 | 名称 | 行为 |
|---|---|---|
| 🔵 Blue Circle | Constant | 无需关键词，始终激活（仍受 Budget 约束，仍受 Probability 过滤，sticky 期间 Probability 跳过） |
| 🟢 Green Circle | Conditional | 需要关键词命中才激活（默认） |
| 🔗 Chain Link | Vectorized | 允许向量相似度激活（仍可被关键词激活，不互斥） |

**避坑**：
- Constant 条目会消耗 token budget，大量 Constant 条目可能挤占 Conditional 条目空间
- 🔗 标记不禁用关键词激活；只想纯向量激活必须同时删除 Key 字段

来源：docs.sillytavern.app + world-info.js ｜ 适用版本：v1.12.x+ ｜ 置信度：high `[文档·多源 + 运行时源码]`

## 4. 关键词匹配机制

### 4.1 基础规则

**核心结论（high）**：

- Key 字段逗号分隔，默认大小写不敏感（全局 Case-sensitive Keys 默认关）
- 支持 JavaScript regex（需 `/pattern/flags` 格式，所有 flags 允许）
- regex key 与 plaintext key 可在同一条目共存
- plaintext key 内部不能有逗号（逗号视为分隔符）；regex 内部可有逗号

### 4.2 v1.12.6+ \\x01 消息边界分隔

**核心结论（high）**：v1.12.6 起扫描缓冲区中每条消息前额外追加 `\x01`（ASCII 1）分隔符，可用 regex 精确限定发言者。

```
# 范式：仅匹配用户消息中包含「keyword」，AI 说的不触发
/\x01{{user}}:[^\x01]*?keyword/i
```

来源：SillyTavern-Docs/main (commit ca6a171) ｜ 适用版本：v1.12.6+ ｜ 置信度：high `[文档·单源]`

### 4.3 全词匹配（Match Whole Words）

**核心结论（high）**：默认启用，单词 key 只在完整独立单词出现时匹配（空格边界判断）。**对中文/日文有破坏性影响**，写非英文条目时应在条目级或全局关闭此设置。

来源：docs.sillytavern.app + worldinfo.md ｜ 适用版本：v1.12.x+ ｜ 置信度：high `[文档·多源]`

### 4.4 Include Names（包含名称）

**核心结论（high）**：开启（默认）时，消息前缀加入角色名（如 `Alice: `），角色名本身可作为关键词触发条目。

## 5. Optional Filter（次级关键词逻辑）

### 5.1 四种逻辑的精确定义

**核心结论**——来源交叉验证（docs + world-info.js release 实现完全一致）：枚举值↔逻辑名映射 = `high [文档·多源 + 运行时源码]`；但 NOT_ANY/NOT_ALL 的**激活行为**列 = `medium·推导未验证`（其门控实现位置 + 历史 bug #2009，见 §21 U2）：

| 枚举值 | 名称 | selectiveLogic JSON 值 | 行为 | 短路优化 |
|---|---|---|---|---|
| AND_ANY | 任意次级匹配 | 0（默认） | 主键命中 + 次级至少一个命中 | ✅ 找到即返回 |
| NOT_ALL | 非全部次级 | 1 | 主键命中 + 次级未全部命中时激活（全部命中则阻止） | ✅ 找到不匹配即返回 |
| NOT_ANY | 无次级匹配 | 2 | 主键命中 + 次级全部未命中时激活 | ❌ 必须完整遍历 |
| AND_ALL | 全部次级匹配 | 3 | 主键命中 + 次级全部命中 | ❌ 必须完整遍历 |

**重要**：主键命中是所有模式的必要前提，次级 key 为空时 `selectiveLogic` 无效。

```json
// 范式：NOT_ANY 排除词
{
  "key": ["角色名"],
  "keysecondary": ["战斗", "敌对"],
  "selective": true,
  "selectiveLogic": 2
}
// 含义：角色名出现、且战斗/敌对词均未出现时，激活此条目
```

### 5.2 getScore 与 NOT_ANY/NOT_ALL 的分数关系

**核心结论（high）**：`getScore()` 函数的 switch-case 只处理 AND_ANY 和 AND_ALL；NOT_ANY/NOT_ALL 不加分，直接返回 `primaryScore`。这意味着 Group Scoring 模式下负向逻辑条目的竞争分数仅取决于主键匹配数，次级关键词匹配情况不提升分数。

来源：world-info.js release (L428-473)；本地 v1.14.0 L461-473 逐字核实 ｜ 置信度：getScore **内部计分逻辑**（AND_ANY/AND_ALL 加分、NOT 系列仅返回 primaryScore）= `high [运行时源码]`；但"计分是否等于**激活门槛**" = `medium·推导未验证`（getScore 之外是否另有布尔门控未定，见 §21 U1/U2）

⚠️ **分歧标注**：DeepWiki 来源指出 AND_ALL 当次键未全中时 getScore 返回 primaryScore（非 0），与官方文档"ALL secondary keys required"语义存在潜在矛盾——实际激活门槛是否在 getScore 外层另有布尔检查，**悬而未决，见§21**。

## 7. Insertion Order 与排序机制

### 7.1 Prompt 构建阶段的排序（影响注入位置）

**核心结论（high）**——来源：world-info.js L88 + L5082（源码直接读取）：

```javascript
const sortFn = (a, b) => b.order - a.order;  // 单一比较器，order 降序，无 uid tiebreak
```

高 order 值条目排在前面（sort 后），由于 prompt 构建时用 `unshift`，**order 越大，在最终 context 中越靠近最新消息（越靠末尾，影响力越强）**。

**直觉反转警告**：order 100 出现在 context 中 order 250 的**前面**（更远离最新消息），order 250 更靠近最新消息端，影响力更强。

> ⚠️ **置信度修正**：`sortFn` 比较器本身源码直读（high）；但「order 越大 → 越靠末尾 → 影响力越强」这一步经过了「sort → unshift → 最终 prompt 排布」的推导链，**未独立真机验证 prompt 实际排布**，方向性结论按 medium 对待。重建时若依赖此方向，先用 Prompt Manager 预览实测一次（参 §17.2 的 splice+reverse 推导）。

### 7.2 UI 编辑器显示排序（与 prompt 构建无关）

`sortWorldInfoEntries` 函数有三级排序（primarySort → order 降序 → uid 升序），uid tiebreak 只影响**编辑器显示顺序**，不影响 prompt 构建和激活判断。

来源：world-info.js release (L88, L5082-5084, sortWorldInfoEntries) ｜ 适用版本：v1.12.x+ ｜ 置信度：high `[运行时源码]`（sortFn 比较器本地核实；§7.1 的方向性推导已自降 medium）

## 8. Token 预算（Budget）机制

**核心结论（high）**：

- `world_info_budget`：百分比，默认 25%，基于 API max-context 计算
- `world_info_budget_cap`：绝对 token 上限（0=不限）
- budget 耗尽后即使关键词命中也不再激活新条目
- Constant 条目优先插入（先消耗 budget）
- Max Entries（向量模式）只限结果数量，不影响 token budget

**避坑**：大量 Constant 条目可能耗尽 budget 导致 Conditional 条目无法插入。

来源：docs.sillytavern.app + world-info.js exports ｜ 适用版本：v1.12.x+ ｜ 置信度：high `[文档·多源 + 运行时源码]`

## 9. Inclusion Group 完整机制

### 9.1 基础逻辑

**核心结论（high）**：同一 Group 标签的多条目同时激活时，只有一条实际插入 prompt。选择算法分三个串行阶段：

**阶段① filterGroupsByTimedEffects**：sticky 条目保留，cooldown/delay 条目移除

**阶段② filterGroupsByScoring**（仅 group scoring 开启时）：按 getScore 分数保留最高分子集，同分全部保留

**阶段③ 主竞争**：
- 有 `groupOverride=true`（Prioritize Inclusion）的条目 → 按 `sortFn`（order 降序）取第一个，确定性胜出
- 无 groupOverride → 加权随机（Group Weight 默认 100）：

```javascript
// 精确实现（来自 world-info.js release L5325-5354）
const totalWeight = group.reduce((acc, item) => acc + (item.groupWeight ?? 100), 0);
const rollValue = Math.random() * totalWeight;
let currentWeight = 0;
for (const entry of group) {
    currentWeight += (entry.groupWeight ?? 100);
    if (rollValue <= currentWeight) { winner = entry; break; }
}
```

### 9.2 Group Weight 理解

**核心结论（high）**：Group Weight 是**相对权重**而非百分比。权重 100 和 300 的两条目，后者被选中概率为 300/400 = 75%，不是 300%。

### 9.3 Use Group Scoring 评分规则

**核心结论（high）**——来源：docs + world-info.js getScore 函数：

| 情况 | 得分 |
|---|---|
| 主键 1 个匹配 | +1 分 |
| AND_ANY 次级 1 个匹配 | +1 分 |
| AND_ALL 次级全部匹配 | 每个次键 +1 分 |
| NOT_ANY / NOT_ALL 次级 | 不加分（仅返回 primaryScore） |

```
# 范式：通用条目 vs 特化条目
Entry A: keys=[song, sing],            group=songs, use_group_scoring=ON
Entry B: keys=[song, sing, Black Cat], group=songs, use_group_scoring=ON

输入「sing me a song」→ AB 各得 2 分 → 加权随机选一
输入「sing a Black Cat song」→ B 得 3 分，A 得 2 分 → B 确定性胜出
```

### 9.4 多组成员资格

**核心结论（high）**：一个条目可属于多个 Group（逗号分隔）。该条目胜出后，它所属所有组的其他成员均被禁用（跨组级联禁用）。

⚠️ **避坑**：条目 A 属于 group1,group2，A 胜出后会同时禁用 group1 和 group2 的其他所有条目，可能意外屏蔽无关条目。

### 9.5 probability 与 Inclusion Group 的交互

**核心结论（high，来源：world-info.js checkWorldInfo 顺序）**：Inclusion Group 竞争淘汰发生在 probability 检查**之前**（L4893 filterByInclusionGroups vs L4900+ probability check）。胜出者仍可能被 probability 过滤掉，其余同组落败条目不会补位。

来源：world-info.js release L5188-5354 ｜ 适用版本：v1.12.x+ ｜ 置信度：Inclusion Group 竞争**发生在** probability 检查之前的源码顺序 = `high [运行时源码]`；"胜出者被 probability 淘汰后是否补位" = `medium·推导未验证`（倾向不补位，见 §21 U4）

## 10. Timed Effects（定时效果）

### 10.1 三种效果

**核心结论（high）**——来源：docs + worldinfo.md 双源一致：

| 效果 | 字段 | 含义 |
|---|---|---|
| Sticky | `sticky=N` | 激活后保持 N **条消息**有效（sticky 期间跳过 Probability 检查） |
| Cooldown | `cooldown=N` | 激活后 N 条消息内不可再激活 |
| Delay | `delay=N` | 聊天消息数少于 N 时不可激活（阈值检查，非倒计时） |

### 10.2 五条铁律

**核心结论（high）**：

1. 时间单位是**消息数**（不是对话轮次/问答对）
2. 效果仅在激活该条目的那个聊天内有效；分支聊天继承父状态
3. 最后消息被 swipe 或删除 → 效果清除（聊天未推进）
4. 对条目做任何修改 → 强制清除当前 timed effect
5. 关键词再次命中不刷新已生效的 duration

### 10.3 组合时间线示例

**核心结论（high）**——来源：worldinfo.md 范例（两源一致）：

```
配置：sticky=3, cooldown=2, delay=2

Message 0：delay（聊天消息数不足，不触发）
Message 1：首次激活
Message 2：sticky（强制插入，跳过 Probability）
Message 3：sticky
Message 4：sticky
Message 5：cooldown（不可激活）
Message 6：cooldown
Message 7+：可再次激活
```

**说明**：delay=2 表示"至少有 2 条历史消息才能激活"，Message 0 时有 0 条历史。

来源：SillyTavern-Docs/main + sillytavern.wiki ｜ 适用版本：v1.12.2+ ｜ 置信度：high `[文档·多源]`

## 11. 递归扫描（Recursion）

### 11.1 基本机制

**核心结论（high）**：已激活条目的 content 文本追加到 recurseBuffer，下一轮扫描时此 buffer 也参与关键词匹配，实现链式激活。全局开关 `world_info_recursive` 控制启用。

```
范式：角色 → 关系 → 场所 三级链
Entry#1: keys=[贝西], content=「贝西是一头奶牛，是鲁弗斯的朋友」
Entry#2: keys=[鲁弗斯], content=「鲁弗斯是一只狗，住在老农场」
Entry#3: keys=[老农场], delay_until_recursion=true

消息提及「贝西」→ #1 激活 → content 提到鲁弗斯 → #2 激活 → content 提到老农场 → #3 激活
```

### 11.2 三种递归控制选项

**核心结论（high）**：

| 选项 | 说明 |
|---|---|
| Non-recursable | 不被其他条目递归激活（保护静态信息） |
| Prevent further recursion | 该条目激活后不再触发其他条目（阻断链） |
| Delay until recursion | 仅在递归检查阶段参与，不在初始扫描中激活 |

### 11.3 Delay until Recursion Level（递归层级，v1.12.6+）

**核心结论（high）**：支持为 delay_until_recursion 条目指定层级数字。初始只匹配最小层级编号的条目；该层无命中后，下一层级开放；依次类推。

```
Entry A: delay_until_recursion=1（层级1，由主扫描内容触发）
Entry B: delay_until_recursion=2（层级2，需A先激活内容出现B的key才开放）
```

来源：SillyTavern-Docs/main (1.12.6 release notes) ｜ 适用版本：v1.12.6+ ｜ 置信度：high `[文档·单源]`

⚠️ **注意**：`Delay until recursion` 与 `Min Activations` 互斥，不能同时使用。

### 11.4 Max Recursion Steps

**核心结论（high）**：限制递归总扫描轮数。

| 值 | 含义 |
|---|---|
| 0 | 无步数限制（仅受 budget 约束） |
| 1 | 禁用递归（只做初次扫描） |
| 2 | 允许递归触发一次 |
| N | 允许递归触发 N-1 次 |

**与 Min Activations 互斥**，UI 层强制一个为 0。

来源：world-info.js release (L4654-4658) ｜ 适用版本：v1.12.5+ ｜ 置信度：high `[运行时源码]`

### 11.5 Min Activations

**核心结论（high）**：非零时无视 Scan Depth 限制，向上扫描整个聊天记录直到激活足够数量条目（仍受 Max Depth 和 Budget 限制）。追加扫描不检查递归新增的条目。与 Max Recursion Steps 互斥。

**注意**：`scan_state.MIN_ACTIVATIONS` 状态下 `#recurseBuffer` 不被纳入扫描上下文（world-info.js get() 内有守卫），递归触发词在此状态下不生效。

## 12. @@activate / @@dont_activate 装饰器

⚠️ **此功能在官方文档中未记载，来源为 world-info.js release 源码直接读取，置信度 high（源码层面），但属于隐藏实现细节。**

**核心结论（high，源码直读）**：两个装饰器写在条目 **content 字段顶部**（不是 key 字段），由 `parseDecorators()` 函数预处理。

| 装饰器 | 效果 |
|---|---|
| `@@activate` | 无条件激活该条目（跳过关键词检查） |
| `@@dont_activate` | 无条件跳过该条目 |

```
// content 字段示例
@@activate
这里是正文内容，第一个非 @@ 开头的行即为正文开始。

// 转义（不触发装饰器，输出字面量 @@activate）
@@@activate
```

**执行顺序约束**（装饰器**存在性** = `high [运行时源码]`；下方与 disable/timed effect 的**交互结论** = `medium·推导未验证`，见 §21 U11）：
- 装饰器处理在 `disable` 检查**之后** → `disable=true` 的条目 `@@activate` 无法复活
- 装饰器处理在 timed effect 检查（sticky/cooldown/delay）**之后** → delay 期间（聊天消息数未达阈值）的条目 `@@activate` 也无效
- ⚠️ **v1.14.0 回源补正**：sticky/cooldown 等前置检查带 `!isSticky` 短路（sticky 期间会绕过这些检查），但 **delay 检查（L4568）无 `!isSticky` 守卫**，故"delay 期间 @@activate 无效"这一方向在 v1.14.0 成立；其余交互未逐一真机验证。
- 同一条目同时写 `@@activate` 和 `@@dont_activate` 时，`@@activate` 先被检查并短路，胜出

> 来源：world-info.js release **commit 5e68410**（采集快照）+ 本地 v1.14.0 回源（L4568 delay / L4594 装饰器判断）— L100 `KNOWN_DECORATORS`、L4540 `parseDecorators()` ｜ 置信度：装饰器**存在 / 写在 content 首行 / `@@@` 转义** = `high [运行时源码]`（二次回源确认存在）；与 disable / timed effect 的**执行顺序交互** = `medium·推导未验证`（源码顺序推断 + 部分 v1.14.0 验证，见 §21 U11）

## 13. Outlet（插槽注入）

**核心结论（high）**——来源：SillyTavern-Docs/main + docs.sillytavern.app 双源一致：

### 13.1 工作原理

### 13.3 五条限制（高置信，双源一致）

## 14. 向量化匹配（Vector Storage）

### 14.1 激活路径

**核心结论（high）**——来源：docs + vectors/index.js staging 双源：

1. 扩展总开关 `enabled_world_info=true`
2. 条目 `vectorized=true`（🔗 标记）或全局 `enabled_for_all=true`
3. 扩展用最近 `settings.query`（默认 2）条消息文本做查询
4. 经 `queryMultipleCollections()` 返回相似度超过 `score_threshold`（默认 0.25）的结果
5. 通过 `WORLDINFO_FORCE_ACTIVATE` 事件写入 `WorldInfoBuffer.externalActivations` Map
6. 下游 `checkWorldInfo` 按正常 WI 流程处理（仍受 Probability/Character Filter/Inclusion Group 约束）

### 14.2 关键参数默认值

| 参数 | 默认值 | 说明 |
|---|---|---|
| `enabled_world_info` | false | 总开关 |
| `enabled_for_all` | false | 批量模式 |
| `max_entries` | 5 | 每次最多匹配条目数 |
| `score_threshold` | 0.25 | 相似度下限（0~1） |
| `query` | 2 | 查询消息数（不是 Scan Depth） |

来源：vectors/index.js staging ｜ 适用版本：Vector Storage 扩展启用后 ｜ 置信度：参数默认值与激活路径 = `high [运行时源码]`（staging 快照）；其中第 5 步强制激活**事件名** `WORLDINFO_FORCE_ACTIVATE` = `medium·推导未验证`（来自 DeepWiki 描述，未在截断代码直接确认拼写，见 §21 U8）

### 14.3 关键约束

- **Scan Depth 对向量模式无效**，查询深度由 `Query messages` 控制（两者是不同设置入口）
- `vectorized` 标记只替换关键词检查，其余激活条件（trigger%、char filter、inclusion group）仍然全部生效
- `vectorized=true` + 有关键词 → 两种方式都能独立触发（非互斥）；纯向量激活必须删除 key
- 向量激活走相同 token budget 管道，Max Entries 不绕过 budget
- 与 LLM 提示缓存（KV cache）不兼容，频繁重排 prompt prefix 导致缓存失效，付费场景成本上升

来源：docs.sillytavern.app + vectors/index.js ｜ 适用版本：Vector Storage 扩展启用后 ｜ 置信度：high `[文档·多源 + 运行时源码]`

## 15. Automation ID 与 STscript 联动

### 15.1 机制

**核心结论（high）**：WI 条目设置 Automation ID，与 Quick Replies 扩展中同 ID 的 QR 命令绑定。条目激活时自动执行对应 STscript，执行时机为**消息生成之前**（pre-generation）。

### 15.3 注意事项

- 依赖 Quick Replies 扩展已安装并配置，扩展未启用则无效
- 多个同 Automation ID 条目同时激活时，文档称脚本只执行一次

**⚠️ 分歧：Automation ID 去重保证**

- **官方文档（medium）**：同 ID 多条目激活时脚本只执行一次
- **源码分析（medium，AutoExecuteHandler.js）**：`automationIds` 数组无 Set 去重；若两个 QR Set 各有一个不同 QR 实例携带相同 automationId，两者均会执行

**⚠️ 分歧：执行时机是否支持 post-generation**

- **当前实现（high）**：Automation ID 在生成**前**执行，无法读取同轮 AI 生成内容
- **GitHub Issue #3419（medium）**：用户请求 post-generation 执行，2025-02-03 提出，当前状态（2026-06-27）未知

来源：SillyTavern-Docs/main + AutoExecuteHandler.js release + Issue #3419 ｜ 适用版本：Quick Replies 扩展版本相关 ｜ 置信度：pre-generation 时机 = `medium·单源孤证 [文档·单源]`（AutoExecuteHandler.js 本地无、当前实现是否仍纯 pre-gen 见 §21 U7）；去重保证 = `medium·推导未验证`（官方文档与源码分析分歧，见正文 ⚠️）

## 16. 多 Lorebook 来源与优先级

### 16.1 来源优先级（v1.17.0+）

**核心结论（high）**——来源：SillyTavern-Docs/main (commit bad9492, c5f7a56)：

```
Chat Lore → Persona Lore → Character Lore 或 Global Lore
```

后两者顺序由全局策略决定：

| 策略 | 行为 |
|---|---|
| Sorted Evenly（默认） | 全部混合，按 Insertion Order 统一排序 |
| Character Lore First | Character Lore 按 Order 插入后，再 Global |
| Global Lore First | 反之 |

**源码级实现（world-info.js L4494-4513）**：

```javascript
switch (Number(world_info_character_strategy)) {
  case evenly:   entries = [...globalLore, ...characterLore].sort(sortFn); break;
  case char_first: entries = [...characterLore.sort(sortFn), ...globalLore.sort(sortFn)]; break;
  case global_first: entries = [...globalLore.sort(sortFn), ...characterLore.sort(sortFn)]; break;
}
entries = [...chatLore.sort(sortFn), ...personaLore.sort(sortFn), ...entries];
```

Chat Lore 和 Persona Lore 始终放在最前面（优先扫描），与策略无关。

### 16.2 Persona Lorebook 与 Chat Lorebook（v1.17.0+）

**核心结论（high）**：

- **Persona Lorebook**：绑定到 Persona，无论打开哪个角色/聊天，只要该 Persona 激活就参与 WI 评估。已绑定时重新弹出分配弹窗需 Shift+Click/Alt+Click（桌面）或长按（移动端）。
- **Chat Lorebook**：绑定到单个聊天，条目仅在该对话中激活。

### 16.3 重命名自动重链接（v1.17.0）

重命名 WI 文件时弹出确认提示，可选择自动更新引用该文件的所有角色绑定到新文件名。

## 17. WI 在 Prompt 中的精确位置（OpenAI path）

**核心结论（high）**——来源：world-info.js + openai.js + script.js release (commit 51ad27fb) 源码直读。

### 17.1 atDepth 条目的四步流水线

1. `checkWorldInfo()` 收集激活条目到 `WIDepthEntries`（按 {depth, role} 分组）
2. `script.js` 调用 `setExtensionPrompt(id, content, IN_CHAT=1, depth, false, role)` 注册
3. `openai.js populationInjectionPrompts()` 在循环 depth=i 时调用 `getExtensionPrompt(IN_CHAT, i, sep, role)` 读取
4. `splice` 进 newest-first messages 数组，最后 `reverse()`

### 17.2 depth 语义（splice + reverse 双步逻辑）

messages 初始为 newest-first（最新消息在 index 0）：

```
depth=D 时 splice(D, 0, injected) → reverse()

推导：
depth=0 → 插在 index 0 → reverse 后落在整个数组最末尾（最新消息之后）
depth=1 → 插在 newest 和 2nd 之间 → reverse 后紧靠在 newest 之前
depth=N → 最终在 prompt 中位于倒数第 N+1 位（距末端 N 条消息）
```

**默认值**：条目未设置 depth 时取 `DEFAULT_DEPTH=4`。

### 17.3 同 depth 多条目合并

相同 depth + role 的多条 WI 条目在 world-info.js 层面已合并为单字符串（`join('\n')`），注入时是**一条消息**，不是多条分离消息。

### 17.4 worldInfoBefore / worldInfoAfter 的路径差异

`position=0/1`（Before/After Char Defs）的条目走 PromptManager/RELATIVE 路径，作为固定位置的 system 消息，**不经过 depth 注入管线**，不进入聊天历史区。

### 17.5 flushWIInjections

每次 `getWorldInfoPrompt` 调用前执行，删除 `extension_prompts` 中所有 `customDepthWI` 和 `customWIOutlet_` 前缀的键，防止旧一轮条目残留叠加。

来源：openai.js + script.js + world-info.js release ｜ 适用版本：v1.18.0（本地为 v1.14.0，openai.js 未逐一回源）｜ 置信度：atDepth 四步流水线 / flushWIInjections 等**调用链** = `high [运行时源码]`；§17.2 depth 的 **splice+reverse 最终落位** = `medium·推导未验证`（推导链未真机，见 §21 U5/U6）

## 20. 工程实践与范式代码

### 20.1 随机事件（低概率 Probability）

```
Key: 长老神名字
Content: 长老神被唤醒，开始干预当前场景...
Probability: 1
Strategy: 🟢
```

每次关键词命中时有 1% 概率插入，实现随机事件触发。

### 20.2 Inclusion Group 回退序列

```
Entry_Specific: keys=[common_key, specific_key], group=scene_pool
  order=200, groupOverride=ON (Prioritize Inclusion)
Entry_Default:  keys=[common_key],               group=scene_pool
  order=100, groupOverride=ON

→ specific_key 也出现时：Entry_Specific (order=200) 确定性胜出
→ 只有 common_key 时：Entry_Default (order=100) 胜出
```

## 21. 悬而未决：重建前需真机验证的问题

> **本章 U1–U11 = `low / 悬案级`，禁止据此施工**；正文凡引用某 U 项的断言，其置信度已按铁规②不高于 `medium·推导未验证` 并交叉引用。

以下问题文档/源码未明确或存在矛盾，重建时遇到相关功能须先实测确认。

---

### U1. AND_ALL 激活门槛（高优先级）

**问题**：`getScore()` 函数（world-info.js）在 AND_ALL 且次键未全中时返回 `primaryScore`（非 0）。若激活判断仅依赖 `score > 0`，则次键未全中时主键命中也会激活——这与官方文档"必须全部次键都存在"相矛盾。

**验证路径**：设置 AND_ALL 条目（2 个次键，只满足 1 个），确认条目是否激活。若激活 = 有外层布尔门控；若不激活 = getScore 之外另有检查。

---

### U2. NOT_ANY / NOT_ALL 的完整实现位置（高优先级）

**问题**：`getScore()` switch-case 只处理 AND_ANY 和 AND_ALL，NOT_ANY/NOT_ALL 不进 switch。否定门控逻辑是在 getScore 外层还是另有函数？DeepWiki 提及 GitHub issue #2009 记录过 NOT_ANY 未按预期工作的历史 bug。

**验证路径**：设置 NOT_ANY 条目（主键+次键），分别测试次键存在/不存在时是否按文档预期激活。

---

### U3. Recursion Level 同层多条目行为（中优先级）

**问题**：同一层级（如 recursion_level=1）有多个条目时，是全部参与匹配还是只匹配最高优先级？某层有命中后是否完全跳过后续层，还是继续评估？官方文档仅有简略描述。

**验证路径**：在同一层级放 2 个条目，验证是否都能被触发；放 2 层后在第 1 层有命中时验证第 2 层是否被跳过。

---

### U4. Inclusion Group 竞争失败后 probability 补位（中优先级）

**问题**：Group 竞争（filterByInclusionGroups）发生在 probability check 之前。胜出条目被 probability 过滤掉后，其余同组条目是否有机会补位？

**当前源码倾向（参 §9.5，high）**：源码顺序里竞争淘汰在 probability 之前、且无重新竞争逻辑 → **倾向「不补位」**（胜出者被 probability 淘汰后该组本轮直接落空）。下方实测仅为最终确认。

**验证路径**：同组 2 条目，probability=1（近似 0%），重复触发多次观察是否偶有其他条目插入。

---

### U5. 同 depth 多 role 注入顺序（低优先级）

**问题**：同一 depth 下 system/user/assistant 三种 role 按 `['system','user','assistant']` 顺序 splice，reverse 后相对顺序的推导结论（assistant 比 system 更靠近最新消息）未真机实测。

**验证路径**：同 depth 各设一条 system 和 assistant WI 条目，用 Prompt Manager 预览确认实际顺序。

---

### U6. depth 超出聊天消息数时的行为（低优先级）

**问题**：openai.js 无 `Math.min` clamp，`Array.prototype.splice` 对超出长度的 index 静默追加到末尾；reverse 后这些超大 depth 条目可能落在 oldest 消息之前（最开头）。

**验证路径**：聊天只有 2 条消息时设 depth=10 的条目，验证其在 prompt 中的实际位置。

---

### U7. Automation ID post-generation 执行（中优先级）

**问题**：GitHub Issue #3419（2025-02-03）请求 post-generation Automation ID 执行，2026-06-27 状态未知。

**验证路径**：检查 AutoExecuteHandler.js 当前 release 分支是否有 `handleAfterGeneration` 类型的 WI 处理逻辑。

---

### U8. externalActivations 事件名拼写（中优先级）

**问题**：向量激活通过 `eventSource.emit(WORLDINFO_FORCE_ACTIVATE, ...)` 写入 externalActivations，事件名来自 DeepWiki 描述，未在截断代码中直接确认。

**验证路径**：在 vectors/index.js 完整源码中搜索 `emit` 调用，确认事件名拼写。

### U9. Outlet 宏在 STscript /genraw 中是否可用（低优先级）

**问题**：文档仅说 Outlet 宏在 Prompt Manager 和 Advanced Formatting 中可用，未明确 STscript 场景（如 `/genraw` 命令内）是否支持。
