## 1. 裁决、定义与反定义

### 1.1 本册研究的对象

**数据库兼容同层前端**是一个待验证的组合架构：

- [C5 同层前端](./C5_同层前端.md)负责把完整游玩体验集中在一个消息楼层；
- 独立消息生成 / 楼层桥负责创建、读取和恢复真实聊天楼层；
- [C7 数据库二创前端](./C7_数据库二创前端.md)负责把数据库状态做成作品 UI；
- [C4 数据库表格模板](./C4_数据库表格模板.md)负责表结构、填表协议、持久化和公开 API；
- 本册只研究这几条链如何额外协调，不改变任何一册的职责。

目标不是让数据库“变成同层引擎”，而是让同层应用在保留真实消息链的同时，安全消费数据库状态。

### 1.2 默认裁决

默认结论必须写清楚：

1. **SP·数据库常规用于正常多楼层角色卡。**
2. **数据库不产出楼层信息。**
3. **数据库不产出同层完整应用所需的剧情正文、选项和路由。**
4. **数据库更新回调不等于“新 assistant 楼已完整提交”。**
5. **数据库默认不兼容 C5。**
6. 兼容同层需要另一条独立消息 / 楼层桥路线，且当前待验证。

这不是措辞差异，而是责任边界。若省略这些前提，后续实现很容易把表格变化误判成完整回合。

### 1.3 本册不是什么

本册不是：

- 不是“把 C7 放进首楼”就自动完成的部署教程；
- 不是 SP·数据库官方支持同层的证明；
- 不是让 `AutoCardUpdaterAPI` 返回剧情正文或消息 ID 的 API 设计；
- 不是用数据库表格重建一套伪聊天记录；
- 不是用状态字段代替 SillyTavern 的真实消息、Swipe 和分支；
- 不是让数据库负责调用模型、创建楼层或管理路由；
- 不是已验证的 DC 成品案例清单；
- 不是对任何未公开作品内部实现的推测；
- 不是当前可直接用于生产角色卡的成熟方案。

### 1.4 与 C5 的关系

C5 的功能和量级远高于开局页、状态栏或数据库面板。它要在单个可见楼层内闭合：

- 玩家输入；
- 正常 AI 生成；
- 剧情正文阅读；
- 选项与自由行动；
- Swipe、重生成和历史回看；
- 状态、背包、任务、地图等游戏系统；
- 路由、设置、恢复和异常降级。

数据库只覆盖其中的“结构化游戏状态”部分。即使 C7 已经有完整的状态栏和控制中心，也没有因此获得 C5 的楼层生成、剧情镜像和消息生命周期能力。

## 2. 为什么数据库默认不兼容同层

### 2.1 正常多楼层的常规链路

SP·数据库的常规角色可以抽象为：

```text
玩家在正常聊天中发送消息
→ SillyTavern 创建真实 user 楼
→ 主模型生成真实 assistant 回复
→ SP·数据库监听 / 消费该回复
→ 解析填表指令或执行对应更新
→ 表格状态持久化
→ 普通消息状态栏或 C7 前端读取新状态
```

这里的剧情正文和楼层身份来自 SillyTavern 的正常消息链，不来自数据库。

### 2.2 C5 额外要求的链路

C5 把后续真实消息隐藏在统一首楼体验之后，但仍需可靠取得：

```text
本回合玩家输入
本回合 user 楼 ID
本回合 assistant 楼 ID
当前激活 Swipe
完整且稳定的 assistant 正文
生成中 / 停止 / 失败状态
本回合对应的数据库更新状态
首楼应该提交的组合视图
```

这些字段不能从数据库快照推导。表格中“体力减少 5”并不能告诉首楼：

- 哪个楼层触发了变化；
- 对应哪次 Swipe；
- 剧情正文是什么；
- 回复是否仍在流式生成；
- 用户是否刚刚编辑或删除了相关楼层；
- 表格变化属于 AI 填表、玩家手动 CRUD、导入还是聊天恢复。

### 2.3 状态更新不等于回合提交

数据库更新可能来自多种来源：

| 更新来源 | 是否一定有新 assistant 楼 | 是否一定有新剧情正文 |
|---|---:|---:|
| 正常 AI 回复后的自动填表 | 通常有，但仍需真机确认时序 | 有，正文属于消息层 |
| C7 前端手动 CRUD | 否 | 否 |
| 导入数据库数据 | 否 | 否 |
| 切聊天后的数据装载 | 否 | 否 |
| 恢复 / 重放日志 | 否 | 否 |
| SQLite 批量写入 | 否 | 否 |

因此，`registerTableUpdateCallback()` 只能说明“表发生了可观察更新”，不能单独作为“同层新回合完成”的信号。

### 2.4 无变化填表也是回合

某一轮剧情可能没有改变任何表格。若同层应用只等数据库更新回调：

- assistant 楼已经生成；
- 剧情正文已经可读；
- 数据库快照与上一轮相同；
- 数据库可能没有产生同层前端期待的更新通知；
- 首楼会永久停在“等待状态”。

所以“收到数据库更新”不能成为唯一提交条件，“没有数据库变化”也不能被当成生成失败。

## 3. 缺失的输出与关联契约

### 3.1 数据库不提供的核心契约

截至本册列出的两个已核源码快照，不能把公开数据库 API 当成以下信息的来源：

| 同层需要的信息 | 数据库是否负责 | 正确责任层 |
|---|---:|---|
| 新 assistant 消息 ID | 否 | 消息 / 楼层桥 |
| 当前激活 Swipe ID 或索引 | 否 | 消息 / 楼层桥 |
| 完整剧情正文 | 否 | 真实消息层 |
| 流式正文是否结束 | 否 | 生成 / 消息事件层 |
| 本次动作对应哪个 user 楼 | 否 | 消息 / 楼层桥 |
| 路由应该进入哪个剧情页面 | 否 | 同层应用 |
| 选项属于哪个正文版本 | 否 | 同层应用 + 消息 / 楼层桥 |
| 表格快照 | 是 | SP·数据库 |
| 表格 CRUD 结果 | 是 | SP·数据库公开 API |
| 表格是否发生更新 | 部分可观察 | 数据库更新回调 |

**数据库不产出楼层信息**，也没有义务为每个表格提交附带消息 ID。

### 3.2 当前缺少的跨层关联键

理想接口会提供类似：

```typescript
interface HypotheticalTurnCommit {
  chatId: string;
  userMessageId: number;
  assistantMessageId: number;
  swipeId: number;
  generationId: string;
  databaseRevision: string;
}
```

这只是用于说明缺口的假想类型，**不是现有 API**。不得把它复制进项目并声称插件会返回这些字段。

当前实验必须自己维护本地 `turnToken`，并通过消息层证据和数据库快照变化做关联。关联是否足够稳定，是本册最关键的待验证项目。

### 3.3 填表开始不等于填表结束

已核快照存在填表开始回调，但不能据此虚构公开的 fill-end 事件。实验实现必须承认：

- fill-start 只能证明数据库开始某段填表流程；
- table-update 可能由非填表写入触发；
- 无变化填表未必带来可观察的数据差异；
- 多次表更新可能属于同一轮；
- 回调先后与消息楼层可读时点仍待真机验证。

在没有可靠结束契约之前，“数据库本回合已稳定”只能是带超时和重读的推导状态，不能包装成确定事实。

## 4. 实验架构与责任分层

### 4.1 五层责任图

```text
┌──────────────────────────────────────────────┐
│ 首楼应用 C5                                  │
│ 输入、剧情、选项、路由、回看、错误与降级 UI  │
└───────────────────┬──────────────────────────┘
                    │ action / composite snapshot
┌───────────────────▼──────────────────────────┐
│ 回合协调器 Turn Coordinator（本册实验层）     │
│ turnToken、状态机、时序门、超时、恢复、降级   │
└──────────────┬──────────────────┬────────────┘
               │                  │
┌──────────────▼────────────┐ ┌───▼──────────────────────┐
│ 消息生成 / 楼层桥          │ │ 数据库适配器             │
│ 真实 user/assistant 楼、   │ │ 快照、订阅、CRUD、       │
│ 正文、Swipe、生成状态       │ │ revision hint、表契约     │
└──────────────┬────────────┘ └───┬──────────────────────┘
               │                  │
┌──────────────▼──────────────────▼────────────┐
│ SillyTavern 真实聊天 + SP·数据库持久状态      │
└──────────────────────────────────────────────┘
```

### 4.2 消息生成 / 楼层桥

楼层桥必须独立承担：

- 把玩家 action 写入正常消息链；
- 触发正常主模型生成；
- 等待目标 assistant 楼成为真实可读消息；
- 读取当前激活 Swipe 的正文；
- 区分流式中、完成、停止和失败；
- 监听消息新增、更新、删除和聊天切换；
- 在首楼 iframe 重绘后恢复当前回合；
- 把楼层信息交给协调器，而不是写入数据库冒充状态。

具体楼层 API 与显示策略以 C5 和目标版本类型声明为准。本册不复制一套未经真机核验的固定事件名。

### 4.3 数据库适配器

数据库适配器只承担：

- 运行时发现 `AutoCardUpdaterAPI`；
- 读取并 clone 当前数据库快照；
- 归一化 `mate / sheet_* / content`；
- 订阅数据库更新；
- 执行受支持的 CRUD / SQL；
- 记录可比较的快照指纹或插件 revision 信息；
- 在聊天切换时清空旧引用并重新读取；
- 把“数据更新证据”交给协调器。

它不能返回虚构的楼层 ID，也不能把“数据库已更新”转换成“剧情已完成”。

### 4.4 回合协调器

协调器是本册新增的实验责任层：

```typescript
interface ExperimentalTurnToken {
  epoch: number;               // 每次切聊天 / 恢复递增
  localTurnId: string;         // 本应用生成，不是插件返回值
  startedAt: number;
  userMessageId?: number;
  assistantMessageId?: number;
  swipeIndex?: number;
  floorReadable: boolean;
  generationSettled: boolean;
  dbObserved: 'unknown' | 'filling' | 'changed' | 'unchanged-timeout';
}
```

这段类型是实验应用自己的状态，不代表数据库公开协议。`epoch` 用来让旧异步结果失效，避免切聊天后把旧回合提交到新聊天。

### 4.5 首楼应用

首楼只消费协调器给出的组合快照：

```typescript
interface ExperimentalCompositeSnapshot {
  turn: ExperimentalTurnToken;
  narrative: {
    messageId: number;
    swipeIndex: number;
    text: string;
  };
  gameState: unknown;          // 数据库 adapter 的归一化结果
  consistency: 'observed' | 'timed-out' | 'degraded';
}
```

首楼不得自己并行猜测消息和数据库状态，否则同一 UI 会出现两个互不一致的提交路径。

## 5. 优先路线：真实楼层先生成，再镜像到首楼

### 5.1 为什么优先真实楼层

优先路线必须保留 SillyTavern 正常消息链：

```text
首楼 action
→ 楼层桥创建真实 user 楼并触发正常生成
→ SillyTavern 创建真实 assistant 楼
→ 楼层桥确认正文可读且生成结束
→ SP·数据库按正常方式消费 AI 回复并填表
→ 数据库适配器观察更新或超时重读
→ 协调器形成组合快照
→ 首楼提交剧情 + 状态 UI
```

优势：

- 不绕过角色卡、世界书、预设与正常提示词链；
- 真实 assistant 楼仍是剧情权威来源；
- Swipe、历史、聊天存档仍有宿主语义；
- SP·数据库仍在它熟悉的多楼层消息链上工作；
- 同层只是显示和交互层，不伪造数据库能力。

### 5.2 “先生成”指责任顺序，不预判事件顺序

“真实楼层先生成，再镜像”表示首楼必须先拿到真实楼层证据，不能表示所有目标版本一定按固定顺序发出事件。

真机中可能观察到：

- assistant 消息先出现、正文随后流式更新；
- 数据库 fill-start 与消息完成接近；
- 一次填表触发多次表更新；
- 表更新早于某个楼层 UI 的最终重绘；
- 无表变化时只有消息完成证据。

因此协调器必须收集两条链的证据，而不是把某个回调排在代码前面就当成宿主保证。

### 5.3 最小提交门

建议实验提交条件：

```text
必须满足：
1. epoch 仍属于当前聊天
2. assistantMessageId 已确定
3. 当前 Swipe 正文可读
4. 正常生成已停止或确认完成

数据库侧满足其一：
A. 观察到本轮开始后的数据库变化，并在短稳定窗内不再变化
B. 超过有上限的等待窗后重读快照，标记 unchanged-timeout

然后：
提交组合快照，并把 consistency 明确显示为 observed 或 timed-out
```

这个门仍是待验证算法。稳定窗和超时值必须通过实际卡、模型和数据库版本测量，不能在指南里宣称存在通用常数。

### 5.4 不要阻塞剧情到无限等待

数据库更新慢、无变化或失败时，首楼不能永远不显示已完成的真实剧情。达到有界等待后应：

1. 先展示真实剧情；
2. 状态区显示“数据库状态仍在核对”；
3. 后续若观察到安全的新快照，再做一次状态补提交；
4. 无法确认归属时停止自动合并并进入降级；
5. 保留返回正常多楼层聊天的入口。

## 6. 其他候选路线及风险

### 6.1 路线 B：截取流式文本直接渲染

设想：生成时把流式片段直接镜像到首楼，完成后再确认真实楼层。

风险：

- 流式文本不是最终 Swipe；
- 中止、重试或宿主修订会让首楼残留旧片段；
- 数据库可能只在最终文本阶段处理；
- 片段与表格更新没有可靠关联；
- 恶意或未闭合 HTML 的增量渲染风险更高。

结论：可以作为视觉优化实验，但不能代替真实楼层提交门，待验证。

### 6.2 路线 C：`generate()` / `generateRaw()` 后手动造楼

设想：在首楼直接调用额外生成接口，拿到文本后手动创建 assistant 楼，再让数据库处理。

风险：

- 可能绕过正常主回合的提示词、事件或消息语义；
- 数据库是否会消费这种后建楼层取决于目标实现；
- Swipe、停止生成、聊天保存和错误恢复都要自行重建；
- 很容易形成“文本生成成功但数据库没填表”的分叉。

结论：不是默认路线。只有在目标版本逐项验证等价性后，才可保留为专用适配，不得推广成通用方案。

### 6.3 路线 D：让数据库表格保存剧情正文

设想：把每轮正文、选项和路由都写进数据库，再由首楼读取。

风险：

- 把状态表变成第二套聊天历史；
- 与真实消息、Swipe、编辑、删楼产生双事实源；
- 长正文进入表格会恶化 token、性能和迁移；
- 表格仍然没有宿主的楼层与分支语义。

结论：禁止作为兼容方案。数据库存业务状态，不存替代真实聊天的完整事件流水。

### 6.4 路线 E：只凭数据库更新触发“新回合”

设想：每次 `registerTableUpdateCallback()` 触发就向首楼追加一轮剧情卡片。

风险：

- 回调不携带剧情正文；
- 同一轮可能多次更新；
- 手动 CRUD 会被误判为新回合；
- 无变化回合完全丢失；
- 切聊天装载可能生成假回合。

结论：架构错误，不进入真机实验。

### 6.5 路线 F：数据库解析完成后再找“最后一楼”

设想：收到表更新后读取聊天尾部，把最后一个 assistant 楼当成本轮正文。

风险：

- 并发动作、重生成和聊天切换会改变尾部；
- 延迟更新可能对应更早一轮；
- 手动 CRUD 没有对应楼层；
- 多 tab 或重复触发会产生竞态。

结论：只可作为诊断对照，不能作为生产关联算法。

## 7. 回合关联与组合快照

### 7.1 本地 `turnToken` 的作用

协调器发起玩家 action 时，立即创建本地 token：

```typescript
function beginExperimentalTurn(epoch: number): ExperimentalTurnToken {
  return {
    epoch,
    localTurnId: crypto.randomUUID(),
    startedAt: Date.now(),
    floorReadable: false,
    generationSettled: false,
    dbObserved: 'unknown',
  };
}
```

`localTurnId` 只用于应用内部去重和日志。它不能注入数据库后再声称成为插件事务 ID。

### 7.2 必须记录的证据

每轮至少记录：

- 聊天 epoch；
- action 创建时间；
- user 楼 ID；
- assistant 楼 ID；
- 当前 Swipe；
- 消息最后更新时间或内容指纹；
- 生成开始、停止、失败时点；
- 数据库 fill-start 时点（若可观察）；
- 数据库 update 回调次数；
- 更新前后快照指纹；
- 最终提交状态与超时原因。

日志中不得输出完整私密正文或完整数据库快照；使用 ID、计数、表名、哈希和耗时即可。

### 7.3 组合快照不是新事实源

组合快照只是一份渲染投影：

```text
narrative 仍由真实 assistant 楼决定
gameState 仍由 SP·数据库决定
turnToken 只记录本次协调证据
UI router 只记录本机呈现状态
```

刷新后应从真实消息和数据库重新构建。不能只靠内存中的组合快照恢复全部游戏。

### 7.4 快照指纹只用于观察

可以对 clone 后的归一化数据库快照计算稳定哈希，用来判断“看起来是否变化”。但它：

- 不是数据库 revision；
- 不能证明没有内部提交；
- 不能证明变化属于当前楼层；
- 不能代替公开 CRUD 返回值；
- 不能代替 V2 日志的真实 revision 语义。

哈希只用于实验观测和 UI 去重。

### 7.5 同一轮多次数据库更新

一轮可能更新多个表或分批提交。协调器应：

1. 把同一观察窗内的回调合并为 dirty 标记；
2. 每次都从公开 API 重读新快照；
3. 不直接使用回调给出的活动对象作为 UI 可变 store；
4. 在短稳定窗结束后才标记 `observed`；
5. 后续迟到更新无法确定归属时，显示状态修订，而不是重放剧情。

稳定窗策略仍待验证。

## 8. 事件、状态机与提交条件

### 8.1 实验状态机

```text
IDLE
  │ 玩家 action
  ▼
ACTION_QUEUED
  │ 真实 user 楼已创建
  ▼
GENERATING
  │ assistant 楼可读
  ▼
FLOOR_READABLE
  │ 生成已停止 / 完成
  ▼
WAITING_DATABASE
  ├─ 观察到更新 ──> DB_RECONCILING
  ├─ 有界超时 ────> DB_UNCHANGED_TIMEOUT
  └─ 错误 ────────> DEGRADED

DB_RECONCILING / DB_UNCHANGED_TIMEOUT
  │ 组合快照提交
  ▼
READY

任意状态：切聊天 / epoch 变化 → CANCELLED → 恢复新聊天
任意状态：证据冲突 → DEGRADED → 显示正常多楼层入口
```

### 8.2 状态含义

| 状态 | 含义 | 首楼允许做什么 |
|---|---|---|
| `IDLE` | 没有活动回合 | 可提交 action |
| `ACTION_QUEUED` | action 已锁定，尚未确认 user 楼 | 禁止重复提交 |
| `GENERATING` | 正常主模型生成中 | 显示进度，可停止 |
| `FLOOR_READABLE` | assistant 楼存在，但生成状态仍需确认 | 可预览，不最终提交 |
| `WAITING_DATABASE` | 剧情已稳定，数据库证据未收齐 | 显示剧情，状态区提示核对中 |
| `DB_RECONCILING` | 观察到表变化，等待短稳定窗 | 只更新状态草稿 |
| `DB_UNCHANGED_TIMEOUT` | 没观察到变化，已重读快照 | 允许带标记提交 |
| `READY` | 组合视图已提交 | 可进入下一 action |
| `DEGRADED` | 证据冲突或关键能力缺失 | 恢复正常多楼层显示 |
| `CANCELLED` | 旧 epoch 的异步任务失效 | 禁止写入当前 UI |

### 8.3 并发门

一个聊天 epoch 默认只允许一个活动的剧情回合 token。以下行为必须被拒绝或排队：

- 生成中再次点选项；
- fill-start 后又提交第二个 action；
- 旧 assistant 楼尚未确认就开始重生成；
- 切聊天过程中继续写回旧数据库；
- 首楼重绘导致重复绑定后再次发送同一 action。

若作品确实需要并行生成，必须为每条生成链建立宿主可验证的独立关联键；当前本册不声称已有这种能力。

### 8.4 迟到事件

每个异步回调在提交前都检查：

```typescript
function isCurrentTurn(epoch: number, localTurnId: string): boolean {
  return coordinator.epoch === epoch
    && coordinator.activeTurn?.localTurnId === localTurnId;
}
```

旧 epoch、旧 token 或已取消任务的结果只能写诊断日志，不能更新首楼 store。

## 9. 数据库回调、竞态与生命周期

### 9.1 更新回调的正确定位

数据库更新回调适合：

- 标记数据库 store dirty；
- 触发一次去抖后的权威重读；
- 更新 C7 状态 UI；
- 给协调器增加“数据库发生变化”的证据。

不适合：

- 创建 assistant 楼；
- 推断剧情正文；
- 推断当前 Swipe；
- 宣告生成结束；
- 自动把最后一楼绑定到当前 token；
- 直接触发下一轮剧情。

### 9.2 同一函数引用注销

如果目标版本提供 `registerTableUpdateCallback()` / `unregisterTableUpdateCallback()`，必须保留同一个函数引用：

```typescript
const onTableUpdate = () => {
  coordinator.markDatabaseDirty();
};

api.registerTableUpdateCallback(onTableUpdate);

function disposeDatabaseSubscription() {
  api.unregisterTableUpdateCallback(onTableUpdate);
}
```

首楼 iframe 会重绘。每次 mount 创建新匿名函数却不注销，会造成重复回调、重复重读和错误的“多次填表”证据。

### 9.3 fill-start 的泄漏风险

在已核 `spv5.5.6` 快照中可见公开 fill-start 注册入口，但没有与之对称的公开注销入口。若目标安装版本仍是这种能力形状：

- 不要让短命首楼 iframe 每次 mount 注册新闭包；
- 把 fill-start 监听收口到常驻脚本 bridge；
- bridge 内只注册一次，再向首楼发布自有可注销事件；
- 或完全不依赖 fill-start，只用消息状态与表更新做实验观测；
- 切聊天时清空 bridge 内的当前 token，不重复注册插件回调。

升级版本可能改变该能力，必须运行时检测并对实际安装版本真机复核。

### 9.4 没有公开 fill-end 时怎么办

不得发明 `registerTableFillEndCallback()`。缺少可靠结束信号时只能：

1. 以消息生成结束作为剧情链完成证据；
2. 用数据库 update 回调触发重读；
3. 用有上限的稳定窗观察是否继续变化；
4. 超时后带 `timed-out` 标记提交；
5. 后续迟到变化作为状态修订，不重复追加剧情；
6. 把实际回调序列记录进真机报告。

这仍是待验证折中，不是强一致事务。

### 9.5 手动 CRUD 与自动填表冲突

当数据库正在核对本回合更新时，C7 的手动编辑应：

- 默认禁用对同一业务记录的提交；
- 或把草稿排到回合协调完成后；
- 提交前重读最新快照并按业务键重新定位行；
- 检查公开 CRUD 返回值；
- 写后再次读取权威快照；
- 冲突时显示明确错误，不静默覆盖。

数据库 UI 锁不是跨层事务锁，不能解决剧情生成、AI 填表和手动 CRUD 的全部竞态。

## 10. Swipe、重生成、编辑、删楼与回滚

### 10.1 Swipe 是独立剧情版本

切换 Swipe 后，首楼必须至少重新读取：

- assistant 消息 ID；
- 当前 Swipe 索引；
- 当前正文；
- 该楼之后的有效消息链；
- 当前数据库快照；
- 数据库是否已随 Swipe 回滚或重算。

不能只切换首楼的文本标签。数据库能否自动回到与该 Swipe 对应的状态、何时完成、V2 日志怎样映射，必须逐目标版本真机验证。

### 10.2 重生成

重生成可能保持消息位置但产生新 Swipe，也可能触发不同事件序列。实验实现必须：

1. 取消旧 token 的待提交任务；
2. 创建新的 `localTurnId`；
3. 记录重生成前数据库指纹；
4. 等待新 Swipe 真实可读；
5. 重新观察数据库更新；
6. 无法证明数据库与新 Swipe 一致时进入降级。

禁止沿用上一 Swipe 的数据库“已观察”标志。

### 10.3 编辑 user 或 assistant 楼

编辑历史消息会改变上下文，但不保证数据库自动重放。需要分别验证：

- 编辑 user 楼是否触发任何数据库动作；
- 编辑 assistant 正文是否重新解析填表；
- 历史编辑是否使后续数据库状态过期；
- 数据库是否有恢复到某 revision 的公开、安全路径；
- 首楼怎样标记“剧情和状态可能不一致”。

在答案不明确前，历史编辑后应停止同层自动状态合并并显示正常聊天 / 数据库检查入口。

### 10.4 删除楼层

删楼后必须假设以下内容都可能失效：

- 当前活动 token；
- 已缓存 message ID；
- Swipe 选择；
- 后续剧情镜像；
- 数据库与消息链的对应关系。

首楼不能只从回看列表移除一条记录。它应取消活动任务、重读整段消息链、重读数据库，并在无法证明回滚一致时进入降级。

### 10.5 回滚与 checkpoint

数据库自身的 V2 checkpoint / log 语义由 C4 负责。本册不得：

- 通过前后快照 diff 伪造数据库 operation；
- 把 UI 的“返回上一页”当数据库回滚；
- 用导入旧 JSON 冒充与 Swipe 对齐的事务回退；
- 在不清楚 revision 父子关系时重放写入。

真正的“消息分支与数据库 revision 对齐”是 C8 转正前必须解决的待验证核心。

### 10.6 失败后的用户选择

发生 Swipe / 编辑 / 删楼冲突时，至少给用户三种可理解的操作：

1. 返回正常多楼层聊天检查真实消息；
2. 打开数据库原生或 C7 检查界面确认状态；
3. 放弃本次同层镜像并从已确认节点重新开始。

不得只弹出“未知错误”后继续自动写表。

## 11. 切聊天、刷新与恢复

### 11.1 聊天 epoch

切聊天时必须原子地：

1. `epoch += 1`；
2. 取消旧聊天的 active turn；
3. 清空旧消息缓存；
4. 丢弃旧数据库对象引用；
5. 等待新聊天消息链可读；
6. 等待新聊天数据库数据就绪；
7. 分别重建 narrative 和 gameState；
8. 完成健康检查后再决定是否启用同层显示。

旧回调迟到时因为 epoch 不匹配而被丢弃。

### 11.2 首楼 iframe 重绘

首楼 iframe 不是常驻进程。重绘后：

- 不重新向插件注册无法注销的 fill-start 闭包；
- 从常驻 bridge 获取当前 epoch 和活动回合摘要；
- 重新读取真实消息，而不是相信旧 DOM；
- 重新 clone 数据库快照；
- 恢复 UI router 之前先验证对应聊天 ID；
- action 按钮直到恢复完成前保持禁用。

### 11.3 页面刷新

页面刷新后内存 token 会丢失。恢复逻辑不能假装知道刷新前某个 fill-start 属于哪一楼：

- 若没有持久的、可验证的回合关联证据，标记为 `recovered-uncertain`；
- 读取当前真实聊天尾部和数据库快照；
- 不自动补做 CRUD；
- 不自动重触发生成；
- 让用户选择继续同层观察或回到正常多楼层。

### 11.4 新聊天与空数据库

新聊天可能先出现首楼，而数据库仍未装载或尚无表数据。启动必须区分：

- API 尚未出现；
- API 已出现但数据未就绪；
- 合法空表；
- 模板尚未导入；
- 数据库加载失败；
- 当前插件版本没有所需方法。

这些状态不能统一显示成“没有角色数据”。

## 12. 正常多楼层降级与止损

### 12.1 降级原则

兼容同层失败时，系统必须回到数据库本来支持的正常多楼层玩法，而不是让整张卡不可用：

```text
同层健康
→ 首楼统一呈现，真实楼层仅做可逆显示隐藏

同层不健康
→ 解除显示隐藏
→ 恢复正常消息流
→ 保留普通 C7 状态栏 / 控制中心
→ 禁止协调器继续自动提交
```

显示隐藏不得修改提示词语义，不得把真实消息永久标成不参与上下文。

### 12.2 触发降级的条件

任一条件成立即降级：

- 无法发现消息 / 楼层桥；
- 数据库公开 API 缺少必要能力；
- assistant 楼在有界时间内不可读；
- 聊天 epoch 在回合中变化；
- 收到无法归属的迟到数据库更新；
- Swipe 后数据库一致性无法确认；
- 回调重复注册或次数异常；
- 首楼 mount 失败；
- 数据库写入返回失败；
- 真实消息和首楼镜像内容指纹不一致；
- 用户主动选择“返回正常聊天”。

### 12.3 失败时禁止的动作

进入降级后禁止：

- 自动重发玩家消息；
- 自动重新生成 assistant 回复；
- 自动导入旧数据库快照；
- 猜测最后一楼并继续填表；
- 隐藏错误并把状态标成 READY；
- 在未知聊天上继续执行旧 CRUD。

### 12.4 工程止损

本路线若在同一核心问题上连续调试 2–3 轮仍无法稳定复现，应暂停继续堆补丁，做架构复审：

- 宿主是否能提供足够的消息完成证据；
- 数据库是否有可用的提交 / revision 关联信息；
- 是否应保留正常多楼层 + C7，而砍掉同层兼容；
- 是否需要插件上游新增正式桥接契约；
- 是否应把实验重写成只读镜像，而不是可交互完整 C5。

未解决关联键和回滚前，不得因 UI 看起来能跑就转正。

## 13. 版本政策与运行时能力检测

### 13.1 不写死插件基线

SP·数据库会持续更新。本册不规定“只有某个固定版本才是基线”，也不承诺未来版本保留当前方法和时序。

截至 2026-07-14，本册可列出的只是两份**已核源码快照**：

| 快照 | commit | 用途 |
|---|---|---|
| 发布标签 `spv5.5.6` | `2020b14c0672e74409a3d98c8a9899e13069ca3f` | 对照公开 API 与历史实现 |
| 2026-07-14 所见 upstream `main` | `708e66ba58dfe0a08d34cba7cb8730a8f8ff1e80` | 检查主线相对发布标签的能力变化 |

这两项不是永久基线，也不等于“已验证支持 C8”。实际施工必须以用户当前安装版本为准。

### 13.2 启动时能力检测

示意：

```typescript
function inspectDatabaseCapabilities(api: unknown) {
  const x = api as Record<string, unknown> | null;
  return {
    exportSnapshot: typeof x?.exportTableAsJson === 'function',
    subscribeUpdate: typeof x?.registerTableUpdateCallback === 'function',
    unsubscribeUpdate: typeof x?.unregisterTableUpdateCallback === 'function',
    subscribeFillStart: typeof x?.registerTableFillStartCallback === 'function',
  };
}
```

能力检测只回答“方法是否存在”，不证明：

- 签名没有变化；
- 回调时序适合当前协调算法；
- 方法在当前 provider 模式行为一致；
- 消息 iframe 可以直接访问宿主 API；
- 同层兼容已经验证。

### 13.3 记录实际环境

每份真机报告至少记录：

- SillyTavern 版本；
- Tavern Helper / 酒馆助手版本；
- SP·数据库实际安装来源、标签或 commit；
- 原生模式或 SQLite 模式；
- 角色卡与数据库模板版本；
- 浏览器 / WebView 与设备；
- 是否使用流式；
- 模型来源与是否经过代理；
- 首楼构建版本、bridge 协议版本；
- 实际检测到的 API 能力表。

不能只写“最新版”。

### 13.4 版本不匹配策略

若运行时能力与本册实验预期不一致：

1. 不继续猜 API 名；
2. 不读取插件内部全局变量补洞；
3. 关闭同层协调器；
4. 恢复正常多楼层玩法；
5. 保留只读诊断信息；
6. 对照实际版本源码后再更新 C8 文档版本。

## 14. 最小实验原型

### 14.1 P0：只读双源观测器

第一阶段禁止隐藏正常消息流，只做：

- 首楼显示当前真实 assistant 楼 ID、Swipe 和正文摘要；
- 显示数据库快照指纹、表数量和更新时间；
- 记录消息事件与数据库回调时间线；
- 手动生成仍从正常 ST UI 发起；
- 不执行数据库写入；
- 不宣称回合已经关联。

验收目标：证明两条链都能稳定观察，且刷新、切聊天不串数据。

### 14.2 P1：首楼发起正常回合

在 P0 稳定后增加：

- 首楼 action → 真实 user 楼；
- 正常主模型生成；
- assistant 楼真实可读；
- 有界等待数据库；
- 组合快照只显示，不隐藏原生楼层；
- 所有 action 有幂等键和重复提交保护。

验收目标：同一 action 只产生一条预期消息链，数据库无变化回合不会卡死。

### 14.3 P2：健康门控下的可逆显示模式

只有 P1 通过后才增加：

- 首楼健康时隐藏普通楼层的显示；
- 真实楼层仍参与上下文与存档；
- 任一健康检查失败立即解除隐藏；
- 用户始终有显式“显示正常聊天”按钮；
- 刷新后默认先显示正常聊天，健康确认后再切同层。

验收目标：显示适配失败不影响角色卡继续游玩。

### 14.4 P3：高风险生命周期

最后才验证：

- Swipe；
- 重生成；
- 编辑历史消息；
- 删除楼层；
- 数据库回滚 / 恢复；
- 切聊天；
- 长聊天；
- 移动端；
- 原生 / SQLite 双模式。

P3 任一主链无法证明一致性，C8 继续保持待验证状态。

### 14.5 原型不包含什么

最小原型阶段不要同时开发：

- 大型地图和复杂动画；
- 多套皮肤；
- 跨设备云同步；
- 批量数据库编辑器；
- 第二 API；
- 多模型并行剧情；
- 自定义数据库持久化替代方案。

先证明“真实楼层与数据库状态能安全协调”，再扩展产品功能。

### 15.2 组件不得直接碰两套 API

UI 组件只能读取组合 store、发送 action。禁止一个组件同时：

- 调用楼层 API；
- 读取 `AutoCardUpdaterAPI`；
- 修改 router；
- 猜测生成结束；
- 写本地恢复状态。

否则任何一次重绘都可能产生重复监听和交叉写入。

### 15.3 诊断时间线

建议统一记录：

```typescript
interface DiagnosticEvent {
  at: number;
  epoch: number;
  localTurnId?: string;
  source: 'shell' | 'chat' | 'generation' | 'database' | 'coordinator';
  type: string;
  messageId?: number;
  swipeIndex?: number;
  snapshotHash?: string;
  detail?: string;
}
```

需支持导出脱敏时间线，以便判断真实顺序，而不是靠 UI 观感猜测。

### 15.4 必备不变量

测试至少断言：

1. 同一 `localTurnId` 最多提交一份最终剧情；
2. 旧 epoch 永远不能更新新聊天 store；
3. 每个更新回调注册都有可证明的单例或注销路径；
4. 未确认 assistant 楼时不能标记 READY；
5. 数据库回调不能创建剧情楼；
6. 超时必须有上限；
7. 降级必须恢复正常多楼层显示；
8. 手动 CRUD 不得被识别为新剧情回合；
9. 无数据库变化的回合仍能展示真实剧情；
10. 刷新恢复不能自动重发 action。

### 15.5 安全边界

- AI 正文、数据库单元格和错误信息都按不可信输入渲染；
- 禁止把正文或表格值直接交给 `innerHTML` / `v-html`；
- 远程前端资源使用可审计版本和失败壳；
- 诊断导出默认脱敏角色名、正文和用户输入；
- bridge 对 shell 暴露最小方法，不透传整个宿主窗口；
- 写操作必须检查聊天 epoch 和业务权限；
- 错误降级不应删除、编辑或隐藏真实数据。

## 16. 真机验证矩阵

### 16.1 基础回合

| 编号 | 场景 | 必查证据 | 通过条件 |
|---|---|---|---|
| B01 | 新聊天第一轮 | user/assistant 楼、生成状态、数据库时间线 | 只提交一次，剧情与状态可读 |
| B02 | 连续三轮 | 每轮 token、楼层 ID、快照指纹 | 不串轮、不重复、不丢轮 |
| B03 | 数据库有变化 | update 次数与重读结果 | 状态最终与数据库一致 |
| B04 | 数据库无变化 | assistant 楼与超时路径 | 剧情正常显示，不无限等待 |
| B05 | 自动填表失败 | 插件错误与首楼降级 | 剧情保留，状态明确报错 |
| B06 | 生成中止 | 消息内容、停止事件、token 状态 | 不把半成品标记 READY |
| B07 | 模型报错 | 无 assistant 楼或错误楼 | 不写假剧情、不重复发送 |

### 16.2 数据库写入与回调

| 编号 | 场景 | 必查证据 | 通过条件 |
|---|---|---|---|
| D01 | 单表多字段更新 | 回调次数、最终快照 | 去抖后读取正确值 |
| D02 | 多表连续更新 | 时间线与稳定窗 | 不把一次回合拆成多轮剧情 |
| D03 | C7 手动 CRUD | 消息链与数据库链 | 不产生假 assistant 回合 |
| D04 | 数据导入 | chat epoch、加载事件 | 不追加假剧情，不泄漏旧快照 |

| D06 | fill-start 多次 | 注册次数与回调次数 | 无 mount 累积泄漏 |
| D07 | 迟到数据库更新 | token/epoch 检查 | 不覆盖新聊天或新回合 |

### 16.3 消息生命周期

| 编号 | 场景 | 必查证据 | 通过条件 |
|---|---|---|---|
| M01 | 当前楼 Swipe | Swipe、正文、数据库状态 | 无法确认一致时降级 |
| M02 | 历史楼 Swipe | 下游消息链与数据库 | 不静默保留错误状态 |
| M03 | 重生成 | 新 token、新 Swipe、旧任务取消 | 旧结果不能迟到提交 |
| M04 | 编辑 user 楼 | 后续上下文与数据库 | 明确标记需核对或降级 |
| M05 | 编辑 assistant 楼 | 填表是否重跑 | 行为有证据，不做假设 |
| M06 | 删除尾楼 | 当前 token、数据库 | 重新读取，不只改 DOM |
| M07 | 删除中间楼 | 分支、后续楼、数据库 | 无法证明一致则停止同层 |
| M08 | 原生回滚操作 | 消息和数据库 revision | 两条链一致或明确失败 |

| 编号 | 场景 | 必查证据 | 通过条件 |
|---|---|---|---|

## 17. 转正门槛

### 17.1 文档状态门

C8 当前是 `v0.1.0`、待验证。只有同时满足以下条件，才可以讨论升为生产指南：

- 实际安装版本和声明支持版本均有完整环境记录；
- 消息 / 楼层桥的生成完成、楼层可读和 Swipe 契约有真机证据；
- 数据库回调真实时序已记录，不再依赖虚构 fill-end；
- 无变化填表回合不会卡死；
- 手动 CRUD、导入和恢复不会被误判为剧情回合；
- 切聊天、刷新和 iframe 重绘无回调泄漏；
- Swipe、重生成、编辑、删楼和回滚有明确一致或安全降级结果；
- 原生 / SQLite 的声明范围分别通过；
- 正常多楼层降级在所有严重错误下可用；
- 至少一张真实角色卡完成长聊天回归，而非只跑合成 demo；
- 关键链路经过独立审查；
- 所有“推导未验证”项要么获得证据，要么明确移出支持范围。

### 17.2 强一致性门

若仍无法回答下面三个问题，禁止转正：

1. 当前数据库状态如何证明对应当前 assistant 楼与 Swipe？
2. Swipe / 编辑 / 删楼后，数据库如何证明已回到一致 revision？
3. 无数据库变化时，协调器如何区分“合法无变化”和“填表流程失效”？

允许的答案可以是“目标上游新增了正式契约”，也可以是“项目收窄功能，禁用无法一致处理的操作并可靠降级”；不允许用时间猜测冒充证据。

### 17.3 产品体验门

技术链路通过后，还需确认：

- 首楼完整游玩体验达到 C5 的功能和量级，不是开局页或数据库面板；
- 加载、生成、核对、超时、冲突和降级状态玩家能理解；
- 返回正常聊天的入口始终可用；
- 移动端输入、键盘、滚动和焦点可用；
- 长剧情与大量表格不会阻塞渲染；
- 错误不会让玩家误以为存档已安全提交。

### 17.4 转正文档流程

满足门槛后也不能直接删掉“待验证”字样。应：

1. 汇总逐项真机证据；
2. 明确支持的 ST、酒馆助手和数据库版本范围；
3. 把仍未验证的路线移到悬案；
4. 由总监确认是否升版与改名；
5. 同步 C4、C5、C7、批次说明和收集蓝图；
6. 保留本册 `v0.x` 的历史裁决记录。

在总监确认之前，本册继续标注：**待验证，禁止生产引用。**
