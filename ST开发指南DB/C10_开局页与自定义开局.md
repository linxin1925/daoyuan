> 文档版本：v1.0.0 ｜ 最后更新：2026-07-15
>
> **定位**：从普通 `first_mes`、`alternate_greetings` 到一次性自定义开局向导，建立可验证的加载、草稿、预览、提交、失败恢复和正常消息链。

## 1. 三层能力与边界

### 1.1 三类开局

| 类型 | 生命周期 | 玩家交互 | 提交后 |
|---|---|---|---|
| 普通 `first_mes` | 展示一条首消息 | 使用 ST 原生输入 | 正常多楼层 |
| `alternate_greetings` | 首消息的 Swipe/候选 | 选择喜欢的开场 | 正常多楼层 |
| 自定义开局页 | 一次性初始化向导 | 选择、编辑、预览、确认 | 发送真实玩家开局消息后退出 |

### 1.2 与同层前端的硬判定

| 能力 | 自定义开局 | C5 同层前端 |
|---|---|---|
| 首楼职责 | 初始化配置 | 整场游玩应用 |
| 玩家输入 | 最终提交一次真实开局发言 | 每回合由首楼应用代理 |
| 后续 AI 楼层 | ST 正常显示 | 首楼长期镜像/组织 |
| 路由 | 只服务开局步骤 | 剧情、角色、背包、地图、设置等完整路由 |
| Swipe/回看 | 提交后交还 ST | 应用内长期代理 |
| 退出条件 | 已确认真实玩家消息楼创建成功 | 聊天结束或应用卸载 |

“界面看起来像大型开局页”不能作为分类依据，职责和生命周期才是。

### 1.3 与控制中心的边界

开局向导可以复用控制中心的世界书、身份、工坊和变量能力，但不应让控制中心成为隐藏的第二套提交真值。所有选项最终必须由一个草稿模型和一个提交事务统一解释。

## 2. ST 原生首消息语义

### 2.1 First Message 是真实消息

`first_mes` 不是仅展示的欢迎页，它会成为当前聊天中的真实 assistant 首消息，参与：

- 用户看到的叙事起点；
- 后续上下文与角色语气示范；
- 消息编辑、Swipe、分支和导出；
- Markdown/HTML 渲染；
- 宏替换与相关正则处理。

因此首消息不能只按 UI 营销页写法设计。即使可见层被扩成开局页面，送入模型的版本也必须短、稳定、可理解。

### 2.2 Alternate Greetings

`alternate_greetings` 是首消息的额外候选。在单聊中通常作为首消息 Swipe；群聊可能从默认首消息与 alternates 中随机选择，具体行为随版本与群聊逻辑核验。

适合 alternate 的情况：

- 只需不同场景、时间、关系或叙事视角；
- 每个开局都能直接进入正常聊天；
- 不需要跨步骤组合复杂初始配置；
- 不需要持久化多个世界因子或身份字段。

如果只是三种固定开场，不要为了“更像产品”制造整套远程开局页。

### 2.3 可见消息与送模内容

复杂 UI 应把“玩家看到的可见 mount”和“模型看到的短说明”分开。常见路线是两条正则：

```text
first_mes 短标记
  ├─ display-only → loading/error mount
  └─ prompt-only  → 简短叙事/状态说明
```

具体正则顺序、`markdownOnly` 和 prompt-only 行为属于版本敏感项，必须在目标 ST/正则环境实测；不能只凭预览认为模型没有收到整页 HTML。

## 3. 普通开局写作与 Alternate Greetings

### 3.1 普通开局四要素

一个可直接游玩的首消息至少建立：

1. **场景**：时间、地点、正在发生什么；
2. **角色声音**：说话和行为符合角色，而不是系统说明；
3. **关系与边界**：双方如何称呼、已知信息与未知信息；
4. **行动钩子**：给玩家明确可回应的情境，不替玩家做关键决定。

### 3.2 多开局一致性

每个 alternate 都要检查：

- 角色和玩家身份是否一致；
- 世界状态是否互斥；
- 变量初始化假设是否相同；
- 世界书常驻条目是否能解释该场景；
- 后续同一套提示词是否能承接；
- 不同开局是否误写成不同作品。

需要真正不同的世界规则时，应在提交时持久化对应世界因子，而不是只在首消息文字里暗示。

### 3.3 宏与副作用

展示宏和写入宏不是一回事。首消息/alternate 在预览、Swipe、编辑、重渲染时可能多次经过处理；任何带写入副作用的宏都必须确认执行时机与幂等性。无法确认时，只在提交事务中进行显式持久化。

## 4. 自定义开局的状态分层

推荐拆成四层：

```text
DraftState            用户尚未确认的选择与自由输入
UiState               当前步骤、折叠、滚动、错误提示
PersistentSetup       身份、世界因子、工坊包等确认后持久数据
OpeningMessage        最终可读的真实玩家开局发言
```

### 4.1 草稿不等于游戏变量

草稿可以频繁修改、撤销、刷新恢复；MVU/数据库游戏变量通常应在正常消息/生成链中初始化。不要每改一个下拉框就修改正式存档变量，否则用户取消开局也会留下半套状态。

### 4.2 UI 状态不污染业务数据

“当前第 4 步”“某 accordion 展开”“预览滚动位置”不应写进世界书、角色卡或 MVU。它们可以存于内存或聊天草稿的 `ui` 子对象，并允许丢失后安全回默认。

### 4.3 持久设置

只有明确确认的内容才进入：

- Persona/身份；
- 聊天级 metadata；
- 角色绑定世界书；
- 工坊来源条目；
- 后续提示词需要的稳定配置。

每类写入都需要所有权、预览、失败策略和回滚，不把“点击开始”变成不可审计的一串跨系统副作用。

## 5. 推荐生产架构

```text
first_mes 唯一短标记
  → 显示正则生成可见 mount（loading/error/retry）
  → prompt-only 正则避免 UI 污染模型上下文
  → 受信本地 binder 找到 mount
  → 拉取声明式 opening-page.html
  → 协议/版本/hash/结构/净化校验
  → 注入 HTML
  → binder 绑定交互和草稿
  → 用户预览并确认
  → 提交事务持久化配置
  → 组装可读真实玩家消息
  → 再核对聊天与输入框
  → 走 ST 正常发送链
  → AI 生成真实下一楼
```

### 5.1 短 mount 的意义

- 角色卡首消息保持小而稳定；
- 远程视觉可以更新；
- loading/error 不依赖远程页面自身；
- prompt-only 版本可控制送模体积；
- 失败时仍能重试或人工进入普通聊天。

### 5.2 声明式远程页面

远程 `opening-page.html` 只提供结构与样式，不承担可信业务逻辑。加载器应移除 `script/iframe/object/embed/base/link/meta/form` 等危险或不需要的节点，交互由角色卡内本地 binder 绑定。

这样远程页面更新不会自动获得任意宿主 API，也减少 CSP、脚本执行时序和供应链风险。

### 5.3 本地 binder

binder 负责：

- 检查页面协议和必需节点；
- 绑定输入、按钮、Tab 和可访问状态；
- 加载/保存当前聊天草稿；
- 调用已审计的世界书/身份/工坊接口；
- 执行校验和预览；
- 发起唯一提交事务；
- 切聊天、Swipe、编辑、卸载时销毁。

## 6. 远程页面加载与供应链安全

### 6.1 多源不是无条件信任

主源失败可尝试备用源，但每个源都必须满足同一协议和完整性要求：

```javascript
const sources = [PRIMARY_URL, FALLBACK_URL];
let lastError = null;

for (const url of sources) {
  const controller = new AbortController();
  try {
    const html = await fetchWithTimeout(url, 10_000, {
      signal: controller.signal,
    });
    await verifySha256(html, EXPECTED_HASH);
    const document = parseAndSanitize(html);
    assertOpeningProtocol(document, EXPECTED_PROTOCOL);
    return document;
  } catch (error) {
    lastError = error;
  } finally {
    controller.abort();
  }
}

throw new Error('所有开局页来源均加载失败', { cause: lastError });
```

每个源独立捕获失败并清理自己的 timeout/AbortController，只有全部失败才统一报错。不同源内容不应“谁先成功就信谁”；若 hash 随发布变化，必须由受信本地配置一并更新。

### 6.2 必验内容

- HTTPS 或本地可信资源；
- 响应大小上限；
- 超时；
- SHA-256 或等价完整性；
- 页面协议版本；
- 必需根节点和 action ID；
- HTML 净化；
- 禁止远程脚本、嵌套 iframe、base URL 劫持和自动表单；
- 失败时保留宿主 mount 的错误与重试。

### 6.3 结构校验

```javascript
function assertOpeningDocument(doc) {
  const root = doc.querySelector('[data-opening-protocol="1"]');
  if (!root) throw new Error('开局页协议不匹配');
  for (const id of ['opening-next', 'opening-preview', 'opening-submit']) {
    if (!doc.getElementById(id)) throw new Error(`缺少节点: ${id}`);
  }
}
```

不要遇到结构不匹配还“尽量 bind”。按钮绑定到错误节点会比直接失败更危险。

### 6.4 iframe 不是天然安全沙盒

若选择 iframe，仍需考虑 `allow-same-origin`、`allow-scripts`、postMessage origin/source、宿主 bridge 权限和导航。**同源内容同时获得 `allow-scripts` 与 `allow-same-origin` 会显著削弱 sandbox 隔离**，不能把这一组合当普通默认值。对于声明式开局页，净化后注入受控容器通常比允许远程脚本更简单。

## 7. 向导 UX 与依赖门

### 7.1 依赖先检查

进入正式编辑前展示依赖状态：

- Tavern Helper 是否存在；
- 目标世界书 API 是否存在；
- 角色是否绑定目标世界书；
- 控制中心/运行时版本是否兼容；
- MVU/数据库是否真的属于本开局提交链；
- 工坊包 schema 与 cardScope 是否匹配；
- 远程页面是否通过完整性检查。

不使用的依赖不要硬设为必需。工坊桥若只写世界书，就不应仅因 MVU 缺失而阻断。

### 7.2 步骤设计

建议：

```text
依赖与说明
→ 身份/关系
→ 场景/时间/年级
→ 世界因子与功能包
→ 自由正文/高级项
→ 预览
→ 最终确认
```

步骤数取决于真实决策，不为了“像产品”强行六步。允许返回修改；跳步限制必须解释缺失项。

### 7.3 预览必须可追溯

预览展示：

- 最终真实玩家消息；
- 将写入的身份/世界配置摘要；
- 将安装/升级的工坊内容；
- 仅本次生效项；
- 不能自动完成、需用户后续处理的项目。

预览使用与提交相同的 `buildOpeningMessage(draft)`，不能另写一个“看起来差不多”的模板。

### 7.4 移动端与无障碍

- 320px 不横向溢出；
- 软键盘弹出仍能看到当前输入与下一步；
- 按钮热区至少约 44px；
- 每步有标题和当前进度；
- 错误使用 `aria-live` 或等价可读状态；
- 焦点在步骤切换后落到新标题或首个错误；
- 不只靠颜色表示已完成/错误；
- Reduced Motion 下取消大幅页面转场。

## 8. 草稿、聊天隔离与迁移

### 8.1 每聊天唯一键

草稿必须绑定稳定聊天身份，而不是只用角色名：

```javascript
function draftKey({ characterId, chatId, chatUuid }) {
  if (chatUuid) return `opening-draft:v2:${chatUuid}`;
  // 仅供旧环境无稳定 UUID 时降级；聊天重命名后需迁移。
  return `opening-draft:v2:fallback:${characterId}:${chatId}`;
}
```

稳定 `chatUuid` 是主键；`characterId/chatId` 记录在草稿正文中用于校验、显示和无 UUID 环境的 fallback，不与 UUID 拼成会随重命名改变的新键。分支应产生新 UUID，并可记录 `originUuid`；聊天重命名和群聊需要单独测试。

### 8.2 草稿 schema

```javascript
const draft = {
  schemaVersion: 2,
  chatUuid: '...',
  updatedAt: '...',
  identity: {},
  world: {},
  packages: [],
  openingBody: '',
  ui: { step: 1 },
};
```

保存前做 schema 校验和大小上限，避免把 DOM、函数、Blob 或完整包体塞进 metadata。

### 8.3 debounce + flush

输入时防抖保存，以下时机同步 flush：

- 切步骤；
- 页面 `pagehide`；
- 切聊天前；
- 打开预览；
- 最终提交；
- 运行时销毁。

### 8.4 损坏与迁移

读取旧草稿：

1. 校验 JSON 和 schemaVersion；
2. 逐版本迁移；
3. 迁移失败时隔离原记录；
4. 提示用户重置当前聊天草稿；
5. 不用空对象覆盖所有聊天；
6. 记录不含敏感正文的诊断。

## 9. 校验、预览与唯一真相源

### 9.1 业务校验

至少覆盖：

- 必填身份/场景；
- 空白正文；
- 正文字节上限；
- 允许/未知宏；
- 身份与年级/世界设定的兼容；
- 工坊包重复、冲突和 cardScope；
- 依赖是否仍可用；
- 当前聊天是否仍是草稿所属聊天。

字符数不等于 UTF-8 字节数：

```javascript
const byteLength = new TextEncoder().encode(text).byteLength;
```

### 9.2 唯一消息构建器

```javascript
function buildOpeningMessage(draft) {
  return [
    `我的身份：${draft.identity.summary}`,
    `开局条件：${draft.world.summary}`,
    draft.openingBody.trim(),
  ].filter(Boolean).join('\n\n');
}
```

预览、字节校验、最终发送全部调用它。默认值也要先写回 draft，再由构建器读取；不能 UI 显示一个默认值而提交器又偷偷使用另一个。

### 9.3 预览不是写入

预览阶段禁止：

- 安装世界书条目；
- 修改 Persona；
- 写 MVU 正式变量；
- 向聊天插入隐藏消息；
- 自动开始生成。

如需计算 diff，只读当前状态并展示。用户确认后再进入事务。

## 10. 提交事务

### 10.1 冻结上下文

提交开始时捕获：

```javascript
// 概念适配层：禁止照抄为未声明的 iframe 全局函数。
const token = OpeningContextAdapter.capture({
  draftHash: hashDraft(draft),
});
```

适配层应从实际运行上下文组合证据：例如 `SillyTavern.getContext()` 的当前聊天标识、聊天 metadata 中的 draft UUID，以及确需时对 0 号楼当前 Swipe/消息修订的读取。`getCurrentMessageId()` 属于消息楼 iframe 便利 API，控制中心/宿主 binder 不得假定它是任意上下文的全局函数。

最终发送前再次比较。用户在确认对话框期间切聊天、修改草稿或首消息 Swipe，旧提交必须取消。

### 10.2 推荐顺序

```text
freeze context
→ flush draft
→ validate draft and dependencies
→ build preview/message
→ show final diff/confirm
→ snapshot persistent targets
→ persist identity/world/workshop configuration
→ verify writes by reread
→ rebuild message from frozen draft
→ recheck chat/input/runtime
→ place text into ST input
→ trigger normal send
→ confirm a real user message was created
→ mark opening committed / archive draft
```

### 10.3 跨系统事务

浏览器、Persona、世界书和聊天发送通常没有真正原子事务。应用层需要：

- 变更前快照；
- 每步结果记录；
- 可逆步骤的回滚；
- 不可逆步骤放在最后；
- 失败时明确哪些已生效；
- 重试幂等，不能重复安装条目或重复发送。

建议先完成可验证的身份/世界书写入，最后才发送真实玩家消息。若持久化失败，默认停止；可允许用户显式选择“仅本次继续”，但必须说明相关配置不会常驻。

### 10.4 输入框仲裁

若通过 `#send_textarea` / `#send_but` 走宿主发送链：

- 这是版本敏感私有 DOM 路线；
- 写入前记录输入框原内容；
- 若用户已经修改输入框，取消自动覆盖；
- 触发正常 input/change 事件；
- 点击发送失败时保留文字，让用户手动发送；
- 不直接 `generate()` 生成一段游离文本冒充消息。

## 11. 回到正常生成链

正确完成链：

```text
真实 user 消息
  → ST 正常生成请求
  → 真实 assistant 下一楼
  → 变量框架按正常回复链初始化/更新
  → 状态栏与控制中心读取正式状态
  → 开局页退出使命
```

### 11.1 为什么不直接 generate

直接调用裸生成可能：

- 不创建真实 user 楼；
- 绕过正常事件、正则或变量更新链；
- 产生不能 Swipe/编辑/导出的游离文本；
- 在切聊天竞态中落错上下文；
- 让首回合和后续回合使用不同路径。

开局页的最终动作应尽可能等同于用户在 ST 输入框发送一条正常消息。

### 11.2 提交标记

成功确认真实 user 消息存在后，写入当前聊天的“开局已提交”标记。该标记用于避免刷新后重复提交，不应用来禁止用户编辑或分支。若用户回到首消息重新开局，应明确提供“重置开局”流程并提示影响。

## 12. 失败降级与人工接管

| 失败 | 默认行为 | 可选恢复 |
|---|---|---|
| 远程页面断网 | mount 显示错误，不白屏 | 手动重试/简化原生开局 |
| hash/协议不匹配 | 拒绝注入 | 更新受信资源后重试 |
| **必需**依赖缺失 | 禁止正式提交 | 只读预览/说明安装步骤；未使用依赖不阻断 |
| 草稿损坏 | 隔离当前记录 | 重置当前聊天草稿 |
| 身份持久化失败 | 停止 | 重试/显式仅本次继续 |
| 世界书事务失败 | 只回滚本事务拥有且 revision 未变化的可逆写入 | 并发/不可逆时停止并展示部分提交 diff |
| 确认后切聊天 | 取消旧提交 | 在新聊天重新确认 |
| 输入框被用户修改 | 不覆盖 | 复制文本或重新预览 |
| 自动点击发送失败 | 保留文本 | 用户手动发送 |
| 页面卸载 | abort 请求和提交 | 重新挂载后恢复草稿 |

所有错误都应保留原生 ST 的可用性。开局页不能把聊天锁在“必须远程页面成功才可退出”的死状态。

## 13. 生命周期与重复挂载

### 13.1 会触发重绑的场景

- 首消息 Swipe；
- 编辑/保存首消息；
- 单条消息重渲染；
- 切聊天/Persona/分支；
- 浏览器刷新；
- 脚本热重载；
- 远程页面重试；
- DOM 被第三方扩展替换。

### 13.2 单例 binder

每个 mount 使用稳定实例键：

```javascript
const BIND_KEY = '__openingBinder';

function bindMount(mount, descriptor) {
  const current = mount[BIND_KEY];
  if (current?.isCompatible?.(descriptor)) return current;
  current?.destroy?.();
  mount[BIND_KEY] = createBinder(mount, descriptor);
  return mount[BIND_KEY];
}
```

`isCompatible` 至少比较 owner/runtime、页面协议和内容 revision。重复扫描发现同一有效实例时直接复用；只有 mount 已失效或协议/revision 变化时才销毁重建，避免定时扫描反复取消请求、丢焦点或打断提交。

destroy 清理：

- fetch AbortController；
- debounce/timeout；
- DOM listeners；
- window message；
- Tavern Helper event disposer；
- observers；
- 打开的模态框；
- 正在进行的提交锁；
- 对旧 chat/draft 的闭包引用。

### 13.3 不依赖单一 DOMContentLoaded

卡内脚本和消息 mount 可能在页面已加载后创建。初始化应使用当前运行时推荐的就绪方式，并对目标 mount 做幂等发现/重绑，而不是只监听一次 `DOMContentLoaded`。

### 14.1 页面协议

至少约定：

- `protocolVersion`；
- 必需根节点；
- 稳定 `data-action`；

- 错误/状态容器；
- 页面内容 hash；
- 可选能力标记；

## 16. 高频错误

| 错误 | 后果 | 修复方向 |
|---|---|---|
| 把开局页当 C5 | 边界和后续职责无限膨胀 | 以提交后退出为硬门 |
| 只是几个 alternate 却造远程应用 | 成本和故障面过大 | 优先原生首消息/Swipe |
| 整页 HTML 送模 | token 膨胀、提示词污染 | display/prompt 分离 |
| 远程 `<script>` 执行业务 | 供应链可任意控制宿主 | 声明式 HTML + 本地 binder |
| 只看 HTTP 200 | 结构错误仍绑定 | hash/协议/结构/净化 |
| 草稿按角色名存 | 多聊天串档 | chat UUID 隔离 |
| UI 默认值不进 draft | 预览和提交不一致 | 唯一草稿真相源 |
| 预览时写正式数据 | 取消后留下半套状态 | 预览只读，确认后事务 |
| 直接改 MVU 作为开局草稿 | 正常变量链被绕过 | 正常回复链初始化 |
| 直接 `generate()` | 没有真实 user 楼 | 走 ST 正常发送 |
| 确认后不复核聊天 | 写进旧聊天 | context token 二次检查 |
| 自动发送覆盖用户输入 | 丢失用户文本 | 输入框仲裁 |
| 加载失败白屏 | 无法继续聊天 | 本地 mount 错误/重试/降级 |
| 重绑不销毁 | 多次提交、监听泄漏 | mount 单例 + destroy |
| 把私有 DOM 当 API | ST 更新后失效 | 能力检测 + 兼容矩阵 |
