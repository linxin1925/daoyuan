# 独立前端：把单张角色卡作品做成 Web / App / PC 产品

> 文档版本：v1.0.0 ｜ 最后更新：2026-07-14

> **第一原则：程序本身就是这一张角色卡。**
>
> 独立前端不是“另做一个能导入角色卡的聊天客户端”，也不是把 SillyTavern 页面包进另一个容器。角色设定、世界观、初始场景、提示词规则、变量或数据库结构、剧情规则、UI、资产、模型接入和存档协议，全部是该作品应用工程内部的版本化源码与内建资源。开发仓库即使把它们编译成内部只读内容包，那也只是程序模块，不是供玩家导入、导出或独立流通的“角色卡封装体”。玩家运行程序后创建的是这一个作品的存档、角色实例或周目，不是导入另一张 PNG / JSON 卡。

本册讨论单张角色卡的**产品化独立运行**。Web、移动 App、PC 软件只是同一作品的三个发行壳；共享内容域与运行时核心，不依赖 SillyTavern、Tavern Helper、MVU 或 ST-Prompt-Template 才能工作。

相关边界：

- ST 楼层内的常规状态栏、控制中心见 [C2_前端应用-状态栏与控制中心.md](./C2_前端应用-状态栏与控制中心.md)。
- 全部游玩界面集中在 ST 首楼的“同层前端”见 [C5_同层前端.md](./C5_同层前端.md)。
- 本册的独立前端**完全脱离 ST 运行时**；ST 只可能在开发期作为旧作品素材的迁移来源。

## 0. 置信度与裁定口径

| 标记 | 含义 |
|---|---|
| **high · 规范** | 固定版本公开规范中有明确字段或行为定义 |
| **high · 源码** | 固定 commit 的成熟项目配置或源码可以直接验证 |
| **high · 总监裁定** | 本工作区对“独立前端”的项目专用定义 |
| **medium · 工程建议** | 本册根据目标整理的推荐架构，需由具体作品落地验证 |
| **待验** | 与模型、平台、商店政策或真机行为有关，必须实测 |

版本敏感说明：

- Character Card V3 只用于**开发期迁移旧素材时理解原字段**，不是独立应用的运行时容器。固定参考 commit：`f3a86af019fbd99f788f7a1155f399655b34ab35`。
- RisuAI 只用于证明“共享前端核心 + Web / Android / PC 壳”在成熟项目中可行。它是通用客户端，**不是本架构的产品模型**。固定参考 commit：`63c59c15336cc6da865218ff3717756557206fcb`。
- 本册中的目录、接口与里程碑均为建议范式；具体作品可以换技术栈，但不能改变“程序本身就是角色卡”的边界。

## 1. 定义与反定义

### 1.1 正确定义

独立前端是将一张角色卡从“宿主中的内容包”升级为“可独立发行的互动作品”：

```text
单张作品内容
  ├─ 角色、世界、剧情规则
  ├─ Prompt、Lore、宏、变量 Schema
  ├─ UI、玩法系统、音画资产
  └─ 内容版本与迁移
          │
          ▼
共享运行时核心
  ├─ Prompt 编译与 Lore 扫描
  ├─ 模型网关与流式生成
  ├─ 状态事务与存档
  ├─ 消息分支 / Swipe / 检查点
  └─ 安全渲染
          │
          ├─ Web 壳
          ├─ 移动 App 壳
          └─ PC 壳
```

三个壳运行的是**同一作品、同一内容版本、同一存档语义**。平台层只处理文件、密钥、网络、更新、通知和窗口等能力差异。

### 1.2 明确不是

- 不是 SillyTavern 内的 iframe、扩展、脚本或首楼页面。
- 不是套壳打开一个 ST 网址。
- 不是“导入任意角色卡后聊天”的通用客户端。
- 不是角色卡 PNG / JSON 的查看器或启动器。
- 不是运行时再解析一份卡片封装体。
- 不是把 Tavern Helper、MVU、EJS 或酒馆正则原样复制出去。
- 不是把模型回复当 HTML 或 JavaScript 直接执行。

### 1.3 独立性的硬门

满足以下条件才可以称为独立前端：

1. 在没有安装或启动 ST 的设备上能新建周目、游玩、保存、退出和恢复。
2. 生产构建不读取 ST 的目录、数据库、全局对象、事件或插件 API。
3. 作品内容作为源码或经过编译的内容包随应用发布。
4. 运行时只管理本作品的存档；没有“角色卡导入”作为主入口。
5. Web、App、PC 的同一存档遵循同一 Schema 和迁移序列。
6. 模型不可绕过 Schema 校验直接改状态，也不可直接执行宿主代码。

## 2. 目标玩家体验

独立前端的产品体验应从“打开一张卡”改写为“打开一个作品”：

1. 首次启动：阅读作品说明、隐私与模型配置。
2. 创建玩家 Persona：名称、称谓、头像、代词和作品允许的自定义项。
3. 选择开局：难度、路线、预设身份、可选内容或随机种子。
4. 创建周目：由内建初始化器生成状态，不依赖聊天楼层变量。
5. 进入主界面：剧情、状态、地图、角色、背包、任务等模块按作品设计呈现。
6. 玩家输入文本或点击玩法动作。
7. 引擎编译 Prompt、选择 Lore、请求模型并流式显示。
8. 结构化状态变更经过校验后与本回合消息原子提交。
9. 玩家可以重试、Swipe、分支、回到检查点或另开周目。
10. 应用更新后，旧存档经过显式迁移继续使用；失败时保留可回滚备份。

“角色卡感”来自作品内容与玩法闭环，不来自是否保留 ST 的界面。

## 3. 总架构：内容域与引擎域分离

### 3.1 内容域：这张卡是什么

内容域只描述本作品：

- `work.json`：作品 ID、名称、内容版本、最低引擎版本、语言和分级。
- `characters/`：主角、NPC、关系锚点、台词约束。
- `world/`：世界观、地点、组织、事件和 Lore 条目。
- `prompts/`：系统契约、文风、行为规则、输出协议和装配计划。
- `state/`：状态 Schema、初值工厂、迁移、可写路径和约束。
- `rules/`：玩法动作、剧情门槛、数值联动、结局条件。
- `scenes/`：开场、教程、固定事件、可选章节。
- `ui/`：本作品专属的页面、面板、文案和主题 Token。
- `assets/`：角色立绘、背景、音频、字体、视频等受控资源。
- `locales/`：面向玩家的本地化文本；Prompt 本地化应另设版本。

内容域应该能被构建工具做静态检查：缺失引用、重复 ID、无效状态路径、超预算 Prompt、孤立资产和迁移断档都应在发布前失败。

### 3.2 引擎域：这张卡怎样运行

引擎域提供与作品内容解耦的运行能力：

- 内容注册与版本检查。
- 周目和 Persona 初始化。
- Prompt 编译、宏替换、Lore 扫描和 Token 预算。
- 模型供应商适配、流式事件、取消、重试和错误归一化。
- 消息树、Swipe、分支、检查点和摘要。
- 状态操作解析、Schema 校验、权限检查和事务提交。
- 存档、备份、迁移、导出和同步冲突处理。
- Markdown / 富文本安全渲染。
- 音画资产解析、缓存和生命周期。
- 平台能力端口。

引擎可以在多个内部项目复用，但**发行产物锁定到当前作品内容**。复用引擎不等于在产品里提供多卡导入。

### 3.3 平台域：同一作品怎样发行

平台域只实现端口：

- 文件选择、保存和分享。
- 密钥安全存储。
- HTTP / SSE / 流式请求。
- 数据库、原子写和备份目录。
- 通知、剪贴板、窗口、深链和系统主题。
- 更新检查、签名校验和崩溃恢复。

作品规则不能散落在平台壳中；否则三个端会产生不同剧情和不同存档语义。

## 4. 内建内容包与构建产物

### 4.1 内容清单

建议每次发布生成不可变清单：

```ts
export interface WorkManifest {
  workId: string;
  contentVersion: string;
  schemaVersion: number;
  promptVersion: string;
  minEngineVersion: string;
  defaultLocale: string;
  entrySceneId: string;
  assetManifestHash: string;
  contentHash: string;
}
```

`workId` 一经发布不可改。显示名称可以改，但存档、云同步和更新定位都依赖稳定 ID。

### 4.2 构建期而非运行时装配

构建工具应完成：

1. 读取本作品内容源码。
2. 校验所有 ID、引用和 Schema。
3. 编译 Prompt 模板和 Lore 索引。
4. 为资产计算哈希并生成清单。
5. 生成初始状态的确定性测试样本。
6. 验证从所有受支持旧 Schema 到当前 Schema 的迁移链。
7. 输出 Web / App / PC 共用的只读内容包。

生产运行时不应临时猜测条目关系，也不应从一张外部角色卡中动态拼出作品规则。

### 4.3 资源寻址

内部资源使用作品自己的逻辑 URI：

```text
work://character/lily/portrait/default
work://background/academy/night
work://audio/bgm/home
```

构建器将逻辑 URI 映射为带哈希的 Web URL、App bundle path 或 PC asset path。剧情和状态只保存逻辑 URI，不保存某个平台的绝对路径。

### 4.4 内容可更新但不可漂移

- 内容包与代码包分别版本化。
- 每个内容包声明 `minEngineVersion`。
- Prompt、状态 Schema、资产清单分别有版本或哈希。
- 在线增量包必须签名并校验哈希。
- 已开始的周目记录创建时的内容版本；更新后由迁移器显式升级。
- 不能在服务器无版本地热改 Prompt，让同一存档第二天出现不可追溯的行为变化。

## 5. 一回合的主链路

推荐把一回合实现为可观察事务：

```text
玩家输入 / UI 动作
  → Command 规范化
  → 读取当前周目快照
  → 规则前置校验
  → Lore 扫描与 Prompt 编译
  → ModelGateway 流式生成
  → 回复正文与结构化操作分离
  → 状态操作 Schema + 权限校验
  → 消息、状态、生成记录原子提交
  → UI 投影刷新
```

必须保留以下证据：

- 本次使用的内容、Prompt、Schema 和模型配置版本。
- 编译后的 Prompt Trace，而不一定长期保存完整敏感 Prompt。
- 命中的 Lore ID 与裁剪原因。
- 请求 ID、开始和结束时间、取消或失败原因。
- 原始结构化操作、校验结果和最终提交差异。

若正文成功但状态操作非法，默认策略应由作品明确选择：

- **严格模式**：整回合不提交，提示重试。
- **正文保留模式**：提交正文但拒绝状态操作，并标记该回合需要修复。
- **修复模式**：向专用修复器请求一次受限重写，再校验；禁止无限重试。

不能静默吞掉非法操作，否则玩家看到的剧情和真实状态会长期分叉。

## 6. 状态、数据库与多周目

### 6.1 一个作品，多份实例

运行时的主要实体建议为：

```text
Work（唯一作品）
  ├─ Persona（玩家身份，可跨周目复用）
  ├─ Playthrough / 周目
  │    ├─ 当前状态快照
  │    ├─ 消息与分支
  │    ├─ 检查点
  │    ├─ 摘要与长期记忆
  │    └─ 生成审计记录
  └─ Settings（模型、显示、无障碍）
```

“新建角色”在这里通常是新建周目中的玩家实例或剧情角色实例，不是把另一张角色卡导入角色库。

### 6.2 状态 Schema

Schema 是本作品状态的唯一事实源。除类型外，还应声明：

- 默认值与首次创建规则。
- 玩家、规则引擎、模型各自允许写的路径。
- 数值范围、枚举、引用完整性和容量上限。
- 哪些字段是派生值，不允许持久化或由模型写。
- 哪些字段涉及敏感信息，需要加密或导出脱敏。

推荐把模型写入限制为显式操作：

```ts
export type StateOperation =
  | { op: 'set'; path: string; value: unknown }
  | { op: 'increment'; path: string; delta: number }
  | { op: 'append'; path: string; value: unknown }
  | { op: 'remove'; path: string };

export interface StatePolicy {
  canWrite(actor: 'player' | 'engine' | 'model', op: StateOperation): boolean;
  validate(nextState: unknown): ValidationResult;
}
```

不要允许模型提交任意 JavaScript、SQL 或表达式。

### 6.3 初始状态

初始状态由纯函数创建：

```ts
createInitialState({
  persona,
  routeId,
  difficulty,
  seed,
  contentVersion,
}): WorkState
```

相同输入应得到可重放的基线。需要随机性的部分使用记录在存档中的种子。

### 6.4 存档事务

单次成功提交至少原子写入：

- 新消息或新 Swipe。
- 激活分支指针。
- 状态版本和差异。
- 摘要失效标记。
- 生成元数据。

推荐采用“快照 + 受控事件日志”：

- 快照用于快速载入。
- 事件用于审计、检查点和迁移调试。
- 定期压缩，避免无限增长。
- 事件日志不是让模型自由写入的流水账。

### 6.5 迁移与回滚

```ts
export interface SaveMigration {
  from: number;
  to: number;
  migrate(input: unknown): unknown;
  verify(output: unknown): ValidationResult;
}
```

规则：

1. 迁移只向前逐级执行，不能跳过中间版本。
2. 迁移前创建备份。
3. 新 Schema 校验失败时不覆盖原档。
4. 迁移器必须幂等或明确拒绝重复运行。
5. 发行前用匿名化真实旧档回放。
6. 降级打开高版本存档时只读或拒绝，禁止猜测性回写。

## 7. Prompt 编译器

### 7.1 Prompt 是编译产物

不要在 UI 点击发送时临时字符串拼接。编译器接收确定输入并返回消息列表、预算报告和来源轨迹：

```ts
export interface PromptCompiler {
  compile(input: {
    manifest: WorkManifest;
    persona: Persona;
    state: WorkState;
    branch: MessageBranch;
    command: PlayerCommand;
    model: ModelProfile;
  }): Promise<PromptCompilation>;
}

export interface PromptCompilation {
  messages: Array<{ role: 'system' | 'user' | 'assistant'; content: string }>;
  trace: PromptTraceItem[];
  estimatedInputTokens: number;
  reservedOutputTokens: number;
}
```

### 7.2 推荐装配顺序

下列顺序是**作品工程建议，不是 Character Card 规范强制顺序**：

1. 引擎安全契约和结构化输出契约。
2. 本作品总规则、角色核心和文风锚点。
3. 当前场景和玩法模式规则。
4. 当前 Persona 与经过过滤的状态投影。
5. 本回合命中的 Lore。
6. 长期摘要和关键记忆。
7. Token 预算内的近期消息。
8. 当前玩家输入或 UI 动作描述。
9. 状态操作允许路径与本回合特殊约束。

作品可以调整，但装配顺序必须配置化、版本化、可做 Golden Test，不能藏在多个页面组件里。

### 7.3 Token 预算

建议先计算：

```text
可用输入 = 模型上下文上限
         - 预留输出
         - 协议安全余量
```

裁剪优先级由作品明确：

1. 安全契约、输出协议和当前玩家输入不可裁。
2. 角色核心和当前状态只允许使用预先定义的紧凑投影。
3. Lore 按优先级、剧情相关性和预算选择。
4. 近期历史按完整回合裁剪，不从中间截断角色消息。
5. 较旧历史先进入摘要，再移出近期窗口。

每次编译输出 `PromptTrace`，至少列出“加入、裁剪、原因、估算 token”。没有 Trace 的 Prompt 优化只能靠猜。

## 8. 内建 Lore、宏与剧情规则

### 8.1 Lore 扫描器

独立作品应定义自己的稳定 `LoreEntry`，而不是在运行时借用 ST 世界书对象：

```ts
export interface LoreEntry {
  id: string;
  enabled: boolean;
  constant?: boolean;
  keys?: string[];
  secondaryKeys?: string[];
  matchMode?: 'any' | 'all' | 'regex';
  scanTurns?: number;
  priority: number;
  insertion: 'before-core' | 'after-core' | 'history-depth' | 'state';
  contentTemplateId: string;
  recursive?: boolean;
}
```

扫描流程：

1. 取配置范围内的玩家输入、近期消息和允许参与匹配的状态文本。
2. 激活 constant 条目。
3. 做主键、次键和条件匹配。
4. 需要递归时，用已激活内容继续扫描；设置 visited 集合与最大轮数。
5. 去重，每个条目一次。
6. 按优先级与插入位点排序。
7. 在独立 Lore 预算内裁剪。
8. 输出命中与裁剪轨迹。

正则关键词使用 RE2 类引擎或 Worker 超时；不得让一个恶意或失控模式冻结 UI。

### 8.2 宏系统

宏只读显式上下文：

```ts
export interface TemplateContext {
  char: CharacterView;
  user: PersonaView;
  state: Readonly<WorkStateProjection>;
  turn: number;
  locale: string;
  seed: string;
}
```

原则：

- 使用白名单注册的纯函数宏。
- 相同种子下的随机宏可重放。
- 未知宏在构建期报错，或在运行时以可见诊断降级。
- 不提供任意文件、网络、DOM、数据库或代码执行能力。
- Prompt 模板和玩家显示模板分开，避免敏感状态意外显示。

### 8.3 剧情与玩法规则

“满足条件才能解锁地点”“使用物品扣除数量”这类确定性规则优先由引擎执行，不交给模型自由判断。推荐分三层：

- **确定性层**：状态机、数值、权限、冷却、任务门槛。
- **生成约束层**：本回合必须表现或避免的内容。
- **叙事层**：模型在边界内自由生成。

模型负责叙事，不应同时充当唯一数据库和唯一规则裁判。

## 9. 模型网关与流式生成

### 9.1 统一端口

```ts
export interface ModelGateway {
  generate(
    request: ModelRequest,
    options: { signal: AbortSignal; requestId: string },
  ): AsyncIterable<ModelEvent>;
}

export type ModelEvent =
  | { type: 'text-delta'; text: string }
  | { type: 'usage'; inputTokens?: number; outputTokens?: number }
  | { type: 'structured-output'; value: unknown }
  | { type: 'finish'; reason: string }
  | { type: 'error'; code: string; retryable: boolean; message: string };
```

供应商适配器只负责：

- 请求格式转换。
- 鉴权头。
- SSE / chunk 流解析。
- 错误归一化。
- Tokenizer 选择。
- 能力声明，例如 system role、JSON Schema、图片、工具调用。

作品规则不得写进供应商适配器。

### 9.2 取消、重试与迟到数据

- 每次生成有唯一 `requestId`。
- 玩家取消时触发 `AbortController`，并把尝试标记为 cancelled。
- 取消后到达的 chunk 必须丢弃，不能写入下一次请求。
- 自动重试只用于确定可重试的网络错误，并有上限和退避。
- 已收到正文后不得隐式全量重试；由玩家选择继续、重试或保留部分内容。
- 同一个重试产生新的生成尝试，但可以挂在同一 Swipe 组下。

### 9.3 BYOK、代理与本地模型

可以支持：

- 玩家自带 API Key 直连。
- 作品方受控代理。
- OpenAI-compatible 本地或远程端点。
- PC 端本地模型。

不同接入方式必须显示数据流向、计费主体和隐私边界。应用“可离线启动”不等于“无模型也能生成”；离线能力要拆成 UI/存档可用与本地推理可用两项说明。

## 10. 消息、Swipe、分支与一致性

### 10.1 数据模型

不要只存一条可变字符串数组。建议：

```ts
export interface TurnNode {
  id: string;
  parentId: string | null;
  role: 'user' | 'assistant' | 'system-event';
  activeVariantId: string;
  variants: MessageVariant[];
}

export interface MessageVariant {
  id: string;
  content: string;
  status: 'streaming' | 'complete' | 'cancelled' | 'failed';
  generationId?: string;
  stateCommitId?: string;
  createdAt: number;
}
```

### 10.2 一致性规则

- Swipe 创建新 variant，不覆盖旧文本。
- 从旧消息改写并继续时创建分支，不偷偷改写原分支后代。
- 每个 assistant variant 绑定它提交的状态版本。
- 切换 Swipe 时，如果后续状态依赖不同，必须切换对应分支或明确重放。
- 删除消息前列出将失效的后代、摘要、检查点和状态提交。
- 编辑历史消息后，旧摘要与 Prompt cache 标记失效。
- 每次生成保存模型、采样参数、Prompt 版本和内容版本，便于复现。

### 10.3 检查点

检查点至少包含：

- 活跃分支末端。
- 状态快照 ID。
- 摘要版本。
- 随机种子状态。
- 内容与 Schema 版本。

回到检查点要创建新分支，不破坏原时间线。

## 11. UI 与玩法系统

### 11.1 UI 是作品的一部分

独立前端不必模仿聊天软件。可以按作品设计：

- 剧情阅读区与输入框。
- 地图、任务、背包、角色关系、日历。
- 视觉小说式立绘和背景。
- 战斗、调查、经营等专用操作面板。
- 开局向导、章节选择、结局图鉴。
- 存档管理、分支树和 Prompt 诊断页。

UI 只通过 selector 读状态，通过 command dispatch 改状态；页面组件不得直接写数据库。

### 11.2 玩家动作统一为 Command

```ts
export type PlayerCommand =
  | { type: 'send-text'; text: string }
  | { type: 'choose-option'; optionId: string }
  | { type: 'use-item'; itemId: string; targetId?: string }
  | { type: 'travel'; locationId: string }
  | { type: 'system'; action: 'retry' | 'continue' | 'checkpoint' };
```

每个 Command 经过相同的规则、Prompt、生成和事务链路，避免按钮玩法与文字输入形成两套互相不同步的状态。

### 11.3 安全渲染

- 模型回复先作为不可信文本处理。
- Markdown 转换后再用严格白名单清洗。
- 默认删除 `script`、事件属性、危险 URL 和可执行嵌入。
- 结构化玩法组件由应用根据受信任 DSL 渲染，不执行模型返回的 HTML。
- 外链打开前提示域名；远程图片默认可禁用或走代理。
- CSS 使用组件作用域或 Shadow DOM，不能让回复修改整站样式。

### 11.4 无障碍与移动体验

- 键盘可完成发送、取消、切换 Swipe 和打开面板。
- 流式区域使用节制的 live region，避免每个 token 都朗读。
- 支持系统字体缩放、减少动画、高对比度与触控目标。
- 输入法组合期间不得误发送。
- 移动端软键盘弹出时保证输入框和取消按钮可见。

## 12. Web / App / PC 三壳

### 12.1 共享边界

共享：

- 内容包和内容版本。
- Prompt、Lore、宏、规则和状态引擎。
- 消息树、存档模型和迁移。
- 模型网关协议。
- 绝大多数 UI 组件与主题。

不强行共享：

- 密钥存储实现。
- 文件、数据库和原子写适配器。
- 更新、通知和系统菜单。
- 原生 HTTP、深链和窗口行为。
- 后台执行与生命周期。

### 12.2 Web 壳

建议能力：

- IndexedDB 保存存档和资源缓存。
- PWA 缓存应用壳与内建内容，可离线打开和管理存档。
- File System Access API 仅作为增强，保留下载/上传兜底。
- Service Worker 更新必须显示“等待重启”，不要在生成中途换版本。
- 浏览器直连供应商受 CORS 限制；失败时使用受控代理或明确告知不支持。
- 浏览器端 API Key 不应放在普通 LocalStorage；至少使用隔离存储、会话模式与清除入口，并明确其安全上限。

### 12.3 移动 App 壳

可以使用 Capacitor、React Native、Flutter 或其他方案；选择不改变共享 core 边界。

需要处理：

- iOS / Android 安全密钥库。
- 系统分享、文件导入导出和照片权限。
- App 进入后台时的网络暂停、WebView 回收和流式中断。
- 前台恢复后的生成状态核对；不能假设长连接仍存在。
- 大资源解压、低内存和磁盘配额。
- 商店隐私清单、内容分级和账户删除要求。

后台被杀时，未完成生成标记为 interrupted；恢复后让玩家选择重试，不能把半截流当成完整回合提交。

### 12.4 PC 壳

可以使用 Tauri、Electron 或原生 UI：

- 系统 Keychain / Credential Manager 保存密钥。
- 原子文件备份、可选择的存档目录和自动备份。
- 原生 HTTP 可绕过浏览器 CORS，但仍要做目标域名限制和 SSRF 防护。
- 支持本地模型、GPU 能力检测和下载管理时，要与云端适配器走同一网关协议。
- 自动更新必须签名；失败能回滚到上一版本。
- 文件关联只应面向本作品自己的存档或内容包格式，不应把产品变成通用角色卡导入器。

### 12.5 成熟项目实证的正确用法

固定 commit 的 RisuAI 同时存在 Vite 前端、Capacitor Android 配置和 Tauri 桌面配置，证明共享前端向多壳发行可行。它同时也是通用 AI 客户端，因此：

- 可参考其跨平台工程拆分和平台能力适配。
- 不照搬其“导入多角色、多供应商通用客户端”的产品边界。
- 不将其插件权限模型直接视作安全标准。

## 13. 从旧 ST 角色卡迁移：一次性开发流程

### 13.1 迁移不是运行时功能

若作品最初在 ST 中开发，可以写**开发期一次性转换工具**提取文本、世界书、状态 Schema、正则意图和资产。转换结果经人工审查后提交到本应用的 `content/` 源码。

生产应用不得携带“导入任意 ST 卡并执行”的入口。迁移工具也不应成为运行时依赖。

### 13.2 能迁什么，必须重写什么

| 旧项目内容 | 独立作品中的去向 | 处理方式 |
|---|---|---|
| 角色描述、世界观、开场白 | `content/characters`、`world`、`scenes` | 可抽取文本，人工拆分和版本化 |
| 世界书条目 | 内建 `LoreEntry` | 转字段后复核激活、递归、预算和插入语义 |
| system prompt / PHI / 预设 | `prompts` 与装配计划 | 重排并建立 Golden Test |
| ST 宏 | 自有白名单宏 | 逐个映射；未知宏构建失败 |
| MVU Schema / InitVar | `state/schema`、初值工厂 | 迁移数据模型，不携带 MVU 运行时 |
| MVU 更新规则 | StateOperation 协议与规则引擎 | 重写写权限、校验和事务 |
| EJS 条目 | 类型化模板、selector 或规则函数 | 禁止原样 eval |
| 酒馆正则 | 渲染转换、输出解析或 UI 组件 | 按“隐藏/解析/美化”意图重写 |
| Tavern Helper 脚本 | 引擎 service、command、selector | 逐功能重实现，不执行原脚本 |
| 卡内 HTML | 作品 UI 组件 | 按页面功能重建并做安全审查 |
| 图片、音频、视频 | `assets` | 检查授权、格式、大小和哈希 |
| 旧聊天 | 可选的开发期存档迁移器 | 只服务本作品，转换后过当前 Schema |

### 13.3 兼容等级

独立作品不应用“能导入卡”衡量兼容性，而用行为迁移等级：

| 等级 | 目标 |
|---|---|
| M0 · 文本迁移 | 角色、世界观、开场、示例和资产进入内容域 |
| M1 · 基础游玩迁移 | Prompt、历史、流式生成、Persona、存档闭环一致 |
| M2 · 玩法迁移 | Lore、宏、变量、UI 动作、正则意图完成原生重实现 |
| M3 · 行为对齐 | 分支、群像、复杂更新规则和旧存档达到作品验收清单 |
| M4 · 原生增强 | 利用独立应用能力重做通知、离线、音画、更新和跨端同步 |

“字段读进来了”只等于 M0，不代表玩法兼容。

### 13.4 ST 专属机制不能凭空运行

- Tavern Helper API 依赖它自己的宿主和事件系统。
- MVU 依赖消息变量、解析和更新生命周期。
- EJS 条目依赖对应模板执行环境与注入上下文。
- 酒馆正则依赖 ST 的消息、Prompt 和 Markdown 执行阶段。
- ST 世界书扩展字段依赖 ST 扫描器的具体实现。

独立应用必须为每项选择：原生重实现、构建期转换、明确降级或不支持。禁止在 README 写“兼容”却没有测试矩阵。

Character Card V3 规范中的字段、Lorebook、宏和资源 URI 可帮助理解旧内容；它不要求本作品继续以卡片封装体运行。**high · 规范**

## 14. 安全与隐私

### 14.1 信任边界

| 输入 | 默认信任 |
|---|---|
| 随签名发行的代码与内容包 | 校验签名和哈希后可信 |
| 玩家输入 | 不可信文本 |
| 模型输出 | 不可信文本和不可信结构化数据 |
| 外部 API 响应 | 不可信网络数据 |
| 远程图片、音频、链接 | 不可信资源 |
| 导出的旧存档 | 导入前不可信 |
| Mod / 插件 | 不可信代码 |

### 14.2 API Key

- Web 端优先会话密钥或作品方代理；不得记录在分析日志和崩溃报告。
- App / PC 使用系统安全存储。
- 设置页能查看密钥归属、测试连接、替换和彻底删除。
- 导出存档默认不包含密钥。
- 代理必须鉴权、限流、限制上游、过滤敏感日志并防 SSRF。
- 自定义端点要显示完整域名和数据风险，默认拒绝内网与元数据地址。

### 14.3 状态写入安全

- 对模型操作做 JSON 结构校验、路径白名单和范围校验。
- 单回合操作数量、数组增长和文本长度设上限。
- 禁止原型污染键。
- 规则引擎计算派生字段，模型无权覆盖。
- 提交前比较 base revision；并发或迟到结果不能覆盖新状态。

### 14.4 内容与更新供应链

- 代码、内容包和增量资产都签名。
- 清单记录每个文件的哈希和大小。
- 解压内容包防路径穿越、压缩炸弹和同名覆盖。
- 更新前确认磁盘空间，失败保留旧版本。
- 内容热更新与 Schema 迁移必须绑定，不能只换 Prompt 不迁存档。

### 14.5 插件与 Mod

单作品 MVP 建议不提供任意代码插件。若确实需要：

- 使用权限清单：网络域、存档只读/写、文件、UI 插槽分别授权。
- 默认在 Worker 或受限进程运行，不给 DOM、密钥和完整数据库。
- 每个插件独立 KV 命名空间。
- 用户可撤销权限、禁用并查看来源与签名。
- 插件崩溃不能破坏主存档。
- 作品内容包中的脚本也遵守同一权限边界，不能因为“随作品发布”就无限制执行。

### 14.6 数据与遥测

- 本地存档默认属于玩家，可完整导出和删除。
- 云同步前明确加密、冲突策略和保留期。
- 敏感 Persona、聊天和状态支持设备级加密。
- 遥测默认最小化并可关闭；不得上传完整 Prompt、聊天或 API Key。
- 崩溃日志先脱敏，再询问是否上传。

- 构建期迁移工具不进入生产依赖图。

## 17. 最小可用里程碑

### P0 · 作品内容冻结

产物：

- 内容域清单和稳定 ID。
- 状态树、初值、可写路径。
- Prompt 装配顺序和输出协议。
- 页面与主玩法闭环。
- 从旧卡迁移的“保留、重写、放弃”对照表。

稳定门：

- 所有 ST 专属依赖均有处置结论。
- 没有运行时卡片导入器。
- 新周目的唯一来源是本作品初始化器。

### P1 · 单端垂直切片

建议先选 Web 或 PC：

- 新建 Persona 和周目。
- 一条完整生成链。
- 流式、取消、重试。
- 状态操作校验并原子提交。
- 保存、退出、恢复。

稳定门：

- 断网、取消、刷新、崩溃不产生“消息成功但状态丢失”。
- Prompt Trace 与状态 diff 可检查。

### P2 · 玩法与分支

- Lore、宏、摘要。
- Swipe、分支和检查点。
- 作品专属地图、背包、任务等核心系统。
- 安全渲染与无障碍。

稳定门：

- 分支切换带着正确状态版本。
- Lore 与 Prompt Golden Test 稳定。

### P3 · 三壳共享

- Web、App、PC 接入同一 core。
- 平台密钥、文件、更新和生命周期适配。
- 同一匿名测试存档跨壳读取。

稳定门：

- 同样输入和固定模型 mock 产生同样 Prompt Trace、状态 diff 和分支结构。
- 平台特有错误都有可恢复路径。

### P4 · 发行

- 签名安装包和内容清单。
- 更新、迁移、回滚。
- 隐私、许可、分级和第三方资产清单。
- 崩溃恢复、备份和玩家导出。

稳定门：

- 从所有公开旧版本升级的自动化回放通过。
- 生产构建不包含开发期迁移器、测试密钥或调试后门。

## 18. 测试矩阵

| 面 | 最低用例 |
|---|---|
| 内容构建 | 重复 ID、断引用、缺资产、Prompt 超预算、无迁移链必须失败 |
| 新周目 | 不同 Persona / 路线 / 难度；固定 seed 可复现 |
| Prompt | Golden Test、裁剪顺序、Prompt Trace、不同模型能力降级 |
| Lore | constant、主/次键、递归环、预算、无效正则、最大轮数 |
| 宏 | 未知宏、转义、随机种子、敏感状态不可见 |
| 模型 | 正常流、碎片边界、空流、限流、超时、取消、迟到 chunk |
| 状态 | 越权路径、类型错、范围错、超大操作、revision 冲突 |
| 消息 | Swipe、编辑、分支、删祖先、检查点恢复、摘要失效 |
| 存档 | 原子提交、断电模拟、备份、损坏检测、逐级迁移、降级拒绝 |
| Web | IndexedDB 配额、PWA 更新、CORS、刷新恢复、多标签页冲突 |
| App | 后台/前台、进程回收、低内存、分享导出、安全密钥库 |
| PC | 原子文件、Keychain、签名更新、路径权限、本地模型中断 |
| 渲染 | XSS、危险 URL、超长 Markdown、恶意图片、CSS 污染 |
| 网络 | SSRF、代理鉴权、证书错误、DNS 失败、离线 |
| 更新 | 内容哈希错、签名错、空间不足、迁移失败、回滚 |
| 跨壳 | 同一存档结构、同一 Prompt Trace、同一状态事务语义 |

模型行为不稳定时，用固定 mock 流验证引擎确定性，再单独做真实模型验收；不要把两种失败混在一起。

## 19. 验收清单

### 19.1 定义对齐

- [ ] 开篇和产品入口都明确“程序本身就是角色卡”。
- [ ] 没有运行时角色卡导入、通用卡解析器或多卡角色库。
- [ ] Web / App / PC 是同一作品的发行壳。
- [ ] 多存档表示同一作品的不同周目。

### 19.2 内容与引擎

- [ ] 角色、世界、Prompt、Lore、状态、规则、UI 和资产全部版本化内建。
- [ ] 内容构建可静态校验并生成哈希清单。
- [ ] Prompt 装配可追踪、可做 Golden Test。
- [ ] 状态写入经过 Schema、权限和事务校验。
- [ ] UI Command 与文字输入走同一主链路。

### 19.3 脱离 ST

- [ ] 无 ST、Tavern Helper、MVU、EJS 运行时依赖。
- [ ] 旧世界书、宏、正则和脚本均已重实现或明确降级。
- [ ] 旧卡迁移工具只存在于开发工具，不进入生产包。
- [ ] 没有用“字段已导入”代替玩法兼容验收。

### 19.4 数据与平台

- [ ] 消息和状态原子提交。
- [ ] Swipe、分支和检查点绑定正确状态版本。
- [ ] 所有公开旧存档有迁移和备份路径。
- [ ] 三壳共享 Schema 与 core，平台差异封装在 adapter。
- [ ] 后台、断网、取消、更新失败都有恢复策略。

### 19.5 安全与发行

- [ ] 模型输出不执行 HTML / JavaScript。
- [ ] API Key 不进入存档、日志和遥测。
- [ ] 内容与更新包校验签名和哈希。
- [ ] 导入旧存档、外链和远程资源按不可信输入处理。
- [ ] 生产包无调试密钥、迁移临时脚本和无限权限插件。

## 20. 悬案与真机验证项

以下不能仅靠架构文档判定：

1. **作品范围**：首版必须支持哪些玩法系统，哪些留到后续单功能迭代。
2. **状态输出协议**：目标模型在 JSON Schema / tool call / 文本标签之间的稳定性。
3. **Prompt 顺序**：不同模型对系统消息、状态投影和后置约束的响应差异。
4. **移动后台**：iOS / Android 在真实长流、锁屏和低内存下的中断行为。
5. **Web BYOK**：目标供应商的 CORS、浏览器鉴权和合规限制。
6. **跨端同步**：是否需要云同步；需要时的端到端加密和冲突策略。
7. **本地模型**：最低硬件、模型大小、量化、首 token 延迟与电量。
8. **内容更新**：旧周目是否锁内容版本，还是强制迁移到新版剧情规则。
9. **Mod 能力**：是否真的需要；若需要，首版允许到“数据 Mod”还是“代码插件”。
10. **发行合规**：模型供应商条款、应用商店分级、成人内容和用户生成内容政策。

每项必须给出“测试设备 / 应用版本 / 内容版本 / 模型 / 结果 / 结论”，不以单次成功代替稳定验收。
