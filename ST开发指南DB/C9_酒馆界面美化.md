# 酒馆界面美化

> 文档版本：v1.0.0 ｜ 最后更新：2026-07-15
>
> **定位**：美化 SillyTavern 宿主界面本身，包括主题、Custom CSS、聊天画布、输入区、菜单、抽屉、弹窗和移动端宿主布局。
>
> **硬边界**：宿主主题是用户级配置，通常影响所有角色和聊天。角色卡不得静默接管或永久改写用户全局主题。消息正文、卡内 HTML、正则渲染和 iframe 内 CSS 归 **《C3_HTML美化与CSS.md》**；需要 JS 动态行为时升级为 **《D5_ST扩展开发完整指南.md》** 的 UI Extension。

## 1. 定义与三条实现路线

### 1.1 什么叫“酒馆美化”

本册的“酒馆”指 ST 顶层宿主 UI，而不是某条消息或某张卡片的前端：

```text
SillyTavern 宿主 document
  ├─ 顶栏 / 菜单
  ├─ 左右抽屉
  ├─ 聊天画布与消息外壳
  ├─ 输入与发送区
  ├─ Popup / Modal / Toast
  └─ 用户主题和 Custom CSS
```

因此它的影响范围更大、回滚要求更高，也更容易与其他扩展和主题冲突。

### 1.2 三条路线与选择顺序

| 路线 | 适合 | 优先级 |
|---|---|---:|
| ST 原生 UI Theme | 颜色、字体、宽度、模糊、消息布局等官方可配置项 | 首选 |
| Theme 内 Custom CSS | 原生设置无法覆盖的静态样式 | 次选 |
| UI Extension | 动态 DOM、设置面板、事件、运行时切换、复杂行为 | 仅必要时 |

原则：能用主题令牌解决，不写深层选择器；能用 CSS 解决，不注入 JS；必须动态操作时才转 D5。

### 1.3 与 C3 的判定表

| 判断项 | C9 酒馆美化 | C3 卡内 HTML/CSS |
|---|---|---|
| DOM | ST 顶层宿主 | `.mes_text` 或卡内 iframe |
| 影响范围 | 用户主题，通常跨聊天 | 某卡/某楼层 |
| 首选载体 | UI Theme / Custom CSS | 正则 HTML / iframe |
| 动态行为 | UI Extension | Tavern Helper / iframe 脚本 |
| 分发 | 用户显式导入/启用 | 可随角色卡携带 |
| 回滚 | 恢复主题/删除 Custom CSS | 关闭正则/脚本/重导卡 |

卡内 CSS 能否穿透宿主不是授权问题。即使技术上能访问父文档，也不得把角色卡加载等同于用户同意修改全局主题。

## 2. 影响范围与权限边界

### 2.1 用户显式启用

主题、美化扩展或 Custom CSS 包必须要求用户主动：

- 预览影响范围；
- 导出现有主题或生成回滚点；
- 导入/安装；
- 选择启用；
- 确认外部字体、图片和网络来源；
- 知道如何恢复默认。

角色卡可以在 creator notes 或开局依赖门中推荐主题，但不能：

- 静默写入用户 Custom CSS；
- 把全局主题改成角色专属配色后不恢复；
- 隐藏设置、扩展管理或发送按钮阻止退出；
- 用远程样式在用户不知情时追踪或加载资源；
- 把“显示最佳”写成“必须授予任意宿主 DOM 权限”。

### 2.2 主题是全局表面

在不同 ST 配置中，主题可能是用户级、账号级或当前实例级。不要假定它只影响当前角色。设计时必须用其他普通卡、群聊、长聊天和设置页验证，避免角色专属视觉污染全局。

### 2.3 何时用卡片专属样式

若要求仅影响某张卡的消息内容，应留在 C3；若必须影响宿主，但只在目标卡启用，应使用按目标 ST 版本声明 `activate/enable/disable/clean/delete` 等真实 manifest hooks 的 UI Extension，并以当前聊天身份为作用域。扩展内部可以另设幂等 `destroy()` 统一清理监听、DOM 和样式，但它不是 manifest hook。纯 Custom CSS 没有可靠的卡片生命周期，不适合实现复杂卡片专属宿主换肤；精确 hook 版本边界见 D5。

## 3. 宿主 DOM 稳定性分级

### 3.1 三层稳定度

| 层级 | 示例 | 策略 |
|---|---|---|
| A：官方主题变量/设置 | `--SmartTheme*`、字体缩放、模糊强度、聊天宽度 | 优先依赖 |
| B：长期存在的语义区域 | 聊天根、消息、输入区、抽屉、Popup | 小范围选择器 + 兼容矩阵 |
| C：内部结构/自动生成类 | 深层 nth-child、图标内部节点、第三方扩展 DOM | 避免；必要时能力检测 |

不要因为某个选择器在一版里存在，就把它写成永久公共 API。

### 3.2 选择器纪律

```css
/* 较稳：根令牌 */
:root {
  --my-theme-radius: 14px;
}

/* 可接受：限定宿主区域 */
#chat .mes {
  border-radius: var(--my-theme-radius);
}

/* 脆弱：依赖深层位置和主题内部结构 */
#sheld > div:nth-child(2) > div:nth-child(4) span:nth-child(1) {
  display: none;
}
```

Custom CSS 直接进入宿主样式层；消息渲染中的 `custom-*` 类改名机制属于卡内净化/格式化链，不应当成宿主主题选择器规范。

### 3.3 不复制整份官方 CSS

只覆写需要改变的表面。复制整份 `mobile-styles.css`、核心主题或扩展 CSS 会冻结一份旧结构，后续 ST 更新时既覆盖新修复，又让冲突难以定位。

## 4. 主题令牌与视觉系统

### 4.1 令牌先行

先定义语义令牌，再映射到具体区域：

```css
:root {
  --my-bg: #11131a;
  --my-surface: rgba(26, 29, 39, 0.92);
  --my-surface-strong: #202431;
  --my-text: #f3f5fb;
  --my-text-muted: #aeb5c7;
  --my-accent: #9bb8ff;
  --my-danger: #ff8c9a;
  --my-border: rgba(255, 255, 255, 0.12);
  --my-radius-sm: 8px;
  --my-radius-lg: 16px;
  --my-shadow: 0 12px 34px rgba(0, 0, 0, 0.34);
}
```

需要覆盖：

- 背景、表面、强调表面；
- 主文字、次文字、禁用文字；
- 链接、强调、成功、警告、错误；
- 边框、焦点环、选中态；
- 小/中/大圆角；
- 阴影、模糊、间距；
- 字体族、字号比例和行高。

### 4.2 对接 ST 变量

ST 主题已经暴露多组 `--SmartTheme*`、字体缩放、模糊强度、聊天宽度等变量。实际字段集合随版本变化，导出目标版本主题后再映射；不要凭旧版变量表硬写。

推荐：

```css
:root {
  --my-text: var(--SmartThemeBodyColor, #f3f5fb);
  --my-bg: var(--SmartThemeBlurTintColor, #11131a);
  --my-shadow-strength: var(--shadowStrength, 1);
}
```

使用 fallback，避免变量缺失时整块文字透明或背景消失。

### 4.3 Flat / Bubble / Document 模式

官方聊天显示模式会改变消息外壳、头像和宽度。美化包至少明确：

- 只支持某一种模式；或
- 为三种模式分别提供选择器；或
- 仅改令牌，不假定具体外壳结构。

不要在 Bubble 模式调好后宣称“全模式兼容”。

## 5. 宿主各区域美化策略

### 5.1 背景与聊天画布

- 背景图必须有纯色 fallback；
- 保证文字区域有足够遮罩，不让高对比图片穿透；
- `background-attachment: fixed` 在移动端可能掉帧，应降级；
- 超大图片按目标分辨率压缩；
- 明确全局背景与角色背景的叠加规则。

### 5.2 用户/AI 消息

检查：

- 用户与 AI 是否仍可区分；
- 编辑、复制、Swipe、删除等消息操作是否可见；
- 代码块、引用、表格、长 URL 是否溢出；
- 系统消息、错误消息、群聊角色名是否有单独语义；
- hover-only 操作在触摸设备是否可达。

### 5.3 头像

头像形状、尺寸、边框和在线/说话状态要独立。不要通过隐藏头像父节点的方式省空间，可能连带隐藏角色名和消息操作区。

### 5.4 输入与发送区

输入区是逃生主链，不允许为了视觉而牺牲：

- 输入文字对比度；
- placeholder 可读性；
- 发送、停止生成、附件、快捷回复按钮；
- 多行扩展与软键盘；
- focus ring；
- 禁用/生成中状态。

### 5.5 抽屉、顶栏、Popup 与 Toast

这些区域通常使用独立层级和 z-index。美化时需验证：

- 左右抽屉不会被聊天画布盖住；
- Popup 遮罩和主体对比足够；
- Toast 不被固定底栏遮挡；
- 魔棒/扩展菜单仍能滚动和点击；
- 第三方扩展设置面板继承的变量不会变成黑底黑字。

## 6. Custom CSS 层叠与冲突控制

### 6.1 分层组织

```css
@layer my-theme.reset, my-theme.tokens, my-theme.layout,
       my-theme.components, my-theme.utilities;
```

若目标浏览器支持 cascade layers，可用层控制优先级；否则仍按“令牌→布局→组件→修补”顺序组织文件。避免从头到尾堆 `!important`。

### 6.2 作用域

纯主题通常直接作用宿主。若是 UI Extension 动态启用，先给根节点加标志类：

```javascript
document.documentElement.classList.add('my-theme-enabled');
```

```css
.my-theme-enabled #chat .mes { /* ... */ }
```

禁用/卸载时移除标志类和注入 style。这样比给每条规则盲目提高 specificity 更可控。

### 6.3 冲突定位

遇到样式不生效，按顺序检查：

1. 目标元素是否仍存在；
2. 规则是否命中正确宿主 document；
3. computed style 的胜出来源；
4. 主题切换是否替换了 Custom CSS；
5. 第三方扩展是否后注入更高优先级；
6. 移动断点是否覆盖桌面规则；
7. 属性内联样式是否来自运行时 JS。

不要先加 `!important`，否则只会掩盖错误选择器或错误层级。

### 6.4 禁止隐藏关键控件

任何主题都不应全局隐藏：

- 用户设置与主题恢复入口；
- 扩展管理；
- 输入框、发送、停止生成；
- Popup 关闭/取消；
- 消息编辑/删除的唯一入口；
- 无障碍焦点指示。

如果设计确需视觉隐藏，仍要保留可访问替代与“安全模式”恢复路径。

## 7. 响应式、动态视口与安全区

### 7.1 断点不是永久常量

当前核验快照中 ST 核心移动样式存在约 1000px 主断点，但这属于版本敏感实现，不能作为永远不变的 API。自定义主题应根据自身内容设置少量断点，并在目标 ST 版本复核核心布局。

### 7.2 `dvw/dvh` 与 visualViewport

宿主固定表面优先使用动态视口单位，并在必须 JS 定位时读取 `visualViewport`。尤其验证：

- 手机浏览器地址栏收放；
- 软键盘；
- PWA standalone；
- 横屏；
- 浏览器缩放；
- 桌面窄窗。

### 7.3 Safe Area

```css
.my-bottom-surface {
  padding-bottom: max(8px, env(safe-area-inset-bottom));
  padding-left: max(8px, env(safe-area-inset-left));
  padding-right: max(8px, env(safe-area-inset-right));
}
```

不要只测竖屏底部 Home 条；横屏左右刘海和 PWA 顶部区域同样需要验证。

### 7.4 移动端不复制核心布局

只覆盖颜色、间距、圆角等必要表面。若重新定义整个抽屉宽度、顶栏定位和发送区高度，等于自行 fork ST 移动布局，后续升级成本极高。

## 8. 可访问性与低性能模式

### 8.1 必验项

- 正文和背景对比度；
- 链接与普通文字不仅靠颜色区分；
- 清晰 focus-visible；
- 200% 字体下不截断关键按钮；
- 粗指针热区至少约 44px；
- 错误/成功不只用红绿；
- 动画不会导致眩晕或阻塞操作。

### 8.2 尊重用户设置

官方提供 Reduced Motion、No Blur、No Text Shadow 等低性能/可访问设置。主题不应通过更高优先级把这些效果强行加回来。

```css
/* ST 内部 Reduced Motion 开关 */
body.reduced-motion *,
body.reduced-motion *::before,
body.reduced-motion *::after {
  animation: none !important;
  transition-duration: 1ms !important;
  scroll-behavior: auto !important;
}

/* 操作系统偏好 */
@media (prefers-reduced-motion: reduce) {
  .decorative-animation {
    animation: none !important;
  }
}
```

纯 UI Theme 不依赖 `.my-theme-enabled` 这类只有 UI Extension 才会动态添加的根类。若用扩展控制主题，可额外加作用域，但仍须响应 `body.reduced-motion`。

### 8.3 模糊降级

```css
.my-surface {
  background: rgba(25, 28, 37, 0.96);
}

@supports (backdrop-filter: blur(8px)) {
  .my-surface {
    background: rgba(25, 28, 37, 0.76);
    backdrop-filter: blur(8px);
  }
}

body.no-blur .my-surface { backdrop-filter: none !important; }
body.noShadows .my-surface { box-shadow: none !important; }
```

低端机长聊天滚动掉帧时，先关闭 blur、阴影、固定背景和粒子；不能以降低文字对比度换性能。

## 9. 字体、图片和外部资源

### 9.1 字体

推荐本地系统字体栈或用户显式安装的字体。外部字体要考虑：

- 加载失败 fallback；
- CORS/CSP；
- 首屏闪烁；
- 字形是否覆盖中日韩；
- 许可；
- 外部请求暴露 IP、Referer 和使用时间。

```css
font-family: "Noto Sans SC", "Microsoft YaHei", system-ui, sans-serif;
```

### 9.2 不默认推荐 `@import`

远程 `@import` 隐藏了供应链和加载顺序；在当前 ST 实现中，**导入含 `@import` 的主题**会弹出安全确认。优先把审计后的 CSS 随主题包分发；确需远程加载时列出固定来源、版本、完整性策略和离线 fallback。

### 9.3 图片

- 背景图压缩并提供多尺寸；
- 使用稳定 HTTPS；
- 断网时不影响关键 UI；
- 不用外链像素作为追踪；
- 不把含敏感 token 的 URL 写入主题；
- 明确作者、许可和再分发权。

## 10. 打包、导入导出与版本迁移

### 10.1 包内容

建议包含：

```text
my-theme/
  README.md
  theme.json              # ST 导出的主题快照
  custom.css              # 如需单独审计
  assets/
  compatibility.md
  CHANGELOG.md
  preview/
```

README 必须写：目标 ST 快照、启用方式、外部资源、影响范围、已知冲突、恢复步骤。

### 10.2 导入前备份

官方主题管理支持保存、导入、导出。导入第三方主题前先：

1. 保存当前设置为主题；
2. 导出备份；
3. 截取关键设置；
4. 再导入新主题；
5. 验证关键控件；
6. 有问题立即回滚。

主题切换会使用目标主题记录的 Custom CSS 替换当前 Custom CSS，这是预期行为。重要修改要先保存回主题或导出，不应当成随机丢失。

### 10.3 版本迁移

每次 ST 大版本更新：

- 比较官方主题字段；
- 检查已删除/重命名变量；
- 用 DevTools 找失效选择器；
- 复测移动断点、抽屉和 Popup；
- 记录 `ST版本 / 主题版本 / 核验日期 / 已知冲突`；
- 不用“最新版通用”代替兼容矩阵。

## 11. 安全回滚

### 11.1 发行必须提供逃生路径

- 导入前备份原主题；
- 提供一键关闭美化扩展；
- 不隐藏用户设置；
- 保留纯 CSS 文件，便于定位坏规则；
- UI Extension 实现 `destroy` 并移除标志类/style；
- 外部资源失败回退系统字体和纯色；
- 文档列出恢复默认主题步骤。

### 11.2 页面已被美化“弄坏”

恢复顺序：

1. 尝试进入用户设置切回已知可用主题；
2. 禁用对应 UI Extension；
3. 清空/替换 Custom CSS；
4. 刷新并确认原生输入、发送和设置恢复；
5. 逐段重新启用 CSS，定位最小故障块；
6. 不在故障状态继续堆覆盖规则。

具体文件级恢复依部署方式而异，见 D4；本册不建议直接修改 ST 核心 CSS。

## 12. 高频错误

| 错误 | 后果 | 修复方向 |
|---|---|---|
| 把卡内 CSS 当宿主主题 | 影响范围和权限混乱 | C3/C9 分层 |
| 卡片静默改全局主题 | 污染所有聊天 | 用户显式导入/启用 |
| 复制整份核心 CSS | 升级后大量冲突 | 令牌和最小覆写 |
| 深层 nth-child | 小更新即失效 | 语义区域 + 能力检测 |
| 无脑 `!important` | 无法维护、扩展互相打架 | 分层与 computed style 定位 |
| 隐藏设置/发送 | 用户无法恢复或聊天 | 关键控件禁止隐藏 |
| 只在 Bubble 测试 | Flat/Document 破版 | 明确支持矩阵 |
| 只测桌面 | 手机软键盘/安全区遮挡 | 真机移动矩阵 |
| 强制 blur/动画 | 低端机掉帧、违背用户偏好 | No Blur/Reduced Motion 降级 |
| 远程字体无 fallback | 断网白字/错位 | 系统字体栈 |
| 主题切换后 CSS“丢失” | 未保存的修改被替换 | 先保存/导出主题 |
| 把宿主选择器当永久 API | ST 更新后失效 | 版本快照 + 兼容测试 |

## 13. 验收矩阵

| 维度 | 必验场景 |
|---|---|
| 主题生命周期 | 新建、保存、更新、导出、导入、切换、删除、恢复默认 |
| 表面覆盖 | 背景、三种聊天模式、头像、消息、输入区、抽屉、Popup、Toast、Wand |
| 桌面 | 1366/1920、窄聊天宽度、全宽、浏览器缩放 |
| 移动 | 320/375/430、横屏、软键盘、PWA safe-area |
| 可访问性 | 键盘焦点、对比度、200% 字体、44px、Reduced Motion |
| 性能 | No Blur、低配机滚动、长聊天、动画关闭 |
| 失败 | 外部字体/背景断网、CSS 语法错误、第三方主题/扩展冲突 |
| 回滚 | 始终能进入设置并恢复；关键交互未隐藏 |
| 隔离 | 切角色/聊天不出现角色卡私有样式意外污染 |
| 兼容 | 最低支持版与当前核验版各跑一次并记录日期 |

稳定完成门：只有在“启用、正常使用、禁用、恢复默认”四条链都可复现时，主题才可分发；视觉截图不能替代交互、性能和回滚验收。

2026-07-14 核验时，SillyTavern release 页面显示 `1.18.0`。这只是当日快照，不是本册永久基线。既有指南中的 `1.14.0` 结论应视为历史观察，施工必须以目标实例为准。

- 宿主 DOM ID/class、Popup 和抽屉结构；
