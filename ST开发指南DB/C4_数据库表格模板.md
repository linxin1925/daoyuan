## 1. 定义、边界与选型

### 1.1 它是什么

SP·数据库是一套运行在 SillyTavern 中、由自身脚本管理的**二维表格变量系统**：

```text
剧情正文 / 世界书 / 人设
          ↓
独立填表提示词与 API
          ↓
AI 输出表格编辑指令
          ↓
原生二维数组服务 或 SQLite 运行时
          ↓
V2 checkpoint + operation log
          ↓
写回当前聊天消息，并可同步到世界书
```

它的核心不是一棵 JSON 变量树，而是多张有表头、数据行、建表说明和增删改触发语义的表。

来源：`src/shared/models/table-data.ts`、`src/shared/table-storage-provider.ts` ｜ 置信度：high `[已核快照源码]`

### 1.2 它不是 MVU

| 维度 | SP·数据库 | MVU |
|---|---|---|
| 基本模型 | `mate + sheet_* + Sheet.content` 二维表 | 消息变量中的 `stat_data` 对象树 |
| 结构约束 | 表头、DDL、Note、SQL 约束 | zod schema / 变量规则 |
| AI 更新 | 独立填表提示词，输出 DSL、SQL 或严格 JSON | 主回复或额外解析模型输出变量命令/JSONPatch |
| 持久化 | 插件自有聊天字段、V2 checkpoint/log | MVU 消息变量快照 |
| 前端入口 | 宿主窗口的 `AutoCardUpdaterAPI` | `window.Mvu` / Tavern Helper 变量接口 |
| 世界书 | 可把表格视图同步成插件生成条目 | 通常用 EJS/宏读取消息变量 |

因此，本文严禁把表格称为“MVU schema”，也不使用 `registerMvuSchema`、`defineMvuDataStore`、`stat_data` 或 JSONPatch 来解释 SP·数据库的内部机制。

两套系统可以同时装在一张卡上，但必须明确谁拥有哪一份状态。不要让同一项数值同时由 MVU 和数据库写入，否则会出现双写、回滚不一致和提示词互相覆盖。

来源：两套系统已核源码接口对照 ｜ 置信度：high `[源码边界]`

### 1.3 什么时候选数据库表格

适合：

- 角色、背包、任务、纪要、阵营等天然按“行 × 列”组织的数据；
- 作者希望在可视化编辑器里直接查看和修改；
- 需要 SQL 查询、聚合、排序、条件更新或多表关联；
- 希望使用独立填表模型，而不是让剧情模型同时承担变量更新；
- 需要把指定表或索引同步到世界书。

不适合：

- 只有少量嵌套开关，二维表反而增加复杂度；
- 必须与 MVU 状态栏、MVU 更新事件完全同链；
- 没有能力维护表结构、提示词、持久化和迁移；
- 把所有剧情正文无限塞进单元格，导致提示词与存档膨胀。

## 2. 一份数据库到底由什么组成

### 2.1 顶层结构

当前合并后的运行时数据形态是：

```ts
type TableDataObject = {
  mate: Mate;
  [sheetKey: `sheet_${string}`]: Sheet | Mate;
};
```

典型布局：

```text
currentJsonTableData_ACU
├─ mate
├─ sheet_global_state
├─ sheet_inventory
├─ sheet_characters
└─ sheet_custom_xxx
```

`sheet_*` 是内部键；给人和 AI 看的表名是 `sheet.name`。模板制作时建议让对象键与 `sheet.uid` 保持一致并长期稳定，不要用中文表名充当内部键。

来源：`TableDataObject_ACU`、内置表默认定义 ｜ 置信度：high `[已核快照源码]`

### 2.2 `mate` 元信息

```ts
interface Mate {
  type: string;                 // 当前模板为 chatSheets
  version: number;
  updateConfigUiSentinel: number;
  globalInjectionConfig: {
    readableEntryPlacement: PlacementConfig;
    wrapperPlacement: PlacementConfig;
  };
}
```

`mate` 描述整个模板，而不是某张表。`importTemplateFromData()` 的当前校验要求至少满足：

- 有 `mate`；
- `mate.type === "chatSheets"`；
- 至少有一个 `sheet_*`；
- 每张表有 `name`、`content`、`sourceData`。

来源：`API_DOCUMENTATION.md`、`template-preset-api.ts` ｜ 置信度：high `[已核快照源码 + 项目 API 文档]`

### 2.3 `Sheet` 完整结构

```ts
interface Sheet {
  uid: string;
  name: string;
  sourceData: {
    note: string;
    initNode: string;
    insertNode: string;
    updateNode: string;
    deleteNode: string;
    ddl?: string;
  };
  content: (string | null)[][];
  updateConfig: {
    uiSentinel: number;
    contextDepth: number;
    updateFrequency: number;
    batchSize: number;
    skipFloors: number;
    sendRowsSqlTemplate?: string;
  };
  exportConfig: SheetExportConfig;
  orderNo: number;
  seedRows?: (string | null)[][];
}
```

### 2.4 `content` 是二维表

```js
content: [
  ['row_id', '物品名称', '数量', '描述/效果', '类别'], // 表头
  ['1', '铁剑', '3', '锋利的铁制长剑', '武器'],       // 第 1 条数据
  ['2', '治疗药水', '10', '恢复体力', '消耗品'],      // 第 2 条数据
]
```

铁律：

1. `content[0]` 是表头，不是数据。
2. 第一条数据是 `content[1]`，外部 CRUD 的 `rowIndex=1` 也表示第一条数据。
3. SQLite 模式仍保留这份 JSON 二维视图，供上层 UI 与兼容 API 使用。
4. SQLite 表的首列必须是 `row_id`；不要让业务字段冒充内部行号。
5. 表头中文名、DDL 中文注释、Note 中的列名必须一致。

来源：`models/table-data.ts`、`syntax-reference.md` ｜ 置信度：high `[已核快照源码 + 项目文档]`

### 2.5 `sourceData` 是 AI 的表格说明书

| 字段 | AI 何时使用 | 应写什么 |
|---|---|---|
| `note` | 每次把该表发给填表 AI | 一行代表什么、每列含义、范围与格式 |
| `initNode` | 表为空或初始化时 | 初始应创建什么；不需要时明确禁止 |
| `insertNode` | 判断是否新增行 | 新实体判定条件；SQLite 模式附 INSERT 示例 |
| `updateNode` | 判断是否改已有行 | 变化条件、定位键、增量/覆盖语义；附带 WHERE 的示例 |
| `deleteNode` | 判断是否删除行 | 只在什么情况下删除；禁止时直接写“禁止删除。” |
| `ddl` | SQLite 建表和中英文映射 | 英文物理表名、字段类型、约束、中文注释 |

这些内容是填表提示词的一部分，不是仅供作者阅读的备注。

## 3. 原生 DSL 与 SQLite 双模式

### 3.1 共同上层，两个存储 Provider

插件以 `ITableStorageProvider` 统一两种模式：

| 能力 | 原生模式 `native` | SQLite 模式 `sqlite` |
|---|---|---|
| 运行时 | JSON 二维数组 | sql.js 运行时数据库 + JSON 兼容视图 |
| AI 看到的结构 | 表 ID、列号、Note、Trigger、当前行 | DDL、SQL 注释、Note、当前数据 |
| AI 旧式编辑 | `insertRow/updateRow/deleteRow` DSL | `INSERT/UPDATE/DELETE` SQL |
| 复杂查询 | 前端遍历 `content` | SQL、ORM 模板变量、JOIN/聚合 |
| 外部 CRUD | 四个 API 均支持 | 四个 API 均支持，并翻译中英文名 |
| 原始 SQL API | 不支持 | 支持 |

两种模式的模板外壳一致；SQLite 多用 `sourceData.ddl` 建立物理表和 NameMapper。

来源：`table-storage-provider.ts`、`syntax-reference.md` ｜ 置信度：high `[已核快照源码]`

### 3.2 原生模式发给 AI 的摘要

```text
[4:背包物品表]
  Columns: [0:物品名称], [1:数量], [2:描述/效果], [3:类别]
  - Note: ...
  - Insert Trigger: ...
  - Update Trigger: ...
  - Delete Trigger: ...
  [0] 铁剑, 3, 锋利的铁制长剑, 武器
```

AI 依赖表 ID、行号和列号生成 DSL。

### 3.3 SQLite 模式发给 AI 的摘要

```sql
CREATE TABLE inventory ( -- 背包物品表
  row_id INTEGER PRIMARY KEY, -- 行号
  item_name TEXT NOT NULL UNIQUE, -- 物品名称
  quantity INTEGER NOT NULL DEFAULT 1 CHECK(quantity > 0), -- 数量
  description TEXT, -- 描述/效果
  category TEXT -- 类别
);
-- Note: ...
-- INSERT: ...
-- UPDATE: ...
-- DELETE: ...
-- 当前数据 (... rows)
```

DDL 注释同时承担中文显示名到英文物理名的映射。漏写注释会导致自定义表的中文查询和 CRUD 翻译失败。

### 3.4 模式切换不是换变量名

切换模式会重建或重载 Provider；它不是把 DSL 文本机械改成 SQL 文本。生产卡应先确定主模式，再围绕该模式写模板与测试。需要双模式兼容时：

- `note/init/insert/update/delete` 都要避免只在一种模式可理解；
- SQLite `insert/update/delete` 中附 SQL 示例；
- 原生模式仍需确保列语义和触发条件不依赖 SQL 才能读懂；
- 前端优先使用四个跨模式 CRUD API，只有复杂事务才直接写 SQL。

## 4. AI 填表协议与提示词流水线

### 4.1 旧式 `<tableEdit>`：原生模式

默认设置 `strictJsonTableFillEnabled=false` 时，原生模式要求 AI 输出：

```text
<tableEdit>
insertRow(2, {"0":"月影佣兵团","1":"中立","2":"0"})
updateRow(2, 1, {"1":"友好","2":"15"})
deleteRow(2, 3)
</tableEdit>
```

语义：

- `insertRow(tableIndex, {"colIndex":"value"})`；
- `updateRow(tableIndex, rowIndex, {"colIndex":"value"})`；
- `deleteRow(tableIndex, rowIndex)`；
- 表 ID、行号是数字；对象列号是双引号字符串；
- `rowIndex` 对当前发给 AI 的数据快照负责，不能凭显示排序后的 UI 猜。

来源：`DEFAULT_CHAR_CARD_PROMPT_ACU`、DSL parser ｜ 置信度：high `[已核快照源码]`

### 4.2 旧式 `<tableEdit>`：SQLite 模式

```text
<tableEdit>
INSERT INTO faction_relations
  (row_id, faction_name, relation_status, favor_level)
VALUES
  ((SELECT COALESCE(MAX(row_id), 0) + 1 FROM faction_relations), '月影佣兵团', '中立', 0);

UPDATE faction_relations
SET relation_status = '友好', favor_level = favor_level + 15
WHERE faction_name = '月影佣兵团';
</tableEdit>
```

规则：

- 表名、列名使用 DDL 的英文物理名；
- `UPDATE`、`DELETE` 必须带 `WHERE`；
- 定位优先使用 `UNIQUE` 业务主键，单行表才固定 `WHERE row_id = 1`；
- 不输出 `BEGIN/COMMIT/ROLLBACK`，事务由插件管理；
- 不在 AI 编辑里执行 `CREATE/DROP/ALTER`；结构变化走模板/可视化编辑器；
- 多条语句都以分号结尾。

来源：`DEFAULT_CHAR_CARD_PROMPT_SQL_ACU`、`prompt-prepare.ts` ｜ 置信度：high `[已核快照源码]`

### 4.3 严格 JSON 是可选的新协议

已核快照还提供“严格 JSON 填表响应”，默认值为关闭。开启后 AI 不直接输出 `<tableEdit>`：

原生模式：

```json
{
  "format": "table_edit_ops_v1",
  "ops": [
    {
      "op": "update",
      "sheet": "阵营关系表",
      "where": { "阵营名称": "月影佣兵团" },
      "set": { "关系状态": "友好", "好感度": 15 }
    }
  ]
}
```

SQLite 模式：

```json
{
  "format": "table_edit_sql_v1",
  "sql": "UPDATE faction_relations SET relation_status = '友好' WHERE faction_name = '月影佣兵团';"
}
```

解析器会把通过校验的严格 JSON 归一化到内部编辑文本再进入提交链。严格模式与旧式 `<tableEdit>` 不应混写；开启严格模式后，裸 `<tableEdit>` 会被拒绝。

来源：`strict-json-table-fill.ts`、`table-edit-parser.ts`、默认设置 ｜ 置信度：high `[已核快照源码]`

### 4.4 提示词槽位和数据流

默认填表提示词把信息分成：

```text
$U  用户基础设定
$C  角色/卡片设定
$4  世界书内容
$1  正文数据
$0  当前表格数据
$8  手动额外提示
```

实际流程：

```text
收集剧情上下文与世界书
→ 根据更新频率/手动选择确定目标表
→ 格式化当前表格与 sourceData
→ 解析该入口支持的模板变量
→ 调用填表 API/profile
→ 提取严格 JSON 或 <tableEdit>
→ Provider 应用 DSL/SQL
→ V2 提交并保存聊天
→ 刷新合并视图、世界书与 UI 回调
```

不要把 Note、DDL、输出协议重复塞入剧情世界书；它们本来就由数据库提示词流水线提供。

## 5. 自定义表设计流程

### 5.1 先回答六个问题

1. 一行代表什么实体？
2. 是单行表还是多行表？
3. 多行表用哪个业务字段唯一定位？
4. 哪些字段必须非空、唯一、限定范围或限定枚举？
5. 初始化、新增、更新、删除各在什么剧情条件触发？
6. 这张表是否需要同步到世界书；按整表、按行还是只输出索引？

任何一项答不出来，都先不要写 DDL。

### 5.2 定中文显示名与英文物理名

建议：

- 中文表名：名词短语，以“表”结尾，例如“阵营关系表”；
- 英文表名：小写蛇形，例如 `faction_relations`；
- 英文字段：小写蛇形，例如 `relation_status`；
- DDL 每列行尾写精确中文注释；
- DDL 注释中文名必须与 `content[0]` 表头、Note 列名完全一致。

### 5.3 DDL 硬规则

```sql
CREATE TABLE english_table_name ( -- 中文表名
  row_id INTEGER PRIMARY KEY, -- 行号
  business_key TEXT NOT NULL UNIQUE, -- 业务主键
  status TEXT NOT NULL CHECK(status IN ('选项A', '选项B')), -- 状态
  score INTEGER NOT NULL DEFAULT 0 CHECK(score BETWEEN -100 AND 100), -- 数值
  note TEXT -- 备注
);
```

必须满足：

- 第一列为 `row_id INTEGER PRIMARY KEY`；
- 多行表至少有一个真正的 `UNIQUE` 业务主键；
- 枚举用 `CHECK(... IN (...))`；
- 数值范围用 `CHECK`；
- 必填字段用 `NOT NULL`；
- 可推导默认值用 `DEFAULT`；
- 每列都写中文注释；
- 最后一列不留多余逗号；
- 保存后在 UI 执行 DDL 校验。

### 5.4 Note 的标准格式

```text
记录主角阵营与其他已知阵营之间的关系；每个对方阵营占一行。
- 列1: 阵营名称 - 对方阵营的正式名称，必须唯一。
- 列2: 关系状态 - 只允许“友好”“敌对”“中立”“同盟”。
- 列3: 好感度 - 整数，范围 -100 到 100。
- 列4: 首领姓名 - 当前首领；未知时留空。
- 列5: 备注 - 补充信息，不超过 150 字。
```

`row_id` 是内部定位列，业务 Note 从列1开始写最不容易混淆。需要与现有模板合并时，以该模板的列号惯例为准，不要同一批表混用两套编号。

### 5.5 Trigger 必须有正反边界

好的新增规则：

```text
剧情中出现之前未记录、且与主角发生明确互动的新阵营时添加。
仅被路人提及、没有剧情相关性的阵营不要添加。
SQL示例: INSERT INTO ...;
```

好的更新规则：

```text
已有阵营的关系、好感度、首领或实力发生变化时更新。
好感度变化使用加减表达式，不要把“本次变化量”误当作新总值。
SQL示例: UPDATE ... WHERE faction_name = '...';
```

好的删除规则：

```text
仅在阵营被明确彻底消灭或正式解散时删除。
暂时失势、潜伏或被占领只更新状态，不删除。
SQL示例: DELETE FROM ... WHERE faction_name = '...';
```

单行表的 `insertNode` 写“禁止操作。”，`deleteNode` 写“禁止删除。”；不要同时给出禁止文字和可执行示例。

### 5.6 数据量和拆表

单表建议控制在 7～8 个业务字段以内。超过后按实体或更新频率拆表：

- “角色基本档案”与“角色关系变化”分表；
- “任务当前状态”与“任务历史记录”分表；
- 高频数值与低频长文本分表；
- 不要一列塞入难以稳定解析的多层 JSON，失去二维表优势。

此处为项目建表指南的经验门槛，不是 SQLite 引擎硬限制。置信度：medium-high `[项目维护文档]`

## 6. 完整可导入表格模板

下面是一张只有“阵营关系表”的完整模板。它包含 `mate`、`sheet_*`、空数据表头、完整 `sourceData`、更新配置与世界书导出配置，可以作为 `importTemplateFromData()` 的基底。

```json
{
  "mate": {
    "type": "chatSheets",
    "version": 1,
    "updateConfigUiSentinel": -1,
    "globalInjectionConfig": {
      "readableEntryPlacement": {
```

## 7. 模板变量与条件表达式

### 7.1 这是“发给 AI 前的字符串模板”，不是持久变量

模板语法写在插件支持的提示词入口，发送前由数据库解析器替换。它与前端直接调用 `AutoCardUpdaterAPI` 是两条独立链路。

支持的主要入口：

1. 酒馆预设提示词；
2. 填表预设提示词；
3. 正文优化提示词；
4. 剧情推进世界书与最终指令；
5. 剧情推进任务内容。

不同入口支持的语法子集不同。尤其填表入口和剧情推进最终指令不运行 `<calc>/<max>/<min>`；不要因为某个标签在酒馆预设里能用，就假定它在所有入口都能用。

来源：`syntax-reference.md`、对应 prompt pipeline ｜ 置信度：high `[已核快照行为]`

### 7.2 双模式共同语法

```html
<random id="luck" min="1" max="100" />
<calc id="double" expr="cell:背包物品表/铁剑/数量 * 2" />
<max id="top" values="3, 5, $calc:double" />

<if seed="战斗">本轮发生战斗。</if>
<if cell="背包物品表/铁剑/数量 >= 1">仍持有铁剑。</if>
```

注意：

- `random` 的 `min/max` 只接受非负整数字面量；
- `<calc>` 结果会 `Math.floor`；
- 标签变量只在当前消息处理过程中有效，不跨消息持久化；
- `<if>` 是否执行还取决于对应入口和条件模板开关。

### 7.3 SQLite 专属读取语法

ORM：

```text
{[db.背包物品表.where('物品名称','铁剑').get('数量')]}
{[db.inventory.where('item_name','铁剑').get('quantity')]}
{[db.重要角色表.where('是否离场','否').count()]}
```

原生 SELECT：

```text
{[sql "SELECT SUM(quantity) FROM inventory"]}
```

临时变量：

```text
{[db.inventory.where('item_name','铁剑').get('quantity') as sword_count]}
铁剑数量：$v:sword_count
```

边界：

- `{[db...]}`、`{[sql...]}` 只在 SQLite 模式替换；原生模式会原样保留；
- `{[sql ...]}` 是只读查询，不能拿它执行 `UPDATE/INSERT/DELETE`；
- `as` 后的变量名只能使用英文、数字和下划线，并且不能以数字开头；
- `$v:` 是当前单条消息的临时值，不是数据库持久字段。

### 7.4 处理顺序

一条支持完整模板语法的消息大致按以下顺序处理：

```text
1. <random>，再替换 $random:
2. <calc>
3. <max>/<min>
4. 替换 $calc/$max/$min
5. SQLite 模式替换 db、sql 与 $v
6. 递归解析 <if>
```

因此 `<calc expr="$v:x + 1">` 不会按预期工作：`$v:` 在 calc 之后才替换。SQLite 中应改用 `db.calc()`，或者重新组织模板。

## 8. 前端读取、CRUD 与 SQL API

### 8.1 唯一公开入口

```js
const hostWindow = window.parent;
const API = hostWindow.AutoCardUpdaterAPI;
if (!API) throw new Error('SP·数据库尚未就绪');
```

所有外部功能统一挂在数据库识别出的宿主窗口 `AutoCardUpdaterAPI` 上。Tavern Helper 消息 iframe 不是宿主窗口，应在确认同源后从 `window.parent.AutoCardUpdaterAPI` 取，不能默认读取消息 iframe 自身的 `window.AutoCardUpdaterAPI`，也不能用 Tavern Helper 的 `waitGlobalInitialized()` 等待它。若前端内部还有更深层子 iframe，由上层应用注入受控 adapter，不要继续猜 `parent.parent`。API 对象出现也不代表表数据已完成加载，正式前端应有限时轮询并处理空对象。

来源：`api-registry.ts` ｜ 置信度：high `[已核快照源码]`

### 8.2 读取全部表格

```js
// 返回值是数据库内部活动对象的引用，不是 clone；先复制再交给 UI。
const tableData = structuredClone(API.exportTableAsJson() ?? {});

const factionSheet = Object.values(tableData).find(
  value => value && typeof value === 'object' && value.name === '阵营关系表',
);

const headers = factionSheet?.content?.[0] || [];
const rows = factionSheet?.content?.slice(1) || [];
```

`exportTableAsJson()` 同步返回当前合并数据的**活动对象引用**，尚未加载时可能是 `{}`；更新回调拿到的也是活动引用。不得直接修改它。先 `structuredClone()` 或在 adapter 中归一化成自己的只读 DTO，再交给组件；正式写入走 CRUD、SQL 或导入 API。

### 8.3 跨模式 CRUD

```js
await API.updateCell('阵营关系表', 1, '好感度', 20);

await API.updateRow('faction_relations', 1, {
  relation_status: '友好',
  favor_level: 20,
});

const newRowIndex = await API.insertRow('阵营关系表', {
  阵营名称: '白塔议会',
  关系状态: '中立',
  好感度: 0,
  大致实力: '强大',
});

await API.deleteRow('阵营关系表', newRowIndex);
```

四个方法都支持原生和 SQLite 模式，并支持中英文名称翻译。关键返回值：

| 方法 | 成功 | 失败 |
|---|---:|---:|
| `updateCell` | `true` | `false` |
| `updateRow` | `true` | `false` |
| `insertRow` | 新 `rowIndex` | `-1` |
| `deleteRow` | `true` | `false` |

`rowIndex=0` 是表头，禁止改删；第一条数据是 1。

### 8.4 对象参数与批量静默

```js
await API.updateCell({
  tableName: '阵营关系表',
  rowIndex: 1,
  colIdentifier: '好感度',
  value: 25,
  skipNotify: true,
});
```

`skipNotify/silent` 只跳过外部更新回调，不跳过写入。批量操作可以逐项静默，最后主动刷新一次：

```js
await API.refreshDataAndWorldbook();
```

`skipChatSave/skipSave/isImportMode` 会跳过聊天保存，正常游玩写入不要开启；它更接近导入或临时运行时操作，不是性能开关。

### 8.5 SQLite 只读查询

```js
const result = API.querySql(
  'SELECT faction_name, favor_level FROM faction_relations WHERE favor_level > ?',
  [0],
  { limit: 50, offset: 0 },
);

console.table(result?.rows || []);
```

也可使用声明式单表查询：

```js
const result = API.queryTableRows({
  tableName: '阵营关系表',
  columns: ['row_id', '阵营名称', '好感度'],
  where: { 关系状态: ['友好', '同盟'] },
  orderBy: { column: '好感度', direction: 'DESC' },
  limit: 20,
});
```

只读入口拒绝多语句和写操作。原生模式没有 SQL 引擎，调用会返回空值并记录错误。

### 8.6 SQLite 写入

优先用四个 CRUD。确实需要复杂批量更新时：

```js
const result = await API.executeSqlMutation(
  `UPDATE faction_relations
   SET favor_level = MIN(100, favor_level + ?)
   WHERE relation_status IN ('友好', '同盟')`,
  [5],
  { targetSheetKeys: ['sheet_faction_relations'] },
);

if (result?.errors?.length) console.error(result.errors);
```

多表或多语句使用 `executeSqlBatch`，并尽量显式提供 `targetSheetKeys`，让事务锁和保存范围准确。参数化 SQL 优先于字符串拼接；不要把玩家输入直接拼进 SQL。

## 9. 回调、锁与并发边界

### 9.1 更新回调

```js
const onTableUpdate = tableData => {
  // 它只表示“数据可能变化”，不是“本次 AI 填表成功结束”。
  renderFromDto(normalizeTableData(structuredClone(tableData ?? {})));
};

API.registerTableUpdateCallback(onTableUpdate);

// 页面销毁时必须用同一个函数引用注销
API.unregisterTableUpdateCallback(onTableUpdate);
```

`registerTableUpdateCallback()` 是通用失效通知：CRUD、导入、重载、可视化保存和 AI 填表都可能触发，一次填表也可能触发多次；它没有任务 ID、成功状态或错误信息，不能当作 fill-end。

在 2026-07-14 两个已核快照中，`registerTableFillStartCallback()` 有公开注册方法，但**没有对应注销方法，也没有公开 fill-end 回调**。会随楼层重绘销毁的消息 iframe 不应直接注册匿名闭包，否则旧闭包会留在宿主回调数组。确实需要开始信号时，只能由常驻 bridge 以单例方式注册，再把受控事件转发给短生命周期 UI；否则用更新回调重新读取数据，并以超时、重新读取和错误态完成 UI 对账。升级插件后必须重新检查回调方法集，不能把这一缺口假定为永久存在。

不要在更新回调里无条件再次写表，否则会形成“写入 → 回调 → 写入”的循环。

### 9.2 UI 表格锁不是事务锁

公开的 `lockTableRow/Col/Cell`、`setTableLockState` 保存可视化编辑层的行、列或单元格锁状态，作用域为“当前聊天 + isolationKey”。在 2026-07-14 两个已核快照中，普通锁只在原生模式 AI DSL 的 `updateRow` 路径执行检查；它不阻止 AI `insertRow/deleteRow`、公开 CRUD、raw SQL，SQLite AI SQL 路径也不能据此视为受保护。自定义前端若把某字段定义为不可编辑，必须在自己的 adapter/UI 写入门同时校验，不能把公开锁当权限或数据库约束；升级后重新核验执行点。

插件内部另有多粒度事务锁：

```text
maintenance lock
→ sheet mutation lock
→ isolation commit lock
```

前者只是原生 AI `updateRow` 的有限保护和 UI 状态，后者才保护运行时数据、V2 log、checkpoint 和 `saveChat`。不要以为调用 `lockTableRow()` 就能代替只读规则、权限或并发控制，也不要绕过公开写 API 直接修改 `content`。

两套索引也不同：CRUD 的第一条数据是 `rowIndex=1`，数字 `colIdentifier` 包含 `row_id`；UI 锁的第一条数据行是 `rowIndex=0`，业务列 `colIndex=0` 排除了 `row_id`。不可把 CRUD 行列坐标直接传给锁 API。

来源：`table-lock-api.ts`、`table-write-transaction.ts`、并发设计文档 ｜ 置信度：high `[已核快照结构]`

### 9.3 并发写入规则

- AI 请求可以并发收集，但应用编辑和提交进入统一事务；
- 同一 sheet 的 CRUD/AI/SQL 写入必须串行 mutation；
- 不同 sheet 可并行准备，但同一 isolation 的 V2 commit 必须串行；
- raw SQL 无法确定目标表时会扩大锁范围；外部调用应显式给 `targetSheetKeys`；
- 清理楼层、compaction、导入、Provider 重载会改变恢复链，应使用排他维护事务；
- 外部前端只调用公开 API，不自行构造 V2 operation、revision 或锁 key。

### 9.4 冲突时不要静默覆盖

V2 日志带 `baseRevision/parentRevision/commitRevision/writeSet`；提交端可对不相交写集 rebase，对同一行/单元格或 schema 冲突应失败或重试。前端看到 `false/-1/errors` 必须提示用户并重新读取快照，不要拿旧 UI 草稿立即覆盖新数据。

## 10. 聊天持久化、V2 日志与隔离

### 10.1 数据保存在聊天链里

SQLite 是运行时引擎，不是独立永久数据库文件。当前设计的持久化主链是：

```text
SQLite / 原生 Provider
→ 导出 TableDataObject JSON 视图
→ 写入 SillyTavern 聊天消息的数据库字段
→ 切换聊天时从消息链恢复
```

因此导出聊天、删除楼层、清理本地字段、切换 isolation 都可能影响恢复结果。不能把“浏览器内 SQL 查询成功”当作“已经持久化成功”。

来源：`ITableStorageProvider.saveToChat/loadFromChat`、chat service ｜ 置信度：high `[已核快照源码]`

### 10.2 V2 帧

```ts
interface TableStorageFrameV2 {
  version: 2;
  headRevision?: string | null;
  checkpoint?: {
    kind: 'full';
    reason: 'init' | 'periodic' | 'manual' | 'schema_change' |
            'compaction' | 'import' | 'migration';
    data: TableDataObject;
    scheduleSummary?: Record<string, {
      lastFilledAiFloor?: number;
      lastChangedAiFloor?: number;
    }>;
  };
  logEntries: TableMutationLogEntryV2[];
}
```

`checkpoint.data` 是可恢复全量快照；`logEntries[].operations` 是 checkpoint 之后的可重放增量。两者不能互相冒充。

### 10.3 operation 必须来自真实执行动作

| 写入来源 | 正确 operation |
|---|---|
| SQLite AI/SQL | `sql_batch`，保留语句和绑定参数 |
| 原生 DSL | `table_edit_dsl` 或结构化行 patch |
| 前端 CRUD | 行/单元格级结构化 operation |
| 整表导入 | `sheet_replace` |
| 全库导入/灾难恢复 | `data_replace` |

禁止通过执行前后快照 diff 猜造 operation；普通 CRUD 或 AI 填表也不得为了“保存成功”静默退化成 `data_replace`。

来源：`v2-storage-commit-constraints.md` ｜ 置信度：high `[已核快照约束文档 + 实现类型]`

### 10.4 checkpoint 必须保留调度摘要

压缩日志前要把历史填表/变更楼层汇总到 `scheduleSummary`。状态面板恢复自动填表节奏时应读：

- `lastFilledAiFloor`：上次尝试填表；
- `lastChangedAiFloor`：上次确有数据变化。

只看当前 `logEntries` 或 `checkpoint.event` 会把已填过的表误判成未初始化。

### 10.5 数据隔离

V2 数据以：

```text
当前聊天 + 当前 isolationKey
```

为恢复和迁移单位，存放在消息的 `TavernDB_ACU_IsolatedData[isolationKey]` 分槽中。不同标签可保存不同模板/数据链；切换标签本质上会重新加载当前聊天下相应槽位。

隔离标签不是表名命名空间。切换后前端必须重新获取快照，不能继续持有旧标签对象引用。

## 11. 导入、恢复、迁移与备份

### 11.1 模板和数据是两种东西

| 操作 | 对象 | 典型 API |
|---|---|---|
| 模板导入 | 表结构、Note、Trigger、DDL、默认配置 | `importTemplateFromData` |
| 数据导出 | 当前合并后的表格内容 | `exportTableAsJson` |
| 数据导入 | 覆盖当前数据库并持久化 | `importTableAsJson` |
| 运行时恢复 | 删除楼层后的内存恢复，不制造新持久化事件 | `restoreTableAsJson` |

不要用模板导入代替存档恢复，也不要把当前数据 JSON 当作全局模板到处覆盖。

### 11.2 外部导入与恢复语义不同

```js
const backup = JSON.stringify(API.exportTableAsJson());

// 正常外部导入：写入当前聊天持久化
await API.importTableAsJson(backup);

// 删除楼层/回滚后的运行时恢复：不新增 data_replace/checkpoint/log
await API.restoreTableAsJson(backup);
// 等价：await API.importTableAsJson(backup, { persist: false });
```

把恢复误用成普通导入，会在刚删除楼层后又创建新的持久化事件；把普通导入误用成恢复，则刷新或重开聊天后可能丢失。

来源：`API_DOCUMENTATION.md`、`core-data-api.ts` ｜ 置信度：high `[已核快照源码]`

### 11.3 旧聊天迁移

加载当前聊天和 isolationKey 时，旧格式会被合并成 V2 full checkpoint：

```text
legacy 顶层字段 / V1 隔离槽
→ 按旧语义合并可见数据
→ reason: migration 的 full checkpoint
→ scheduleSummary 迁移历史楼层状态
→ logEntries: []
→ 成功后清理当前槽位旧字段
```

旧数据没有真实 V2 operation 语义，所以迁移不反向伪造 SQL、DSL 或 patch。迁移失败时不应初始化空库或清理旧字段。

来源：`docs/v2-storage-legacy-migration.md` ｜ 置信度：high `[已核快照约束文档]`

### 11.4 安全备份流程

结构升级前：

1. 固定当前脚本版本与模板预设名；
2. `exportTableAsJson()` 保存当前数据；
3. 导出当前表格模板；
4. 记录 storage mode 和 isolation code；
5. 在复制聊天中导入新模板并迁移数据；
6. 对账表数、表头、行数、业务主键和关键字段；
7. 重载聊天，确认不是仅运行时成功；
8. 再同步世界书并检查旧生成条目是否清理。

不要只备份 SQLite 查询截图；永久恢复需要可导入的模板和 JSON 数据。

## 12. 世界书同步与 0TK 边界

### 12.1 世界书是派生视图

表格数据可以通过 `exportConfig` 生成世界书条目：

- `enabled`：启用该表的自定义导出；
- `injectIntoWorldbook`：是否生成主条目；
- `splitByRow`：按行拆分；
- `injectionTemplate` / `sqlInjectionTemplate`：自定义内容；
- `extraIndexEnabled`：生成附加索引；
- `entryPlacement` / `extraIndexPlacement`：位置、深度、顺序；
- `preventRecursion`：控制递归。

主事实源仍是数据库聊天持久化；世界书条目是可删除、可重建的派生物。不要反过来手改生成条目，再期待它自动回写表格。

### 12.2 同步 API

```js
await API.syncWorldbookEntries({ createIfNeeded: true });

// 重新加载聊天表格、合并数据、同步世界书并通知前端
await API.refreshDataAndWorldbook();
```

`deleteInjectedEntries()` 只删除插件生成条目。不要用模糊名称批量删除用户自己维护的世界书条目。

### 12.3 0TK 的准确含义

```js
await API.setZeroTkOccupyMode(true);
```

在当前 API 语义中：

- `true`：把对应数据库生成世界书条目设为禁用；
- `false`：允许条目启用；
- 它控制世界书条目的 `enabled` 状态，不会删除数据库、不停止聊天持久化，也不等于关闭填表；
- 纪要索引与其他特殊模式可以保留/更新内容，但 0TK 仍控制条目是否启用。

所以“0TK”是**提示词占用控制**，不是“数据库处于零数据”或“完全不运行”。

来源：`worldbook-ai-api.ts`、`injection-engine-custom.ts` ｜ 置信度：high `[已核快照源码]`

### 12.4 开场白阶段抑制

当前世界书流水线在没有用户消息的新对话开场阶段，会抑制自动创建/更新，并执行旧生成条目清理；用户开始对话后再解除。开场白刚加载时看不到新条目，不应立即判定同步失败。

来源：`worldbook/pipeline.ts` ｜ 置信度：high `[已核快照源码]`

### 12.5 Token 控制

- 不需要进主提示词的表不要启用世界书导出；
- 大表按行拆分并设置关键词，避免整表常驻；
- 索引只选真正用于召回的短列；
- 注入模板不要重复 DDL、Note 和完整 Trigger；
- 默认 `preventRecursion:true`，除非明确设计了递归召回；
- 数据为空时应清理旧派生条目，防止陈旧状态继续进提示词。

## 13. 性能与维护

### 13.1 控制填表输入

`updateConfig` 可按表控制上下文深度、更新频率、批大小和跳过楼层。先做低频、分表更新，再考虑把所有表每轮全送给 AI。

原则：

- 高频小状态与低频长档案分表；
- 手动更新只选择需要的 `sheetKey`；
- 纪要类设置明确的合并/压缩策略；
- 不把 UI 展示缓存写回业务表；
- 先记录“本轮尝试填表”与“本轮确有变化”的差异，避免重复触发。

### 13.2 前端批量写入

大量逐格写入会产生通知风暴。推荐：

1. 尽量合并为 `updateRow`；
2. 批次内使用 `skipNotify:true`；
3. 最后 `refreshDataAndWorldbook()`；
4. 多表原子业务用参数化 SQL batch；
5. 提供准确的 `targetSheetKeys`；
6. 检查返回值后再更新 UI 的成功状态。

### 13.3 DDL 变更是迁移，不是随手改名

字段改名或拆表会同时影响：

- DDL 英文物理名；
- DDL 中文注释；
- `content[0]` 表头；
- Note 列说明；
- init/insert/update/delete 示例；
- 模板变量与 SQL；
- 前端 CRUD；
- 世界书导出模板；
- 已有聊天数据与 V2 checkpoint。

先设计旧→新映射，在复制聊天中验证，再应用正式存档。只改表头或只改 DDL 都会造成映射错位。

### 13.4 不依赖内部全局变量

`currentJsonTableData_ACU`、Provider、V2 frame 类型是理解机制的源码真相源，不是稳定的外部 API。角色卡前端只依赖宿主窗口 `AutoCardUpdaterAPI` 的文档化方法；消息 iframe 从同源 `window.parent` 访问，版本升级时重新核对方法签名。

## 14. 高频错误

### 错误 1：继续按 MVU 设计

症状：写 zod、`stat_data`、JSONPatch，却没有 `mate/sheet/content/sourceData`。

处理：停止；这不是 SP·数据库模板。重新按二维表与填表协议设计。

### 错误 2：DDL 中文注释、表头和 Note 不一致

症状：中文 CRUD 找不到列，AI 把值写错列。

处理：三处逐字对齐；所有列都写注释。

### 错误 3：多行表没有业务 `UNIQUE`

症状：同一角色/物品/阵营反复 INSERT。

处理：为稳定业务主键加 `NOT NULL UNIQUE`，并在 Trigger 明确“未记录才新增”。

### 错误 4：UPDATE/DELETE 没有 WHERE

症状：整表被更新或删除。

处理：所有示例都带 WHERE；多行表按业务主键，单行表按 `row_id=1`。

### 错误 5：把 `{[sql "UPDATE..."]}` 当写入口

症状：模板文字被替换为空或写入没有发生。

处理：模板 SQL 只读。AI 写入走编辑协议，前端写入走 CRUD/SQL mutation API。

### 错误 6：直接修改 `exportTableAsJson()` 返回对象

症状：直接污染数据库内部活动对象，页面看似改变，但没有形成受支持的提交和持久化链。

处理：读取后先 clone/归一化；写入用公开 CRUD、SQL mutation 或数据导入 API。

### 错误 7：普通写入打开 `skipChatSave`

症状：UI 更新成功，重开聊天恢复旧值。

处理：正常游玩不要跳过持久化；该选项只用于受控导入/运行时场景。

### 错误 8：用 UI 行锁代替并发事务

症状：公开 CRUD、SQL、插入或删除仍能越过锁，两路写入也可能互相覆盖或日志冲突。

处理：所有写入走插件 API；自定义 UI 另做只读规则和写前校验。已核快照中的普通锁只对原生 AI DSL `updateRow` 提供有限保护，不能代替权限、数据库约束或事务锁；升级后重新核验。

### 错误 9：导入与恢复混用

症状：删除楼层后又生成新持久化事件，或导入后刷新丢失。

处理：外部数据导入默认持久化；楼层回滚用 `restoreTableAsJson`。

### 错误 10：把世界书当主数据

症状：手改生成条目后，下一次同步被覆盖。

处理：改表格主数据或导出模板，再重新同步。

### 错误 11：原生模式使用 SQLite 模板语法

症状：`{[db...]}`、`{[sql...]}` 原样发给 AI。

处理：切 SQLite，或改用原生支持的 `cell:`/共同语法。

### 错误 12：严格 JSON 与 `<tableEdit>` 混写

症状：解析器拒绝输出。

处理：确认 `strictJsonTableFillEnabled`；严格模式只输出对应 JSON 根对象，旧模式才输出 `<tableEdit>`。

## 15. 验收清单

### 15.1 模板静态验收

- [ ] 顶层有 `mate`，且 `type="chatSheets"`。
- [ ] 每张表键以 `sheet_` 开头，`uid` 稳定且不重复。
- [ ] 每张表都有 `name/sourceData/content/updateConfig/exportConfig/orderNo`。
- [ ] `content[0]` 是表头，SQLite 首列为 `row_id`。
- [ ] DDL 使用英文物理表名和字段名。
- [ ] DDL 每列有与表头一致的中文注释。
- [ ] 多行表有业务 `UNIQUE`，枚举/范围有 `CHECK`。
- [ ] Note 说明一行实体、每列语义、格式和上限。
- [ ] init/insert/update/delete 都非空；禁止操作明确写禁止。
- [ ] 所有 UPDATE/DELETE 示例都有 WHERE。
- [ ] 世界书导出按需开启，默认不无意常驻大表。

### 15.2 运行验收

1. 在复制聊天导入模板，确认 DDL 校验通过。
2. 初始化空表，确认单行/多行语义正确。
3. 各制造一次 INSERT、UPDATE、DELETE 剧情。
4. 查看 AI 原始输出，确认协议与当前模式一致。
5. 查看 API 快照，确认值落在正确表、行、列。
6. 重载聊天，确认数据从持久层恢复。
7. 切换 isolationKey，确认数据不串槽。
8. 注册更新回调，确认一次业务提交不会形成回调循环。
9. 若启用世界书，检查条目内容、位置、递归与 0TK enabled 状态。
10. 导出备份，在复制聊天分别验证“外部导入”和“运行时恢复”。

离线模板检查不能替代真实 ST 中的 AI 输出、持久化重载和世界书注入验收。

### 16.2 版本敏感事实

- `window.AutoCardUpdaterAPI` 方法集会随发布变化；本册只声明最低能力契约，并记录核验过的快照，不指定永久基线。
- V2 checkpoint/log、严格 JSON、事务锁、SQL API 都属于版本敏感实现，升级必须重新对照源码。
- 世界书位置枚举、0TK、向量索引与交火模式可能继续演进，生产卡要锁定脚本版本。
- `sourceData`、`content`、`mate` 的结构是当前模板兼容层；不能假设任意未来版本永久不变。
- 上游标签编号和创建日期不保证简单单调；不要只比较字符串大小来判断“最新版”，应核对发布时间、commit、发行说明和实际安装来源。
