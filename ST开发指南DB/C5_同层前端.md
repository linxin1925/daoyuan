# 同层前端 — 在单个消息楼层内完成完整游玩体验

> 文档版本：v1.0.1 ｜ 最后更新：2026-07-14
>
> **本册术语裁定（2026-07-14）**：“同层前端”特指在一个消息楼层——通常是首楼 / 0 号楼层——闭合完整游玩主链；玩家始终在这一楼完成输入、生成、剧情阅读、选项、角色状态、背包、地图、设置、路由与回看，而不是只看到一张开局页或一个状态面板。
>
> “同层”按**完整游玩闭环所在楼层**定义，不按 DOM 是否只有一层定义。首楼完整应用内部可以有路由、Tab、Modal、组件树，甚至受控子 iframe；与此同时，SillyTavern 的真实用户消息和 AI 消息仍继续生成，承担提示词上下文、Swipe、分支和聊天存档。首楼应用负责驱动、组织并镜像这些真实楼层。
>
> 这不是把 Tavern Helper 脚本的 DOM 挂到 SillyTavern 主页面，也不是把后续楼层删掉、覆盖成一条假记录。后台脚本可以充当桥接服务，但它不是“同层前端”本身。
>
> 事实基线：`酒馆助手开发完全指南/@types.txt` 的楼层、显示与事件接口；本库 `C1_前端基础-TavernHelper与iframe.md`；星月 3.4.9 开局页的“短标记 → 远程挂载 → 本地绑定”局部生产样本。星月样本只证明挂载链路，不是完整同层玩法成品。
>
> 职责导航：本册负责在首楼 / 0 号消息内完成输入、生成、状态、路由、回看等全部体验；只在开局时收集配置、确认真实玩家消息后退出的一次性初始化器归 **《C10_开局页与自定义开局.md》**。**《C7_数据库二创前端.md》**只讨论正常多楼层角色卡里的数据库状态栏、控制中心和数据面板；SP·数据库默认在正常 AI 回复楼层生成后填表，本身不产出同层应用所需的剧情、选项、路由和楼层信息，因此不能直接作为本册的成熟状态后端。实验性兼容研究另见 **《C8_数据库兼容同层前端-待验证.md》**，在真机验收前禁止当成已支持能力。

## 1. 定义、反定义与适用边界

### 1.1 一句话定义

**同层前端 = 以单个真实消息楼层为唯一可见应用 shell，在 shell 内完成全部玩家交互，同时让 ST 的真实聊天楼层在后台正常增长。**

默认将 0 号楼作为 shell，是因为它随聊天创建、天然属于该存档，并可用楼层 iframe API 确认身份：

```typescript
const SHELL_MESSAGE_ID = 0;

$(() => {
  const current = getCurrentMessageId();
  if (current !== SHELL_MESSAGE_ID) {
    throw new Error(`同层前端必须运行在 ${SHELL_MESSAGE_ID} 号楼，当前为 ${current}`);
  }
  bootShell();
});
```

`getCurrentMessageId()` **只能在消息楼层 iframe 内调用**；在脚本 iframe 中调用会抛错。来源：`@types.txt` 类型声明 ｜ 置信度：high `[类型声明]`

### 1.2 “同层”不等于 DOM 只有一层

以下结构仍然是同层前端：

```text
0 号消息楼层（唯一可见 shell）
└─ App
   ├─ Router / 页面状态
   ├─ 剧情回看
   ├─ 角色 / 背包 / 地图 / 任务
   ├─ Modal / Drawer / Toast
   └─ 受控子 iframe（可选）
```

判断标准不是“嵌套了几层 DOM”，而是：玩家是否始终在同一个消息楼层提供的应用界面内完成游玩。

### 1.3 明确不属于同层前端的形态

| 形态 | 为什么不是本册的同层前端 |
|---|---|
| 脚本把按钮、浮窗或面板插进 ST 宿主页面 | UI 承载面是宿主界面，不是单个消息楼层 |
| 每个 AI 楼层各带一份状态栏 | 可见前端随消息楼层分散，不是单 shell |
| 只把首楼做成开局向导 | 只覆盖开局阶段，不等于完整游玩体验 |
| 删除后续消息，只在首楼变量里模拟剧情 | 丢失 ST 原生上下文、Swipe 与存档语义 |
| Web / App / PC 客户端直接运行角色卡 | 已脱离 ST，属于 `C6_独立前端.md` |

### 1.4 适用与不适用

适用：

- 要把角色卡做成“游戏应用”，不希望玩家滚动原生消息流；
- 仍希望复用 ST 的角色卡、世界书、预设、模型连接、聊天历史和 Swipe；
- 有多个游戏子系统，需要统一路由和稳定入口；
- 愿意维护一个后台桥接服务处理消息事件、shell 恢复和显示模式。

不适用：

- 只要一个轻量状态栏或单次开局页；
- 无法接受对 ST 显示层做版本适配；
- 希望彻底摆脱 ST 的运行环境与依赖；
- 项目没有能力做长聊天、重绘、Swipe 和失败恢复的真机回归。

本节定义来自总监裁定 ｜ 置信度：high `[总监裁定·2026-07-14]`

## 2. 三层架构：真实消息、首楼应用、桥接服务

### 2.1 总体结构

```text
┌──────────────────────────────────────────────┐
│ 真实消息数据层（SillyTavern chat）            │
│ 0 shell / 1 user / 2 assistant / 3 user ... │
│ 上下文、Swipe、分支、楼层 data、存档          │
└───────────────────┬──────────────────────────┘
                    │ Tavern Helper API / events
┌───────────────────▼──────────────────────────┐
│ 桥接服务层（常驻脚本，可选但完整实现强烈建议）│
│ 事件汇聚、消息读取、生成状态、显示适配、远程加载│
└───────────────────┬──────────────────────────┘
                    │ 共享接口 / 自定义事件 / 持久状态
┌───────────────────▼──────────────────────────┐
│ 首楼应用层（0 号消息 iframe）                 │
│ 唯一可见 shell、路由、渲染、输入、回看、设置   │
└──────────────────────────────────────────────┘
```

### 2.2 真实消息数据层

这一层是权威聊天记录，不能为了界面效果被“前端状态”取代：

- 玩家每次行动都创建真实 `user` 消息；
- 正常主模型回复创建真实 `assistant` 消息；
- 当前激活 Swipe 决定该楼实际进入上下文的内容；
- 消息 `data` 可以携带楼层快照；
- ST 负责聊天文件、角色绑定、提示词历史和原生分支语义。

首楼本身也是一条真实消息，但其正文只应保存短标记或轻量启动 mount，不应用来反复写入完整剧情镜像；完整同层应用由显示期展开和常驻逻辑共同维持。

### 2.3 首楼应用层

首楼应用只做三件核心事：

1. **读取**真实聊天、变量和生成状态；
2. **呈现**剧情、菜单与游戏子系统；
3. **发起**玩家行为，让行为回到真实消息链路。

首楼不应自创第二套“剧情真相”。刷新后，它必须能仅凭持久数据重新构建界面。

### 2.4 桥接服务层

完整同层前端建议使用常驻 Tavern Helper 脚本承担：

- 监听 `CHAT_CHANGED`、消息增删改、Swipe、生成开始/结束；
- 聚合事件，防止首楼 iframe 重绘期间漏事件；
- 在首楼被销毁时仍保存生成状态和待恢复信息；
- 发现首楼 mount、加载远程/构建产物、绑定交互；
- 只在 shell 健康时启用“仅显示首楼”模式；
- 首楼重绘后重新接线。

它可以通过 `initializeGlobal` / `waitGlobalInitialized` 暴露受控 API，或通过自定义 `eventEmit` / `eventOn` 传递通知。桥接脚本是服务层，不因它接触宿主 DOM 就改变“同层前端”的定义。

### 2.5 单向数据流

```text
真实聊天 / 变量后端
→ bridge 读取并归一化为 snapshot
→ shell store
→ router / components 渲染
→ 玩家 action
→ createChatMessages + /trigger
→ 真实聊天产生新楼层
→ bridge 再读取 snapshot
```

不要让组件直接在各处调用楼层 API。所有调用应收口到 `chat-service` / `game-state-adapter`，否则重绘、并发和测试都无法统一处理。

架构属于基于裁定和 API 能力形成的工程基线 ｜ 置信度：medium·推导未验证 `[架构推导]`；主链路 API 存在性为 high `[类型声明]`

## 3. 首楼 shell 的短挂载与产物加载

### 3.1 为什么首楼只放短标记

把几百 KB 的完整 HTML 直接写进开场白会同时带来：

- 角色卡和聊天记录膨胀；
- 开场白正文可能进入提示词上下文；
- 每次楼层重绘都重复处理大段 HTML；
- 版本升级必须重导开场白；
- 正则、DOMPurify 与类名前缀处理成本增大。

推荐链路：

```text
first_mes / alternate greeting
→ 只存唯一短标记
→ 显示正则扩成轻量 mount + loading/error fallback
→ bridge 发现 mount
→ 加载固定版本构建产物或远程 HTML
→ 注入 shell
→ 本地 binder / bundle 绑定交互
```

短标记必须唯一、稳定、不会被模型正常复读，例如项目专属标签或文本哨兵。不要使用可能自然出现在剧情中的普通句子。

### 3.2 星月 3.4.9 的局部生产样本

当前样本链路为：

1. `first_message_opening.html` 只有短标记 `*星月·开局页*`；
2. `opening_expand_regex.html` 把它扩展成带 `data-xy-opening-remote` 的加载 mount；
3. 当前 `rolecard-diy-workshop/runtime/xingyue/3.4.9/control-center.js` 集成多源 `fetch`、完整性/结构校验、净化、注入与 `bindOpeningPage`；
4. `rolecard-diy-workshop/runtime/xingyue/3.4.9/opening-page.html` 提供声明式开局页结构；
5. 聊天切换、消息渲染、更新和 Swipe 后重新扫描。

这证明“短消息正文 + 显示期展开 + 远程内容 + 本地绑定”可用于首楼复杂界面。它当前只覆盖一次性开局页，**不得冒充已验证的完整同层玩法实现**。开局页完整职责和提交链见 **《C10_开局页与自定义开局.md》**。

来源：上述四个本地文件 ｜ 置信度：high `[运行时源码·局部生产样本]`

### 3.3 两种部署路线

#### 路线 A：固定版本的单文件构建产物

- HTML、CSS、JS 构建为可部署产物；
- 角色卡或正则只保留 mount；
- bridge 加载锁定版本的资源；
- 发布时同步 bump shell 版本与缓存键。

优点是可回滚、产物边界清晰。缺点是网络失败时必须有本地 fallback。

#### 路线 B：远程声明式 HTML + 本地 binder

- 远程 HTML 只负责结构和样式；
- 交互、消息 API、密钥边界留在本地受信 binder；
- 远程失败时 mount 保留重试按钮和错误说明。

不要指望 `mount.innerHTML = html` 自动可靠执行其中的 `<script>`；交互应由已加载的 binder 明确接管。

### 3.4 远程加载最低安全线

- URL 锁定发行版本或内容哈希，禁止无审计地追 `latest`；
- 主源、备源必须指向同一构建版本；
- 设置超时、失败态、手动重试和离线降级；
- 异步结果提交前核对当前 chat id 与 mount 是否仍存活；
- HTML、JSON、图片分别限定允许来源；
- 不在远程 HTML 内放模型 API key、用户 token 或高权限逻辑；
- 新版本结构变化时由 binder 做协议版本检查，不能静默错绑。

### 3.5 mount 契约

建议 mount 至少提供：

```html
<div
  data-same-floor-shell="1"
  data-shell-protocol="1"
  data-shell-state="loading"
  aria-busy="true"
>
  <p role="status" aria-live="polite">游玩界面加载中……</p>
  <button type="button" data-shell-retry hidden>重新加载</button>
</div>
```

加载状态只能从 `loading` 进入 `ready` 或 `error`。只有进入 `ready` 且完成首轮聊天镜像后，桥接服务才可以隐藏原生楼层显示。

## 4. 启动、重绘与恢复

### 4.1 启动状态机

```text
detect-floor
→ locate-mount
→ connect-bridge
→ load-assets
→ read-chat-and-state
→ render-accessible-transcript
→ ready
→ enable-single-floor-display
```

任一步失败都进入可见 `error`，并保持原生聊天可访问。禁止在 shell 尚未 ready 时先把其他楼层隐藏。

### 4.2 首楼权威读取

首楼内容和楼层数据可以直接读取：

```typescript
const [shellMessage] = getChatMessages(0);
if (!shellMessage) throw new Error('0 号楼不存在，无法启动同层前端');

const shellData = shellMessage.data ?? {};
```

`getChatMessages(0)` 返回 0 号楼当前被采用的消息页。若要读取全部开局 Swipe，使用 `{ include_swipes: true }`。来源：`@types.txt` ｜ 置信度：high `[类型声明]`

### 4.3 什么会让 shell 重建

以下动作都必须按“旧 shell 可能已经销毁”设计：

- `refreshOneMessage(0)`；
- `setChatMessages(..., { refresh: 'affected' | 'all' })` 影响 0 号楼；
- 0 号楼 Swipe 切换；
- 聊天切换或整个消息列表重载；
- 正则配置变化导致聊天重新渲染；
- 浏览器刷新、扩展重载、热更新。

消息楼层 iframe 的内存 store、定时器、请求和事件监听都不能视为持久状态。

### 4.4 受控重绘

当首楼显示损坏或必须应用新 mount 时，可以：

```typescript
await refreshOneMessage(0);
```

该函数只刷新显示，不修改真实消息内容；如果 0 号楼当前没有显示在网页上，它会什么也不做。来源：`@types.txt` ｜ 置信度：high `[类型声明]`

调用后当前 shell 自身可能被销毁，因此：

- 重绘前先把路由、草稿和滚动锚点写入可恢复存储；
- 最好由 bridge 发起重绘；
- shell 自己发起时，把它作为当前调用栈最后一步，不依赖调用后的代码；
- 新实例启动后重新读取真实聊天，不继承旧闭包对象。

### 4.5 恢复优先级

```text
真实聊天与游戏数据
> chat / message 级持久 UI 状态
> localStorage 中的本机偏好
> 当前 iframe 内存
```

如果内存路由与持久聊天状态冲突，以当前 chat id 对应的持久状态为准。所有异步任务携带启动时的 chat id / epoch；切聊天后旧任务即使返回，也不得提交到新 shell。

### 4.6 失败恢复要求

- shell 加载失败：恢复原生楼层显示；
- bridge 不可用：允许只读回看，并清楚禁用发送按钮；
- 资源源全部不可达：显示来源和重试，不留白屏；
- 数据后端不可用：显示诊断，不把空对象写回覆盖真实数据；
- 0 号楼不在当前 DOM：不得假装 `refreshOneMessage(0)` 已成功；
- 热重载：旧请求、observer、listener、interval 全部取消。

## 5. 玩家输入到真实 AI 回复的主链路

### 5.1 唯一推荐主链路

```text
玩家在首楼输入 / 点击选项
→ shell 校验并生成 action_id
→ createChatMessages 创建真实 user 楼层
→ triggerSlash('/trigger') 触发 ST 正常生成
→ ST 创建真实 assistant 楼层
→ 变量框架按其规则处理该楼
→ bridge 收到消息 / 生成事件
→ 重新读取真实聊天
→ 首楼镜像最新剧情和状态
```

这条链路保留 ST 的预设、世界书、角色定义、聊天历史和原生存档语义。

### 5.2 最小发送骨架

```typescript
let sending = false;
let activeActionId: string | null = null;

async function submitPlayerTurn(rawText: string): Promise<void> {
  const text = rawText.trim();
  if (!text || sending) return;

  sending = true;
  activeActionId = crypto.randomUUID();
  renderGenerationState('sending');

  try {
    await createChatMessages([{
      role: 'user',
      message: text,
      data: { same_floor_ui: { action_id: activeActionId } },
    }]);

    renderGenerationState('generating');
    await triggerSlash('/trigger');
  } catch (error) {
    sending = false;
    activeActionId = null;
    renderGenerationError(error);
  }
}
```

`createChatMessages` 只创建消息；创建用户消息后仍需 `/trigger` 才触发 AI。来源：`@types.txt` 与 `references/api.md` ｜ 置信度：high `[类型声明]`

不要仅依赖 `triggerSlash` Promise 的返回时机解除 busy；以生成结束/停止事件和真实消息重读为准，并准备事件漏收后的超时对账。

### 5.3 生成结束后的权威提交

推荐至少监听：

| 事件 | shell 行为 |
|---|---|
| `MESSAGE_SENT` | 重读用户楼层，确认 action 已落库 |
| `GENERATION_STARTED` | 显示生成态、锁定重复提交 |
| `MESSAGE_RECEIVED` | 重读完整聊天，提交 assistant 正文 |
| `MESSAGE_UPDATED` | 延迟重读变量/消息 data，等待解析器落库 |
| `GENERATION_ENDED` | 结束流式预览，做最终对账 |
| `GENERATION_STOPPED` | 解锁并保留用户消息，提示可重试 |
| `MESSAGE_EDITED` / `MESSAGE_DELETED` / `MESSAGE_SWIPED` | 全量重建受影响的回看索引 |
| `CHAT_CHANGED` | 取消旧任务，重启 shell 会话 |

同一轮可能连续触发多个事件。用 debounce + 单飞锁聚合读取，不要每个事件都并发跑一遍完整渲染。

### 5.4 为什么不把 `generate()` 当普通回合

`generate()` / `generateRaw()` 自行生成文本时不会自动创建真实消息楼层，MVU 也不会因此自动按普通楼层管线解析。它们适合第二 API、预览、摘要等专门任务，不适合替代同层前端的正常剧情回合。

正常玩法绕开真实楼层，会导致回看、Swipe、上下文、变量解析与存档出现两套真相。

### 5.5 选项和自由输入统一成 action

```typescript
type PlayerAction = {
  id: string;
  source: 'free_text' | 'choice' | 'system_panel';
  text: string;
  created_at: number;
};
```

无论按钮还是输入框，最终都应生成玩家能看懂的真实 user message。不要只发送隐藏控制码；必要的结构化元数据放 `data`，正文保留可回看语义。

## 6. 消息回看、Swipe 与分支

### 6.1 首楼镜像的是消息，不是复制一份存档

活动聊天读取：

```typescript
function readActiveTranscript(): ChatMessage[] {
  return getChatMessages('0-{{lastMessageId}}')
    .filter(message => message.message_id !== 0);
}
```

shell 可以把这些真实消息渲染为聊天气泡、章节、视觉小说对话或战报，但每个 UI 节点必须保留：

- `message_id`；
- `role`；
- 当前 `swipe_id`（若读取全部 Swipe）；
- 内容版本或更新时间；
- 是否仍处于流式临时态。

不要用数组下标代替 `message_id`，因为删除、插入和历史操作会改变顺序。

### 6.2 读取全部 Swipe

```typescript
const floors = getChatMessages('0-{{lastMessageId}}', {
  include_swipes: true,
});

for (const floor of floors) {
  const activeText = floor.swipes[floor.swipe_id];
  const alternatives = floor.swipes.map((text, swipeId) => ({ swipeId, text }));
  // 显示当前页和可选页；不要在这里修改真实聊天
}
```

`ChatMessageSwiped` 直接提供 `swipe_id`、`swipes`、`swipes_data`、`swipes_info`。来源：`@types.txt` ｜ 置信度：high `[类型声明]`

### 6.3 切换 Swipe

```typescript
async function selectSwipe(messageId: number, swipeId: number): Promise<void> {
  await setChatMessages([{ message_id: messageId, swipe_id: swipeId }], {
    refresh: 'affected',
  });
  await reconcileFromRealChat();
}
```

切换必须写回 ST 的真实 Swipe 状态，不能只改 shell 的选中样式。历史楼层换 Swipe 后，下游分支如何处理属于 ST 版本和操作语义敏感项；执行后必须重读整段聊天，不能假定后续消息仍有效。

### 6.4 “消息树”能力边界

`include_swipes:true` 能得到“每楼有哪些候选页、当前选了哪页”，但类型声明没有直接给出一个完整的跨楼层父子图。因此：

- 可稳定实现：当前活动线、单楼 Swipe 选择、按楼层回看；
- 需要额外验证：历史 Swipe 对后续楼层的裁剪/分叉行为；
- 若产品要完整可视化分支树，需要自建 branch metadata，并与 ST 原生操作做一致性对账；
- 不得仅凭 `swipes[]` 宣称已恢复全部历史分支拓扑。

当前边界 ｜ 置信度：medium·推导未验证 `[类型能力推导]`

### 6.5 长聊天性能

- 初次只渲染最近 N 楼，回看时向前分页；
- 归一化缓存以 `message_id + swipe_id` 为键；
- `MESSAGE_UPDATED` 只失效对应楼，删除/重排/历史 Swipe 才全量重建；
- 大段消息使用虚拟列表或分章折叠；
- 流式预览单独维护，不在每个 token 后重建全部历史；
- 任何摘要都只能是显示索引，不能替换真实聊天。

## 7. 状态划分、已支持后端与数据库实验边界

### 7.1 四类状态不能混放

| 状态 | 示例 | 推荐位置 |
|---|---|---|
| 真实聊天 | user / assistant 文本、Swipe | ST 消息楼层 |
| 游戏状态 | 属性、背包、任务、关系 | 已验证主线使用变量或 MVU；数据库兼容另列待验证路线 |
| 可恢复 UI 状态 | 当前路由、未发草稿、回看锚点 | namespaced chat 变量或 0 号楼 data |
| 本机偏好 | 主题、字号、动效 | `localStorage`，带卡片与版本前缀 |

当前 iframe 的 Pinia/React state 只是缓存，不是任何一类持久真相源。

### 7.2 原生变量后端

适合不采用 MVU、结构较轻的卡：

- chat 变量保存当前存档共享状态；
- message 变量保存楼层快照；
- `updateVariablesWith` 用于受控读改写；
- 正式字段统一放在项目命名空间下，避免与其他脚本冲突。

前端读取后先做 schema 校验和默认值归一。不得把空对象当成“加载成功”再覆盖原值。

### 7.3 MVU 后端

适合已有 `stat_data`、变量更新规则和 schema 的卡：

```typescript
await waitGlobalInitialized('Mvu');

const latest = Mvu.getMvuData({
  type: 'message',
  message_id: 'latest',
});

const statData = latest.stat_data ?? {};
```

主回合继续走真实 user → AI → assistant → MVU 解析链；普通玩家操作不要由 shell 直接改 `stat_data`。前端直接写只保留给有明确事务、schema 校验、审计和回滚的管理工具。

### 7.4 SP·数据库不是默认兼容后端

SP·数据库是 MVU 之外的表格变量 / AI 填表方案，常规流程是在正常多楼层聊天中读取已经生成的 AI 回复，再更新数据库。它本身不向同层应用提供剧情正文、选项、路由或楼层信息输出契约，所以不能仅靠“接一个数据库 adapter”就完成本册主链。

任何数据库兼容尝试都必须另加消息/楼层桥，验证“真实回复生成 → 楼层可读 → 数据库填表 → 首楼渲染”的完整时序，以及 Swipe、重生成、编辑、删楼、回滚和切聊天恢复。该路线仅在 [C8_数据库兼容同层前端-待验证.md](./C8_数据库兼容同层前端-待验证.md) 中讨论；通过转正门前，本册不把它列为可用后端。

### 7.5 统一适配器

```typescript
type Unsubscribe = () => void;

interface GameStateAdapter<T> {
  readonly kind: 'variables' | 'mvu';
  init(): Promise<void>;
  read(): Promise<T>;
  subscribe(onChange: () => void): Unsubscribe;
  dispose(): void;
}
```

组件只消费归一化 snapshot；不要在组件中用 `if (Mvu)` 到处分叉。后端切换必须是项目级选型，不是运行时把多套数据悄悄合并。待验证的数据库兼容层不得提前塞进这个稳定接口冒充第三种已支持实现。

### 7.6 UI 恢复状态示例

```typescript
type ShellUiState = {
  protocol_version: 1;
  route: string;
  draft: string;
  transcript_anchor?: number;
  last_seen_message_id?: number;
};
```

建议写到独立命名空间，例如 `same_floor_ui`。游戏状态后端迁移不应顺带覆盖 UI 状态；UI 协议升级需要显式 migration。

## 8. 只隐藏显示，不改变提示词语义

### 8.1 最高优先级警告

> **严禁为了“玩家只看首楼”而把后续真实消息设置为 `is_hidden: true`。**

`is_hidden` 不是 CSS 显隐，它会改变消息是否发送给 LLM。把真实 user / assistant 楼层设为 hidden，会直接破坏提示词上下文和连续剧情。

来源：`ChatMessage.is_hidden` 语义、`references/api.md` ｜ 置信度：high `[类型声明]` + `[本库接口核对]`

同样禁止为视觉效果删除、清空、旋转或改写真实消息。

### 8.2 正确做法：展示适配层

只改变 ST 页面上原生消息容器的**视觉呈现**，不修改聊天文件字段：

```text
bridge 检测 shell ready
→ 给宿主聊天视图开启项目专属 presentation mode
→ 视觉折叠 / 隐藏非 shell 消息容器
→ 保留 0 号 shell 可见
→ 新楼层渲染后重新应用
→ shell error / destroy / chat change 时立即撤销
```

这层可以使用项目专属 class、样式节点或受控 DOM 操作，但必须：

- 选择器加项目命名空间；
- 只作用当前聊天的消息显示区；
- 不修改 `is_hidden`、role、message 或 data；
- 不屏蔽 ST 系统弹窗、输入区和错误提示；
- 有“一键显示原生消息”逃生开关；
- shell 心跳丢失时自动恢复原生显示；
- ST 版本变化导致选择器失效时安全失败为“都可见”，不能失败为“全消失”。

### 8.3 可访问性前提

若原生消息容器被 `display:none`，屏幕阅读器也无法访问它们。因此 shell 必须在进入单层模式前提供：

- 可键盘导航的完整活动线回看；
- role / 说话者语义；
- 新消息通知；
- 跳到最新与向前加载；
- 显示原生聊天的恢复按钮。

### 8.4 自动滚动与 shell 位置

ST 生成新楼层时可能自动滚到消息列表底部，而 shell 位于 0 号楼。展示适配层必须真机解决：

- shell 是否固定在可视区域；
- 原生自动滚动会不会把玩家带到空白底部；
- 长聊天未加载 0 号楼时如何恢复；
- 移动端软键盘打开后 shell 是否仍可操作。

这些不能只靠离线 CSS 宣称完成，见悬案 U1/U4。

## 9. 套娃式路由、组件与子 iframe

### 9.1 单 shell 内部路由

推荐用应用内状态路由，不把 ST 楼层号当页面：

```typescript
type ShellRoute =
  | { name: 'story' }
  | { name: 'characters'; selectedUid?: string }
  | { name: 'inventory' }
  | { name: 'map'; locationId?: string }
  | { name: 'quests' }
  | { name: 'settings' };
```

路由切换只改变首楼视图；真实消息继续按回合增长。需要刷新恢复的 route 写入 `ShellUiState`。

### 9.2 推荐层级

```text
ShellFrame
├─ TopBar：角色 / 时间 / 资源 / 生成状态
├─ Viewport
│  ├─ StoryView：真实消息镜像、选择栏、输入
│  ├─ CharacterView
│  ├─ InventoryView
│  ├─ MapView
│  └─ SettingsView
├─ ModalHost：详情、确认、诊断
└─ RecoveryRail：原生聊天、重载 shell、导出诊断
```

剧情视图必须始终可达，不能让玩家进入设置页后失去生成进度或新消息提示。

### 9.3 Modal 与 Drawer

- 在 shell 根容器内统一管理 z-index；
- 只允许一个顶层交互焦点；
- ESC 关闭最上层，不穿透到 ST；
- 打开时记录焦点，关闭后归还；
- 不用依赖宿主 viewport 的 `position: fixed` 解决楼层内弹窗；
- shell 重绘前关闭或持久化关键确认流程。

### 9.4 受控子 iframe

子 iframe 适合：地图渲染器、复杂编辑器、第三方可视化、重 CSS 子应用。它仍属于首楼 shell 的内部子视图，所以产品体验仍是同层。

最低要求：

- 显式 `sandbox`，按需开放最小能力；
- 父子通信使用版本化 `postMessage` 协议；
- 校验 `event.source`、`event.origin`、channel 和 schema；
- 子 iframe 不直接接触 Tavern Helper 高权限 API，由 shell/bridge 提供白名单 RPC；
- 用 `ResizeObserver` 或明确尺寸协议同步高度；
- route 离开时销毁 listener、observer、worker 与媒体资源。

不要把“嵌了子 iframe”误判为 `C6` 独立前端；只要运行仍依赖 ST，且完整游玩闭环仍在首楼内完成，它就是本册范围。

## 10. 流式、并发与重复提交

### 10.1 生成状态机

```typescript
type GenerationPhase =
  | 'idle'
  | 'sending'
  | 'generating'
  | 'streaming'
  | 'reconciling'
  | 'stopped'
  | 'error';
```

按钮文案、输入禁用、停止/重试和流式预览都由同一状态机驱动。不要让每个组件各维护一个 `isLoading`。

### 10.2 流式显示

- 正常 `/trigger` 生成监听 Tavern 生成/流式事件；
- `iframe_events.STREAM_TOKEN_RECEIVED_*` 是 `generate()` 系列的事件，不要和正常 ST 回合无条件混用；
- token 只更新临时 preview，不写进真实聊天或游戏状态；
- `MESSAGE_RECEIVED` 后丢弃 preview，重新读取持久 assistant 消息；
- 流式区使用节流，避免每 token 重排整棵组件树；
- `aria-live` 不逐 token 朗读，完成句或完成回合再通知。

事件范围随生成入口而异 ｜ 置信度：high `[类型声明]`；具体正常 `/trigger` 流式矩阵需真机确认，见 U2。

### 10.3 重复提交防护

至少同时具备：

1. UI busy 锁；
2. 持久 `action_id`；
3. 提交前检查是否已有同 action 的真实 user 消息；
4. bridge 维护当前 generation epoch；
5. 网络/重绘后通过真实消息对账，而不是盲目重发；
6. 超时只显示“状态未知，先检查聊天”，不能自动再发一次。

### 10.4 异步竞态

```typescript
let refreshEpoch = 0;

async function reconcile(): Promise<void> {
  const epoch = ++refreshEpoch;
  const snapshot = await readAllSources();
  if (epoch !== refreshEpoch) return;
  if (snapshot.chatId !== currentChatId()) return;
  applySnapshot(snapshot);
}
```

`MESSAGE_RECEIVED`、`MESSAGE_UPDATED`、变量更新结束可能很接近。旧读取结果不得覆盖新状态。

### 10.5 停止、失败与重试

- 生成停止后保留真实 user 消息并标出“未获回复”；
- 重试前让玩家选择继续生成、重新生成或编辑输入；
- 不自动删除上一条用户消息；
- assistant 已落楼但变量后端仍在处理时显示 `reconciling`，不要过早解锁依赖状态的操作；
- shell 刷新后从最后楼层 role 和生成事件状态恢复，不仅看内存 busy。

## 11. 移动端、可访问性与安全

### 11.1 消息 iframe 布局

- 禁止把 `100vh` 当作可靠可视高度；ST 外层、浏览器工具栏和软键盘都会改变实际空间；
- 以内容流 + 内部可控滚动区设计，避免无限撑高整个消息楼层；
- 通过 `ResizeObserver` 观察 shell 主区域，不要对每个子节点各开 observer；
- 窄屏 Tab 使用横向滚动或底部导航，不维护两套业务 DOM；
- 触控目标约 44×44 CSS px，输入按钮不依赖 hover；
- 尊重安全区和 `prefers-reduced-motion`。

### 11.2 键盘与焦点

- shell 根提供“跳到剧情”“跳到输入”“打开恢复工具”；
- 发送后焦点回到合理位置，不因全量重渲丢失；
- Modal 焦点陷阱只限制在当前 Modal；
- ESC 不冒泡触发 ST 的无关快捷键；
- 键盘导航能完成选项、Swipe、回看和发送；
- 路由变化更新标题或 live region，不能只靠视觉动画表达。

### 11.3 AI 内容全部不可信

- 默认 `textContent` / 框架转义插值；
- 禁止把 AI 正文、变量值、数据库单元格直接交给 `innerHTML` / `v-html`；
- 若需要 Markdown/正则显示效果，优先走 `formatAsDisplayedMessage`，仍对最终插入点做边界审计；
- URL 协议白名单至少限制为项目需要的 `https:` / 受控媒体协议；
- 移除事件属性、脚本标签、危险 SVG 与任意 iframe；
- 导入存档、工坊包和远程 JSON 与 AI 文本同等不可信。

### 11.4 远程供应链

- 锁版本、可回滚、保留本地最小错误壳；
- 发布清单记录 HTML/JS/CSS 哈希；
- 资源协议版本不匹配时拒绝绑定；
- 不让远程展示产物直接获得世界书写入、角色卡覆盖或模型密钥；
- 高权限动作通过 bridge 白名单方法，并在 UI 明确确认。

### 11.5 隐私与日志

- 诊断导出默认脱敏 API 地址、token、persona 私密字段；
- 不把完整聊天自动上传到远程分析服务；
- localStorage key 带项目/角色/协议前缀，正式游戏状态不得只存本机；
- console 默认不打印完整提示词、世界书和玩家隐私内容。

### 12.1 shell 与 bridge 的最小协议

```typescript
interface SameFloorBridge {
  protocolVersion: 1;
  getSnapshot(): Promise<SameFloorSnapshot>;
  submit(text: string, actionId: string): Promise<void>;
  selectSwipe(messageId: number, swipeId: number): Promise<void>;
  requestShellRefresh(): Promise<void>;
  setPresentationMode(enabled: boolean): Promise<void>;
  subscribe(listener: (event: BridgeEvent) => void): () => void;
}
```

bridge 不应把整个 `window.SillyTavern` 或 `TavernHelper` 原样暴露给 shell 子组件。协议越小，重绘恢复和权限审计越清楚。

### 12.2 版本号至少三处

- shell 构建版本；
- bridge 协议版本；
- UI 持久状态 schema 版本。

三者不是一个概念。只 bump 文件名而不迁移 UI 状态，会在升级后读到旧草稿结构并崩溃。

## 13. 高频错误

1. **把宿主 DOM 浮窗叫同层前端**：完整交互闭环不在单个消息楼层；一个宿主按钮或浮窗不构成本册所说的同层前端。
2. **首楼只是开局向导却宣称完整同层玩法**：必须覆盖整个回合、回看和恢复链。
3. **后续剧情只写 shell store**：没有真实消息，提示词与存档都断链。
4. **用 `is_hidden` 隐藏原生楼层**：改变 LLM 上下文语义，是最高风险事故。
5. **直接 `generate()` 当正常回合**：不自动创建楼层，变量管线和回看分裂。
6. **点击选项只改变量不发 user 消息**：剧情 AI 看不到玩家行为，审计也没有记录。
7. **每 token 全量重渲历史**：长聊天立即卡顿。
8. **事件一来就并发读取**：旧 snapshot 反盖新状态。
9. **重绘前不保存草稿和 route**：`refreshOneMessage(0)` 后全部丢失。
10. **shell 未 ready 就隐藏其他楼层**：远程失败时玩家只剩空白。
11. **远程 HTML 内塞高权限逻辑**：供应链一旦被替换即可改卡、读聊天或发消息。
12. **把 `swipes[]` 当完整分支树**：它只直接证明单楼候选页和当前页。
13. **只适配桌面**：移动端软键盘、自动滚动和 iframe 高度会让核心输入不可用。
14. **原生消息隐藏后没有无障碍回看**：屏幕阅读器用户失去剧情。
15. **切聊天后旧请求仍提交**：A 存档数据污染 B 存档 shell。
16. **桥接脚本被当作 UI 真相源**：脚本重启或热更新后状态无法恢复。
17. **错误态仍保持 presentation mode**：shell 崩了，原生消息也看不见。
18. **巨大首楼 HTML 直接进开场白**：提示词、渲染与升级成本同时爆炸。

## 14. 验收清单

### 14.1 术语与架构

- [ ] 玩家全程只需使用一个消息楼层的 shell
- [ ] “同层”按完整游玩闭环所在楼层定义，不限制应用内部 DOM/iframe 嵌套
- [ ] 真实 user / assistant 楼层持续存在并作为上下文真相源
- [ ] bridge 明确是服务层，不冒充同层 UI 本体
- [ ] 开局页样本只被引用为局部挂载证据

### 14.2 启动与恢复

- [ ] 0 号楼读取使用 `getChatMessages(0)` 并处理不存在
- [ ] 入口确认 `getCurrentMessageId() === 0`
- [ ] shell 未 ready 时原生聊天仍可见
- [ ] `refreshOneMessage(0)` 后 route、草稿、回看锚点可恢复
- [ ] 远程源全部失败时有本地错误壳和重试
- [ ] 聊天切换后旧异步结果不能写入新聊天

### 14.3 回合主链

- [ ] 玩家行为创建真实 user 消息
- [ ] `createChatMessages` 后显式触发 `/trigger`
- [ ] AI 回复落为真实 assistant 消息
- [ ] `MESSAGE_RECEIVED` 后以真实消息覆盖流式 preview
- [ ] 稳定主线只选定并适配变量或 MVU 后端
- [ ] 若研究数据库兼容，明确标为 C8 待验证路线，不并入生产验收
- [ ] 普通玩家行为不绕过变量框架直接写正式游戏状态

### 14.4 回看与分支

- [ ] 回看节点保留 `message_id` 与 Swipe 标识
- [ ] Swipe 切换写回 ST，再重读聊天
- [ ] 编辑、删除、Swipe、重排后索引能恢复
- [ ] 长聊天有分页、虚拟化或分章策略
- [ ] 未把单楼 `swipes[]` 误称完整分支图

### 14.5 显示模式

- [ ] 没有为视觉目的写 `is_hidden:true`
- [ ] 没有删除、清空或改写真实楼层正文
- [ ] presentation mode 只改显示 DOM/CSS
- [ ] shell error / destroy / chat change 会恢复原生消息
- [ ] 有玩家可操作的“显示原生聊天”逃生口
- [ ] 新楼层渲染后显示模式能重新对齐

### 14.6 并发、安全与可访问性

- [ ] busy 锁、action_id、epoch 对账三者都有
- [ ] 停止/失败不会静默重复发消息
- [ ] AI、数据库、工坊和远程内容均按不可信输入处理
- [ ] 子 iframe 使用最小 sandbox 和校验后的消息协议
- [ ] 键盘、焦点、屏幕阅读器、减弱动效可用
- [ ] 移动端软键盘、窄屏、旋转、自动滚动已真机检查

10. 切换聊天再切回；
11. 远程源断网；
12. shell 加载失败时恢复原生聊天；
13. 100+ 楼长聊天回看；
14. 手机窄屏、软键盘、横竖屏；
