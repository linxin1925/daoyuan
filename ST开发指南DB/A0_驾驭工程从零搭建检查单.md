# A0 · 驾驭工程从零搭建检查单（Harness Engineering · From Zero）

> **给谁用**：任何想让 AI 助手（Claude Code 等）长期、稳定替自己干活的人——**不需要会写代码**。
>
> **解决什么问题**：同一个 AI 模型，有人用得像老搭档，有人天天翻车返工。差别通常不在模型，在**驾驭层（harness）**——你围绕 AI 搭起来的工作环境：规则文件、记忆、检查流程、硬护栏。一句话公式：**AI 助手的实际表现 ＝ 模型能力 × 驾驭层质量**。
>
> **怎么用**：按阶段推进，每阶段末尾有「✔ 完成标志」。**搭完一层就受益一层，不必一次全建**；卡住了就先停在当前层用着，等痛点出现再往上搭。

---

## 第 1 层 · 地基：常驻规则文件（AI 的"岗位说明书"）

> AI 每轮对话都会自动读的那份文件（Claude Code 里叫 `CLAUDE.md`）。没有它，你每次开工都要口头重新交代一遍自己是谁、要什么。

- [ ] 在项目根目录建规则文件，先只写三件事：**你是谁**（背景、会不会代码）、**项目是什么**（一句话）、**沟通偏好**（语言、要不要先给结论、术语要不要解释）。
- [ ] 保持短：**60 行以内起步**。它每轮都被读一遍，越长越贵、越容易被忽略。
- [ ] 立一条元规则：**这文件里只放"每轮都要考虑"的规则**；只在特定场景用的知识放别的文档，需要时再让 AI 去读。
- [ ] 心法：**每条规则都应该能指着一次真实翻车说"就是因为那次"**。还没翻过的车，先别写规则防它——想象出来的规则会稀释真规则。

**✔ 完成标志**：新开一个对话，不做任何交代，AI 的第一反应就符合你的偏好（语言对、口吻对、知道项目是什么）。

**✘ 常见坑**：把规则文件写成"风格指南"或愿望清单。它的正确定位是**飞行员检查单**——短、硬、每条都救过命。

---

## 第 2 层 · 开工同步：动手前的三格检查（防"方向就错了"）

> 大部分翻车不是 AI 干得差，是它干的根本不是你想要的。开工前 30 秒把三件事说清，能挡掉一半返工。

每次让 AI 开工前，过这三格（**任何一格含糊 ＝ 先补齐再开工**）：

- [ ] **目标**：这次要做成什么？一句话、做完能验收的那种。顺带说清**不做**什么。
- [ ] **红线**：这个任务有什么绝对不能碰的？（没有就明说"没有"）
- [ ] **验收**：怎么算做完？你打算怎么验证？

然后走最关键的一步：

- [ ] **复述纠偏**：让 AI 用三句话回读它的理解（目标是 X / 红线是 Y / 验收是 Z），你逐句确认或纠正，**对上了才开工**。同步的成色不看你说了多少，看 AI 复述得对不对。

**✔ 完成标志**：连续几次任务，AI 交付的东西不再"能用但不是我要的"。

**✘ 常见坑**：跳过复述。你说得再全，AI 理解偏了照样白干——复述是唯一能提前暴露理解偏差的手段。

## 第 3 层 · 记忆与交接：跨会话不失忆

> AI 的对话是有长度上限的，关掉就忘。不搭这层，每次新会话都从零开始，你就是它唯一的记忆体。

- [ ] 建一份**进度文档**（一个 md 文件就行）：当前做到哪、下一步是什么、有什么坑。让 AI 每次干完活自己更新它。
- [ ] 立规矩：**进度只写这一份**。多份文档各记各的进度 ＝ 很快没有一份是真的。
- [ ] 每次会话结束前，让 AI 留一段**交接摘要**：本次目标 / 完成了什么 / 遗留什么 / 下一步从哪继续。
- [ ] 每次新会话开工：先让 AI 读进度文档 + 上次交接，**复述现状**，对了再继续。
- [ ] 进阶（可选）：如果工具支持自动化（如 Claude Code 的 hooks），把"会话结束存交接、会话开始读交接"做成自动的——机器兜底，不靠 AI 自觉。

**✔ 完成标志**：隔了三天回来开新会话，两分钟内 AI 就能准确接上进度，不需要你重新讲历史。

**✘ 常见坑**：让 AI"记在心里"。它没有心，只有文件——**没写进文件的等于没发生**。

---

## 第 4 层 · 硬护栏：把"反复犯的错"交给机器拦（可选，需要一点配置）

> 第 1 层的规则靠 AI 自觉遵守，偶尔还是会忘。对反复出现的同一种错，升级成**机器强制**：工具在 AI 动手时自动检查/拦截（如 Claude Code 的 hooks 机制），AI 忘了也犯不成。

- [ ] 盘点：过去一个月，AI 在你项目里**重复犯过 2 次以上**的错有哪些？
- [ ] 其中"机器能判定"的（写错地方、用错命令、改完没检查语法），挑最痛的一两个做成自动拦截/自动检查。不会配置就让 AI 自己给自己写——描述清楚"什么情况该拦"，让它实现，然后你测一次。
- [ ] 原则：**能机器强制的，不靠自觉**。规则文件管"该怎么想"，护栏管"不许怎么做"。

**✔ 完成标志**：那个反复犯的错，再也没成功犯出来过（AI 试图犯的瞬间被拦下）。

---

## 第 5 层 · 验证闭环：完工的定义不由 AI 说了算

> AI 有个天性：倾向于宣布自己成功了。不搭这层，你会收到大量"已完成✅"但一用就坏的交付。

- [ ] 立规矩：**AI 说"做完了"之前，必须先证明它生效**——跑起来看结果、截图、过测试，拿证据说话，不接受"应该没问题"。
- [ ] 立规矩：**离线检查通过 ≠ 交付**。凡是最终跑在真实环境里的东西（网页、卡片、脚本），必须在真实环境里验过才算数。
- [ ] **阶段门控**：大活完工时，AI 不许自己宣布完成，必须列出「验收对齐项」清单（做了什么 / 怎么验证的 / 哪些还没验），逐项等你确认。
- [ ] 进阶（可选）：关键改动让**另一个独立的 AI 视角**做对抗审查（换个会话或子 agent，专门挑毛病）。生成和评估分开，比让它自查自纠有效得多。
- [ ] 止损规矩：同一个问题让 AI 修了 **2–3 轮**还不稳，停——别再让它打补丁，改问"要不要推倒重写或砍掉这个功能"。越修越烂是强信号，说明方向或架构错了。

**✔ 完成标志**："已完成"和"实际能用"这两件事重新变成同一件事。

## 第 6 层 · 版本恢复：能回到"上一个好状态"

> AI 会改坏东西，这无法避免；能避免的是改坏之后回不去。

- [ ] 给项目目录开版本管理（git；不会用就让 AI 帮你初始化并代管）。
- [ ] 让 AI 养成习惯：**每完成一个可验收的小块就存一次版本**，附一句人话说明。
- [ ] 你只需要记住一句话的能力：「回到昨天那个还能用的版本」——有版本管理，这句话就能被执行。
- [ ] 没条件上 git 的最低配：重要文件改动前先复制一份带日期的备份。

**✔ 完成标志**：某次改崩了，你们在五分钟内回到了已知好状态，而不是花一晚上手工找回。

---

## 第 7 层 · 棘轮：让系统只进不退

> 棘轮＝只朝一个方向转的齿轮。前六层是设施，这一层是习惯：**每次翻车都变成设施的一部分，同一个坑不踩第二次**。

- [ ] 每次翻车后问一句："这次的教训，落到哪一层？"——多数答案是：规则文件加一条（第 1 层）/ 护栏加一道（第 4 层）/ 验收清单加一项（第 5 层）。
- [ ] **口头提醒不算落地**。"下次注意"是最贵的四个字——下次它一定不注意。
- [ ] 给重复犯的错记个数。犯得越多的，越优先升级成硬护栏。
- [ ] 反向操作也要做：**模型升级变强后，定期删掉不再需要的护栏**。驾驭层里的假设会过时，臃肿的护栏和缺失的护栏一样有害。

**✔ 完成标志**：回看三个月前的翻车类型，现在一个都犯不出来了——不是因为你俩变小心了，是因为系统不允许了。

## 成熟度速查

| 层 | 你有了 | 效果 |
|---|---|---|
| L1 | 常驻规则文件 | 不用每次重新自我介绍 |
| L2 | 开工三格 + 复述 | 干的是你要的活 |
| L3 | 进度文档 + 交接 | 跨会话不失忆 |
| L4 | 硬护栏 | 重复错犯不出来 |
| L5 | 验证闭环 + 门控 | "完成"就是真完成 |
| L6 | 版本恢复 | 改崩了回得去 |
| L7 | 棘轮习惯 | 系统随翻车变强 |

## 三条心法（全单浓缩）

1. **规则出自翻车，不出自想象**——没踩过的坑不立规则，踩过的坑必须立。
2. **能机器强制的，不靠自觉**——写进文件不如挂上钩子，口头提醒等于没提醒。
3. **检查单要短**——长到没人走完的检查单，等于没有检查单。
