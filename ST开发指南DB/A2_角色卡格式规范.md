### 1.1 版本对照表

| 版本 | `spec` 值 | `spec_version` | 关键新增 |
|---|---|---|---|
| V1 | 无（字段即顶层） | 无 | name/desc/personality/scenario/first_mes/mes_example 六字段 |

| V3 | `"chara_card_v3"` | `"3.0"`（接受 ≥3.0 <4.0） | ccv3 PNG chunk 优先 + data.assets（CharX 格式新增）；ST 服务端对 data 内容验证极宽松 |

来源：`src/validator/TavernCardValidator.js`、`src/character-card-parser.js` | 适用版本：ST v1.14.0 | **置信度：high** `[运行时源码]`

### 1.2 双 spec 并存问题

ST 的 `charaFormatData` 函数在写入时**只写 V2**（`spec: 'chara_card_v2'`），但 `character-card-parser.js` 的 `write()` 函数同时嵌入 chara（V2）和 ccv3（V3）两个 PNG chunk；`read()` 时 ccv3 优先。

> 源码校准注：`write()` 函数头注释写"ccv3 is not supported and removed"，但实际代码（行 32-40）确实写入了 ccv3 chunk（try/catch 包裹，失败则跳过），注释与实现矛盾，以代码实际行为为准。

**关键行为**：
- 磁盘上 JSON 文件通常是 V2 格式（`spec: 'chara_card_v2'`）
- 卡内 PNG 同时包含两个 chunk，V3 chunk 仅添加 `spec='chara_card_v3'` + `spec_version='3.0'` 两个字段
- 本地实战卡（星月/交错）**直接存 V3 JSON**（`spec: 'chara_card_v3'`），ST 验证器对 data 内容检查极宽松（只验 data 存在且是对象）

来源：`src/character-card-parser.js` write()/read() 函数 | 适用版本：ST v1.14.0 | **置信度：high** `[运行时源码]`

## 3. data 层字段参考

### 3.1 V2 data 层标准字段（完整列表）

```
data: {
  // ── V1 同名字段（全部保留在 data 内，顶层也复制一份）──
  name, description, personality, scenario, first_mes, mes_example,

  // ── V2 新增字段 ──
  creator_notes,              // 创作者备注，不注入提示词
  system_prompt,              // 替换全局 system prompt；支持 {{original}} 占位
  post_history_instructions,  // 替换 UJB/jailbreak；支持 {{original}}
  alternate_greetings,        // string[]，开场消息 swipe 列表
  tags,                       // string[]，大小写不敏感
  creator,                    // 创作者署名
  character_version,          // 版本字符串，供前端排序
  extensions,                 // object，第三方扩展命名空间
  character_book,             // 可选，内嵌世界书

  // ── V3 新增字段（ST validator 未强制校验）──
  group_only_greetings,       // string[]（spec 定义，本地生产卡未使用）

}
```

**V2 必须通过 ST 验证器的字段**（`TavernCardValidator.js #validateDataV2()`）：

`name, description, personality, scenario, first_mes, mes_example, creator_notes, system_prompt, post_history_instructions, alternate_greetings, tags, creator, character_version, extensions`

其中 `alternate_greetings` 和 `tags` 必须是 Array，`extensions` 必须是 object。

**V3 验证器**（`#validateDataV3()`）仅验证 `data` 存在且 `typeof data === 'object'`，data 内部字段**不检查**，这是"ST 验证极宽松"的准确范围（仅对 V3 成立，V2 有14字段强校验）。

来源：`src/validator/TavernCardValidator.js` | 适用版本：ST v1.14.0 | **置信度：high** `[运行时源码]`

## 4. character_book（内嵌世界书）

### 4.1 书级字段

```json
{
  "name": "世界书名称",
  "description": "描述（可空）",
  "extensions": {},
  "entries": []
}
```

**ST 验证器要求**：`extensions`（object）+ `entries`（array）必须存在；`name`、`description` 为可选。

`scan_depth`、`token_budget`、`recursive_scanning` 是 spec_v2.md 中定义的书级字段，但 ST 实际存储在 `extensions` 内而非书顶层（见 12.2 悬案）。

来源：`TavernCardValidator.js #validateCharacterBookV2()` | 适用版本：ST v1.14.0 | **置信度：high** `[运行时源码]`

### 4.2 entry 核心字段（spec 标准）

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | number | 条目 ID（内部序号） |
| `keys` | string[] | 主触发关键词数组 |
| `secondary_keys` | string[] | 次级关键词（配合 selective + selectiveLogic 使用） |
| `comment` | string | 人类可读备注/标题 |
| `content` | string | 注入提示词内容（可含 Markdown、HTML、自定义标签） |
| `constant` | boolean | 常驻触发（无需关键词激活） |
| `selective` | boolean | 是否启用次级关键词过滤 |
| `insertion_order` | number | 同 position 组内插入顺序（数值越小越前） |
| `enabled` | boolean | 是否启用 |
| `position` | string | 插入位置字符串枚举，见 4.3 |
| `use_regex` | boolean | 关键词匹配是否允许正则 |
| `extensions` | object | ST 扩展字段，见 4.4 |

来源：`src/endpoints/characters.js convertWorldInfoToCharacterBook()` + 本地实战卡 | 适用版本：ST v1.14.0 | **置信度：high** `[运行时源码]`

### 4.3 position 字段枚举（双轨机制）

`position` 字段有**两套值**，必须同时正确设置，是最高频踩坑点。

**字符串形式**（character_book 条目顶层 `position` 字段，仅两个值）：

| 字符串值 | 含义 |
|---|---|
| `"before_char"` | 角色定义之前（position 数字 0） |
| `"after_char"` | 角色定义之后（所有其他位置的通用兜底值） |

**数字形式**（`extensions.position`，ST 内部完整 8 位置枚举）：

```javascript
export const world_info_position = {
    before: 0,       // "before_char" / Before Char Defs
    after: 1,        // "after_char" / After Char Defs
    ANTop: 2,        // Author's Note 顶部
    ANBottom: 3,     // Author's Note 底部
    atDepth: 4,      // @ Depth（需配合 extensions.depth 和 extensions.role）
    EMTop: 5,        // Example Messages 顶部
    EMBottom: 6,     // Example Messages 底部
    outlet: 7,       // Outlet（通过 {{outlet::Name}} 宏手动插入）
};
```

**转换规则**：`convertWorldInfoToCharacterBook()` 将 position 0 映射为 `"before_char"`，其他全部映射为 `"after_char"`——因此字符串形式**仅能表达两种位置**，完整的 8 位置体系存于 `extensions.position`（数字）。

**本地实战卡验证**（交错宙域 v2.6.0 实测精确数据）：
- 顶层 `position: "before_char"` 的共 19 条，其中 `extensions.position: 4`（atDepth）9 条，`extensions.position: 0` 10 条
- 顶层 `position: "after_char"` 的共 9 条，这 9 条的 `extensions.position` 全部为 **0**（before_char 数字值）
- **重要发现**：生产卡的顶层 position 字符串与 `extensions.position` 数字存在**系统性不对应**——生产卡是直接组装的 JSON，未经过 `convertWorldInfoToCharacterBook()` 转换映射。ST 读取时以 `extensions.position` 数字为实际语义，顶层字符串字段形同虚设。
- **施工规范**：直接手写卡时，顶层 `position` 字符串只需写合法值（`"before_char"` 或 `"after_char"` 任一），真实位置**只看 `extensions.position` 数字**。

来源：`src/endpoints/characters.js` + `public/scripts/world-info.js` + 本地实战卡 | 适用版本：ST v1.14.0 | **置信度：high** `[运行时源码]` `[本地实战]`

**关键字段语义**：

## 5. PNG tEXt chunk 嵌入格式

### 5.1 读写机制

**写入（`write()` 函数）**：
1. 删除所有现有 `chara` 和 `ccv3` chunk
2. 将完整 JSON 字符串 UTF-8 → Base64 编码
3. 插入 `chara` chunk（V2）：`PNGtext.encode('chara', base64Data)`，插入 IEND 之前
4. 尝试解析 JSON，加 `spec='chara_card_v3'`、`spec_version='3.0'`，再插入 `ccv3` chunk（V3）

**读取（`read()` 函数）**：
1. 提取所有 tEXt chunk
2. 优先匹配 `ccv3`（V3），其次 `chara`（V2），均大小写不敏感
3. Base64 解码 → UTF-8 字符串 → JSON.parse

```javascript
// V2 chunk key: 'chara'
// V3 chunk key: 'ccv3'
// 编码: Buffer.from(jsonString, 'utf8').toString('base64')
```

**当前仅支持 PNG**（WEBP 格式未实现），非 PNG 格式抛出错误。

来源：`src/character-card-parser.js` | 适用版本：ST v1.14.0 | **置信度：high** `[运行时源码]`

### 5.2 导入格式支持

| 格式 | 函数 | 关键逻辑 |
|---|---|---|
| PNG | `importFromPng` | 读 tEXt chunk → `getCharaCardV2()` |
| JSON | `importFromJson` | `spec !== undefined` → readFromV2；`name !== undefined` → V1；`char_name` → Pygmalion/gradio |
| YAML | `importFromYaml` | YAML 解析后同 JSON 流程 |
| CharX | `importFromCharX` | ZIP 包含 card.json + assets/ 目录 |
| BYAF | `importFromByaf` | ZIP 包含聊天背景、角色图像等资源 |

来源：`src/endpoints/characters.js` | 适用版本：ST v1.14.0 | **置信度：high** `[运行时源码]`

## 6. V3 新增特性：data.assets（CharX 格式）

CharX（.charx）是 ZIP 包装格式，`card.json` 内的 `data.assets` 数组支持以下结构：

```json
{
  "uri": "embedded://sprites/happy.png",
  "type": "emotion",
  "name": "happy",
  "ext": "png"
}
```

**`type` 合法值**：`"icon"` / `"user_icon"` / `"emotion"` / `"expression"` / `"background"` / 其他（归为 misc）

**URI 前缀**：`"embeded://"` 或 `"embedded://"` 或 `"__asset:"` 均支持（拼写容错）

**图像扩展名支持**：png、jpg、jpeg、webp、gif、apng、avif、bmp、jfif

**group_only_greetings**：spec 定义存在，但 ST 服务端代码无处理逻辑，本地生产卡未使用（见 12.1 悬案）。

来源（assets）：`src/charx.js` | 来源（group_only_greetings）：spec_v2.md | 适用版本：ST v1.14.0 | **置信度：high**（assets）`[运行时源码]` / **medium·单源孤证**（group_only_greetings）`[文档·单源]`

## 11. 高频避坑

### 11.1 顶层 spec 与 data 层必须同步

ST 在 `readFromV2()` 时读 `data` 层，在直接访问时读顶层，两层均要写且保持一致。本地生产卡直接存 V3 JSON（`spec: "chara_card_v3"`），但 ST 服务端的 `charaFormatData()` 写 V2；PNG 嵌入时 `write()` 会在 V2 基础上追加 V3 chunk。

**置信度：high** `[运行时源码]` `[本地实战]`

### 11.2 character_book.extensions 必须存在

`TavernCardValidator` 的 `#validateCharacterBookV2()` 要求 `character_book` 必须同时包含 `extensions`（object）和 `entries`（array），否则验证失败导致导入报错。手写卡常见遗漏 `character_book.extensions: {}`。

> 实测补充：星月 3.0.2 生产卡的 `character_book` 顶层只有 `{entries, name, description}`，**缺少 `extensions`**——但该卡是 V3（`spec: "chara_card_v3"`），V3 验证器（`#validateDataV3`）只验 data 存在，不调用 `#validateCharacterBookV2`，因此不会报错。此陷阱仅对 V2（`spec: "chara_card_v2"`）导入路径有效；V3 卡绕过此检查。

**置信度：high** `[运行时源码]`

### 11.3 position 字段的双轨坑

`entry.position`（字符串）只有 `"before_char"` 和 `"after_char"` 两个值，**永远**不应写 `"atDepth"` 等字符串。atDepth 等高级位置必须通过 `entry.extensions.position = 4` 来指定，同时 `entry.position` 写 `"before_char"` 作为兼容值。

**置信度：high** `[运行时源码]` `[本地实战]`

### 11.4 正则 findRegex 必须是完整正则字符串（含 `/flags`）

ST 的 `RegexProvider.instance.get(regexString)` 期望传入类似 `/pattern/gi` 形式的字符串，而非裸 pattern。本地实战卡全部使用该格式：`"findRegex": "/<tag>/gi"`。

**置信度：high** `[本地实战]` `[运行时源码]`

### 11.5 `markdownOnly: true` + `placement: [2]` 是 AI 输出美化的标准组合

`promptOnly: true` 的正则**不在**渲染层执行，只作用于发给 AI 的提示词。状态栏/气泡等显示类正则必须 `markdownOnly: true, promptOnly: false, placement: [2]`。

**置信度：high** `[运行时源码]` `[本地实战]`

> 校准补充：`engine.js` 中 `getRegexedString` 逻辑已验证：`markdownOnly && isMarkdown` 和 `promptOnly && isPrompt` 分别控制执行路径，源码与生产卡实践一致。

### 11.6 `substituteRegex: 0` 是默认安全值

生产卡默认用 0（不替换宏），仅当 findRegex 里需要 `{{char}}` 等宏替换时才改为 1 或 2。值 2 会转义所有正则元字符，导致复杂正则失效。

**置信度：high** `[运行时源码]` `[本地实战]`

### 11.7 extensions 命名空间应隔离

spec_v2.md 规定：第三方工具写入 `extensions` 时必须用命名前缀（如 `agnai_xxx` 或 `{"myapp": {...}}`），不得直接写裸字段。ST 官方字段：`talkativeness`、`fav`、`world`、`depth_prompt`、`regex_scripts`。

**置信度：medium·单源孤证** `[文档·单源]`

### 11.8 `json_data` 字段的特殊地位

ST 服务端存储时会在顶层附加 `json_data` 字段（包含完整原始 JSON），用于保留未知字段。`charaFormatData()` 第一步就从 `json_data` 解析出 `char` 对象。导出时会 `unsetPrivateFields()` 移除该字段，不会出现在导出卡中。

**置信度：high** `[运行时源码]`

## 12. 悬案与不确定项

### 12.1 group_only_greetings 的实际 ST 实现状态

**现象**：spec_v2.md 定义了该字段，本地生产卡（星月/交错）的 `data` 层有该字段但为空数组，ST 服务端代码（`characters.js`）无任何处理逻辑。

**两种可能**：(a) ST 客户端侧（`public/scripts/`）有处理，服务端不需要；(b) 该字段仅为 spec 预留，ST 尚未实现。

**真机验证路径**：在 ST 中创建 `group_only_greetings` 非空的角色卡，加入群聊，观察开场消息是否切换。

**置信度：low** `[文档·单源]`

### 12.2 character_book.scan_depth / token_budget / recursive_scanning 的存储位置

spec_v2.md 定义这些书级字段，但本地实战卡的 `character_book` 顶层只有 `name`、`description`、`entries`（星月）或加 `extensions`（交错）。

**回源结论（源码+生产卡已明确）**：
- `convertWorldInfoToCharacterBook()` 返回值只有 `{entries, name}`，不输出这三个字段
- 交错宙域 2.6.0 的 `character_book.extensions` 只含 `world/export_source/mvu_worldbook_name`，无 `scan_depth/token_budget/recursive_scanning`
- 两生产卡顶层均无这三字段
- **结论**：这三个书级字段在 ST v1.14.0 的卡内 `character_book` 中**不被写入或保存**，由 ST 全局世界书设置覆盖管理，不随卡导出

仍存疑：客户端 `world-info.js` 读取 character_book 时是否会用到这些字段（若存在的话），需真机在 character_book 中手动写入这三字段后验证 ST 是否识别。

**置信度：medium·推导有佐证** `[运行时源码]` `[本地实战]`（字段不写入已双重确认；客户端读取行为未验证）

### 12.3 atDepth position 的 extensions.role 取值 1/2 的实际作用

`role: 0/1/2`（对应 system/user/assistant）仅在 `extensions.position = 4`（atDepth）时有意义。本地实战卡的 atDepth 条目全部使用 `role: 0`（system）。其他 position 值下 role 字段是否被忽略未经源码验证。

**真机验证路径**：创建 atDepth + role=1(user) 的条目，查看 ST prompt 组装时该条目的 message role。

**置信度：medium·推导未验证** `[本地实战]`（role=0 已实测；role=1/2 推断有效但未真机确认）

### 12.4 ccv3 chunk 与 chara chunk 同时存在时的客户端行为

服务端 `read()` 明确 ccv3 优先，客户端 `public/scripts/utils.js` 的 `extractDataFromPng(data, identifier = 'chara')` 是独立实现，默认 identifier 为 `'chara'`，**不读 ccv3**。

**回源结论（客户端源码已明确）**：
- 服务端（`character-card-parser.js read()`）：ccv3 优先 → chara 回退
- 客户端（`utils.js extractDataFromPng()`）：只读 `'chara'`（V2），**不识别 ccv3**

实际影响：ST 中 PNG 导入走服务端路径（服务端解析后存 JSON），客户端的 `extractDataFromPng` 主要用于本地预览或前端操作场景，通常不直接影响导入流程。真机影响未验证。

**置信度：high** `[运行时源码]`（两端行为已分别在源码中确认）

### 12.5 世界书绑定（data.extensions.world）与 character_book 的关系

本地实战卡两者均存在且同名。`charaFormatData()` 在有 `data.world` 时读取世界书文件后赋给 `data.character_book`。

**回源结论（源码已明确）**：两个分支（`file.originalData` 和 `file.entries`）均使用 `_.set(char, 'data.character_book', ...)` 直接**覆盖**，不合并。若 `data.world` 指向有效文件，原有 `character_book` 内容会被完全替换。

仍存疑：`readWorldInfoFile` 第三个参数 `false` 的含义（是否控制是否读 `originalData`），以及两个分支 if 之间是否互斥——需阅读 `readWorldInfoFile` 实现确认。

**置信度：high** `[运行时源码]`（覆盖逻辑已直接读取源码确认）
