## 1. 定义与主链

### 1.1 一句话定义

**数据库二创前端**是在正常多楼层角色卡中，以 SP·数据库保存的表格为状态后端，把人物、背包、任务、关系、地图或经营数据呈现为作品专属状态栏、控制中心或消息内面板的前端。

它在体验上近似“数据库版的 MVU 状态栏/控制中心”，但这只是产品形态类比：

- 数据来自 SP·数据库的 `mate`、`sheet_*` 和 `content`；
- 读写入口是运行时提供的 `AutoCardUpdaterAPI`；
- 不读取 MVU 的 `stat_data`；
- 不使用 MVU 的变量更新协议；
- 不把二维表数据伪装成另一套变量树。

### 1.2 正常游玩的权威主链

本册只讨论以下常规链路：

```text
玩家在 ST 正常发送消息
        ↓
ST 调用模型并生成一条真实 AI 回复楼层
        ↓
SP·数据库在 AI 回复后按自身配置触发填表
        ↓
数据库完成或提交本轮表格变更
        ↓
状态栏 / 控制中心重新读取数据库并渲染
        ↓
玩家继续下一轮正常聊天
```

这里有三个不可打乱的事实：

1. **剧情正文属于真实 AI 回复楼层。**数据库前端不是剧情正文的替代品。
2. **数据库填表发生在正常回复链之后或其配置的生命周期点。**前端只能观察与呈现状态，不能假定数据库先于回复产出完整楼层信息。
3. **下一轮仍由 ST 正常聊天推进。**前端不得把对话主链改造成只在面板内部循环的私有协议。

### 1.3 C7 可以负责什么

- 显示当前数据库状态；
- 在表更新后刷新局部 UI；
- 提供经约束的玩家编辑入口；
- 展示加载、填表中、保存中、冲突、失败和空数据状态；
- 将二维表转换为作品可消费的 domain model；
- 为人物、背包、任务、关系等系统提供统一查询与渲染层；
- 在消息 iframe 或长生命周期宿主脚本中安全接入数据库 API。

### 1.4 C7 不负责什么

- 不生成、替代、隐藏或镜像剧情楼层；
- 不接管 ST 的玩家输入、模型生成、Swipe、消息树或聊天历史；
- 不承诺产出剧情正文、选项、路由、楼层号等完整应用协议；
- 不把插件的内部对象直接暴露给每一个 UI 组件；
- 不用直接数组突变冒充持久化；
- 不把一次已核源码快照写成永久版本真相。

### 1.5 最小成立条件

一个前端要称为数据库二创前端，至少满足：

- 以 SP·数据库当前聊天的表数据为权威状态；
- 通过公开 API 或经项目明确审计的兼容层读取数据；
- 通过公开 CRUD/SQL 能力写回，不直接修改活动引用；
- 有明确表契约、业务主键和失败重读策略；
- 能在真实多楼层聊天中随回复与填表更新；
- 插件缺失、能力不足或表结构不兼容时能降级，而不是静默显示旧数据。

## 2. 与数据库模板、MVU 前端及实验路线的边界

### 2.1 C4 与 C7 的职责

| 指南 | 回答的问题 | 主要交付物 |
|---|---|---|
| [C4 数据库表格模板](./C4_数据库表格模板.md) | 数据库表怎样设计、填表、约束和维护 | `mate`、`sheet_*`、列头、填表规则、数据规范 |
| 本册 C7 | 已有数据库怎样被做成玩家可用的作品级 UI | adapter、store、组件、交互、回调、写回与验收 |

推荐顺序：先用 C4 稳定数据契约，再用 C7 消费该契约。前端不能用 CSS 掩盖表结构不稳定，也不能靠猜列名补救没有版本管理的数据模型。

### 2.2 与 MVU 前端的类比只能停在体验层

两者可以拥有相似产品组件：

| 体验组件 | MVU 前端 | 数据库二创前端 |
|---|---|---|
| 状态栏 | 读取 MVU 变量 | 读取数据库表 |
| 控制中心 | 调用 MVU 变量链写回 | 调用数据库 CRUD/SQL 写回 |
| 订阅更新 | MVU 对应事件 | 数据库 table-update callback |
| 数据模型 | 路径化变量树 | 多张二维表及元数据 |
| 持久化边界 | MVU 运行时 | 数据库插件运行时 |

因此可以说“数据库版 MVU 前端”，但不能把它实现为 MVU：

```text
错误：UI → stat_data → JSONPatch
正确：UI → domain adapter → AutoCardUpdaterAPI → SP·数据库
```

### 2.3 数据库兼容同层是另一条待验证路线

SP·数据库的常规公开表能力没有为完整同层应用提供剧情正文、选项、路由和楼层信息输出契约。把数据库状态 UI 直接等同于完整同层应用，会遗漏真实回复、楼层生命周期和生成主链。

相关设想单独见 [C8 数据库兼容同层前端（待验证）](./C8_数据库兼容同层前端-待验证.md)。C8 是实验路线，不是本册的常规兼容模式；在真机链路完成前，不应写进 C7 的成熟能力清单。

## 3. 动态版本政策

### 3.1 为什么不能写死插件基线

SP·数据库仍在更新。标签编号、提交时间和展示名称不是一个可永久排序的版本字符串：2026-07-14 查询公开仓库时，`spv7.9` 的创建时间为 2026-07-12，而编号更大的 `spv8.2` 创建于 2026-07-10。只比较数字会得出错误的“最新版本”。

本册禁止：

- 宣称某个 tag 永久最新；
- 只凭 `display_name` 或 manifest `version` 判断 API；
- 因为 CDN URL 中有版本号，就把所有行为永久绑定到该版本；
- 未检测方法存在便调用新 API；
- 用一次源码核验替代每次发布前的运行时回归。

### 3.2 三层兼容策略

#### 第一层：最低能力契约

C7 的跨版本主链只依赖一组最小能力：

```text
读取：exportTableAsJson
订阅：registerTableUpdateCallback + unregisterTableUpdateCallback
写入：updateCell / updateRow / insertRow / deleteRow（仅编辑型 UI 必需）
```

只读状态栏可以只要求读取与订阅；编辑型控制中心还必须检测 CRUD。

#### 第二层：已核源码快照

本册记录两组可复核证据，而不是把它们当永久唯一基线：

| 核验日期 | 快照 | 用途 |
|---|---|---|
| 历史样本 | tag `spv5.5.6`，commit `2020b14c0672e74409a3d98c8a9899e13069ca3f` | 精确核验核心数据、CRUD、callback、lock、SQL 公开实现 |
| 2026-07-14 | `main` / tag `test10`，commit `708e66ba58dfe0a08d34cba7cb8730a8f8ff1e80` | 检查近期主干；manifest `display_name` 为 `SP·数据库 VII` |

本轮源码差异检查中，C7 使用的核心 export、CRUD、callback、lock API 仍存在，主干同时增加了其他 API 组。这个结论只对表中快照负责。

#### 第三层：运行时能力检测

真正运行时不问“你叫哪个版本”，而问“当前对象有没有所需能力”。但要区分两件事：API 方法有没有挂载，以及当前 storage provider 是否真的允许使用该能力。已核快照会在原生模式也挂载 SQL 方法，原生 provider 调用时才返回失败，因此 SQL 不能只靠 `typeof` 判定：

```js
function detectDatabaseCapabilities(api) {
  return {
    read: typeof api?.exportTableAsJson === 'function',
    subscribe:
      typeof api?.registerTableUpdateCallback === 'function' &&
      typeof api?.unregisterTableUpdateCallback === 'function',
    crud: ['updateCell', 'updateRow', 'insertRow', 'deleteRow']
      .every(name => typeof api?.[name] === 'function'),
    fillStart: typeof api?.registerTableFillStartCallback === 'function',
    manualFill: typeof api?.triggerUpdate === 'function',
    locks: typeof api?.getTableLockState === 'function',
    sqlApiPresent:
      typeof api?.querySql === 'function' &&
      typeof api?.executeSqlMutation === 'function',
  };
}

async function probeDatabaseCapabilities(api) {
  const capabilities = detectDatabaseCapabilities(api);
  if (!capabilities.sqlApiPresent) {
    return { ...capabilities, sqliteReady: false };
  }

  try {
    // 只做无副作用的只读探针；原生 provider 在已核快照中会返回 null。
    const result = await api.querySql('SELECT 1 AS ok');
    const sqliteReady = Boolean(
      result &&
      Array.isArray(result.rows) &&
      Number(result.rows[0]?.ok) === 1,
    );
    return { ...capabilities, sqliteReady };
  } catch {
    return { ...capabilities, sqliteReady: false };
  }
}
```

产品应按能力降级：

- 没有 `read`：显示依赖缺失，停止初始化；
- 有 `read`、无 `subscribe`：允许手动刷新，禁止宣称实时；
- 无 `crud`：进入只读模式；
- 无 `locks`：隐藏锁 UI；
- `sqlApiPresent=false` 或 `sqliteReady=false`：关闭全部 SQL UI，继续使用通用 export + CRUD 主链；
- 出现未知新 API：默认不用，先审计再接入。

### 3.3 发布时保存能力报告

建议在开发日志中保存：

```js
async function logDatabaseCapabilities(api) {
  console.table({
    checkedAt: new Date().toISOString(),
    ...await probeDatabaseCapabilities(api),
  });
}
```

能力报告只用于诊断，不应上传玩家的表数据、聊天内容或鉴权信息。

## 4. 分层架构

### 4.1 推荐五层

```text
┌──────────────────────────────────────┐
│ View：人物卡、背包、任务、关系、地图 │
├──────────────────────────────────────┤
│ Store：加载、就绪、填表、保存、冲突  │
├──────────────────────────────────────┤
│ Domain：Character / Item / Quest     │
├──────────────────────────────────────┤
│ Adapter：表契约、归一化、主键、写回  │
├──────────────────────────────────────┤
│ Host：AutoCardUpdaterAPI             │
└──────────────────────────────────────┘
```

### 4.2 唯一允许的数据方向

```text
读取：Host API → clone → adapter → domain → store → view
写入：view intent → validator → adapter → Host CRUD/SQL → re-read
通知：Host callback → schedule refresh → clone → adapter → store
```

禁止：

```text
view → 直接改 content
组件 A → 自己猜表头
组件 B → 长期缓存 rowIndex
回调参数 → 原地排序或 splice
保存成功 → 只改本地 store、不重读数据库
```

### 4.3 UI 临时状态不进业务表

标签页、折叠状态、搜索词、排序方式、弹窗开关、滚动位置属于前端临时状态。除非它们本身是角色卡规则的一部分，否则不应写入数据库业务表。

## 5. API 宿主发现与能力检测

### 5.1 不要假设 API 一定在当前 window

消息内容经常运行于 iframe。已核源码快照把 API 挂到宿主窗口；从同源消息 iframe 接入时，常见入口是 `window.parent.AutoCardUpdaterAPI`，而插件或宿主脚本中也可能直接使用当前 `window.AutoCardUpdaterAPI`。

跨域访问可能抛异常，因此查找必须防御式执行：

```js
function findDatabaseHost(start = window) {
  const candidates = [];

  try { candidates.push(start); } catch {}
  try { if (start.parent && start.parent !== start) candidates.push(start.parent); } catch {}
  try { if (start.top && !candidates.includes(start.top)) candidates.push(start.top); } catch {}

  for (const candidate of candidates) {
    try {
      if (candidate?.AutoCardUpdaterAPI) {
        return { hostWindow: candidate, api: candidate.AutoCardUpdaterAPI };
      }
    } catch {
      // 跳过不可访问的窗口层级。
    }
  }

  return null;
}
```

### 5.2 有界等待 API 与数据

消息 DOM 出现不代表插件 API 与当前聊天数据已经就绪：

```js
async function waitForDatabase({ timeoutMs = 10_000, intervalMs = 100 } = {}) {
  const deadline = Date.now() + timeoutMs;

  while (Date.now() < deadline) {
    const host = findDatabaseHost();
    const api = host?.api;

    if (typeof api?.exportTableAsJson === 'function') {
      const raw = api.exportTableAsJson();
      if (raw && typeof raw === 'object' && Object.keys(raw).length > 0) {
        return host;
      }
    }

    await new Promise(resolve => setTimeout(resolve, intervalMs));
  }

  throw new Error('数据库 API 或当前聊天表数据在限定时间内未就绪');
}
```

超时应呈现“数据库尚未就绪/请检查插件与表配置”，不能无限转圈。

### 5.3 不要把 API 对象永久缓存到消息 iframe

切换聊天、重载扩展或重建消息 iframe 后，旧引用可能过期。短生命周期视图应在 mount 时发现宿主，在 unmount 时注销回调；长生命周期 bridge 则在聊天切换后重新读取并广播新快照。

## 6. 表契约与只读快照

### 6.1 顶层结构

已核快照中，导出对象通常包含：

```text
mate
sheet_0
sheet_1
...
```

业务表一般位于 `sheet_*`，每张表包含 `name` 与二维 `content`。不要把对象枚举顺序当业务顺序；优先使用稳定 `sheetKey`。

### 6.2 二维 content

```js
const content = [
  ['row_id', '角色ID', '姓名', '生命值'], // rowIndex 0：表头
  ['1', 'char_001', '莉莉', '82'],        // rowIndex 1：第一条数据
  ['2', 'char_002', '阿尔法', '67'],      // rowIndex 2：第二条数据
];
```

关键约束：

- `content[0]` 是表头；
- 已核模板把 `row_id` 放在第 0 列；它是数据库物理标识，不代替作品业务主键；
- 第一条数据的数据库 `rowIndex` 是 `1`；
- `content.slice(1)` 的数组下标 `0` 不能直接传给 CRUD；
- 单元格可能为空、数字字符串、JSON 字符串或普通文本；
- 表头重名、空表头和缺失必需列必须在 adapter 层报错。

### 6.3 export 返回值按活动引用处理

历史已核源码中，`exportTableAsJson()` 直接返回当前活动表对象，而不是深拷贝。即使未来版本改变实现，前端仍应把返回值视为只读，并立即 clone/归一化：

```js
function cloneSnapshot(value) {
  if (typeof structuredClone === 'function') {
    return structuredClone(value ?? {});
  }
  return JSON.parse(JSON.stringify(value ?? {}));
}

function readSnapshot(api) {
  return cloneSnapshot(api.exportTableAsJson());
}
```

直接写 `raw.sheet_0.content[1][2] = 100` 不等于调用了插件持久化，也可能污染插件当前内存状态。

### 6.4 表契约必须显式声明

```js
const TABLE_CONTRACTS = {
  characters: {
    sheetKey: 'sheet_0',
    nameHint: '角色表',
    primaryKey: '角色ID',
    requiredHeaders: ['row_id', '角色ID', '姓名', '生命值'],
  },
  inventory: {
    sheetKey: 'sheet_1',
    nameHint: '背包表',
    primaryKey: '物品ID',
    requiredHeaders: ['row_id', '物品ID', '名称', '数量'],
  },
};
```

建议校验：

- `sheetKey` 存在且为 `sheet_*`；
- `content` 是二维数组；
- 表头非空、无重复；
- 必需列齐全；
- 主键列每行非空且唯一；
- 数值列可解析且范围合理；
- 表名提示仅用于报错，不代替稳定 key。

## 7. 归一化 adapter

### 7.1 通用归一化

```js
function assertTable(raw, contract) {
  const table = raw?.[contract.sheetKey];
  if (!table) throw new Error(`缺少表：${contract.sheetKey}`);
  if (!Array.isArray(table.content)) throw new Error(`${contract.sheetKey}.content 不是数组`);
  if (!Array.isArray(table.content[0])) throw new Error(`${contract.sheetKey} 缺少表头`);

  const headers = table.content[0].map(value => String(value ?? '').trim());
  const duplicates = headers.filter((value, index) => value && headers.indexOf(value) !== index);
  if (headers.some(value => !value)) throw new Error(`${contract.sheetKey} 存在空表头`);
  if (duplicates.length) throw new Error(`${contract.sheetKey} 存在重复表头：${[...new Set(duplicates)].join('、')}`);

  const missing = contract.requiredHeaders.filter(header => !headers.includes(header));
  if (missing.length) throw new Error(`${contract.sheetKey} 缺少列：${missing.join('、')}`);

  const rows = table.content.slice(1).map((cells, index) => {
    const values = Array.isArray(cells) ? cells : [];
    const record = Object.fromEntries(headers.map((header, col) => [header, values[col] ?? '']));
    return {
      rowIndex: index + 1,
      record,
    };
  });

  const seen = new Set();
  for (const row of rows) {
    const key = String(row.record[contract.primaryKey] ?? '').trim();
    if (!key) throw new Error(`${contract.sheetKey} 第 ${row.rowIndex} 行缺少业务主键`);
    if (seen.has(key)) throw new Error(`${contract.sheetKey} 业务主键重复：${key}`);
    seen.add(key);
  }

  return {
    sheetKey: contract.sheetKey,
    name: String(table.name ?? contract.nameHint ?? contract.sheetKey),
    headers,
    rows,
  };
}
```

### 7.2 转换为 domain model

```js
function toInteger(value, fallback = 0) {
  const number = Number.parseInt(String(value ?? ''), 10);
  return Number.isFinite(number) ? number : fallback;
}

function buildCharacterDomain(snapshot) {
  const table = assertTable(snapshot, TABLE_CONTRACTS.characters);

  return table.rows.map(({ rowIndex, record }) => ({
    id: String(record['角色ID']).trim(),
    name: String(record['姓名'] || '未命名'),
    hp: toInteger(record['生命值']),
    source: {
      sheetKey: table.sheetKey,
      observedRowIndex: rowIndex,
    },
  }));
}
```

`observedRowIndex` 只供显示与诊断，不能成为长期写回地址。

### 7.3 一次刷新只导出一次

```js
function buildDomainSnapshot(api) {
  const snapshot = readSnapshot(api);
  return {
    characters: buildCharacterDomain(snapshot),
    inventory: buildInventoryDomain(snapshot),
    readAt: Date.now(),
  };
}
```

所有组件消费同一份 domain snapshot，避免一帧内多次 export 得到交错状态。

## 8. 回调与填表生命周期

### 8.1 table-update 必须同引用注销

已核快照公开：

```js
api.registerTableUpdateCallback(callback);
api.unregisterTableUpdateCallback(callback);
```

注销时必须传入注册时的同一个函数引用：

```js
function mountDatabaseView(api, refresh) {
  let queued = false;

  const onTableUpdate = () => {
    if (queued) return;
    queued = true;
    queueMicrotask(() => {
      queued = false;
      refresh();
    });
  };

  api.registerTableUpdateCallback(onTableUpdate);
  refresh();

  return () => api.unregisterTableUpdateCallback(onTableUpdate);
}
```

错误示例：

```js
api.registerTableUpdateCallback(() => refresh());
api.unregisterTableUpdateCallback(() => refresh()); // 不是同一个函数，注销失败
```

### 8.2 fill-start 不能在短生命周期视图里随意注册

历史快照公开了 `registerTableFillStartCallback(callback)`，但没有对应的公开注销方法。若每次重绘消息 iframe 都注册一个新闭包，可能累积监听和重复渲染。

安全顺序：

1. 优先只依赖可成对注销的 table-update；
2. 确需填表开始状态时，由长生命周期宿主 bridge 最多注册一次；
3. bridge 再用项目自己的可退订事件分发给视图；
4. 插件未来若新增公开注销能力，先做能力检测与真机验证，再接入。

### 8.3 没有公开 fill-end 回调

已核快照没有 `registerTableFillEndCallback()`。table-update 也不能无条件视为填表结束，因为更新可能来自：

- AI 自动填表；
- 玩家 CRUD；
- SQL 写入；
- 导入、重载或其他插件操作；
- 一次任务中的多次提交。

因此 store 应区分：

```text
filling：观察到填表开始，但结束语义未知
refreshing：收到任意更新，正在重读
ready：重读和表契约校验完成
stale：超过项目设定时间仍无可确认结果
error：重读或校验失败
```

不要显示“AI 填表已完成”，除非当前运行时确实提供并验证了对应信号。

### 8.4 triggerUpdate 的返回值要归一化

`triggerUpdate` 属于版本敏感能力。不同快照的文档、类型和实现可能对返回值表达不同，不能把某一版本的 `boolean` 或结果对象写成永久契约。

```js
async function requestManualFill(api) {
  if (typeof api?.triggerUpdate !== 'function') {
    return { status: 'unsupported', ok: false };
  }

  try {
    const raw = await api.triggerUpdate();

    if (raw === true) return { status: 'settled', ok: true, raw };
    if (raw === false) return { status: 'rejected-or-noop', ok: false, raw };
    if (raw && typeof raw === 'object' && typeof raw.success === 'boolean') {
      return {
        status: 'settled',
        ok: raw.success === true,
        raw,
      };
    }

    return { status: 'unknown-result', ok: false, raw };
  } catch (error) {
    return { status: 'failed', ok: false, error };
  }
}
```

即使 Promise settle，也只说明这次调用返回；是否对应最终持久化、是否产生 callback、是否有后续提交，仍需针对当前插件版本真机验证并在返回后重读。

### 8.5 宿主级单例 bridge

需要跨多个消息 UI 共享状态时，可建立长生命周期 bridge：

```js
function createDatabaseBridge(api) {
  const listeners = new Set();
  let snapshot = null;

  const publish = () => {
    snapshot = readSnapshot(api);
    for (const listener of listeners) listener(snapshot);
  };

  const onUpdate = () => publish();
  api.registerTableUpdateCallback(onUpdate);
  publish();

  return {
    getSnapshot: () => cloneSnapshot(snapshot),
    subscribe(listener) {
      listeners.add(listener);
      if (snapshot) listener(cloneSnapshot(snapshot));
      return () => listeners.delete(listener);
    },
    dispose() {
      api.unregisterTableUpdateCallback(onUpdate);
      listeners.clear();
      snapshot = null;
    },
  };
}
```

切换聊天时必须 dispose 旧 bridge，再从新聊天重新发现 API 与读取表。

## 9. CRUD、安全写回与行号漂移

### 9.1 已核公共签名

历史及 2026-07-14 已核快照中的核心调用形式包括：

```js
await api.updateCell(tableName, rowIndex, colIdentifier, value);
await api.updateRow(tableName, rowIndex, data);
await api.insertRow(tableName, data);
await api.deleteRow(tableName, rowIndex);
```

也支持对象参数的快照中，`updateCell` 的字段名是 `colIdentifier`，不是 `column`：

```js
await api.updateCell({
  tableName: '角色表',
  rowIndex: 1,
  colIdentifier: '生命值',
  value: 90,
});
```

其他对象式调用应同样在当前版本核对字段后使用：

```js
const rowUpdated = await api.updateRow({
  tableName: '角色表',
  rowIndex: 1,
  data: { 生命值: 90, 状态: '警戒' },
});

const insertedRowIndex = await api.insertRow({
  tableName: '背包表',
  data: { 物品ID: 'item_017', 名称: '绷带', 数量: 1 },
});

const rowDeleted = await api.deleteRow({
  tableName: '背包表',
  rowIndex: 4,
});
```

在两组已核快照中，更新/删除以 `false` 表示失败，插入以 `-1` 表示失败并在成功时返回新行位置。项目应把这些值封装为自己的统一结果；未来版本若改变返回形状，只改 repository，不让视图跟着漂移。

版本升级时应以运行时 API 和对应源码为准；不要仅凭本节代码假定新版本永不变化。

### 9.2 保存必须是意图驱动

视图不要传裸行号，而传业务意图：

```js
await characterRepository.setHp({
  characterId: 'char_001',
  expectedHp: 82,
  nextHp: 90,
});
```

repository 在写回前：

1. 重新 export 并 clone；
2. 按业务主键定位当前行；
3. 校验 `expectedHp`，检测并发变化；
4. 获取最新 `rowIndex`；
5. 调用公开 CRUD；
6. 检查返回值；
7. 再次重读，以数据库真值覆盖本地乐观状态。

### 9.3 行号会漂移

插入、删除、排序、导入和重新生成表都可能改变 `rowIndex`。长期保存 `rowIndex = 7`，下一次很可能写到另一条记录。

```js
function findRowByBusinessKey(table, primaryKey, wanted) {
  return table.rows.find(row => String(row.record[primaryKey]) === String(wanted)) ?? null;
}
```

每次写回前必须重新解析当前表。业务主键应：

- 创建后稳定；
- 不依赖显示名称；
- 同表唯一；
- 不因本地化而改变；
- 对删除/重建有明确策略。

### 9.4 SQLite row_id 的边界

SQLite 模式若公开稳定 `row_id`，可作为比数组行号更可靠的定位线索；但它不是跨存储模式通用契约。通用 UI 仍应保留业务主键，并在 adapter 中隔离 SQLite 专属定位。

### 9.5 完整安全更新示例

```js
async function updateCellByBusinessKey({
  api,
  contract,
  key,
  column,
  expectedValue,
  nextValue,
}) {
  const snapshot = readSnapshot(api);
  const table = assertTable(snapshot, contract);
  const row = findRowByBusinessKey(table, contract.primaryKey, key);

  if (!row) throw new Error(`记录不存在：${key}`);

  const currentValue = row.record[column];
  if (String(currentValue) !== String(expectedValue)) {
    const error = new Error('数据已被其他操作更新，请刷新后重试');
    error.code = 'STALE_WRITE';
    throw error;
  }

  const result = await api.updateCell({
    tableName: table.name,
    rowIndex: row.rowIndex,
    colIdentifier: column,
    value: nextValue,
  });

  if (result === false || result == null) {
    throw new Error('数据库拒绝或未确认本次更新');
  }

  return readSnapshot(api);
}
```

返回值的成功表达仍要按当前版本归一化；上例保守地拒绝 `false/null`，项目可在源码核验后扩展。

### 9.6 删除和批量操作

- 删除必须二次确认并显示业务对象名称；
- 多行删除应从当前最大 `rowIndex` 向小执行，或使用插件公开批处理；
- 批量静默写入后必须显式重读；
- 不要使用未公开的内部“跳过保存”参数做性能优化；
- 写入中禁用重复提交，但要提供超时恢复；
- 失败后不继续维护本地伪成功状态。

## 10. 锁的坐标与有限语义

### 10.1 锁坐标不能与 CRUD 坐标混用

已核快照提供表、行、列、单元格锁相关 API。读锁的精确调用是 `getTableLockState(sheetKey)`，返回 `{ rows, cols, cells } | null`；它不接收表名、CRUD 行号或列名。

不要在组件里散写：

```js
api.getTableLockState(tableName, rowIndex, columnName); // 未核对签名就猜参数
```

两套坐标在已核快照中明确不同：

- CRUD 第一条数据是 `rowIndex=1`；锁的第一条数据行是 `rowIndex=0`；
- 导出表头与数字 CRUD 列索引包含第 0 列 `row_id`；锁的业务列 `colIndex=0` 从导出表头第 1 列开始；
- 单元格锁 key 形如 `"锁行:锁列"`，例如第一条数据的第一个业务字段是 `"0:0"`。

因此必须先按当前版本审计签名，再封装坐标换算。已核快照可使用：

```js
function isCellLocked(api, sheetKey, crudRowIndex, exportedHeaderIndex) {
  const state = api.getTableLockState(sheetKey);
  if (!state) return false;

  const lockRowIndex = crudRowIndex - 1;
  const lockColIndex = exportedHeaderIndex - 1;
  if (lockRowIndex < 0 || lockColIndex < 0) return false;

  return state.rows.includes(lockRowIndex)
    || state.cols.includes(lockColIndex)
    || state.cells.includes(`${lockRowIndex}:${lockColIndex}`);
}
```

升级后重新核验坐标；不要把此换算外推为未来版本永久契约。

### 10.2 可视锁不是事务锁

历史快照中，普通表锁的内建执行范围有限，主要参与特定原生 AI 更新路径。不能由此推断：

- 公开 CRUD 会自动拒绝所有锁定内容；
- `insertRow/deleteRow` 一定受保护；
- raw SQL 一定受保护；
- SQLite 写入一定服从同一把锁；
- 锁可以解决两个 UI 同时写入的并发冲突。

锁 UI 若面向玩家，自己的写回层必须执行相同规则：读取锁态、禁用控件、提交前复核。并发仍靠 expected-value、重读与冲突提示解决。

### 10.3 锁状态必须随聊天切换重读

已核实现按当前聊天与隔离维度保存锁态。切换聊天后不能沿用旧 store；进入新聊天必须重新读取表与锁。

## 11. SQLite 专属能力

### 11.1 通用主链优先 export + CRUD

如果作品同时支持原生表与 SQLite，前端主链应优先使用跨模式的 export + CRUD。SQL 能力作为检测后的增强，不应成为所有 UI 的隐含前提。

### 11.2 已核 SQL 组

已核快照包括：

- `querySql`：只读 SQL；
- `queryTableRows`：结构化表查询；
- `executeSqlMutation`：参数化写入；
- `executeSqlBatch`：多语句批处理。

这些方法在已核快照的原生模式中也会挂载，**方法存在不等于 SQL 可用**。只有 `probeDatabaseCapabilities` 的只读探针得到 `sqliteReady=true` 后，才能启用查询和写入入口；不能为了探测模式执行试写。

### 11.3 参数化查询

```js
async function queryItemsByOwner(api, capabilities, ownerId) {
  if (!capabilities?.sqliteReady || typeof api?.querySql !== 'function') {
    throw new Error('当前 storage provider 未通过 SQLite 就绪探针');
  }

  const result = await api.querySql(
    'SELECT row_id, item_id, name, quantity FROM inventory WHERE owner_id = ?',
    [ownerId],
  );

  if (!result || !Array.isArray(result.rows)) {
    throw new Error('SQL 查询未返回可识别结果，已停用 SQL UI');
  }
  return result;
}
```

不要把玩家输入或 AI 文本拼进 SQL 字符串。

### 11.4 写后仍要重读

```js
async function decrementItem(api, capabilities, itemId) {
  if (!capabilities?.sqliteReady || typeof api?.executeSqlMutation !== 'function') {
    throw new Error('当前 storage provider 未通过 SQLite 就绪探针');
  }

  const result = await api.executeSqlMutation(
    'UPDATE inventory SET quantity = quantity - 1 WHERE item_id = ? AND quantity > 0',
    [itemId],
  );

  if (result?.errors?.length) {
    throw new Error(result.errors.join('; '));
  }

  return readSnapshot(api);
}
```

SQL 结果也要进入 domain 层，视图不直接绑定数据库行对象。

## 12. UI store 与状态机

### 12.1 推荐 store

```js
const store = {
  phase: 'idle',
  capabilities: null,
  domain: null,
  error: null,
  fillingSince: null,
  savingKeys: new Set(),
  revision: 0,
  chatToken: null,
};
```

### 12.2 状态转换

```text
idle
  → locating-host
  → loading
  → ready

ready
  → refreshing → ready
  → saving → refreshing → ready
  → filling → refreshing → ready/stale
  → incompatible
  → error
```

### 12.3 每种状态都有明确 UI

| 状态 | 玩家看到什么 | 可以做什么 |
|---|---|---|
| locating-host | 正在连接数据库 | 取消或等待 |
| loading | 骨架屏 | 不提交编辑 |
| ready | 当前真实数据 | 正常查看/编辑 |
| filling | 数据库正在处理本轮状态 | 继续阅读剧情；避免重复触发 |
| refreshing | 正在同步最新数据 | 保留布局，局部忙碌态 |
| saving | 具体控件显示提交中 | 禁止同一记录重复提交 |
| stale | 状态可能未完成同步 | 手动刷新/查看诊断 |
| incompatible | 表或 API 不满足契约 | 显示缺失项 |
| error | 可理解错误 | 重试、刷新、复制诊断 |

### 12.4 丢弃过期渲染

```js
async function refreshStore(api, store) {
  const revision = ++store.revision;
  store.phase = 'refreshing';

  try {
    const domain = buildDomainSnapshot(api);
    if (revision !== store.revision) return;
    store.domain = domain;
    store.error = null;
    store.phase = 'ready';
  } catch (error) {
    if (revision !== store.revision) return;
    store.error = error;
    store.phase = 'error';
  }
}
```

切换聊天时递增 revision、清空旧 domain、注销回调并重建宿主连接。

## 14. 正常多楼层集成

### 14.1 消息内状态栏

状态栏可以出现在 AI 回复楼层内，但它的职责仍是显示数据库状态：

```text
真实 AI 回复正文
──────────────
数据库状态栏（读取当前表）
```

它不得：

- 用数据库字段拼出一份替代正文；
- 删除或隐藏原 AI 回复；
- 把后续输入改成私有生成循环；
- 假装数据库已经提供当前楼层的完整叙事结构。

消息 iframe 生命周期短，必须成对注销 table-update callback。

### 14.2 常驻控制中心

常驻脚本或插件面板适合：

- 多张表的导航；
- 复杂编辑表单；
- 宿主级单例 bridge；
- 跨消息共享筛选和布局；
- 大表虚拟列表。

常驻不代表永久有效：切聊天、切角色、切预设和扩展重载都必须重新发现 API 与清空旧数据。

### 14.3 每轮正常聊天的刷新策略

推荐顺序：

1. 玩家发送正常消息；
2. UI 保留上一轮数据，但显示“等待本轮状态更新”；
3. ST 创建真实 AI 回复；
4. 数据库按配置开始填表时，若有可靠信号则进入 filling；
5. 收到 table-update 后合并刷新请求；
6. export 一次、clone、校验并更新 domain；
7. 无 update 或生命周期不明确时，允许玩家手动刷新；
8. 下一轮继续由 ST 正常聊天推进。

### 14.4 Swipe、删楼与切聊天

这些动作可能改变数据库所对应的聊天状态。前端不能只看 DOM：

- Swipe 后重读，不沿用旧回复对应的 store；
- 删除消息后重读并提示数据是否已同步回退；
- 切聊天立即清空旧角色/背包数据；
- 重载后重新注册 callback；
- 所有回退与重放行为留真机验证，因为插件策略可能随版本变化。

## 15. 安全、性能与可访问性

### 15.1 文本一律视为不可信

数据库内容可能来自 AI 或玩家。默认使用 `textContent`：

```js
nameElement.textContent = character.name;
```

若必须渲染富文本，使用可信白名单 sanitizer，禁止事件属性、脚本、危险 URL 和任意 iframe。

### 15.2 写入白名单

不要让 DOM 自由传入任意表、列和 SQL。每个操作都映射到预定义意图：

```js
const EDITABLE_FIELDS = new Set(['生命值', '状态']);

if (!EDITABLE_FIELDS.has(column)) {
  throw new Error('该字段不可由前端编辑');
}
```

### 15.3 回调合并与大表

- 同一事件循环中的多次 update 合并成一次 refresh；
- 一次 refresh 只 export 一次；
- 大表使用分页或虚拟列表；
- domain 层建立主键索引；
- 搜索采用 debounce；
- 不在每次渲染重新解析大段 JSON 字符串；
- 不在 callback 内直接做重型 DOM 全量重建。

### 15.4 移动端

- 宽表改为卡片/详情抽屉，不做整表缩放；
- 触控目标至少保持可点击尺寸；
- 关键操作不依赖 hover；
- 底部弹层避开系统安全区；
- 长名称和长描述可换行；
- 保存中状态紧邻被编辑控件。

### 15.5 可访问性

- 真实按钮使用 `<button>`；
- 输入框有 label；
- 进度条提供文本值；
- 错误状态使用可读提示，不只靠颜色；
- modal 管理焦点并支持 Escape；
- 动态刷新使用克制的 live region；
- 尊重 `prefers-reduced-motion`。

### 15.6 日志隐私

生产环境不要打印整张数据库。诊断日志只记录：

- 时间；
- 能力布尔值；
- sheetKey；
- 操作类型；
- 错误码；
- 去敏后的业务主键。

## 17. 高频错误

### 错误 1：把状态 UI 当成剧情楼层生成器

修复：恢复正常 ST 多楼层主链；数据库前端只消费填表后的状态。

### 错误 2：把体验类比写成 MVU 实现

修复：只使用表契约和 `AutoCardUpdaterAPI`，不接 `stat_data`/JSONPatch。

### 错误 3：把一个 tag 写成永久最新版

修复：记录核验日期与 commit，运行时检测能力，发布前重跑。

### 错误 4：直接修改 export 的 content

修复：立即 clone；写回只走公开 CRUD/SQL。

### 错误 5：每个组件自己解析二维表

修复：统一 adapter 与 domain model。

### 错误 6：把数组下标当数据库 rowIndex

修复：数据行下标 `0` 对应 `rowIndex = 1`。

### 错误 7：长期缓存 rowIndex

修复：按业务主键重新读取并定位。

### 错误 8：用姓名或标题作主键

修复：使用不随显示文本变化的稳定 ID。

### 错误 9：监听匿名函数无法注销

修复：保存同一 callback 引用并在 unmount 调用 unregister。

### 错误 10：每个消息 iframe 都注册 fill-start

修复：不注册，或仅由宿主单例 bridge 注册一次。

### 错误 11：把任意 update 当 fill-end

修复：update 只触发重读；结束语义单独建模。

### 错误 12：用一次 triggerUpdate 返回定义所有版本

修复：能力检测、返回归一化、重读和真机验证。

### 错误 13：锁图标只是装饰

修复：写回层复核锁态，同时保留并发检查。

### 错误 14：把可视锁当事务锁

修复：用 expected-value、最新行定位和冲突提示处理并发。

### 错误 15：原生模式无条件调用 SQL

修复：SQL 只作检测后的 SQLite 增强。

### 错误 16：只 catch 异常，不检查失败返回

修复：按当前快照归一化返回值，失败后重读。

### 错误 17：AI 文本直接 innerHTML

修复：textContent 或经过白名单清洗的富文本。

### 错误 18：切聊天仍显示上一局数据

修复：注销、清 store、重新发现 API、重读。

### 错误 19：拿 DC 截图推断 API

修复：DC 二创只作为需求与产品形态背景；API 事实必须来自源码或真机，不虚构具体作品与实现。

## 18. 验收矩阵

### 18.1 定义与边界

- [ ] 明确这是正常多楼层角色卡的数据库状态 UI；
- [ ] 主链保留玩家发言、真实 AI 回复和后续正常聊天；
- [ ] UI 不替代、隐藏或镜像剧情楼层；
- [ ] “数据库版 MVU 前端”仅作为体验类比；
- [ ] 实验路线只导航到 C8，并标注待验证。

### 18.2 版本兼容

- [ ] 记录核验日期、commit/tag 和能力报告；
- [ ] 不宣称某个 tag 永久最新；
- [ ] 读取、订阅、CRUD、锁分别检测；SQL 另外区分“方法已挂载”和“SQLite 已实测就绪”；
- [ ] 缺失能力有明确降级；
- [ ] 升级插件后重跑源码抽样与真机矩阵。

### 18.3 表契约

- [ ] export 后立即 clone；
- [ ] `sheetKey`、表头、必需列和业务主键显式声明；
- [ ] 空表头、重复列、缺列和重复主键会报错；
- [ ] `rowIndex 0` 只作为表头；
- [ ] 视图只消费 domain model。

### 18.4 生命周期

- [ ] table-update callback 使用同引用注销；
- [ ] 多次 update 会合并刷新；
- [ ] 短生命周期 iframe 不累积 fill-start callback；
- [ ] table-update 不被当成可靠 fill-end；
- [ ] 切聊天、重载和销毁会清理 bridge/store。

### 18.5 写回

- [ ] 组件传业务意图，不传长期缓存行号；
- [ ] 提交前按主键重定位；
- [ ] expected-value 冲突会阻止覆盖；
- [ ] `updateCell` 使用 `colIdentifier`；
- [ ] 写后重读数据库真值；
- [ ] 删除、批量和失败恢复有产品状态。

### 18.6 UI 与安全

- [ ] 插件缺失、加载、空表、填表、保存、冲突、失败都有界面；
- [ ] 玩家和 AI 文本不直接进入 `innerHTML`；
- [ ] SQL 参数化且只在兼容模式启用；
- [ ] 大表使用分页/虚拟化；
- [ ] 移动端、键盘与辅助技术可用；
- [ ] 日志不泄露整表和聊天内容。

### 18.7 必须真机验证

| 场景 | 观察点 |
|---|---|
| 新建聊天首轮 | API 与表数据何时就绪 |
| 正常连续三轮 | AI 回复、填表、table-update、UI 刷新的实际顺序 |
| 本轮无表变化 | 是否有 callback；UI 如何退出等待态 |
| 手动 triggerUpdate | 返回形状、settle、持久化和 callback 次数 |
| updateCell/updateRow | 返回值、保存、重读与错误表现 |
| insert/delete 后写旧对象 | 业务主键能否避免 rowIndex 漂移误写 |
| Swipe | 数据库是否回退/重算，UI 是否刷新 |
| 删除楼层 | 表状态与聊天状态是否一致 |
| 切换聊天 | 旧回调、旧锁态和旧数据是否清空 |
| 重载插件/页面 | API 引用与 bridge 是否安全重建 |
| 锁 + 公共 CRUD | 当前版本是否自动拦截；自定义 UI 是否正确执行规则 |
| SQLite 模式 | 查询、写入、通知、row_id 与原生模式差异 |
| 大表与移动端 | 性能、触控、滚动与错误恢复 |

| export 活动引用应先 clone | high | 历史源码直接返回当前对象；防御策略跨版本仍成立 |

| DC 二创的具体实现 | 未核 | 本册不虚构作品和技术细节 |

### 19.4 每次升级必须复核

- API 挂载窗口与 iframe 可访问性；
- `exportTableAsJson` 返回值与引用语义；
- CRUD 的参数、返回值、错误与通知；
- callback 注册、注销、触发次数与是否新增 fill-end；
- `triggerUpdate` 的返回形状与持久化时点；
- 锁 API 坐标及实际拦截范围；
- 原生/SQLite 模式切换、SQL 方法挂载状态与只读就绪探针；
- Swipe、删楼、切聊天和重载后的数据行为；
- 新增 API 组是否值得纳入最低能力契约。

升级判断顺序：

```text
读取上游 commit/tag 与时间
        ↓
比较公开 API 组
        ↓
运行时能力检测
        ↓
执行最小真机矩阵
        ↓
更新本册“已核快照”和验证状态
```

本册不承诺任何标签永久最新。它承诺的是：以最低能力契约维持跨版本主链，以可追溯快照说明精确事实，以运行时检测和真机回归阻止版本漂移进入玩家体验。
