## 置信度分级体系

本手册采用三级置信度标注，格式为 `置信度 [依据类型]`：

- **high**：多源一致 + 源码直接验证 / 生产卡实战佐证，可直接施工
- **medium**：单源或推断，可参考施工，宜在真机验证后再大规模应用；强制子标签：`medium·单源孤证` / `medium·推导未验证`（须随附真机验证路径）
- **low**：孤证无源，**禁止据此施工**，须先真机验证

依据类型轴：`[运行时源码]` / `[文档·多源]` / `[文档·单源]` / `[社区]` / `[推断]` / `[本地实战]`

收集日期：2026-06-28
回源基准：SillyTavern v1.14.0 + 本地成熟卡（星月 2.5.0 / 交错宙域 2.6.0）

**两条铁规：**

1. **「从源码事实再推一步」的下游推导，即使源码本身已直读，该推导结论最高只能 `medium·推导未验证`**——除非已真机验证；源码内部事实仍标 `high [运行时源码]`。
2. **节级标注 = 节内条目下确界**——禁止整节标 high 却内含悬案条目；冲突即在节内局部降级，并交叉引用悬案章对应条目。

## 1. 消息渲染完整管线

> **核心结论** | 来源：`script.js` `messageFormatting()` 函数直读 + `engine.js`/`chats.js` | 适用版本：v1.14.0+ | **high [运行时源码]**
>
> 注：草稿原标注「engine.js 源码分析 + ST 扩展开发文档」不够精确。v1.14.0 中 `messageFormatting()` 是主渲染函数（script.js:1618），管线可在此直读。`[hook] beforeRegex/afterRegex/afterMarkdown` 为 messageFormatter API（v1.14.0 不存在），故从管线图中区分标注。

消息从 LLM 产生到最终在 `.mes_text` DOM 中渲染，经过以下完整管线（宏求值在管线最前，见第 2 章）：

```
LLM 原始文本
  │
  ▼ prompt-bias 剥离（替换开头的 prompt bias 字串）
  │  ← script.js:1645-1648 直读
  │
  ▼ 正则脚本（placement=[2] AI_OUTPUT 且 markdownOnly=true 时，此阶段执行）
  │  ← getRegexedString(raw, AI_OUTPUT, { isMarkdown: true }) [high 运行时源码]
  │
  ▼ encode_tags 处理（若开启，转义 < >）+ 双引号 <q> 包裹
  │  ← script.js:1685-1736 直读
  │
  ▼ Showdown 转 HTML（markdown → HTML）
  │   开启: simpleLineBreaks / tables / strikethrough / ghCodeBlocks
  │   ← script.js:1745 直读
  │
  ▼ encodeStyleTags（<style>→<custom-style>，DOMPurify 前保护）
  │   ← chats.js:538-543 直读 [high 运行时源码]
  │
  ▼ DOMPurify.sanitize(mes, config)
  │   config: MESSAGE_SANITIZE=true / ADD_TAGS: ['custom-style']
  │   hook A: uponSanitizeAttribute → class 名加 custom- 前缀
  │   hook B: uponSanitizeElement → HTMLUnknownElement 内 \n→<br>
  │   ← chats.js:1906-1977 直读 [high 运行时源码]
  │
  ▼ decodeStyleTags（<custom-style>→<style>，CSS AST 重写选择器前缀）
  │   ← chats.js:553-702 直读 [high 运行时源码]
  │
  ▼ 渲染进 .mes_text DOM
```

**关于 messageFormatter hook（BEFORE_REGEX/AFTER_REGEX/AFTER_MARKDOWN）：**
- 这三个 hook 点在 v1.14.0 **不存在**（getContext() 无此 API）
- 属官方文档记录的扩展点，可能在更高版本引入 | **medium·推导未验证 [文档·多源]**

**关键时序说明（v1.14.0 实测）：**

- `markdownOnly: true` 的正则脚本在 **showdown 之前**、encode_tags 之前执行 | **high [运行时源码]**（script.js:1674-1678 直读）
- `promptOnly: true` 的正则脚本在**发送给模型时**执行，不影响 display 层 | **high [运行时源码]**
- `{{trim}}` 已在宏求值阶段（管线最前）处理，不属于此管线内部 | **high [运行时源码]**
- DOMPurify 是 display 层最后净化关；之后 decodeStyleTags 再 decode CSS | **high [运行时源码]**

## 2. 宏求值管线三阶段

> **核心结论** | 来源：`macros.js` `evaluateMacros()` 源码 | 适用版本：v1.14.0+ | **high [运行时源码]**

宏求值发生在消息渲染管线之前，适用于世界书、系统提示、角色卡字段、消息体所有字符串：

```
原始字符串
  │
  ▼ Phase 1 — Pre-env Macros
  │   包含：骰子/instruct/newline/trim/legacy <USER><BOT> 标签
  │   getVariableMacros() 返回值也在此（变量宏最早执行）
  │
  ▼ Phase 2 — Env Variables
  │   用户传入 macro map，支持函数值
  │
  ▼ Phase 3 — Post-env Macros
  │   包含：token 限制/消息访问/时间/random/pick/banned
  │
  ▼ 求值完成字符串 → 进入消息渲染管线
```

**重要特性：**
- 变量宏（`{{getvar::}}` 等）处于 Phase 1，是最早执行的宏之一 | high [运行时源码]
- 旧引擎中宏求值顺序不稳定；新引擎保证 stable evaluation order | high [运行时源码]

## 3. 宏系统双引擎架构

### 3.1 两套引擎对比

> **medium·推导未验证** | 来源：官方宏文档 [文档·多源] | 适用版本：**v1.14.0 后某版本引入，v1.14.0 本地源码未见** | ~~high~~ → **medium·推导未验证 [文档·多源]**
>
> **降级原因**：本地 v1.14.0 `macros.js` + `st-context.js` 直读——未发现 `experimental_macro_engine` 开关、chevrotain 依赖、新引擎任何痕迹；`getContext()` 不暴露 `macros` 对象。双引擎架构在官方文档中有记录，但属于 v1.14.0 之后引入的特性，本地版本无法验证。**若仍使用 v1.14.0，下表描述不适用。**
>
> 真机验证路径：在目标 ST 版本设置页搜索 "experimental macro"，或 grep `experimental_macro_engine` 确认是否存在。

| | 旧引擎 MacrosParser | 新引擎 MacroEngine（Macros 2.0） |
|---|---|---|
| 存储结构 | 静态 `Map` | chevrotain CST 树 |
| 处理管线 | 正则逐条替换 | Lexer → Parser → CstWalker |
| 嵌套支持 | **不支持** | **支持**：`{{getvar::{{char}}_mood}}` |
| 求值顺序 | 不保证 | stable evaluation order |
| 开关 | 旧安装默认（未打开实验选项） | `power_user.experimental_macro_engine`（新安装默认开） |
| `{{pick}}` 兼容性 | 旧 hash 算法 | 新 hash 算法，结果不兼容 |

**切换风险**：从旧引擎切换到新引擎时，`{{pick}}` 的结果会因 hash 算法不同而改变，同 chat 同位置的"稳定随机"结果将全部刷新。已有依赖 `{{pick}}` 的卡需告知用户此风险。

来源 | 适用版本 | 置信度：medium·推导未验证 [文档·多源]（v1.14.0 本地无此特性）

### 3.2 宏处理技术细节

> 来源：`macros.js` + `variables.js` 源码 | 适用版本：v1.14.0+ | **high [运行时源码]**（v1.14.0 直读，以下各条分别标注）

**`{{trim}}`**
正则 `/(?:\r?\n)*{{trim}}(?:\r?\n)*/gi` → 替换为空字符串，连同前后换行一并清除。在 Phase 1（Pre-env）阶段处理，早于所有其他用户可见宏。| **high [运行时源码]**（macros.js:521 直读）

**`{{pick}}`**
哈希公式：`seedrandom(getStringHash("${chatIdHash}-${rawContentHash}-${offset}"))` — 三因子（chat ID、内容 hash、偏移量）保证同 chat 同位置一致。**注意：v1.14.0 只有一套 hash 实现，「跨引擎不兼容」说法仅适用于新引擎已引入的更高版本。** | **high [运行时源码]**（macros.js:404-432 直读）

**`{{random}}`**
使用 `seedrandom('added entropy.', { entropy: true })` 加密种子真随机，每次求值结果不同，不具备跨轮次一致性。| **high [运行时源码]**（macros.js:391 直读）

**变量宏求值**
变量名不经过宏预处理，直接字符串 trim 后匹配。`resolveVariable()` 查找顺序：
`scoped → local (chat_metadata) → global (extension_settings) → 返回原字符串（静默回退）`
| **high [运行时源码]**

**新引擎嵌套宏与变量速记语法**（需启用 experimental_macro_engine，v1.14.0 无此特性）：| **medium·推导未验证 [文档·多源]**（v1.14.0 无此功能，官方文档记录属更高版本）
- `.varname` = local 变量读
- `$varname` = global 变量读
- 运算符：`++/--/+=/=/-=`
- 空值合并：`||` / `??`
- 比较：`==/!=/>/</>=/<=`（返回字符串 "true"/"false"）

## 5. 正则脚本字段完整 API

### 5.1 字段定义表

> 来源：`extensions/regex/engine.js` + 本地成熟卡实战验证 | 适用版本：v1.14.0+ | **high [运行时源码] + [本地实战]**

| `scriptName` | string | UI 显示名（决定执行顺序中的 beforeScriptName 引用） |

| `replaceString` | string | 替换内容；支持 `$1..$n`、`$<name>`、`{{match}}` |

## 8. DOMPurify 与 custom-style 陷阱

### 8.1 style 属性与 style 标签的 DOMPurify 处理机制

> 来源：`chats.js` 源码直读 + Issue #3844 | 适用版本：v1.14.0+ | 整体 **medium·单源孤证→部分 high**（两种机制需分开标注）

**【重要纠错】草稿原描述「style= 属性名被改为 custom-style=」是错误的。实际存在两套独立机制：**

---

**机制 A：`<style>` 标签 → `<custom-style>` 标签（encode/decode 方案）** | **high [运行时源码]**（chats.js:538-543, 553-702 直读）

流程：
1. `encodeStyleTags()` 在 DOMPurify 前将 `<style>CSS内容</style>` 替换为 `<custom-style>URL编码内容</custom-style>`
2. DOMPurify 运行时 `ADD_TAGS: ['custom-style']`，`<custom-style>` 标签被保留
3. DOMPurify 后 `decodeStyleTags()` 将 `<custom-style>` decode 回 `<style>` 并通过 CSS AST 解析对所有选择器加 `.mes_text ` 前缀，防止跨楼层样式污染

**结论：`<style>` 标签能穿越 DOMPurify，最终在 DOM 中正常工作。** CSS 规则中的 class 选择器（如 `.foo`）会被改写为 `.custom-foo`（见 8.3 节）。

---

**机制 B：`style="..."` 内联属性处理** | **medium·版本敏感 [源码观察]**

源码中未见把属性名重命名为 `custom-style=` 的逻辑，但这不足以推出“DOMPurify 默认禁止所有 style”。现有 C3 与当前 release 观察也不支持把“必然删除”写成跨版本事实。应把内联 style 视为版本/净化配置敏感能力，而不是稳定契约。

> 真机验证路径：在最低支持版与当前版发送同一 `<span style="color:red">x</span>`，F12 同时检查最终属性与 computed style。

**生产结论：**
- 关键视觉使用 `<style>` 块 + class，避免依赖内联 style 是否保留；
- `<style>` 块在已核管线中经 encode/decode 保护；
- 动态数值使用 `data-*` + 受信 JS 或 iframe，不把用户数据拼进 style 字符串。

**绕坑方案（生产卡实战验证）：**

| 方案 | 适用场景 | 成本 | 置信度 |
|---|---|---|---|
| 使用 `<style>` 块 | 静态样式规则 | 低 | **high [运行时源码 + 本地实战]** |
| `data-*` 属性 + JS 后处理 | 需要动态数据的样式 | 中；需要配套 JS | **high [本地实战]** |
| iframe + Blob URL | 完整独立 UI 组件（如状态栏） | 高；完全绕开 ST sanitization | **high [本地实战]** |

来源：`chats.js` `encodeStyleTags()`/`decodeStyleTags()` 直读 + `C:\...\星月 3.0.2\components\*.html` [本地实战]

### 8.2 自定义 XML 标签 + simpleLineBreaks 双重 bug（#3996）

> 来源：GitHub Issue #3996 + 本地实战 | 适用版本：受 Chromium 136+ 影响更严重 | **high [运行时源码]**

**三重叠加机制：**

1. Showdown `simpleLineBreaks: true` → 把标签内部的 `\n` 转为 `<br>`
2. DOMPurify `uponSanitizeElement` hook → 对 `HTMLUnknownElement`（未在 ADD_TAGS 声明的自定义标签）再次把 `\n` 转为 `<br>`
3. Chromium 136+ 对 HTML 解析更严格 → 双重 `<br>` 堆叠，版面破坏

**修复方案（#3996 采纳）：**
```javascript
// 在 DOMPurify ADD_TAGS 中声明自定义标签，避免被当做 HTMLUnknownElement
DOMPurify.sanitize(mes, {
    ADD_TAGS: ['custom-style', 'UpdateVariable', 'JSONPatch', 'analysis', 'content', ...],
    // ...
});
```

**角色卡工程绕坑（星月卡实战）：**
在 AI 回复正文中的 `<UpdateVariable>` 块由正则替换为 HTML，不让裸 XML 标签进入 showdown 处理流程。这是更彻底的方案：正则在 showdown 前消费自定义标签，根本不存在 HTMLUnknownElement 问题。

### 8.3 CSS class 前缀（双层前缀行为：存在且需区分）

> 来源：`chats.js` 源码直读 + Issue #3844 | **high [运行时源码]**（chats.js:1915-1939 直读）

**【重要纠错】草稿「ST 不对 CSS class 名称加前缀」是错误的。v1.14.0 源码有两套 class 前缀机制：**

---

**机制 A：`class` 属性 → `custom-` 前缀（DOMPurify `uponSanitizeAttribute` hook）** | **high [运行时源码]**

在 `MESSAGE_SANITIZE` 模式（消息渲染时）下，DOMPurify hook 将每个 class 名加上 `custom-` 前缀，**例外**：
- 以 `fa-` 开头的 class（FontAwesome 图标）保留原名
- 以 `note-` 开头的 class 保留原名
- `monospace` 保留原名
- 当 `config['MESSAGE_ALLOW_SYSTEM_UI']` 为 true 且节点包含 `menu_button` class 时，整个节点豁免

```javascript
// chats.js:1926-1939（直读）
case 'class': {
    data.attrValue = data.attrValue.split(' ').map((v) => {
        if (v.startsWith('fa-') || v.startsWith('note-') || v === 'monospace') {
            return v;  // 豁免
        }
        return 'custom-' + v;  // 所有其他 class 加前缀
    }).join(' ');
}
```

**实际影响**：正则注入 `<div class="foo">` → DOM 中变为 `<div class="custom-foo">`。**角色卡若在 JS 里用 `.querySelector('.foo')` 或 CSS 用 `.foo {}` 选择器，将全部失效。须改用 `.custom-foo` 或 `data-` 属性选择。**

---

**机制 B：`<style>` 块内的 CSS 选择器前缀（CSS AST 重写）** | **high [运行时源码]**（chats.js:562-598 直读）

`decodeStyleTags()` 在将 `<custom-style>` decode 回 `<style>` 时，通过 CSS AST 将所有选择器加 `.mes_text ` 前缀（如 `.foo` → `.mes_text .foo`）。同时将选择器中的 `.foo` 改写为 `.custom-foo`（与机制 A 一致）。目的是防止跨楼层 CSS 污染。

**「Issue #3844 not planned」的正确解读**：该 Issue 申请的是给 class 加前缀的功能，官方"not planned"指的是拒绝此功能——但实际上相关行为已经通过另外方式（DOMPurify hook + CSS AST 重写）实现，只是与 Issue 描述的方案不同。
