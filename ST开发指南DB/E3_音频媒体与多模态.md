## 1 TTS（文字转语音）系统

### 1.1 机制总览与双队列架构

**核心结论**

ST 的 TTS 系统采用**提供商插件化架构**：中央 TTS Manager 负责文本解析、队列管理、播放控制，各 Provider 实现统一接口对接不同语音引擎。

**双队列机制：**

| 队列 | 作用 |
|------|------|
| `ttsJobQueue` | 等待语音合成的文本消息队列 |
| `audioJobQueue` | 已完成合成、等待播放的音频 blob 队列 |

两队列串行处理：`文本 → 语音合成 → audioJobQueue → 播放`

来源：[运行时源码] | 适用版本：ST v1.14.0+ | 置信度：**high**

### 1.3 Provider 接口规范与 Manager 公开函数

**核心结论**

每个 Provider 必须实现的接口：

| 方法 | 签名 | 说明 |
|------|------|------|
| `loadSettings` | `(settings) => void` | 初始化 Provider 设置 |
| `fetchTtsVoiceObjects` | `() => Promise<VoiceObject[]>` | 获取可用语音列表 |
| `generateTts` | `(text, voiceId) => Promise<Response>` | 生成 TTS 音频，返回 HTTP Response |
| `processText` | `(text) => string`（可选） | 文本预处理/清洗 |

TTS Manager 公开函数：

| 函数 | 用途 |
|------|------|
| `registerTtsProvider()` | 注册自定义 TTS 提供商 |
| `playFullConversation()` | 播放整个对话历史 |
| `tts_preview()` | 预览单个语音样本 |
| `getPreviewString()` | 获取语言特定预览文本 |

Voice Map 结构：

```javascript
voiceMap = {
  "CharacterName": "voiceId",          // 正常映射
  "CharacterName": "[Default Voice]",  // 使用默认语音
  "CharacterName": "disabled"          // 禁用该角色 TTS
}
```

来源：[运行时源码] | 适用版本：ST v1.14.0+ | 置信度：**high**

**避坑**：Voice Map 必须配置，否则 TTS 不知道用哪个语音。`[Default Voice]` 和 `disabled` 是两个特殊标记值，不可写错。

### 1.5 STscript 命令

**核心结论**

### 1.7 TTS 与立绘同步

**核心结论**

当 TTS 播放时，系统调用：

```javascript
globalThis.vrmLipSync(audioBlob, character)
```

用于 VRM 模型口型同步（须安装对应 VRM 扩展）。

表情自动更新通过情绪分类事件驱动，**流式传输时**以 `STREAMING_UPDATE_INTERVAL = 10000ms` 为节流间隔，防止过频率 API 调用。

来源：[运行时源码] | 适用版本：ST v1.14.0+ | 置信度：**high**

**避坑**：表情分类在 streaming 期间以 10 秒为节流间隔，不是每 token 触发——这是性能保护措施，不是 bug。

## 2 表情立绘（Expression/Sprite）系统

### 2.1 机制总览

**核心结论**

ST 的表情系统通过 NLP 情绪分类驱动角色立绘切换，支持本地模型运行或调用 LLM API 进行分类。

分类器可配置为：

| 模式 | 模型 | 情绪数量 |
|------|------|----------|
| Local | `distilbert-base-uncased-go-emotions-onnx`（约 100MB） | 28 个 |
| Local | `bert-base-uncased-emotion-onnx`（via config.yaml） | 6 个 |
| Main API (LLM) | 主模型做分类，可加自定义 prompt | 可变 |
| WebLLM | 本地推理扩展 | — |
| Extras | （已废弃） | — |

来源：[文档·多源] + [运行时源码] | 适用版本：ST v1.14.0+ | 置信度：**high**

## 7 本地实战：media_library.js 架构分析

### 7.1 设计要点与 Tavern Helper 接口

**核心结论**

这是角色卡内置**自管理图像资产库**，不依赖 ST 内置表情系统，通过 Tavern Helper 的变量接口自管理持久化存储：

```javascript
// 与 Tavern Helper 的接口对接（关键！）
function scriptVariableOption() {
  return { type: 'script', script_id: SCRIPT_ID };  // 'crossed-zone-media-library'
}

function readRawLibrary() {
  if (typeof getVariables !== 'function') return defaultLibrary();
  return getVariables(scriptVariableOption()) || {};
}

function writeRawLibrary(value) {
  if (typeof replaceVariables !== 'function') throw new Error('酒馆助手变量接口不可用');
  replaceVariables(value, scriptVariableOption());
}
```

**关键发现**：`getVariables` / `replaceVariables` 是 **Tavern Helper 扩展暴露的全局函数**，通过 `{ type: 'script', script_id: string }` option 对象区分变量命名空间。这是本角色卡访问持久化变量的核心机制。

## 10 高频避坑

**[high][文档·多源]** Extras API 已于 2024 年 4 月停止维护，**不要再基于 Extras 构建新卡功能**（Coqui-TTS、Vosk、Whisper 旧路径、talkinghead 均已死亡）。

**[high][运行时源码]** `getVariables` / `replaceVariables` 是 Tavern Helper 扩展接口，不是 ST 原生接口。卡内脚本调用前必须检查 `typeof getVariables !== 'function'`，否则在未安装 Tavern Helper 的环境中会报错。

**[high][运行时源码]** 媒体库整库大小上限 24MB；单张 data URL 上限 4MB；GIF 额外限制 3MB。超出需清理未使用资产或改用外链 URL（`external: true`）。

**[high][运行时源码]** GIF 图片不进行裁剪/压缩（保留动画帧），但超大 GIF 会被拒绝导入——这是主动防卡死措施，移植时须保留此逻辑。

**[high][文档·多源]** TTS Voice Map 必须配置，否则 TTS 不知道用哪个语音。`[Default Voice]` 和 `disabled` 是两个特殊标记值。

**[high][运行时源码]** TTS 表情分类在 streaming 期间以 10 秒为节流间隔，不是每 token 触发——这是性能保护，不是 bug。

**[high][文档·多源]** `Send inline images` 默认关闭，必须手动在 Chat Completion 设置中开启，且后端模型必须是视觉模型，否则图片会被忽略。

**[medium·单源孤证][文档·单源]** ScreenShare 扩展依赖浏览器 ImageCapture API，**Firefox 不支持**，Safari 支持有限，主要在 Chrome 上可靠运行。

**[high][文档·多源]** Dynamic Audio 的 BGM 情绪命名必须与表情系统的 28 个情绪英文名完全一致（如 `anger_0.mp3`），大小写敏感，拼写错误将导致静默失败。【提升：草稿标 medium·推导未验证，但官方文档 dynamic-audio 页面给出了明确示例（`anger_0.mp3`、`joy_0.mp3`）且属多源文档，非推导，提升至 high。】

**[high][运行时源码]** `importUrlAsset` 失败时（CORS 被拒）会自动降级为外链模式（`external: true`），不会报错，但导出库时会有 `warnings` 列表，外链资产在离线场景无效。
