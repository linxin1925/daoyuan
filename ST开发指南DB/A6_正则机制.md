## 1. 机制总览

SillyTavern regex extension 是运行在客户端 JS 层的文本变换管线。每条正则脚本（regex script）是一个 JSON 对象，存储于角色卡 `data.extensions.regex_scripts`（scoped）或 `settings.json`（global）。执行时，`getRegexedString()` 将所有匹配当前 placement + 上下文条件的脚本**顺序链式应用**，上一条的输出是下一条的输入。

**三类脚本存储位置：**

| 类型 | 存储位置 | 生效范围 |
|---|---|---|
| GLOBAL | `settings.json` | 对所有角色生效 |
| SCOPED | 角色卡 `data.extensions.regex_scripts` | 仅当前角色，需 `allowScopedScripts()` 显式允许 |
| PRESET | API 预设 | 仅绑定该预设时生效 |

**执行顺序：** Global → Scoped → Preset（顺序链式，不并行）

> 源码订正（草稿原写 Global→Preset→Scoped，已更正）：`SCRIPT_TYPES = { GLOBAL: 0, PRESET: 2, SCOPED: 1 }`，`getRegexScripts()` 以 `Object.values(SCRIPT_TYPES).flatMap(...)` 遍历——JS 对整数键按升序排列，实际顺序为 GLOBAL(0) → SCOPED(1) → PRESET(2)。

来源：`[运行时源码]` engine.js `SCRIPT_TYPES` 枚举 + `getRegexScripts()` 实现 | 适用版本：v1.14.0 | **high** `[运行时源码]`

## 2. 字段完整参考表

### 2.1 运行时字段

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | 建议 | UUID 标识符；缺失时自动生成 |
| `scriptName` | string | 是 | 显示名称；为空则无法保存 |
| `findRegex` | string | 是 | 正则表达式字符串，含旗标，格式 `/pattern/flags`；为空则不执行 |
| `replaceString` | string | 否 | 替换字符串；支持 `$1`/`$2`... 捕获组引用、`$<name>` 命名捕获组引用、`{{match}}` 宏（内部转为 `$0`）；**不支持** `$&`/`$$`/`` $` ``/`$'` 标准 JS 替换特殊模式（源码使用函数形式 replacer，这些符号不被 JS 特殊处理，当作字面量传入 `substituteParams`） |
| `placement` | number[] | 是 | 作用域数组（见第 3 章 placement 枚举）；至少一项否则不激活 |
| `disabled` | boolean | 否 | true = 屏蔽自动执行（slash command 仍可手动触发） |
| `markdownOnly` | boolean | 否 | true = 只在渲染显示时运行（UI 对应 "Alter Chat Display"） |
| `promptOnly` | boolean | 否 | true = 只在发给模型的 prompt 时运行（UI 对应 "Alter Outgoing Prompt"） |
| `runOnEdit` | boolean | 否 | true = 用户编辑消息时也执行 |
| `substituteRegex` | number | 否 | 宏替换模式：0=不替换，1=原始替换，2=转义替换（见第 6 章） |
| `minDepth` | number\|null | 否 | 最小楼层深度（0=最新一楼，null/-1=不限）；`depth < minDepth` 则跳过 |
| `maxDepth` | number\|null | 否 | 最大楼层深度；`depth > maxDepth` 则跳过；须 > minDepth 才生效 |
| `trimStrings` | string[] | 否 | 从**捕获组匹配值**（$1/$2/... 的实际值）中逐一 replaceAll 删除的字符串列表；作用于每条捕获组求值阶段，而非整条 replaceString 完成后——**medium·单源孤证** `[运行时源码]`（仅 engine.js 可见，实际效果待真机验证） |

来源：`[运行时源码]` engine.js | 适用版本：v1.14.0 | **high** `[运行时源码]`

## 3. placement 枚举（作用域）

| 数值 | 名称 | 触发时机 |
|---|---|---|
| `0` | MD_DISPLAY | **已废弃**，勿用 |
| `1` | USER_INPUT | 用户发送消息后 |
| `2` | AI_OUTPUT | AI 回复生成后 |
| `3` | SLASH_COMMAND | STscript slash command 返回值 |
| `5` | WORLD_INFO | 世界书条目注入；**必须同时设 `promptOnly: true`**，否则不激活（world-info.js 硬编码传入 `{ isMarkdown: false, isPrompt: true }`，默认两者均 false 的脚本在此路径不满足激活条件） |
| `6` | REASONING | AI 推理块（Gemini / Deepseek 等推理 API） |

来源：`[运行时源码]` engine.js `export const regex_placement` | 适用版本：v1.14.0 | **high** `[运行时源码]`

> 避坑：`placement: []`（空数组）= 脚本永远不自动触发，只能通过 slash command 手动调用。本地实战卡均用 `[2]`（AI 输出）。
>
> `MD_DISPLAY (0)` 标记为 deprecated，源码未完全移除，旧卡含此值时的实际行为不确定——**low** `[推断]`，新卡禁用。

## 4. markdownOnly / promptOnly / Ephemerality 三态逻辑

这是本机制最易混淆的核心。

**UI 字段对应关系（关键）：**

- UI 中的 "Alter Chat Display" = `markdownOnly: true`
- UI 中的 "Alter Outgoing Prompt" = `promptOnly: true`

来源：`[运行时源码]` engine.js 激活条件逻辑 | **high** `[运行时源码]`

**源码激活条件：**

```javascript
(script.markdownOnly && isMarkdown) ||
(script.promptOnly && isPrompt) ||
(!script.markdownOnly && !script.promptOnly && !isMarkdown && !isPrompt)
```

**四种组合语义：**

| `markdownOnly` | `promptOnly` | 含义 | 写回 JSONL |
|---|---|---|---|
| false | false | **默认**：永久写入聊天记录；显示和 prompt 均改变 | 是 |
| true | false | **只改显示**（Alter Chat Display）：仅影响 markdown 渲染 | 否 |
| false | true | **只改 prompt**（Alter Outgoing Prompt）：仅影响发给模型的内容 | 否 |
| true | true | **完全临时**：显示和 prompt 均改变，但不写文件——**medium·推导未验证** `[推断]`（见下方注） | 否（中等置信） |

> 注：`markdownOnly: true, promptOnly: true` 双 true 的"完全临时"行为，其 JSONL 不写回的确切触发逻辑（isSend/saveMessage 调用层）在 engine.js 中未直接体现，可能在 chat.js 上层处理。实际行为待真机验证。——**medium** `[文档·多源] + [推断]`

**实战含义：**

- 星月 / 交错宙域所有美化正则（状态栏、气泡、OMNI-NEXUS wrap）均设 `markdownOnly: true, promptOnly: false`：只影响显示，不污染模型上下文。
- 交错宙域的 prompt 清理正则（`promptOnly: true`）：从历史楼层裁剪变量块，只影响发给模型的内容，用户界面不变。

来源：`[本地实战]` 星月 3.0.2 + 交错宙域 v2.0.9 | **high** `[本地实战]`

## 5. 深度过滤（minDepth / maxDepth）

深度值 = 消息在历史中的位置，**0 = 最新一楼，数值越大越旧**。

**源码逻辑（引自 engine.js）：**

```javascript
if (typeof depth === 'number') {
    if (!isNaN(script.minDepth) && script.minDepth !== null &&
        script.minDepth >= -1 && depth < script.minDepth) {
        return; // 跳过
    }
    if (!isNaN(script.maxDepth) && script.maxDepth !== null &&
        script.maxDepth >= 0 && depth > script.maxDepth) {
        return; // 跳过
    }
}
```

**常用值语义：**

| 值 | 含义 |
|---|---|
| `minDepth: null` 或 `-1` | 无下限，对所有楼层生效（含 Continue 追加的消息） |
| `minDepth: 0` | 从最新楼开始，实际等效于对所有楼层生效 |
| `minDepth: 6` | 只对第 6 楼以深（较旧）的消息生效 |
| `maxDepth: null` | 无上限，对所有历史楼层生效 |
| `maxDepth: 3` | 只对最新 4 楼（深度 0-3）生效 |

**实战用例：** 交错宙域清理正则设 `minDepth: 6`，只清理发给模型的 prompt 中第 6 楼以深的旧变量块，减少 token 消耗，同时不影响界面显示。

来源：`[运行时源码]` engine.js + `[本地实战]` 交错宙域 v2.0.9 | 适用版本：v1.14.0 | **high** `[运行时源码]`

> 悬案：`minDepth: -1` 对 Continue 追加内容的深度计算方式待验证——**medium** `[推断]`，见第 10 章。

## 6. substituteRegex 宏替换模式

控制 `findRegex` 字段中宏（如 `{{user}}`）的处理方式：

| 值 | 名称 | 行为 |
|---|---|---|
| `0` | NONE | 宏当作字面量，不替换（**绝大多数场景用此值**） |
| `1` | RAW | 直接插入宏值（宏值含正则特殊字符会破坏 pattern，危险） |
| `2` | ESCAPED | 先 escape 宏值中的正则元字符再插入（安全） |

来源：`[运行时源码]` engine.js `substitute_find_regex` enum | 适用版本：v1.14.0 | **high** `[运行时源码]`

> 避坑：若要在 `findRegex` 中动态嵌入 `{{user}}` 等宏值，用 `2`（escaped），不要用 `1`（raw）——角色名可能含正则元字符（如括号、点号）导致 pattern 语法错误。

```json
  "replaceString": "<div class=\"xy-omni-mount\" data-xy-omni=\"analysis\"><div class=\"xy-omni-raw\" data-xy-omni-raw style=\"...\">$1</div></div>\n",
  "placement": [2],
  "disabled": false,
  "trimStrings": [],
  "markdownOnly": true,
  "promptOnly": false,
  "runOnEdit": false,
  "substituteRegex": 0,
  "minDepth": 0,
  "maxDepth": null
}
```

来源：`[本地实战]` 星月私立高等学院 MVU 3.0.2.json | **high** `[本地实战]`

### 7.2 范式 B：流式进行中（无闭合标签）美化

流式输出时 AI 尚未生成闭合标签，用负向前瞻匹配**未闭合**的开放标签，提供流式进度展示。

```json
{
  "id": "xingyue-sync-progress-omni-nexus",
  "scriptName": "[美化]变量同步中-OMNI-NEXUS",
  "findRegex": "/<UpdateVariable(?:variable)?>(?![\\s\\S]*<\\/UpdateVariable(?:variable)?>)\\s*([\\s\\S]*)$/gi",
  "replaceString": "<div class=\"xy-omni-mount\" data-xy-omni=\"progress\">...$1...</div>\n",
  "placement": [2],
  "markdownOnly": true,
  "promptOnly": false,
  "runOnEdit": false,
  "substituteRegex": 0,
  "minDepth": 0,
  "maxDepth": null
}
```

关键技巧：`(?![\s\S]*<\/UpdateVariable>)` 负向前瞻确保只匹配**未闭合**的标签（流式中间状态）。

> 配套说明：范式 B（流式中）与范式 A 的闭合版本须并存。闭合版的贪婪匹配用 `[\s\S]*?`（惰性），避免跨块吞掉其他内容。两条脚本缺一会导致流式过程中或完成后显示异常。

来源：`[本地实战]` 星月 3.0.2 + 交错宙域 v2.0.9 | **high** `[本地实战]`

### 7.4 范式 D：占位符插入——多步正则联动

状态栏渲染需要两步：先在变量块旁插入占位符，再将占位符替换为真实 HTML 组件。ST 按数组顺序串行执行，顺序必须正确。

步骤 1（插入占位，须排在步骤 2 之前）：

```json
{
  "scriptName": "[显示]变量更新状态栏占位",
  "findRegex": "/(?:<StatusPlaceHolderImpl\\s*\\/>\\s*)*(<UpdateVariable(?:variable)?>\\s*[\\s\\S]*?\\s*<\\/UpdateVariable(?:variable)?>)(?:\\s*<StatusPlaceHolderImpl\\s*\\/>)*/gi",
  "replaceString": "<StatusPlaceHolderImpl/>\n$1\n",
  "placement": [2],
  "markdownOnly": true,
  "promptOnly": false,
  "minDepth": 0,
  "maxDepth": null
}
```

步骤 2（替换占位为真实组件，须排在步骤 1 之后）：

```json
{
  "scriptName": "状态栏美化 HUD",
  "findRegex": "/<StatusPlaceHolderImpl\\s*\\/>/g",
  "replaceString": "```html\n<!doctype html>...\n```\n",
  "placement": [2],
  "markdownOnly": true,
  "promptOnly": false
}
```

> 排序注意：组件库构建元数据字段 `beforeScriptName` / `afterScriptName` 是打包工具的排序提示，非运行时字段。打包后脚本数组顺序即为执行顺序，需人工或工具确认正确性。

## 8. 执行管线时序

```
用户发送 / AI 回复生成
        ↓
getRegexedString(rawString, placement, { characterOverride, isMarkdown, isPrompt, isEdit, depth })
        ↓
getRegexScripts({ allowedOnly: true })
  →  Object.values(SCRIPT_TYPES) 顺序：GLOBAL(0) → SCOPED(1) → PRESET(2)
  →  flatMap 展开为脚本合集
        ↓
per script: 过滤检查（但注意：placement 检查在最后）
  1. (markdownOnly && isMarkdown) ||
     (promptOnly && isPrompt) ||
     (两者均 false && isMarkdown=false && isPrompt=false)?
     （不满足任一 → 直接跳过，不再检查后续条件）
  2. runOnEdit=false && isEdit=true → 跳过
  3. depth 在 [minDepth, maxDepth] 内?
  4. script.placement.includes(placement)?
     （最后检查 placement，而非最先）
        ↓ 四道全通过
runRegexScript(script, currentString)
  1. substituteRegex 处理 findRegex 中的宏
  2. regexFromString(regexString) 将字符串编译为 RegExp
     （注：v1.14.0 engine.js 中无 RegexProvider/LRU cache，
      草稿原描述有误；每次直接调用 regexFromString）
  3. String.replace(regex, replacerFn)
     - replaceString 中的 {{match}} → $0（字面替换为 $0 字符串，再由 JS replace 解析）
     - $1/$2/... 及 $<name> 命名捕获组替换
     - 每条捕获组值在求值时经 filterString(match, trimStrings) 过滤
       （trimStrings 作用于各捕获组的匹配值，非整体 replaceString 结果）
  4. substituteParams(replaceWithGroups) 处理宏替换
  5. 返回变换后字符串
        ↓
下一条脚本的输入 = 上一条输出
        ↓
最终字符串
```

> 订正说明：①执行顺序为 Global→Scoped→Preset（非原写 Global→Preset→Scoped）；②无 RegexProvider/LRU cache，为草稿虚构内容，已删除；③trimStrings 作用于捕获组值，非整体结果；④placement 检查位于四道关最后，不是第一道。

来源：`[运行时源码]` engine.js `getRegexedString` + `runRegexScript` 逐行阅读 | 适用版本：v1.14.0 | **high** `[运行时源码]`

## 9. 高频避坑

1. **`placement` 必须至少一个值**——空数组 = 脚本永远不自动触发（只能 slash command 手动调）。实战卡均用 `[2]`（AI 输出）。
   来源：`[运行时源码]` | **high**

2. **`markdownOnly: true` 的替换结果不会写回 JSONL**——在做界面美化时安全；若要让模型在下一轮看到替换结果，必须改用默认（两者均 false），或用 `promptOnly: true`。
   来源：`[运行时源码]` script.js 调用架构（isMarkdown=true 仅在渲染路径调用，写回 JSONL 发生在独立的保存流程，不受 getRegexedString 结果影响）+ `[本地实战]` 星月/交错宙域生产验证 | **high** `[运行时源码] + [本地实战]`

3. **多脚本顺序依赖**——占位符先插再替换的两步范式，必须保证数组顺序，ST 按索引顺序串行执行。构建工具的 `beforeScriptName`/`afterScriptName` 排序提示在打包时生效，需校验最终数组顺序。
   来源：`[本地实战]` + `[运行时源码]` | **high**

4. **流式中间状态的正则必须成对**——`progress` 正则（无闭合标签）和 `done` 正则（有闭合标签）须并存；`done` 正则的贪婪匹配要用 `[\s\S]*?`（惰性），避免跨块吞掉其他内容。
   来源：`[本地实战]` | **high**

5. **`minDepth: null` vs `minDepth: 0` 的区别**——`null` 无限制对所有楼层生效（含 Continue 追加）；`0` 从最新楼开始，实际上也对所有楼层生效。要限制只对旧楼生效，用较大的 `minDepth`（如 6）。
   来源：`[运行时源码]` | **high**

6. **`substituteRegex` 推荐用 `0` 或 `2`，禁用 `1`**——绝大多数场景用 `0`（不替换宏）；需动态嵌入 `{{user}}` 等宏值时用 `2`（escaped），不要用 `1`（raw），因角色名可能含正则元字符。
   来源：`[运行时源码]` | **high**

7. **`MD_DISPLAY` (placement=0) 已废弃**——现存旧卡可能含此值，新卡不要用，废弃后的实际行为不确定。
   来源：`[运行时源码]` 标注 | **medium** `[运行时源码]`（实际副作用为 **low** `[推断]`）

8. **`runOnEdit: true` + `promptOnly: true` 联合使用**——如交错宙域清理正则，确保用户编辑历史消息时 prompt 也被清理。`markdownOnly: true` 的脚本通常不需要 `runOnEdit: true`，因编辑后重新渲染会自动触发 markdown 路径。
   来源：`[本地实战]` 交错宙域 v2.0.9 | **high** `[本地实战]`

9. **`trimStrings` 是捕获组值内过滤**——作用于每个 `$1`/`$2`/... 捕获组的**实际匹配值**（在 `filterString()` 中 replaceAll），而非作用于整条 `replaceString` 完成后的最终结果。若 replaceString 中无捕获组引用（$1/$2），trimStrings 实际无效。本地实战卡均为空数组 `[]`（不需要）。
   来源：`[运行时源码]` engine.js `runRegexScript` 实现 | **medium·单源孤证** `[运行时源码]`（描述已订正；原 high 降级：草稿原描述有误，且实际边界行为未真机验证）

10. **Scoped regex 在群聊中无法通过 UI 编辑**——index.js 源码显示，在 `selected_group` 为真时，UI 层对"新增/编辑/批量操作 Scoped 脚本"均弹出 toastr.error 并阻止。但**执行层**（engine.js `getScriptsByType(SCOPED)` + `allowedOnly` 检查）完全依赖 `character_allowed_regex` 的 avatar 列表，与是否群聊无关——若 avatar 已在白名单且存在 scoped 脚本，群聊中也会执行。"群聊默认禁用"的说法**不准确**，应改为"群聊中 UI 禁止编辑 Scoped 脚本（执行准入仍由 avatar 白名单控制，不由群聊状态决定）"。
    来源：`[运行时源码]` index.js `selected_group` 检查 + engine.js `getScriptsByType` 实现 | **medium·推导未验证** `[运行时源码]`（执行路径从源码推导；群聊中 Scoped 脚本是否实际运行待真机验证）
