## 2. Message 对象完整字段参考

**来源：** `script.js`（raw GitHub）、DeepWiki 5.2、SillyTavern.NET-File-Converter  
**适用版本：** v1.14.0  
**置信度：high [运行时源码] [文档·多源]**

### 2.1 顶层字段

```jsonc
{
  "name": "CharacterName",        // 发送者名称（必填）
  "is_user": false,               // 是否用户消息（必填）
  "is_system": false,             // 是否系统/注释消息（/sys /comment）
  "mes": "消息正文",              // 主文本（必填）
  "send_date": "April 26, 2024 6:06am",  // 时间戳（格式不一致，见避坑 8.1 节）
  "swipes": ["版本A", "版本B"],   // 所有划动变体的文本数组
  "swipe_id": 0,                  // 当前显示的 swipe 索引（0-based）
  "extra": { /* 见 2.2 */ },      // 元数据对象
  "variables": {},                // per-message 变量（来自正则/脚本写入）
  "force_avatar": "",             // 强制使用的头像路径
  "title": ""                     // 消息标题
}
```

### 2.2 extra 子字段完整表

```jsonc
{
  "api": "openai",                // 生成来源 API
  "model": "gpt-4o",             // 使用的模型名
  "display_text": "...",          // 格式化后的显示文本（覆盖 mes 渲染）
  "bias": "",                     // 生成偏置
  "token_count": 123,             // token 数
  "gen_started": "...",           // 生成开始时间戳
  "gen_finished": "...",          // 生成结束时间戳
  "reasoning_duration": 0,        // 推理时长（ms）
  "time_to_first_token": 0,      // TTFT（ms）
  "media": [],                    // 附件数组（图片/视频/音频）
  "media_display": "...",         // 媒体显示模式
  "media_index": 0,               // 当前显示媒体索引
  "files": [],                    // 文件附件（Data Bank）
  "title": "",                    // 附加标题
  "bookmark_link": "chatname",    // 链接到的检查点文件名（由 bookmarks.js 写入）
  "branches": [],                 // 从该楼创建的分支列表
  "type": "",                     // 消息类型标记
  "hidden": false,                // /hide 命令写入，true = 不进 LLM prompt
  "tool_invocations": [],         // 工具调用记录
  "uses_system_ui": false         // 是否使用系统 UI
}
```

**extra.hidden 与 is_system 区别：**

- `extra.hidden = true` — 用户主动隐藏（/hide 命令），消息仍显示在 UI 中（有幽灵图标），不发给 LLM
- `is_system = true` — `/sys` 或 `/comment` 发出的中性叙述/注释，也不进 LLM prompt，但 UI 渲染方式不同

**【重要更正】`/messages` 命令的过滤逻辑：** `slash-commands.js:3428` 直读：`if (!includeHidden && msg.is_system)` —— 实际过滤的是 `is_system=true` 的消息，而 **`extra.hidden=true` 的消息不受影响**（除非该消息同时有 `is_system=true`）。加 `hidden=true` 参数后，`is_system=true` 的消息也会被放行。换言之，"两者都会被 `/messages` 排除"的说法有误：hidden 消息（`extra.hidden=true` 但 `is_system=false`）仍会出现在 `/messages` 输出中。

注意 PR #5040 修复了 hidden 消息 `is_system` 被强制清零的 bug，该 bug 会导致深度检测失效，v1.14.0 已修复。

**来源：** `slash-commands.js`（本地直读 3428行）、PR #5040 | **置信度：high [运行时源码]**（过滤逻辑已直读源码确认）

## 5. 分支/检查点机制详解

**来源：** `bookmarks.js`（raw GitHub）  
**适用版本：** v1.14.0  
**置信度：high [运行时源码]**

### 5.1 数据模型

```
主聊天文件：CharacterName - 2024-01-01@12-00-00.jsonl
├── 第 N 楼 message.extra.bookmark_link = "某自定义名 - January 1, 2024 12:00pm"
└── 第 M 楼 message.extra.branches = ["Branch #M - January 1, 2024 12:01pm"]

检查点文件命名：用户输入名称 + " - " + humanizedDateTime()（bookmarks.js 第100行）
└── 第1行 chat_metadata.main_chat = "CharacterName - 2024-01-01@12-00-00"

分支文件命名：`Branch #${mesId} - ${humanizedDateTime()}`（bookmarks.js 第174行直读，mesId 是消息楼层号）
└── 第1行 chat_metadata.main_chat = "CharacterName - 2024-01-01@12-00-00"
```

- `extra.bookmark_link` — 字符串，指向检查点 JSONL 文件名（不含扩展名）
- `extra.branches` — **纯字符串数组**，每条为一个分支文件名（bookmarks.js 第189行直读：`lastMes.extra['branches'].push(name)`，name 是字符串）；**悬案已解除，见第9章**

### 5.2 关键实现函数

```javascript
// 创建检查点（bookmarks.js）
createNewBookmark(mesId, { forceName = null })
// 1. 获取已有聊天名 getExistingChatNames()
// 2. 生成唯一名 "{mainChat} - Checkpoint #{N}"
// 3. 保存为独立 JSONL 文件（含消息到 mesId 为止）
// 4. 在 chat[mesId].extra.bookmark_link 写入检查点文件名
// 5. saveChat()

// 创建分支（bookmarks.js）
createBranch(mesId)
// 1. 生成名称格式：`Branch #${mesId} - ${humanizedDateTime()}`（bookmarks.js 第174行直读，是 mesId 非递增N）
// 2. 保存为独立 JSONL 文件（含消息到 mesId 为止）
// 3. 在 chat[mesId].extra.branches 追加分支文件名（字符串 push）
// 注意：createBranch() 本身不切换聊天，切换由上层 branchChat() 调用 openCharacterChat()/openGroupChat()

// 返回主聊天
backToMainChat()
// 读取 chat_metadata['main_chat'] → 调用 openCharacterChat()/openGroupChat()
```

**避坑：** 分支命名使用 `Branch #${mesId} - humanizedDateTime()`（mesId 为楼层号），不使用 `getUniqueName()`，无递增编号逻辑。若手动删除分支文件，`extra.branches` 数组中仍保留文件名，切换时会出现死链错误。[high，运行时源码]

**避坑：** `/branch-create` 执行后会立即切换聊天，触发 `CHAT_CHANGED` 事件。角色卡的所有状态需在 `CHAT_CHANGED` 回调中重新初始化，否则旧状态残留。[high，推断+真机]

### 5.3 Timelines 扩展的树状视图逻辑

**来源：** SillyTavern-Timelines 扩展（社区）| **置信度：medium [社区]**

SillyTavern-Timelines 通过以下方式构建可视化分支历史：
1. 跟踪 `extra.bookmark_link` 链接（忽略死链）
2. 合并"相同深度、相同内容"的消息节点
3. 递归构建树状结构

### 6.3 message-level variables 与 swipe 的关联

**置信度：high [真机] [社区]**

`getVariables({ type: 'message' })` 返回**当前显示的 swipe** 的变量（跟随 swipe_id 切换而切换）。这是 LALib MessageVariables 扩展的核心机制：变量绑定在每个 swipe 上，切换 swipe 会切换到对应 swipe 的变量集。

实际存储位置：`message.variables`（顶层字段，非 extra 内）。具体写入时机见第 9 章悬案。

### 7.3 扩展 API 事件监听范式（角色卡正则脚本上下文）

**来源：** `control_center.js`（生产卡实战代码，真机验证）  
**置信度：high [真机]**

```javascript
function bindEvents() {
    const on = typeof eventOn === 'function' ? eventOn : null;
    const events = window.tavern_events || {};
    const disposers = [];

    // 监听 swipe 事件（用于同步状态）
    if (on && events.MESSAGE_SWIPED) {
        disposers.push(on(events.MESSAGE_SWIPED, (messageId) => {
            // messageId 为被 swipe 的楼层索引
            handleSwipeChange(messageId);
        }));
    }

    // 监听 AI 消息渲染完成
    if (on && events.CHARACTER_MESSAGE_RENDERED) {
        disposers.push(on(events.CHARACTER_MESSAGE_RENDERED, (messageId) => {
            updateStatusBar(messageId);
        }));
    }

    // 监听聊天切换（切换角色/分支时触发）
    if (on && events.CHAT_CHANGED) {
        disposers.push(on(events.CHAT_CHANGED, () => {
            reinitialize();
        }));
    }

    // 监听 MVU 变量初始化完成（自定义变量系统）
    if (Mvu?.events?.VARIABLE_INITIALIZED) {
        disposers.push(eventOn(Mvu.events.VARIABLE_INITIALIZED, () => {
            refreshDisplay();
        }));
    }

    // 清理（组件卸载时调用返回函数）
    return () => disposers.forEach(d => typeof d === 'function' ? d() : d?.off?.());
}
```

// 读取最近 N 条消息的原始文本（用于状态解析）
function recentMessages(n = 6) {
    if (typeof getChatMessages !== 'function') return '';
    return (getChatMessages(`-${n}--1`, { include_swipes: false }) || [])

}

## 8. 高频避坑

**来源综合：** `slash-commands.js` / `script.js` / `bookmarks.js` / 本地生产卡 / 社区 issue

### 8.1 send_date 格式不一致

**置信度：medium [文档·单源]** | 来源：Issue #2151

ST 的 `send_date` 字段历史上存在多种格式：`"April 26, 2024 6:06am"`、Unix 毫秒时间戳、ISO 8601 等。不要用 `new Date(send_date)` 直接解析，需做格式检测。官方 issue #2151 承认此问题但已 Stale（未修复）。

### 8.2 /addswipe 不能给用户消息或系统消息添加 swipe

**置信度：high [运行时源码]** | 来源：`slash-commands.js`（本地直读 3860-3866行）

【重要更正】`addSwipeCallback` 内部检查的是 `is_user`（行3860）和 `is_system`（行3865），**不是** `extra.hidden`。违反条件时会触发 `toastr.warning` 弹出提示，**并非静默失败**。`extra.hidden=true` 的消息不受此限制。

### 8.3 /delswipe 的 swipe-id 是 1-based，LALib 是 0-based

**置信度：high [运行时源码]** | 来源：`slash-commands.js`（本地直读 1534行 `description: '1-based swipe id'`）

- `/delswipe 1` — 删除第一个 swipe（1-based）
- `/delswipe`（无参数）— 删除**当前**swipe（deleteSwipeCallback 第3910行：无参数时 swipeId=null，deleteSwipe 内部取当前 swipe_id）
- `/swipes-del 0` — 删除第一个 swipe（LALib，0-based，需安装 LALib 扩展）

混用时必须换算，否则删错位置。

### 8.4 extra.hidden 和 is_system 行为差异

**置信度：medium [运行时源码]** | 来源：`slash-commands.js`、PR #5040

两者都排除在 LLM prompt 和 `/messages` 命令之外，但 UI 渲染方式不同。PR #5040 修复了 hidden 消息 `is_system` 被强制清零的 bug（深度检测失效），v1.14.0 已修复。

### 8.5 检查点文件命名：空名自动递增，用户自定义名加时间戳

**置信度：high [运行时源码]** | 来源：`bookmarks.js`（本地直读 82-101行）

【更正】草稿说使用 `getUniqueName()`，实际不是。`getBookmarkName()` 的逻辑：
- 用户输入名称 → 最终名称 = `"用户名 - humanizedDateTime()"`（每次生成新时间戳，几乎不重复）
- 用户输入空字符串 → 自动生成 `"Checkpoint #N"` 循环递增查找未被使用的编号（bookmarks.js 第89-94行）
- 分支命名格式：`"Branch #${mesId} - humanizedDateTime()"`，无去重逻辑

手动删除检查点文件后，空名自动递增编号可能跳跃，但已存在的 bookmark_link 链接会变成死链（跳转时 404/无法打开）。

### 8.6 context.chat 数组是可变引用

**置信度：high [文档·多源]** | 来源：官方扩展 API 文档

扩展 API 文档明确：`"Messages are mutable, so any changes you make to the array will be reflected"`。在 `GENERATION_AFTER_COMMANDS` 事件中修改 `chat` 数组会真实影响发给 LLM 的内容，但修改必须在回调中**同步完成**（异步修改可能无效）。

### 8.7 getChatMessages 返回的字段名不统一

**置信度：high [真机]** | 来源：`control_center.js`（生产卡）

不同版本 ST 或不同消息来源的字段名存在差异，必须做三重兼容：

```javascript
msg?.message || msg?.mes || msg?.text || ''
```

### 8.8 分支切换会触发 CHAT_CHANGED

**置信度：high [推断+真机]** | 来源：本地实战

`/branch-create` 执行后立即切换聊天，触发 `CHAT_CHANGED` 事件。角色卡的所有状态需在 `CHAT_CHANGED` 回调中重新初始化，否则旧状态残留。

### 8.9 swipe 切换只触发 MESSAGE_SWIPED，不触发 MESSAGE_UPDATED

**置信度：medium [推断]**

需要同时监听两个事件才能覆盖所有消息变化场景。

### 8.10 /hide 无参数默认末尾，range 是包含区间

**置信度：high [运行时源码]** | 来源：`slash-commands.js`

`3-7` 表示下标 3、4、5、6、7 共 5 条消息（包含两端）。
