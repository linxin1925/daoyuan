> 源码版本：`spv7.0`（`https://gcore.jsdelivr.net/gh/AlbusKen/shujuku@spv7.0/index.js`，106538 行）
> 分析日期：2026-07-04
> 脚本描述（源码 L5）：*"SillyTavern 数据库自动更新与交火模式索引管理脚本"*

---

## 一、核心定性：交火模式 = 纪要向量索引系统

**重要纠错**：[星河璀璨数据库结构.md](./星河璀璨数据库结构.md) 把交火模式描述为"战斗模式 + `important_non_romance` + `loot_table`"，这一描述对 spv7.0 已**过时/错误**。源码验证：

- `loot_table` 除 DDL 外**零使用**（grep 非DDL 为空）
- `important_non_romance` 只是默认预设里一张普通表（源码 L967），无"交火触发才启用"逻辑
- 英文 `crossfire` 仅 5 次，中文"交火"251 次全部围绕**向量索引**

**交火模式的真实含义**：纪要表（chronicle）条目过多时，用向量检索召回相关概要，替代全量注入——是一个省 token 的长期记忆检索系统。

---

## 二、核心流程

```
填表保存 ──► 防抖归档 ──► embedding 向量化 ──► /user/files 分片存储
                                                          │
                                          累积条数 < 200？─┤
                                                │是          │否（达门槛）
                                          不触发召回     用户发送前触发召回
                                          继续累积       │
                                                        ├─ 生成检索关键词
                                                        ├─ 向量召回概要 chunks
                                                        ├─ 可选 Rerank
                                                        └─ 按纪要原顺序覆盖
                                                           "纪要索引"世界书条目
```

三大阶段：**归档（Archive）→ 门槛（Threshold）→ 召回（Recall）**。

### 完整召回流程（L58350-58431）

| 5. 稠密检索 dense | 余弦相似度匹配较早行的概要 chunks | L58382-58394 |
| 6. BM25 稀疏检索（可选） | 文本匹配补充候选 | L58395-58397 |
| 7. RRF 混合融合 | 融合 dense + sparse | L58399-58401 |
| 8. Rerank | rerank API 重排 | L58407-58409 |
| 9. 选 TopK | 按 row 去重，取 topK 条 | L58410-58416 |
| 10. 合并固定行 + 排序行 | 去重（固定行优先） | L58417-58424 |
| 11. 覆盖纪要索引条目 | 按纪要原 rowOrder 顺序输出 | L58428-58429 |

### 关键词生成提示词详解

**作用**：向量检索的"查询增强器"。在第 2-3 步，用独立 LLM 调用从"用户输入 + 最近上下文"提取 12 个关键词，拼接到 queryText，再整体向量化做检索。

**queryText 构建**（L58360）：

```js
const queryText = [userInput, keywords.join('，')].filter(Boolean).join('\n关键词：');
// 结果：用户输入\n关键词：关键词1，关键词2，...
```

**为什么需要**：用户输入可能很短（如"继续"、"我同意"），直接 embedding 检索语义稀薄、召回质量差。LLM 提取人物、地点、事件、冲突等关键词，补充 queryText 的语义维度，让向量检索更精准。

**关键词优先级**（默认提示词 L3808）：人物、地点、时间、事件、目标、冲突、道具、组织、关系变化、未解决问题。

**输出格式**：`<thinking>` 分析 + `<keywords>关键词1，关键词2</keywords>`（中文逗号分隔，最多 24 个）。

**解析**（`parseKeywords_ACU` L57935）：优先读 `<keywords>` 标签，回退读"关键词："前缀；按中英文逗号/顿号/分号/竖线/换行分割；去重，最多 24 个。

### 关键词生成配置项

| 配置 | 默认 | 作用 | 源码 |
|---|---|---|---|
| `keywordPromptGroup` | 内置 | 提示词内容（system + user + assistant prefill） | L3802 |
| `keywordApiPreset` | `''` | 关键词生成专用 API 预设（空则用当前 API） | L3799 |
| `keywordContextPairCount` | 1 | 读取最近 N 层对话作为上下文 | L3800 |
| `keywordGenerationMaxAttempts` | 3 | 生成失败重试次数 | L3801 |

### 失败兜底

`generateKeywords_ACU` 重试 N 次仍失败 → 返回空数组 `[]`（L57976）→ `queryText` 退化为纯用户输入 → 检索质量下降但不报错，召回流程继续。

> 关键词生成只影响召回质量，不影响归档（归档独立、填表后自动触发）。可针对具体世界观定制提示词（如《道渊》修真世界观专用版见 [V0.3/道渊_交火关键词提示词.md](./V0.3/道渊_交火关键词提示词.md)）。

## 六、召回提示词双系统

源码内置两套召回预设。

### A. time-recall 预设（`_acuBuiltinPresetId: "time-recall"`，L1636）

从「已发生的事件概览」按**内容相关性**选 AM 编码，再按时间分组输出。

- **选优先级**：
  - REAL LINK（因果链 / 人物关系 / 共同经历 / 物品线索 / 矛盾挑战 / 情感主题 / 地点延续）
  - WEAK LINK（同角色 / 同关系对 / 同物品线索 / 同未决剧情线）
  - FALLBACK（补位召回）
- **时间分组**（按 `diff_min` 分钟数）：`now / today / days / weeks / months / seasons / years / old / unknown`
- **输出顺序**：`<all_am>` → `<self_check>` → `<match>` → `<recall>`
- **硬约束**：AM 来源**仅限**「已发生的事件概览」，禁止从 USER_INPUT/PREVIOUS_PLOT/背景设定取 AM；按内容选，不按时间距离选
- `recallCount: 25`（L1465）

### B. recall + supplement 双系统预设（L1656）

| 系统 | 来源 | 数量 | 规则 |
|---|---|---|---|
| recall | 总结大纲（唯一真值来源） | 按需 | 大纲≥zhaohui 条选最相关；大纲为空则输出 0 条 |
| supplement | 背景设定 | **强制 6-8 条** | 每轮必输出，与 recall 完全独立 |

两者来源严格分离：recall 只从总结大纲，supplement 只从背景设定，互不补足。
`recallCount: 20`（L1671）

## 七、与 0TK 模式的关系

0TK（零 Token 占用模式）和交火模式**各管一个世界书条目**，可同时启用（L28614、L32706）。

| 模式 | 控制的条目 | 控制什么 |
|---|---|---|
| 0TK | 大纲注入条目 | 条目的 `enabled` 状态 |
| 交火 | "纪要索引"条目 | 条目**内容**（保留/覆盖召回结果），不管 `enabled` |

设计原则（L23240、L32706-32721）：
- 0TK 持续控制大纲条目 `enabled`，交火模式不应把它重新打开
- 交火模式只控制纪要索引条目内容，不接管 `enabled`
- 两者允许同时启用，互不强制关闭

## 八、纪要索引条目保护逻辑

源码 L28821-28836：

```text
isCrossfireSummaryEntry = (extraIndexSpec.entryName === '纪要索引')

if (交火启用 && 达门槛):
    保留现有纪要索引召回内容，避免覆盖发送前召回结果
else if (交火启用 && 未达门槛):
    纪要索引条目使用普通填表数据，不保护
// 交火只控制内容保护，enabled 仍归 0TK 管
```

- 达门槛时：填表过程不会覆盖发送前已召回的内容（保护召回结果）
- 未达门槛时：用普通填表数据填入纪要索引条目
- `enabled` 状态始终由 0TK 控制，交火不接管（L28836 修复点）

## 十、与道渊 V0.3 的关联

道渊 V0.3 纪要表 `exportConfig` 已配好交火模式接口（[道渊_表格预设.json](./V0.3/道渊_表格预设.json)）：

| 字段 | 值 | 交火模式作用 |
|---|---|---|
| `entryName` | `"纪要"` | 主条目（全量注入） |
| `extraIndexEntryName` | `"纪要索引"` | **交火模式操作的条目** |
| `extraIndexColumns` | `["概览", "编码索引"]` | 概览列向量化；编码索引用于 AM 召回 |
| `extraIndexEntryType` | `keyword` | 关键词触发 |
| `extraIndexKeywords` | `"编码索引"` | — |

启用交火模式后的效果（道渊 V0.3）：

1. 每轮填表后，纪要表「概览」列被向量化归档到 `/user/files`
2. 累积到 200 条纪要后，每次发送前按用户输入召回相关概要，覆盖"纪要索引"条目
3. 替代全量纪要注入，长剧情下大幅省 token

**启用前提**：在脚本设置面板开启交火模式 + 配置 embedding API（`embeddingEndpoint` / `embeddingModel` / `embeddingApiKey`）。
