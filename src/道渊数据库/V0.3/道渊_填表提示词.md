## 关键规则

0. **客观性铁律（最高优先级）**：所有字段填写必须严格依据 `<正文数据>` 与 `<背景设定>` 中已经明确发生的文字事实和 MVU 变量。
   - 禁止修饰：不得添加形容词、情绪渲染、价值判断、文学化润色
   - 禁止联想：不得脑补未在正文中出现的动机、心理活动、因果关系、未来走向
   - 禁止扩写：不得在事实之外补充背景、细节、对话或动作
   - 仅允许：对正文已发生事实进行客观转述与必要压缩；对 MVU 变量进行如实映射
   - 若正文/变量未提及，该字段保持原值或写「（未提及）」/ NULL，不得猜测
1. 必须逐表阅读每个表格的 DDL 注释和 Note 部分，严格遵守其中的约束
2. Note 的约束优先级最高，高于通用填表经验；Note 中若提供了 SQL 示例，必须参照示例的写法
3. 若 Note 要求"禁止修改/格式固定/编码规则"，必须严格执行
4. 修真世界观特有约束（**优先级高于以上所有规则**）：
   - `global_state.cur_time` 必须使用元会历四位年份（`3726-12-23 16:00`），不可写成 `2024` 等现实历法
   - `chronicle.code_index` 必须为 `AM[0-9][0-9][0-9][0-9]`（前缀 AM，4 位数字递增）
   - `chronicle.time_span` 必须为 `YYYY-MM-DD HH:MM ~ YYYY-MM-DD HH:MM` 格式
   - 主角名 `char_name` 永远是「赤羽」，不可更改
   - 主角的 `世界 = 玄天界`、`灵根 = 天品灵根`、`性别外貌` 一般不变（除非剧情明确触发变化）

## SQL 编写原则

### INSERT（添加新行）⚠️ 重要规则

- **永远不要手动指定 row_id 列**，让 SQLite 自动递增。批量 INSERT 用 `(SELECT MAX(row_id)+1 FROM t)` 子查询会因只计算一次导致多行 row_id 相同、触发 UNIQUE 冲突，禁止使用
- **已存在的实体必须 UPDATE，禁止 INSERT**：带 UNIQUE 列的表（`important_characters.name` / `inventory.item_name` / `protagonist_skills.skill_name`）若该实体已存在于 `<当前表格数据>`，必须用 UPDATE 修改，不要 INSERT（会触发 `UNIQUE constraint failed`）。只有新实体才 INSERT。填表前先查 `<当前表格数据>` 确认实体是否存在

✅ 正确写法（单行）：

```sql
INSERT INTO inventory (item_name, quantity, description, category, current_status) VALUES ('紫灵米', 3, '...', '灵草', '主角储物袋中');
```

✅ 正确写法（批量）：

```sql
INSERT INTO inventory (item_name, quantity, description, category, current_status) VALUES
  ('紫灵米', 3, '...', '灵草', '主角储物袋中'),
  ('下品灵石', 50, '...', '灵石', '主角储物袋中');
```

❌ 错误（批量用子查询指定 row_id，会 UNIQUE 冲突）：

```sql
INSERT INTO inventory (row_id, item_name, ...) VALUES
  ((SELECT MAX(row_id)+1 FROM inventory), 'A', ...),
  ((SELECT MAX(row_id)+1 FROM inventory), 'B', ...);
```

❌ 错误（已存在实体 INSERT，会 UNIQUE 冲突）：

```sql
-- 姜梦已在表里，应 UPDATE 却 INSERT → UNIQUE constraint failed
INSERT INTO important_characters (name, ...) VALUES ('姜梦', ...);
-- 正确：UPDATE important_characters SET ... WHERE name = '姜梦';
```

**例外**：chronicle 表的 `code_index` 必须每条手动指定不同编号（AM0001, AM0002, ...），但 `row_id` 列仍省略。

### UPDATE（更新已有行）

- 所有 UPDATE 必须带 WHERE 条件，禁止无条件更新
- WHERE 条件选择优先级（递减）：
  1. 优先参考该表 Note 中的 SQL 示例写法
  2. 使用 DDL 中有 UNIQUE 约束的列（如 `WHERE name = '晏青'` / `WHERE skill_name = '九阳神功'` / `WHERE code_index = 'AM0001'`）
  3. 单行表使用 `WHERE row_id = 1`（`global_state` / `protagonist_info`）
- 过往经历每轮覆盖更新（完整历史由纪要表维护，禁止拼接！）：

  ```sql
  -- 正确：每轮覆盖，≈100字近期摘要
  UPDATE protagonist_info
     SET past_experience = '万宝楼拍得陨铁剑胚并寄拍；与苏清雪、九歌南下烬骨荒原钓鱼执法'
   WHERE row_id = 1;
  
  -- 错误：无限拼接
  -- SET past_experience = past_experience || '；新事件'
  ```

- 多列同时更新：

  ```sql
  UPDATE protagonist_info
     SET sect = '湮丹宗·记名弟子',
         recent_experience = '获晏青引荐拜入湮丹宗，挂记名弟子'
   WHERE row_id = 1;
  ```

### DELETE（删除行）

- 所有 DELETE 必须带 WHERE 条件
- 物品完全消耗/损毁时使用 `DELETE FROM inventory WHERE item_name = 'xxx';`；已赠予/已失去但保留记录的用 `quantity=0` + `current_status` 标记去向，不删除
- **chronicle 与 protagonist_info / protagonist_skills 的核心条目禁止删除**

## 每表填写要点（道渊定制）

### 1. global_state（全局数据表）

- **每轮必更新**：`prev_scene_time = cur_time`，`cur_time` 推进合理时间（半炷香/一炷香/半个时辰/三日…对应分钟数）
- 地点格式：`大区·方位·势力·地点·微观·环境`（如「中央神州·南部·湮丹宗地界·杏临谷外围·紫灵米药田边青石小径·午后暖阳」）

### 2. protagonist_info（主角信息表 · 10 列道渊 v5.2 版）

| 列 | 何时更新 |
|---|---|
| `char_name` | 几乎不动 |
| `gender_age` | 性别年龄变化才改 |
| `appearance` | 容貌变化才改（伤疤/突破） |
| `body_type` | **v5.2 新增** — 身形体态，随炼体突破/血脉觉醒/极度透支更新，≥30字 |
| `clothing` | **v5.2 新增** — 当前衣着穿戴，写明材质与完整度；脱衣时描写裸体及性征，≥30字 |
| `sect` | 入宗/退宗/晋升弟子等级时改（对应 MVU 主角.宗门） |
| `qiyun` | 习得/失去气运时改（标题列表，详情同步到 `protagonist_skills`） |
| `past_experience` | 每轮覆盖更新 — 最近 2~3 轮关键事件摘要（≈100字）。完整历史见纪要表，禁止累积拼接 |
| `recent_experience` | **每轮必更新**，用 ≈50 字概括本轮事件 |
| `personality` | 性格变化才改（如经历重大打击/顿悟） |

### 3. important_characters（重要角色表 · v5.2 新增境界/功法技能列）

- NPC 必须**有名有姓**才入表，路人不记录
- `realm`（**v5.2 新增**）：NPC 当前修仙境界，随剧情推进更新，例：「金丹初期」「筑基大圆满」「元婴中期」
- `skills`（**v5.2 新增**）：NPC 所修炼的功法/剑诀/神通/法术/血脉天赋/本能能力名称，多项用「、」分隔，仅记名称不记境界。世界书描述格式不统一时（「修炼/习得/擅使/身具/血脉觉醒XX」），从中提取并归一为标准名；性格特质、外貌气场、临时增益不计入；无法判定的整列填「（未提及）」。待剧情揭示后 UPDATE
- `past_experience`：每轮覆盖更新为最近 2~3 次交互摘要（完整历史见纪要表）
- 初始 NPC：晏青（男/约二十，金丹初期，湮丹宗宗主亲传）、红蝶（女/约十九，金丹大圆满，绝色榜成员，异火化形散修）、温月清（女/约二十，元婴初期，绝色榜成员）
- NPC 陨落标注「（已陨落）」于姓名，`is_absent = '是'`
- NPC 离场写 `is_absent = '是'`，重新登场写 `'否'`

### 5. inventory（背包物品表 · v5.2 新增状态列）

- 类别枚举：`灵石/丹药/灵草/法器/法宝/符箓/玉简/材料/杂物`
- 物品名带品级（如「下品灵石」「淬体丹·三品」「玄铁飞剑·法器」）
- 数量为 0 仅用于已赠予/已失去但保留记录的物品（`current_status` 标记去向）；完全消耗/损毁的物品 DELETE，不留 quantity=0
- `current_status`（**v5.2 新增**）：物品当前归属/去向，默认「主角储物袋中」，可写「已赠予XX」「寄拍于万宝楼」「待移交XX」「托管于宗门库房」「战斗中掉落未拾取」等

### 6. chronicle（修真纪要表）

- **每轮必 INSERT 一条**，编码 AM 递增
- `chronicle_text` 不少于 300 字，第三方仙史官视角，**只复述正文已发生的事实，禁止主观加戏/升华/结尾总结/心理脑补/文学修饰**
- `summary` 30 字内概括
- 历史纪要禁止 UPDATE / DELETE

## SQL 格式要点

- 字符串值使用单引号包裹，如 `'晏青'`
- 字符串值内部包含单引号时，使用两个单引号转义，如 `'秉持''谁欺负我就打谁''的信念'`
- 数值列直接写数字，不加引号
- 每条 SQL 语句以分号结尾
- 多条语句之间用换行分隔
- 表名和列名使用英文（参照 CREATE TABLE 中的定义）
- 禁止使用 `BEGIN / COMMIT / ROLLBACK` 等事务语句，系统会自动处理事务
- 禁止使用 `DROP TABLE / ALTER TABLE / CREATE TABLE` 等结构变更语句
