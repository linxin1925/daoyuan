# 星河璀璨·数据库 SQLite 数据结构

> **脚本本体**：`SP·数据库 V`，版本 **v2.0.0**（`UNIQUE_SCRIPT_ID = 'shujuku_v120'`，对应酒馆侧脚本壳 v2.1，远程包 `AlbusKen/shujuku@spv5.4.3`）
> **运行模式**：默认 `Mode = normal`，附带 `交火模式 (CrossFire)` 分支（5+ 次"crossfire"提及、239 次"交火"提及）
> **SQLite 引擎**：`sql.js` (initSqlJs)
> **外键**：无（`PRAGMA foreign_keys` 仅检查/重置）
> **唯一性**：实体表统一用 `name TEXT NOT NULL UNIQUE` 作为业务主键，`row_id INTEGER PRIMARY KEY` 是 SQLite 内部自增 PK

---

## 一、表清单（10 张）

| # | 表名 | 中文别名 | 作用 | 主键策略 |
|---|---|---|---|---|
| 1 | `global_state` | 全局数据表 | 时间/地点/经过时长 | 单行（row_id=1）|
| 2 | `protagonist_info` | 主角信息表 | 主角档案 | 单行 |
| 3 | `protagonist_skills` | 主角技能表 | 主动/被动技能 | `skill_name UNIQUE` |
| 4 | `inventory` | 背包物品表 | 物品+数量 | `item_name UNIQUE` |
| 5 | `important_characters` | 重要角色表 | NPC（含恋爱可攻略） | `name UNIQUE` |
| 6 | `important_non_romance` | 重要角色表（非攻略） | **交火模式专用** NPC | `name UNIQUE` |
| 7 | `quests_events` | 任务与事件表 | 主线/支线任务 | `quest_name UNIQUE` |
| 8 | `chronicle` | 纪要表 | 长期记忆（按时间跨度归档） | `code_index UNIQUE` (`AMxxxx`) |
| 9 | `options` | 选项表 | AI 提供给玩家的 4 选项 | 单行 |
| 10 | `loot_table` | 战利品表 | **交火模式专用** 战斗掉落 | `row_id` |

> **模式分歧**：
> - `normal` 模式：用 `important_characters`（含 `key_items`、`is_absent`），无 `loot_table`、`important_non_romance`
> - `crossfire` 模式：用 `important_non_romance`（拆出 `gender/age` 双字段、`outfit_text`、`location_name`、`presence_status`、`relation_text`、`interaction_options`），并启用 `loot_table` 记录掉落
> - 两者**不互斥**于物理 DB，但同一时间只用一套

## 二、表 DDL（按完整定义）

### 1. global_state — 全局数据表

```sql
CREATE TABLE global_state (
  row_id INTEGER PRIMARY KEY,            -- 行号
  current_location TEXT NOT NULL,        -- 主角当前所在地点
  cur_time TEXT NOT NULL                 -- 当前时间
    CHECK(cur_time GLOB '????-??-?? ??:??'),
  prev_scene_time TEXT                   -- 上轮场景时间
    CHECK(prev_scene_time IS NULL OR prev_scene_time GLOB '????-??-?? ??:??'),
  elapsed_time TEXT                      -- 经过的时间
);
```

**AI 视角列名**：`row_id` · 主角当前所在地点 · 当前时间 · 上轮场景时间 · 经过的时间

**时间格式约束**：`YYYY-MM-DD HH:MM`（`????-??-?? ??:??` GLOB），AI 输出不合规则插入失败。

### 2. protagonist_info — 主角信息表

```sql
CREATE TABLE protagonist_info (
  row_id INTEGER PRIMARY KEY,            -- 行号
  char_name TEXT NOT NULL,               -- 人物名称
  gender_age TEXT,                       -- 性别/年龄
  appearance TEXT,                       -- 外貌特征
  occupation TEXT,                       -- 职业/身份
  past_experience TEXT,                  -- 过往经历
  personality TEXT                       -- 性格特点
);
```

**AI 视角列名**：`row_id` · 人物名称 · 性别/年龄 · 外貌特征 · 职业/身份 · 过往经历 · 性格特点

---

### 3. protagonist_skills — 主角技能表

```sql
CREATE TABLE protagonist_skills (
  row_id INTEGER PRIMARY KEY,
  skill_name TEXT NOT NULL UNIQUE,       -- 技能名称
  skill_type TEXT NOT NULL                -- 技能类型
    CHECK(skill_type IN ('被动', '主动')),
  skill_level TEXT,                       -- 等级/阶段
  effect_desc TEXT                        -- 效果描述
);
```

**AI 视角列名**：`row_id` · 技能名称 · 技能类型 · 等级/阶段 · 效果描述
**枚举**：`skill_type ∈ {被动, 主动}`

---

### 4. inventory — 背包物品表

```sql
CREATE TABLE inventory (
  row_id INTEGER PRIMARY KEY,
  item_name TEXT NOT NULL UNIQUE,         -- 物品名称
  quantity INTEGER NOT NULL DEFAULT 1     -- 数量
    CHECK(quantity > 0),
  description TEXT,                       -- 描述/效果
  category TEXT                           -- 类别
);
```

**AI 视角列名**：`row_id` · 物品名称 · 数量 · 描述/效果 · 类别
**约束**：`quantity > 0`（数量归零应 DELETE 行而非置 0）

---

### 5. important_characters — 重要角色表（普通模式）

```sql
CREATE TABLE important_characters (
  row_id INTEGER PRIMARY KEY,
  name TEXT NOT NULL UNIQUE,              -- 姓名
  gender_age TEXT,                        -- 性别/年龄
  brief_intro TEXT,                       -- 一句话介绍
  appearance TEXT,                        -- 外貌特征
  key_items TEXT,                         -- 持有的重要物品
  is_absent TEXT NOT NULL DEFAULT '否'    -- 是否离场
    CHECK(is_absent IN ('是', '否')),
  past_experience TEXT                    -- 过往经历
);
```

**AI 视角列名**：`row_id` · 姓名 · 性别/年龄 · 一句话介绍 · 外貌特征 · 持有的重要物品 · 是否离场 · 过往经历
**枚举**：`is_absent ∈ {是, 否}`

---

### 6. important_non_romance — 重要角色表（交火模式专用）

```sql
CREATE TABLE important_non_romance (
  row_id INTEGER PRIMARY KEY,
  name TEXT NOT NULL UNIQUE,              -- 姓名
  gender TEXT,                            -- 性别（与上表 gender_age 不同：拆开）
  age INTEGER CHECK(age IS NULL OR age >= 0),  -- 年龄
  brief_intro TEXT NOT NULL,              -- 一句话介绍（必填）
  appearance TEXT,                        -- 外貌特征
  outfit_text TEXT,                       -- 穿着打扮
  location_name TEXT,                     -- 所在地点
  presence_status TEXT NOT NULL           -- 在场状态
    DEFAULT '在场' CHECK(presence_status IN ('在场', '离场')),
  relation_text TEXT NOT NULL,            -- 人际关系
  past_experience TEXT,                   -- 过往经历
  interaction_options TEXT                -- 交互选项
);
```

**与 `important_characters` 的差异**（v5.4.3 新增）：
- `gender_age` → 拆为 `gender TEXT` + `age INTEGER`（带 `>= 0` 约束）
- `key_items` → 删除（交火模式下物品归集到玩家 `inventory` / `loot_table`）
- `is_absent (是/否)` → `presence_status (在场/离场)`（语义化）
- 新增 `outfit_text`（穿着）/ `location_name`（所在地点）/ `relation_text`（人际关系，**必填**）/ `interaction_options`（交互菜单）
- `brief_intro` 变为 **NOT NULL**

**枚举**：`presence_status ∈ {在场, 离场}`

---

### 7. quests_events — 任务与事件表

```sql
CREATE TABLE quests_events (
  row_id INTEGER PRIMARY KEY,
  quest_name TEXT NOT NULL UNIQUE,        -- 任务名称
  quest_type TEXT NOT NULL                -- 任务类型
    CHECK(quest_type IN ('主线任务', '支线任务')),
  issuer TEXT,                            -- 发布者
  detail_desc TEXT,                       -- 详细描述
  current_progress TEXT,                  -- 当前进度
  time_limit TEXT,                        -- 任务时限
  reward TEXT,                            -- 奖励
  penalty TEXT                            -- 惩罚
);
```

**AI 视角列名**：`row_id` · 任务名称 · 任务类型 · 发布者 · 详细描述 · 当前进度 · 任务时限 · 奖励 · 惩罚
**枚举**：`quest_type ∈ {主线任务, 支线任务}`

---

### 8. chronicle — 纪要表（长期记忆/历史归档）

```sql
CREATE TABLE chronicle (
  row_id INTEGER PRIMARY KEY,
  time_span TEXT NOT NULL                 -- 时间跨度
    CHECK(time_span GLOB '????-??-?? ??:?? ~ ????-??-?? ??:??'),
  location TEXT NOT NULL,                 -- 地点
  chronicle_text TEXT NOT NULL,           -- 纪要
  summary TEXT,                           -- 概览
  code_index TEXT NOT NULL UNIQUE         -- 编码索引
    CHECK(code_index GLOB 'AM[0-9][0-9][0-9][0-9]')
);
```

**AI 视角列名**：`row_id` · 时间跨度 · 地点 · 纪要 · 概览 · 编码索引

**关键设计**：
- `time_span` 强制格式 `YYYY-MM-DD HH:MM ~ YYYY-MM-DD HH:MM`
- `code_index` 强制 `AM0000` ~ `AM9999`（4 位数字编码，全局唯一）
- 这是**长期记忆的核心存储**——chronicle 满了会被压缩为 `summary`

---

### 9. options — 选项表

```sql
CREATE TABLE options (
  row_id INTEGER PRIMARY KEY,
  option_1 TEXT NOT NULL,                 -- 选项一
  option_2 TEXT NOT NULL,                 -- 选项二
  option_3 TEXT NOT NULL,                 -- 选项三
  option_4 TEXT NOT NULL                  -- 选项四
);
```

**AI 视角列名**：`row_id` · 选项一 · 选项二 · 选项三 · 选项四
**用途**：AI 每回合给玩家的 4 个分支选项，UI 渲染按钮组用

---

### 10. loot_table — 战利品表（交火模式专用）

```sql
CREATE TABLE loot_table (
  row_id INTEGER PRIMARY KEY,
  item_name TEXT,                         -- 物品名称
  quantity INTEGER,                       -- 数量
  time_span TEXT NOT NULL,                -- 时间跨度
  remarks TEXT                            -- 备注
);
```

**与 `inventory` 的区别**：
- `inventory` 是当前持有物品（活动表，量为 0 删除）
- `loot_table` 是**战斗掉落历史日志**（事件流，按 time_span 追加）
- 字段更精简：仅 `item_name / quantity / time_span / remarks`

## 四、AI 写入接口约定

数据库脚本期望 AI 在回复里输出**操作指令**（不是直接写 SQL），脚本侧解析后转化为 SQL。从代码侧看到的写入入口形如：

```js
// 行级数组接口（AI 端实际看到的是中文列名 + 行数组）
{
  table: 'chronicle',
  content: [
    ['row_id', '时间跨度', '地点', '纪要', '概览', '编码索引'],
    [null, '2026-06-20 08:00 ~ 2026-06-20 12:00', '杏临谷', '...', '...', 'AM0001'],
    // ...更多行
  ]
}
```

**写入语义**：
- 实体表（带 UNIQUE 列）：按业务键 `INSERT OR REPLACE` / `UPDATE`
- 单行表（`global_state` / `protagonist_info` / `options`）：固定 `row_id = 1`，按列 `UPDATE`
- 日志表（`chronicle` / `loot_table`）：仅 `INSERT`（chronicle 后续可能被汇总压缩）

## 六、与 MVU 的关系

`星河璀璨·数据库` 是**通用全局脚本**，不依赖任何角色卡的 MVU 变量结构。

它与《道渊》v5.1 MVU 的关系：
- **正交**：MVU `stat_data` 走酒馆助手楼层变量，SQLite 数据库走脚本扩展存储，**互不影响**
- **概念重叠**：
  - MVU `主角` ↔ SQL `protagonist_info` + `protagonist_skills`
  - MVU `人物` ↔ SQL `important_characters`（或交火模式下 `important_non_romance`）
  - MVU `机遇` ↔ SQL `quests_events`
  - MVU `世界.当前时间/当前地点` ↔ SQL `global_state.cur_time / current_location`
  - MVU `主角.储物袋` ↔ SQL `inventory`
- **MVU 没有的概念**：`chronicle`（长期纪要归档）、`options`（4 选项菜单）、`loot_table`（战斗掉落日志）—— 这些是数据库脚本独有的"过程数据"

> 如果同时启用，AI 会被要求**双写**（MVU 命令 + 数据库表）；正文产物则按数据库脚本约定的"行级数组"格式输出。

## 七、运行模式速判

```text
脚本入口
  ├─ Mode = normal
  │   ├─ 用 important_characters
  │   ├─ 不用 important_non_romance
  │   └─ 不用 loot_table
  └─ Mode = crossfire （交火模式）
      ├─ 用 important_non_romance（字段更细）
      ├─ 用 loot_table
      └─ inventory + important_non_romance 双轨记录物品流转
```

`normal` ↔ `crossfire` 切换在 UI 设置里完成，DB 文件可同时持有两套数据（迁移工具 `数据库迁移新聊天` 处理跨聊天导入）。
